## 二、软件架构风格（★★★★★）

### 2.1 风格总览（概念辨析表）

| 一级风格 | 二级风格 | 核心概念 | 典型场景 |
|----------|----------|----------|----------|
| **数据流** | 批处理 | 分步执行，前一步完成**整体传递**数据，**无并行** | 传统数据处理、批处理脚本 |
| **数据流** | 管道-过滤器 | 前一步**输出即后一步输入**，可**并行** | Unix Shell、编译器各阶段 |
| **调用/返回** | 主程序/子程序 | **单线程**控制，分层调用 | 早期结构化程序 |
| **调用/返回** | 面向对象 | 数据与操作**封装**，**消息**传递 | Java/C# 应用 |
| **调用/返回** | 层次型 | 每层为上层服务、作下层客户，**相邻层**可见 | 分层系统 |
| **调用/返回** | 客户端/服务器 | 表示层/功能层/数据层，**网络**交互 | Web、数据库应用 |
| **以数据为中心** | 仓库 | **中央数据存储**，独立构件操作 | IDE、企业数据管理 |
| **以数据为中心** | 黑板 | **分级解空间**，多**知识源**协同 | 语音识别、模式识别 |
| **虚拟机** | 解释器 | 自定义语言**解析执行**，可动态扩展 | 游戏脚本引擎 |
| **虚拟机** | 规则系统 | 规则集+规则解释器+选择器+**工作内存** | 专家系统 |
| **独立构件** | 进程通信 | 构件为**独立进程**，连接件为**消息传递** | 分布式、微服务 |
| **独立构件** | 事件驱动 | **隐式调用**，事件**广播**触发响应 | UI 系统 |
| **特殊** | C2 | 构件经连接件**分层**，**顶部到底部**通信 | 嵌入式、分布式 |
| **特殊** | 过程控制 | **反馈循环**调节物理状态 | 空调、巡航 |

**批处理 vs 管道-过滤器**：批处理需“落盘”（Tape/文件），整体传递；管道-过滤器**数据连续流动**、可并行、更高效。

**进程通信 vs 事件驱动**：进程通信是**显式**消息传递（点对点、RPC 等）；事件驱动是**隐式调用**——构件**注册**到事件，事件触发时系统**自动调用**已注册过程。

**案例（航空运营智能管理平台）**：

- **批处理 vs 管道**：**T+1 日批** 与 **800GB 增量** 偏批处理；**实时超售/设备告警** 用 **管道-过滤器**（采集→清洗→规则→告警）。
- **层次 C/S**：**多终端 + 网关 + 微服务 + 分布式存储** 即多层 **调用/返回**。
- **仓库**：**航信主数据、统一订单与指标** 作中心，服务经治理 API 访问。
- **黑板**：**复杂航变调度** 或多模型协同的 **排故/决策**（考点：分级解空间）。
- **解释器/规则**：**动态定价、退改签规则** 热更新。
- **进程通信**：**RPC/消息**；**事件驱动**：`FlightCancelled` 等 **隐式触发** 通知、库存、分析入湖。

---

### 2.2 若干风格示意

**批处理**：`[步骤1] → Tape → [步骤2] → Tape → [步骤3]`（顺序、整体传递）

**管道-过滤器**：`[Filter1] → Pipe → [Filter2] → Pipe → [Filter3]`（流式、可分支/并行）

**仓库**：多个**独立构件** ↔ **中央数据库**（仅通过中心交换，构件间无直接连接）

**黑板**：**知识源 1～N** ↔ **黑板**（共享解空间）；实线=读写黑板，虚线=访问外部数据/计算

**解释器**：**存储器**（程序+当前状态）↔ **执行引擎**（取指、解释、更新状态）→ 输出

**层次型**：上层 → 调用 → 下层；仅**相邻层**可见，接口一致可替换实现。

**二层/三层 C/S、B/S**：二层=客户端(表示+业务)+数据库服务器（胖客户端）；三层=表示+**应用服务器**(业务)+数据（瘦客户端）；B/S=浏览器+Web 服务器+数据库，零安装、易推广，动态交互与高并发 OLTP 弱于 C/S。

---

### 2.3 典型例题：解释器 vs 面向对象

**题干**：折扣规则可修改、个性化折扣定义灵活、系统性能三方面比较，选更适合的架构。

**结论**：选**解释器**。  
- **可修改性**：解释器规则为独立语法，由解释器解析，**修改更容易**；面向对象需改具体类。  
- **灵活性**：解释器可按用户**灵活解释执行规则**，优于面向对象。  
- **性能**：面向对象一般略优（解释器多一层）。综合前两项优势更大，故选解释器。

**案例（航空运营智能管理平台）**：**票价/退改规则** 需 **频繁变更**、多承运人策略并存时，**解释器/规则引擎 + 表驱动** 比硬编码 **类继承** 更易维护；性能靠 **缓存、编译规则、热点预计算** 补偿。

## 三、面向架构评估的九大质量属性（★★★★★）

### 3.1 九大属性及对策

| 属性 | 描述 | 对策（简） |
|------|------|------------|
| **性能** | 响应能力、处理时间、单位时间处理量 | 优先级队列、增加资源、减少开销、并发、资源调度 |
| **可靠性** | 在错误/异常使用下维持功能；**平均失效时间** | 冗余、错误检测与恢复、异常处理、备份与恢复、负载均衡 |
| **可用性** | **正常运行时间比例**；故障间隔/恢复速度 | **心跳**（检测在线）、冗余/双机热备/集群、**快速恢复**、**故障转移**、内置监控与预警 |
| **安全性** | 合法服务+阻止非授权；机密性、完整性、不可否认、可控 | 认证、授权、加密、防火墙、入侵检测、审计与日志 |
| **可修改性** | 快速、高性价比变更；含可维护、可扩展、可移植等 | 模块化、接口标准化、抽象与封装、配置管理、版本控制 |
| **互操作性** | 与其他系统/环境交互；跨语言、跨系统 | 标准协议与接口（REST、SOAP）、数据格式转换、中间件、服务注册与发现 |
| **可移植性** | 适应不同运行环境；跨 OS/硬件/云 | 抽象层/虚拟机、减少平台依赖、兼容性测试 |
| **可测试性** | 易于发现缺陷、验证功能与性能 | 分层与模块化、测试接口、可执行语句覆盖率等度量 |
| **易用性** | 好学、好找、好操作、少犯错 | 统一交互与视觉、信息架构、即时反馈、快捷操作、多语言与无障碍、引导与帮助 |

**可用性 vs 可靠性**：真题常把“故障后 N 秒内发现并启用备用”“主站断电后 N 秒内重定向”归为**可用性**；二者辨析少见，按题目默认可用性即可。

**案例（航空运营智能管理平台）**：**性能** 对齐 **5000 TPS、≤1 秒、看板延迟≤3 秒、PB 查询≤5～10 秒**；**可用性** 对齐 **99.99%**；**安全** 对齐 **RBAC、TLS、审计、脱敏、民航局核验**；**互操作** 对齐 **支付、核验、承运人接口**；**可测试性** 对齐 **压测与契约测试**；**易用性** 对齐 **多终端一致体验**。

### 3.2 效用树填空思路（例题）

- **效用树结构**：树根 → **质量属性** → 属性分类 → **质量属性场景**（叶子）。  
- **ATAM 默认四类**：**性能、可用性、安全性、可修改性**。  
- 看题干（a）～（k）等描述：**响应时间/吞吐**→性能；**故障恢复/重定向/备用**→可用性；**审计/日志/攻击检测/访问控制**→安全性；**规则可配置/界面可调/扩展人天**→可修改性。  
- 例题答案示例：（1）安全性（2）可修改性（3）e（4）j（5）h（6）k；或（1）性能（2）可修改性（3）e（4）j（5）h（6）i——以题干树形与选项为准。

**案例（航空运营智能管理平台）**：效用树叶子可写 **“春运峰值购票仍满足 P99≤1 秒”**（性能）、**“单 AZ 故障 RTO 内切换”**（可用性）、**“审计日志防篡改”**（安全）、**“新增销售渠道不改核心票务”**（可修改性）。

## 四、质量属性场景描述（六要素）（★★★★★）

### 4.1 公式与语义

**公式**：质量属性场景 = **刺激源** + **刺激** + **环境** + **制品** + **响应** + **响应度量**

| 要素 | 理论 | 白话 | 例 |
|------|------|------|-----|
| **刺激源** | 人、计算机系统或其他刺激器 | 谁触发 | 大量用户、开发人员 |
| **刺激** | 刺激到达时需考虑的条件 | 触发条件 | 同时点击秒杀、请求到达 |
| **环境** | 刺激发生时系统可能处于的状态 | 何时/何状态 | 正常访问、降级模式 |
| **制品** | 受刺激的模块或系统 | 测谁 | 商品展示模块、订单模块 |
| **响应** | 激励到达后采取的行动 | 系统怎么做 | 处理请求、返回结果、切换备用 |
| **响应度量** | 可度量的指标，用于验证 | 如何衡量 | 响应时间 200ms、吞吐 500QPS、故障修复时间 |

**故事串**：**谁**（刺激源）在**什么条件**下（刺激）、**系统处于何状态**（环境）、对**谁**（制品）**做了什么**（响应），并用**什么指标**衡量（响应度量）。

**案例（航空运营智能管理平台）**：**峰值购票**——刺激源：**数十万用户**；刺激：**提交订单**；环境：**国庆 9–11 点**；制品：**票务集群**；响应：**扩容、排队、缓存**；响应度量：**P99≤1 秒、TPS≥5000、错误率阈值。 **实时告警**——响应度量：**采集≤1 秒、计算≤2 秒、推送≤3 秒、准确率≥98%**（与素材一致）。

---

### 4.2 各质量属性六要素简表

| 质量属性 | 刺激源 | 刺激 | 环境 | 制品 | 响应 | 响应度量 |
|----------|--------|------|------|------|------|----------|
| **可用性** | 系统内/外部 | 疏忽、错误、崩溃、时间 | 正常→降级 | 处理器、信道、存储器、进程 | 检测、记录、通知、隔离故障、恢复 | 故障修复时间 |
| **可修改性** | 最终用户、开发、管理员 | 改变功能、扩容等 | 设计时/编译时/构建时/运行时 | 界面、平台、交互系统 | 修改且不影响其他功能 | 成本、努力、对其它功能影响 |
| **性能** | 用户请求、他系统 | 事件到达 | 正常→超载 | 系统 | 处理刺激、改变服务级别 | 等待时间、期限、吞吐量、抖动、缺失率 |
| **可测试性** | 开发、测试、客户 | 分析/架构/设计/集成交付时触发 | 设计时/开发时/编译时/部署时 | 设计、代码段、应用 | 提供状态访问、计算值、测试环境 | 语句覆盖率、缺陷触发概率、测试时间等 |
| **易用性** | 最终用户 | 想学习系统、有效使用 | 运行时或配置时 | 系统 | 友好界面与适应性设计，错误影响最小 | 用户满意度、成功操作占比 |
| **安全性** | 内/外部个人或系统 | 试图访问/修改数据、访问服务 | 网络环境 | 系统服务、数据 | **认证、授权、审计、加密、监控** | 检测、溯源、恢复 |

**可用性简答模板**（24 年案例）：刺激源：系统内部、系统外部。刺激：疏忽、错误、崩溃、时间。环境：正常操作、降级模式。制品：系统处理器、通信信道、持久存储器、进程。系统响应：检测事件并记录、通知相关方、禁止故障源、一段时间不可用后恢复、在正常或降级模式下运行。响应度量：故障修复时间。

**案例（航空运营智能管理平台）**：可用性场景可写 **“支付渠道超时触发降级为只读查询 + 公告”**；**MTTR** 与 **故障转移** 作响应度量。

## 五、系统架构评估（★★★★★）

### 5.1 敏感点、权衡点、风险点、非风险点

| 概念 | 定义 | 句式/例 |
|------|------|----------|
| **敏感点** | **一个或多个构件（及关系）**的特性 | “**…将影响…**”；如：对查询处理时间的要求**将影响**传输协议与处理设计 |
| **权衡点** | **影响多个质量属性**的特性，是多个敏感点的综合 | “**…能提升 A 但会降低 B**”；如：加密级别高→安全性↑、性能↓ |
| **风险承担者** | 影响架构决策的利益相关者 | 财务部门、销售部门 |
| **场景** | 风险承担者角度的交互描述；常用刺激、环境、响应 | 客户在公共 Wi-Fi 下登录网银，10 秒内完成认证并显示账户 |
| **风险点** | **可能**导致架构出问题或不满足质量要求的特性/决策 | “**如果…，可能…，影响…**” |
| **非风险点** | 对质量属性无负面影响或可接受的特性/决策 | 选用成熟 MQ（如 RabbitMQ），经市场验证，当前架构下无明显风险 |

**案例（航空运营智能管理平台）**：

- **敏感点**：“采用 **同步分布式事务** 将 **影响** 购票延迟。”
- **权衡点**：“**全链路 TLS + 字段加密** **提升** 安全性 **但降低** 性能与排障效率。”
- **风险点**：“若 **外部核验** 无超时与熔断，**可能** 导致**购票失败率**飙升。”
- **非风险点**：“采用成熟 **消息队列** 做异步出票，在团队熟悉前提下 **风险可控**。”

### 5.2 三种架构评估方法

| 方法 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **调查问卷/检查表** | 用问卷或检查表，凭经验与知识评估 | 利用经验与知识 | 依赖主观推断 |
| **基于场景的评估** | 分析架构对场景的支持程度；**SAAM、ATAM** 属此类 | 系统化评估对场景的支持 | 需详细定义场景，可能耗时 |
| **基于度量** | 建立质量属性与度量的映射，从架构文档推导 | 客观、可量化 | 需精确度量与映射，专业要求高 |

---

### 5.3 SAAM（★★★☆☆）

- **思想**：用**实际使用场景**对架构“挑刺”，检查是否扛得住。  
- **主要分析**：**可修改性**；协调风险承担者达成共识。  
- **输入**：问题描述、需求声明、架构描述。  
- **过程**：场景开发 → 架构描述 → 单场景评估 → 场景交互评估 → 总体评估。  
- SAAM 是 **ATAM** 的基础。

**案例（航空运营智能管理平台）**：用 SAAM 思路可评估 **“增加行李子域”** 对 **可修改性** 及与 **票务、旅客、通知** 的 **场景交互** 是否可承受。

---

### 5.4 ATAM 与效用树（★★★★★）

- **ATAM**：在 SAAM 基础上，针对**性能、安全性、可修改性、可用性**，在开发前**评价与折中**；用**效用树**等工具做**优先级排序**再评估。  
- **四阶段**：场景与需求收集 → 架构视图与场景实现 → 属性模型构造与分析 → 权衡（迭代）。  
- **效用树**：树根 → **质量属性**（上述四类）→ 属性分类 → **场景**（叶子）。  
- **修剪与优先级**：保留重要场景（通常≤50），按**重要度 H/M/L** 与**实现难易度 H/M/L** 标优先级对，如 (H,L)=重要且易实现。

**案例（航空运营智能管理平台）**：优先评估 **(H,L)** 如 **“春运压测达标”**； **(H,H)** 如 **“多活机房”** 留到架构决策阶段做 **折中**。

---

### 5.5 ATAM 实践四阶段

| 阶段 | 主要活动 |
|------|----------|
| **演示** | 介绍 ATAM 流程、业务目标、待评估架构 |
| **调查和分析** | 确定架构设计方法；**生成质量属性效用树并排序**；分析架构与质量属性匹配度 |
| **测试** | 头脑风暴候选场景；筛选高优先级场景并验证架构支持能力 |
| **报告** | 汇总**风险、非风险、敏感点、权衡点**；提交效用树、场景列表及改进建议 |

**案例（航空运营智能管理平台）**：**演示** 对齐信息中心业务目标；**调查分析** 生成效用树并对照 **逻辑/进程/部署** 视图；**测试** 用 **压测与故障注入** 验证场景；**报告** 输出 **扩容与降级预案、外部依赖 SLA**。

## 六、速记清单（考前扫一眼）

1. **架构风格**：数据流（批处理/管道-过滤器）、调用返回（主程序子程序/OO/层次/C-S）、以数据为中心（仓库/黑板）、虚拟机（解释器/规则系统）、独立构件（进程通信/事件驱动）；批处理整体传递无并行，管道可并行；事件=隐式调用+广播。
2. **九大质量属性**：性能、可靠性、可用性、安全性、可修改性、互操作性、可移植性、可测试性、易用性；各记一条对策（如可用性：心跳、冗余、快速恢复、故障转移）。
3. **六要素**：刺激源、刺激、环境、制品、响应、响应度量；可用性=检测记录通知隔离恢复+故障修复时间；安全性=认证授权审计加密监控+检测溯源恢复。
4. **敏感点**：构件特性，“将影响”；**权衡点**：多属性，“提升 A 降低 B”；风险点：“如果…可能…影响”；非风险点：可接受或无负面影响。
5. **评估方法**：问卷（主观）、场景（SAAM/ATAM）、度量（客观需专业）。
6. **ATAM 四属性**：性能、可用性、安全性、可修改性；效用树=属性→分类→场景；优先级对 (重要度, 难易度)。
7. **ATAM 四阶段**：演示→调查与分析（含效用树）→测试→报告。
