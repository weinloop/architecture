## 二、什么是 SOA（概念简答）

**从应用角度**：SOA 是一种**应用框架**，将日常业务应用划分为**单独的业务功能与流程**（即服务），使用户可构建、部署和整合这些服务，且**不依赖具体应用与运行平台**，从而提升业务灵活性、资产重用、管理简便与开发部署速度。

**从软件原理角度**：SOA 是一个**组件模型**，将应用的不同功能单元（**服务**）通过**定义良好的接口与契约**联系起来。接口以**中立方式**定义，**独立于**硬件平台、操作系统和编程语言，使不同系统中的服务能以**统一、通用**的方式交互。

**默写要点**：组件模型 + 服务 + 良好接口与契约 + 中立/独立于平台与语言 + 统一通用交互。

**案例（航空运营智能管理平台）**：**服务** 如 **出票、库存、行程同步、检修告警**；**契约** 如 **OpenAPI/Protobuf**；**中立** 体现为 **Java/Go 多语言、多渠道终端** 共用同一套接口语义。

## 三、SOA / 微服务 / 单体对比（★★★★★）

### 3.1 微服务 vs SOA（五维度）

| 特性 | 微服务 | SOA |
|------|--------|-----|
| **服务粒度** | **极细**，单服务单业务功能（如商品展示、订单、支付各一服务） | **较大**，一服务可含多业务功能（如 ERP 中采购+库存一服务） |
| **通信方式** | 以 **HTTP/RESTful API** 为主，语言平台无关、易集成 | **多样化**，除 HTTP 外常用 **SOAP** 等，规范与安全性要求高 |
| **数据管理** | **每服务独立数据库**，数据源唯一 | 多服务**共享库**或经 **ESB** 交换整合 |
| **架构风格** | **去中心化**，无中心服务总线，轻量通信 | 可有**中心化 ESB**，负责注册、发现、路由、消息转换等 |
| **复杂性** | 服务独立、去中心，单服务简单；但**服务多**带来治理、监控、运维挑战 | 服务间耦合与 ESB 使设计、开发、维护相对复杂，对团队要求高 |

**口诀**：微细 REST 独库去中心，SOA 粗 SOAP 共享可 ESB。

**案例（航空运营智能管理平台）**：当前建设属 **微服务** 形态——**票务/库存/支付** 等 **细粒度服务**，**HTTP/REST** 为主，**分库分表**；若与 **遗留 SOA** 对比，旧系统可能是 **较粗服务 + 共享库 + ESB** 集成多个航站子系统。

---

### 3.2 SOA vs 单体架构

| 特性 | SOA | 单体架构 |
|------|-----|----------|
| **服务粒度** | 业务功能**分解为多服务**，粒度较细，可复用 | **整体紧耦合**，功能难单独拆分与复用 |
| **通信方式** | **多协议**（SOAP、HTTP 等），标准接口，**分布式部署** | **方法调用**，同一应用内部，无复杂外部协议 |
| **数据管理** | 多服务**共享库**或经 **ESB** 交换 | **单一数据库**，所有模块共一数据源 |
| **架构风格** | **分布式**，服务可独立部署；可有 **ESB** 协调 | **集中式**，功能集中一应用、同一环境 |
| **故障影响** | 单服务故障**隔离较好**，仅影响依赖该服务的功能 | 任一部分故障可能导致**整体不可用** |

**案例（航空运营智能管理平台）**：旧 **单体** 烟囱系统 **春节峰值全站拖死**；**服务化** 后 **购票链** 与 **报表** 故障 **隔离**，契合素材 **高并发与 99.99% 可用** 目标。

## 四、服务注册模式（★★★★★）

### 4.1 目的与三角色

**引入目的**：解决**服务提供者**与**服务请求者**之间的**寻址与发现**。通过注册中心统一发布与存储服务信息，调用方**不直接依赖具体地址**，而是**动态查询**获取调用方式，实现解耦，支持动态发现、负载均衡与版本演进。

| 角色 | 操作内容 |
|------|----------|
| **服务提供者** | 使用 **UDDI** 将 Web 服务**发布**到注册中心，用 **WSDL** 描述服务 |
| **服务注册中心** | **接收并存储**提供者通过 UDDI 发布的服务信息，供请求者**查找** |
| **服务请求者** | 借助 **UDDI** 在注册中心**查找**服务，依据 **WSDL** 了解调用方式，通过 **SOAP** 与提供者建立连接并使用服务 |

**流程示意**：
```
  提供者 --publish/UDDI--> 注册中心 <--bind/UDDI-- 请求者
       |                        |                        |
    WSDL 描述              存储/发现                 WSDL 了解
       |                        |                        |
       +-------- SOAP 调用 -----+------------------------+
```

**案例（航空运营智能管理平台）**：工程上多为 **Consul/Eureka/Nacos + K8s Service** 做 **注册发现**；**WSDL/SOAP** 多见于 **对接传统民航/企业 Web Service** 题干；答题 **按教材写 UDDI/WSDL/SOAP**，实施可写 **“注册中心 + 健康检查 + 客户端负载均衡”** 与之一致思想。

---

### 4.2 UDDI、WSDL、SOAP（必背）

| 协议 | 概念 | 主要作用 |
|------|------|----------|
| **UDDI**（通用描述、发现和集成） | 基于 **XML** 的、用于发现与集成 Web 服务的协议/规范 | **发布与搜索** Web 服务信息；提供者注册，使用者查找与连接 |
| **WSDL**（Web 服务描述语言） | 描述 Web 服务**接口**的 **XML** 格式语言 | 定义功能、输入输出参数、绑定协议、访问端点等，供调用方了解**如何调用** |
| **SOAP**（简单对象访问协议） | 基于 **XML** 的**消息传输**协议，用于 Web 服务间通信 | 定义消息格式、编码规则、RPC 约定；可与 HTTP、SMTP 等结合 |

**口诀**：UDDI 发现注册，WSDL 描述接口，SOAP 传消息。

**案例（航空运营智能管理平台）**：对外 **HTTPS JSON** 为主；若监管/合作方要求 **SOAP/XML**，则 **适配器层** 做 **协议转换**（与 **ESB** 一节呼应）。

---

### 4.3 REST 规范要点（★★★★★）

- **全称**：Representational State Transfer（表述性状态转移）。
- **RESTful**：符合 REST 约束与原则的架构；未创造新技术，而是更好利用现有 Web 准则。

| 主题 | 要点 |
|------|------|
| **资源与 URI** | 资源是任何可被引用的事物（实体或抽象）；**URI** 为资源唯一标识（地址/名称） |
| **统一资源接口** | 使用标准 **HTTP 方法**（GET、PUT、POST、DELETE 等）访问资源 |
| **资源的表述** | 客户端获得的是资源的**表述**，形式多样（如 HTML、XML、JSON） |
| **资源的链接** | 用**超媒体**在表述中加入链接，引导客户端，增强连通性 |
| **状态的转移** | **服务端不保存客户端状态**；每次请求自带所需信息，实现状态转移 |

**案例（航空运营智能管理平台）**：**资源** 如 `/flights/{id}`、`/orders/{id}`；**GET/POST** 语义清晰；**JWT/会话令牌** 放在请求中体现 **无状态**；**HATEOAS** 可选（题干考到再写）。

## 五、ESB 模式（★★★★★）

### 5.1 ESB 概念

**企业服务总线（ESB）**：由**消息中间件**发展而来，采用**“总线”模式**管理与简化应用集成拓扑，基于**开放标准**支持应用在**消息、事件和服务**级别的互联互通。

**ESB 提供**：
- **位置透明**的消息路由与寻址、**服务注册与命名**管理；
- **多种消息传递范型**（请求/响应、发布/订阅等）；
- **多种传输协议**、**多种数据格式及相互转换**；
- **日志与监控**功能。

**与服务注册**：ESB 与服务注册模式可**混用**（如架构图中 ESB 与 UDDI 注册中心并存）。

**案例（航空运营智能管理平台）**：**多协议**（REST、MQ、遗留 SOAP）与 **多格式**（JSON/XML）经 **总线或网关** 转换；**日志与监控** 对齐素材 **操作审计、链路追踪**。

---

### 5.2 SOA 与 ESB 简答（约 200 字）

**SOA**：见第二节“什么是 SOA”的软件原理定义（组件模型、服务、良好接口与契约、中立、统一通用交互）。

**ESB 在 SOA 中的作用与特点**：提供位置透明性的消息路由和寻址服务、服务注册和命名的管理功能；支持多种消息传递范型（如请求/响应、发布/订阅）、多种传输协议、多种数据格式及其相互转换；还提供日志和监控功能。

## 六、SOA 设计原则（★★★★★）

**口诀**：**无单明自粗松重互**（没订单的人，名字粗松丢重要客户）。

| 原则 | 简要概括 |
|------|----------|
| **无状态** | 避免请求者依赖提供者状态 |
| **单一实例** | 防止功能冗余 |
| **明确定义的接口** | 由 **WSDL** 定义，WS-Policy 描述规约，XML Schema 定义消息格式 |
| **自包含和模块化** | 封装组件，功能实体可独立部署与管理 |
| **粗粒度** | 服务数量适度，靠消息交互 |
| **松耦合** | 使用者仅见接口，服务位置、实现不可见 |
| **重用能力** | 服务可重用 |
| **互操作性、兼容和策略声明** | 不同系统、应用、设备或组件能相互通信 |

**案例（航空运营智能管理平台）**：

- **粗粒度**：对外 **“购票服务”** 封装多步内部调用，避免客户端编排过碎。
- **松耦合**：调用方只依赖 **接口契约**，不依赖对方 **部署 IP**（结合注册发现）。
- **无状态**：服务实例 **任意扩缩**，会话外置 **Redis**。
- **重用**：**旅客主数据、通知、支付适配** 多业务复用。

## 七、典型架构

### 7.1 ESB 层次结构（四层）

| 层次 | 作用 |
|------|------|
| **应用工具层** | **流程编排**与 **ESB 管理**；流程编排负责业务流程设计与协调 |
| **流程控制层** | 通过**节点**控制流程走向，对**服务调用顺序、条件分支**等管理，保证流程按预期执行 |
| **数据转换与适配器层** | **适配器**连接不同系统/服务，实现通信；**数据转换**在不同格式间转换，保证数据可被正确识别与处理 |
| **服务总线层** | 基于 **JMS**（Java 消息服务），提供**可靠、异步、安全**的消息传递，为 ESB **基础通信层** |

**示意**：
```
  应用工具层（流程编排、ESB 管理）
       ↓
  流程控制层（节点、调用顺序、分支）
       ↓
  数据转换与适配器层（适配器、数据转换）
       ↓
  服务总线层（JMS、消息传递）
```

**案例（航空运营智能管理平台）**：遗留集成场景下，**流程编排** 可对应 **航变批量处置 SOP**；**适配器** 对接 **民航局核验、承运人**；底层 **消息** 与素材 **异步解耦** 一致。现代云原生可用 **Kafka + 编排引擎** 替代部分 ESB 能力，**考点默写仍以四层为准**。

---

### 7.2 微服务六模式（★★★★★）

| 模式 | 定义 | 示意 |
|------|------|------|
| **聚合器微服务** | 将**多个微服务的 API 组合成统一接口**，整合响应后返回客户端，简化交互 | 客户端 → 聚合器 → 服务A、B、C → 聚合器 → 客户端 |
| **代理微服务** | 在**客户端与微服务间**增加中间层，处理**路由、负载均衡、安全验证**等 | 客户端 → 代理 → 微服务 |
| **链式微服务** | 微服务**按顺序串联**，每服务处理部分功能并传递结果，形成完整流程 | 客户端 → S1 → S2 → S3 → 响应 |
| **分支微服务** | 按**业务子系统**拆成独立微服务，通过 **API 或消息队列**通信，模块化开发 | 多个并列微服务，API/MQ 互联 |
| **数据共享微服务** | 提供**统一数据访问接口或复制机制**，支持多应用/服务共享数据 | 多服务 → 统一数据层/复制数据 |
| **异步消息微服务** | 通过**消息队列**异步通信，**解耦**服务依赖，提升吞吐与响应能力 | 服务 → MQ → 服务 |

**微服务优势**：解耦、独立部署、技术选型灵活、容错性好、易扩展。

**案例（航空运营智能管理平台）**：

| 模式 | 案例 |
|------|------|
| **聚合器** | **BFF/API 网关** 聚合 **航班查询 + 余票 + 价格** 一次返回 App。 |
| **代理** | **网关** 做 **鉴权、mTLS、限流**（素材 **10 万并发**）。 |
| **链式** | **购票链**：校验→锁库→支付→出票→通知。 |
| **分支** | **票务 / 旅客 / 检修 / 数据服务** 并列，**MQ/API** 互联。 |
| **数据共享** | **主数据服务** 或 **读副本**，避免每服务直连乱查库。 |
| **异步消息** | **出票成功 → 消息 → 行程、通知、分析入湖**（素材联动链路）。 |

## 八、速记清单（考前扫一眼）

1. **SOA 定义**：组件模型 + 服务 + 良好接口与契约 + 中立 + 统一通用交互。
2. **微服务 vs SOA**：粒度（细/粗）、通信（REST/SOAP）、数据（独库/共享或 ESB）、架构（去中心/可 ESB）。
3. **服务注册三角色**：提供者（UDDI 发布、WSDL 描述）、注册中心（存储）、请求者（UDDI 查找、WSDL 了解、SOAP 调用）。
4. **UDDI/WSDL/SOAP**：发现注册、描述接口、传消息。
5. **REST**：资源+URI、统一 HTTP 接口、表述、超媒体链接、无状态转移。
6. **架构图填空**：业务层、UDDI、publish、ESB、安全验证与质量管理、组件层。
7. **ESB**：总线、消息路由与寻址、注册与命名、多范型/多协议/多格式转换、日志监控。
8. **SOA 原则**：无单明自粗松重互。
9. **ESB 四层**：应用工具→流程控制→数据转换与适配器→服务总线（JMS）。
10. **微服务六模式**：聚合器、代理、链式、分支、数据共享、异步消息。
