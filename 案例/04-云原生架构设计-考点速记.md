## 二、云原生架构原则（★★★★★）

### 2.1 七条原则一览

| 原则 | 描述 |
|------|------|
| **服务化原则** | 规模大时拆分服务，提升复用与流量治理能力 |
| **弹性原则** | 系统**自动伸缩**，**适配业务需求**（资源动态适配） |
| **可观测性原则** | 通过日志、链路跟踪等掌握系统状态 |
| **韧性原则** | **抵御软硬件异常**，保障系统稳定（故障场景下的保护） |
| **全流程自动化原则** | 自动化交付，提升效率、降低风险 |
| **零信任原则** | 基于身份访问控制，替代传统网络边界安全 |
| **持续演进原则** | 架构适应技术与业务变化，渐进式升级 |

**弹性 vs 韧性（必区分）**：
- **弹性**：侧重**资源动态适配业务需求**（扩缩容、负载适配）。
- **韧性**：侧重**故障场景下的系统保护**（容错、降级、恢复）。

**口诀**：弹性对“业务量”，韧性对“故障”。

**案例（航空运营智能管理平台）**：

- **服务化**：**票务、旅客、航信、数据服务** 等按域拆分，独立发布。
- **弹性**：春运/国庆 **HPA 扩容**、**票务峰值** 自动加副本。
- **可观测**：**全链路追踪、指标、日志** 支撑 **≤1 秒** 与错账排查（日志保留≥1 年）。
- **韧性**：**限流、熔断、降级、多 AZ**；航变洪峰下 **保护核心购票链路**。
- **自动化**：**CI/CD、灰度、回滚**；节前 **压测流水线**。
- **零信任**：**服务间 mTLS、RBAC、API 网关鉴权**（与素材安全章节衔接）。
- **持续演进**：**特性开关**、分阶段上线 **行李、新模型** 等。

## 三、云原生架构模式（★★★★★）

> 多模式组合使用；案例常给场景判断属于哪种模式；易作论文主题。

### 3.1 七种模式核心要点

| 模式 | 核心定义 | 典型场景/工具 |
|------|----------|----------------|
| **服务化架构**（SOA/微服务/小服务） | 按业务域或功能**拆分服务**，通过 API/协议交互；常结合 DDD、容器化与 CI/CD。优点：灵活扩展、独立部署、低耦合；缺点：分布式治理成本高 | 电商订单、支付、库存拆微服务 |
| **Mesh 化**（Service Mesh） | 将 **RPC、负载均衡、限流熔断、安全认证** 等从业务进程剥离，由 **Sidecar/Proxy（Mesh）** 统一处理；业务只写逻辑。优点：统一治理、解耦；缺点：额外性能开销 | Istio、Linkerd |
| **Serverless** | **无需管理服务器**，写业务逻辑即可；平台负责部署、运行、弹性伸缩与关闭；事件/流量触发，按调用量或时长计费。优点：运维成本低、按需伸缩；缺点：冷启动、厂商锁定 | 图片处理、日志分析、IoT 采集 |
| **存储计算分离** | **计算与存储解耦**：计算无状态，数据在分布式/云存储；有状态恢复靠 **WAL + Snapshot**。优点：弹性、成本优化；缺点：分离带来延迟 | Snowflake、TiDB、云数仓 |
| **分布式事务** | 跨库/跨服务下保持一致性。常见：**XA**（两阶段、强一致、性能差）、**消息最终一致**（MQ 异步、性能好）、**TCC**（Try/Confirm/Cancel、侵入强）、**SAGA**（本地事务+补偿）、**SEATA AT**（代理自动回滚） | — |
| **可观测架构** | **Logging**（多级日志）、**Tracing**（分布式链路）、**Metrics**（QPS、响应时间、并发等）；结合 SLO/SLA 做监控与告警 | Prometheus、Grafana、ELK、Jaeger |
| **事件驱动（EDA）** | 以**事件**为核心；事件带 Schema，可发布/订阅；有 QoS 与失败补偿。优点：解耦、异步、可扩展 | Kafka、Pulsar、EventBridge；订单完成触发发货、库变更触发缓存更新 |

**示意**：
```
  服务化：  [订单服务] [支付服务] [库存服务]  ←→ API/协议
  Mesh：    业务进程 + Sidecar(通信/治理/可观测)
  Serverless： 事件/流量 → 云函数 → 按量计费
  存储计算分离：  [计算节点-无状态]  ←→  [分布式存储]  （WAL+Snapshot 恢复）
  可观测：  Logging + Tracing + Metrics → 监控/告警
  EDA：     生产者 → 事件总线 → 消费者（异步、解耦）
```

**案例（航空运营智能管理平台）**（与七模式一一对应）：

| 模式 | 案例落点 |
|------|----------|
| **服务化** | 七大模块微服务 + **API 网关**；独立扩缩 **购票** 与 **看板**。 |
| **Mesh** | **Istio/Linkerd** 统一 **重试、超时、mTLS、流量分割**（春运灰度）。 |
| **Serverless** | 可选 **云函数** 做 **短信模板渲染、轻量规则**；核心账务仍以容器为主。 |
| **存储计算分离** | **无状态服务 + 分布式库/湖仓**；分析 **PB 级** 与在线 **OLTP** 分离。 |
| **分布式事务** | 购票 **跨库存、支付、消息**；**SAGA/TCC/消息最终一致** 与素材 **强一致** 要求对齐选型。 |
| **可观测** | **Prometheus/Grafana + ELK + Jaeger**；SLO 对齐 **TPS、P99、错误率**。 |
| **EDA** | `OrderPaid`、`FlightCancelled` 等 **异步** 驱动 **通知、库存回补、检修阈值、入湖**；解耦峰值。 |

## 四、云原生反模式 / 不好的实践（★★★☆☆）

| 反模式 | 问题描述 | 解决方案 |
|--------|----------|----------|
| **庞大的单体应用** | 依赖隔离不足、代码耦合、模块接口缺乏治理、发布难以协调、单模块故障影响整体 | **服务化适度拆分**：梳理聚合根，明确服务边界与接口，使组织与架构匹配 |
| **单体“硬拆”为微服务** | 过度拆分导致与组织能力不匹配、服务间数据依赖、性能下降 | **合理评估粒度**：考虑组织能力与业务需求，避免过度拆分；优化服务间数据交互与性能 |
| **缺乏自动化的微服务** | 规模大时人工开发/测试/运维导致交付慢、风险高、成本增 | **建立自动化**：自动化测试、发布、环境管理等，以应对复杂度 |

**案例（航空运营智能管理平台）**：

- **大单体**：旧时代 **烟囱系统** 难撑 **联动链路与峰值** → 按域 **适度拆分**。
- **硬拆微服务**：把 **同一聚合** 拆成十几个服务、徒增 RPC → **按 DDD 限界上下文** 与团队对齐粒度。
- **缺自动化**：手工发版在 **99.99%** 目标下不可持续 → **流水线 + 基础设施即代码**。

## 五、容器技术（★★★★★）

### 5.1 容器 vs 虚拟机（必背）

| 对比维度 | 容器 | 虚拟机 |
|----------|------|--------|
| **运行原理** | **OS 层虚拟化**，**共享宿主机内核**，直接运行在 OS 之上 | **硬件虚拟化**，模拟完整硬件，运行**独立 OS** |
| **资源隔离** | **cgroups** 限制资源（CPU、内存、磁盘 I/O），**namespace** 隔离（进程、网络、文件系统等）；共享内核 | 硬件虚拟化，**完全隔离**，每 VM 独立内核与硬件 |
| **启动速度** | **秒级**（毫秒级），无需启完整 OS | **分钟级**，需加载整个 OS |
| **资源占用** | **少**，仅应用及依赖，共享内核 | **多**，每 VM 独立 OS 与硬件模拟 |
| **应用移植性** | **好**，镜像打包应用与依赖，环境一致 | 受虚拟硬件与 OS 影响，迁移需考虑兼容 |
| **隔离性** | **较弱**，共享内核，存在内核级安全风险 | **强**，VM 相互独立 |
| **管理复杂度** | 相对简单，管镜像与容器生命周期，编排工具管集群 | 复杂，需管多 OS 安装、配置、补丁及虚拟硬件 |

**口诀**：容器共享内核轻又快，虚拟机独立 OS 隔离强。

**案例（航空运营智能管理平台）**：生产环境 **票务/数据服务** 等多以 **容器** 交付，提高 **密度与发布速度**；强隔离合规区可仍用 **VM/裸金属** 混部，答题按 **对比表** 选词。

---

### 5.2 容器 vs 容器编排（必背）

| 对比维度 | 容器 | 容器编排技术 |
|----------|------|----------------|
| **定义** | 将**应用及依赖**打包成**独立单元**的技术，可在不同环境运行，**共享宿主机内核** | **自动化部署、管理、扩展**容器化应用的技术，管理**多容器集群** |
| **核心功能** | 隔离运行环境，**快速部署与迁移** | **集群管理、资源调度、服务发现、负载均衡、自动伸缩、故障恢复**等 |
| **代表工具** | **Docker** | **Kubernetes（k8s）** |
| **部署难度** | 单容器简单（拉镜像、启容器） | 编排系统复杂（多组件、集群初始化） |
| **适用场景** | 单一/简单应用，或微服务中**单服务**载体 | **大规模、复杂**容器集群，需资源调度、伸缩、服务治理 |
| **管理粒度** | **单容器**生命周期（创建、启停、删除） | **集群级**：多容器分组、调度、健康监控、动态扩缩 |

**口诀**：容器管“一个盒子”，编排管“一集群盒子”。

**案例（航空运营智能管理平台）**：**Kubernetes** 管理 **成百 Pod**、**滚动升级、健康检查、弹性伸缩**，对应素材 **弹性扩展架构**；单容器仅适合演示或边缘任务。

### 5.5 典型例题要点

**题1**：用 300 字以内说明容器与虚拟机的含义，并论述采纳容器技术的原因。

**答**：  
- **容器**：轻量级隔离技术，多容器**共享主机内核**，每容器运行应用，有自身文件系统与运行环境，实现快速部署与隔离。  
- **虚拟机**：虚拟化软件在物理机上模拟**多台独立虚拟计算机**，每 VM 有独立 OS 与硬件，**完全隔离**。  
- **采纳容器原因**：更轻量、资源占用少、启动快、利于资源利用；安全可通过策略与工具补强；**低成本、高效率**符合升级初衷。

**题2**：性能指标（a）～（h）填入对比表（1）～（8）。  
- （a）分钟级、（b）包含 GuestOS G 量级、（c）跨 OS 迁移、（d）CPU/内存按核按 G 分配、（e）毫秒级、（f）Cgroups 进程级、（g）VM 伸缩手动、（h）实例自动伸缩。  
- **答案**：①b ②d ③a ④e ⑤c ⑥g ⑦h ⑧f（对应题干表格中 VM 与容器的各对比项：如镜像大小、资源分配、启动时间、移植性、伸缩方式、隔离方式等，按表格列顺序填写）。

**案例（航空运营智能管理平台）**（题1 可补一句）：选型容器是为支撑 **多环境一致、快速扩容缩容与 DevOps**，与 **春运弹性、频繁发版** 匹配；安全上配合 **镜像扫描、非 root、网络策略**。

## 六、速记清单（考前扫一眼）

1. **弹性 vs 韧性**：弹性=资源适配业务；韧性=故障场景下保护。
2. **七原则**：服务化、弹性、可观测、韧性、全流程自动化、零信任、持续演进。
3. **七模式**：服务化、Mesh、Serverless、存储计算分离、分布式事务、可观测、事件驱动；各记一句核心。
4. **反模式**：大单体→适度拆分；硬拆微服务→评估粒度与组织能力；缺自动化→建自动化体系。
5. **容器 vs VM**：容器共享内核、轻、快、隔离弱；VM 独立 OS、重、慢、隔离强。
6. **容器 vs 编排**：容器=单实例打包运行；编排=多容器集群管理（k8s）；Docker vs Kubernetes。
7. **Docker/kubectl**：ps、run、stop、logs、exec、build、images、pull/push；get nodes/pods/services、describe、logs。
