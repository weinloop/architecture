## 二、类识别（★★★☆☆）

### 2.1 三类与识别方法（词性法）

设计得好的类至少有一种职责：**属性封装数据，方法封装行为**。类按职责分为三类：

| 类型 | 识别方法（词性） | 作用 | 示例 |
|------|------------------|------|------|
| **实体类** | **通常是名词** | 保存需永久存储的信息，表达属性和关系 | 学员类、课程类、账户、图书、借阅记录、罚金 |
| **控制类** | **动宾短语**（动词+名词 / 名词+动词）或**纯动词** | 控制用例工作，对用例特有的控制行为建模，具有协调性 | 身份验证器、借阅、归还、预订、续借 |
| **边界类** | **通常是名词**（多为固定几类） | 系统接口与外部交互，将系统与外部环境隔离 | 窗口、通信协议、打印机接口、报表、读者证等“连接外部与控制类”的东东 |

**记忆口诀**：
- **实体**：名词语义存数据（名词 → 实体）
- **控制**：动宾/动词管流程（动宾、动词 → 控制）
- **边界**：窗口协议报表面，连外连控是边界

**边界类特殊记忆**：窗口、通信协议、打印机接口、报表 + 其他能“连接外部和控制类”的都可视为边界类；其余不易判断时从“是否沟通外部与内部控制”来判。

**案例（航空运营智能管理平台）**：

- **实体类（名词）**：`航班`、`客票`、`订单`、`旅客档案`、`检修工单`、`设备`、`舱位库存` 等需 **持久化** 的领域对象。
- **控制类（动宾/动词）**：`出票协调`、`航变处置`、`分布式事务协调`（映射为领域服务/用例控制器）、`退改签处理` 等 **协调用例步骤**。
- **边界类（连外连控）**：`移动端购票界面`、`开放 API 网关适配器`、`民航局核验客户端`、`支付网关适配器`、`短信/推送网关`、`运营可视化报表接口` 等 **隔离外部协议与 UI 变更**。

---

### 2.2 三类关系示意

```
  [外部参与者]  ←——→  边界类  ←——→  控制类  ←——→  实体类
     (人/系统)         (界面/协议)    (用例控制)    (持久化数据)
```

- **边界类**：与外部交互的“界面层”。
- **控制类**：协调用例流程，一般一个用例一个（可复用）。
- **实体类**：存数据、表关系，其他类一般不向控制类发消息，而由控制类发出消息。

---

### 2.3 概念简答（真题要求 200 字内）

- **边界类**：描述外部参与者与系统的交互；对系统外部环境与内部运作之间的交互建模，包括转换事件、记录系统表示方式（如接口）的变更。
- **实体类**：数据管理与业务逻辑层面；主要职责是存储和管理系统内部信息，可有与实体对象密切相关的行为。
- **控制类**：描述一个用例的事件流控制行为，控制用例中的事件顺序；控制其他类工作，通常每用例一个控制类，也可多用例共用；其他类一般不向控制类发消息，由控制类发出消息。

**案例（航空运营智能管理平台）**：旅客在 **边界（App）** 提交购票请求 → **控制类** 编排 **锁库存、调支付、调核验、写订单、发消息** → **实体类** 落库；外部系统只与 **边界适配器** 对话，符合 **边界隔离外部环境** 的定义。

---

### 2.4 典型例题思路（图书馆管理系统）

**候选项**：a 系统管理员 b 图书管理员 c 读者 d 读者证 e 账户 f 图书 g 借阅 h 归还 i 预订 j 罚金 k 续借 l 借阅记录  

**分类技巧**：
- **控制类**：动宾/动词 → g 借阅、h 归还、i 预订、k 续借。
- **边界类**：能“连接外部与控制类” → 典型答案为 **d 读者证**（作为与系统交互的凭证/接口）。
- **实体类**：名词且非边界 → a、b、c、e、f、j、l（系统管理员、图书管理员、读者、账户、图书、罚金、借阅记录）。

**注意**：官方答案曾把 (1) 填 j、l（实体），(2) 填 a、b、c、f 等，(3) 填 g、h、i、k（控制）；边界类在不同解析中可能为 d 读者证或其它，做题时以“连接外部与控制类”和题目选项为准。

**案例（航空运营智能管理平台）（对照练习）**：

| 类型 | 可能选项词（示例） |
|------|-------------------|
| **实体** | 旅客、航班、客票、订单、罚金类比 **改签手续费记录**、借阅记录类比 **行程单** |
| **控制** | 购票、退票、改签、候补、出票协调（动宾） |
| **边界** | **电子登机牌/二维码凭证**、**短信网关**、**小程序界面**（连接旅客与内部控制） |

## 三、面向对象设计原则（★★★★★）

### 3.1 七大原则一览

| 原则 | 英文/缩写 | 核心含义 |
|------|-----------|----------|
| **单一职责原则** | SRP (S) | 类应该**只有一个**设计目的 |
| **开闭原则** | OCP (O) | 对**扩展开放**，对**修改关闭**；通过抽象层实现，是其他原则的基础 |
| **里氏替换原则** | LSP (L) | 使用基类对象的地方可替换为子类对象，反之不一定；变化类设计为抽象类或接口 |
| **接口隔离原则** | ISP (I) | 接口提供客户端**需要的行为**，大接口拆成小接口；或把接口视为“角色” |
| **依赖倒置原则** | DIP (D) | **抽象不依赖细节**，针对接口编程；实现开闭的主要机制 |
| **组合/聚合复用原则** | — | 通过**组合或聚合**复用已有对象，比继承更灵活、耦合度低 |
| **最少知识原则** | 迪米特法则 | 软件实体应**尽量少**与其他实体相互作用 |

**口诀**：**SOLID**（S-O-L-I-D） + **组聚复用** + **迪米特**。

**案例（航空运营智能管理平台）**：

- **S**：`售票计算` 与 `库存服务` 分离，避免“一个类又算价又扣库”。
- **O**：新增 **售票渠道** 或 **退改规则** 时 **扩展策略类**，少改核心出票流程。
- **L**：`国内票`/`国际票` 替换抽象 `票` 时不破坏计价与退改契约。
- **I**：拆分 **只读查询接口** 与 **交易写接口**，避免运营大屏依赖购票写权限。
- **D**：业务依赖 **支付、核验** 的 **接口**，实现可替换。
- **组合/聚合**：用 **消息+仓储组合** 完成同步，而非深继承。
- **迪米特**：`旅客应用服务` 通过 **门面** 访问票务与通知，减少跨域类互相引用。

---

### 3.2 原则关系示意

```
开闭原则(O) ←—— 依赖倒置(D) 是实现 O 的主要机制
     ↑
单一职责(S)、里氏替换(L)、接口隔离(I) 等共同支撑可扩展、可维护
组合/聚合复用：优先组合/聚合，少用继承，降低耦合
最少知识：少交互，降低耦合
```

---

### 3.3 开闭原则简答（标准答法，约 150 字）

**题目**：请简要说明什么是面向对象设计的开闭原则。

**答**：面向对象设计的开闭原则（Open-Closed Principle，OCP）指软件实体（如类、模块、函数等）应对**扩展开放**、对**修改关闭**。即：在不修改原有源代码的前提下，通过扩展来增加新行为；能够在不改动原有代码的情况下改变模块行为。实践中常通过**抽象层**（抽象类、接口）实现，依赖倒置原则是实现开闭原则的主要机制。

**案例（航空运营智能管理平台）**（答题时可加一句落地）：例如新增 **承运人退改规则** 时，通过 **实现新的 `退改策略` 接口** 接入，而不修改原有 **出票协调类** 的主干代码。

---

### 3.4 各原则一句话记忆

| 原则 | 一句话 |
|------|--------|
| S 单一职责 | 一个类只干一类事 |
| O 开闭 | 能扩展不能改（加抽象层） |
| L 里氏替换 | 基类能用的地方子类也能用，变化放抽象/接口 |
| I 接口隔离 | 接口只给客户需要的，大接口拆小 |
| D 依赖倒置 | 面向接口编程，抽象不依赖细节 |
| 组合/聚合复用 | 用组合/聚合复用，少用继承 |
| 最少知识 | 少跟别的实体打交道（迪米特） |

## 四、速记清单（考前扫一眼）

1. **类识别**：实体→名词；控制→动宾/动词；边界→窗口、协议、打印机接口、报表、读者证等“连外连控”。
2. **SOLID**：S 单职、O 开闭、L 里氏替换、I 接口隔离、D 依赖倒置。
3. **开闭原则**：对扩展开放、对修改关闭；通过抽象层实现；依赖倒置是实现手段。
4. **组合/聚合复用**：优先组合/聚合，少继承，低耦合。
5. **最少知识**：迪米特法则，少与其他实体交互。
