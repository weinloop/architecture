## 二、层次架构简介与反模式

### 2.1 层次架构要点

- **定义**：系统组成**层次结构**，每一层为**上层服务**，并作为**下层客户**；内部层接口可只对**相邻层**可见。
- **特性**：**关注分离**——本层组件只负责本层逻辑，角色与职责清晰，便于开发、测试、管理与维护。
- **常见三层**：**表现层** → **中间层（业务层）** → **数据访问层**。

**层次关系示意**：
```
  表现层（上层）  —— 使用 ——→  中间层  —— 使用 ——→  数据访问层（下层）
       ↑ 服务对象                    ↑ 服务对象
```

**案例（航空运营智能管理平台）**：**表现层** 负责 **查询航班、展示舱位与价格**；**业务层** 负责 **计价规则、库存锁、与支付/核验编排**；**数据访问层** 负责 **订单持久化、读写分离、分库分表**。相邻层通过 **DTO/接口** 协作，避免表现层直接写 SQL。

---

### 2.2 污水池反模式（Sinkhole Anti-Pattern）

- **现象**：请求**只是穿过**各层，没有在每层做实质处理。例如：界面层把“获得数据”交给业务层，业务层再交给持久层，持久层简单 SQL 查库，数据原路返回，**没有任何二次处理**。
- **后果**：层次隔离失效、结构不清晰、业务规则变更时易遗漏或不一致、可维护性降低。
- **解决**：**层次开放**——部分数据处理可**直达核心层次**，不必层层穿越。

**示意**：
```
  污水池反模式：  [界面] → 透传 → [业务] → 透传 → [持久] → SQL → 原路返回（无加工）
  改进：          部分请求可跨层直达核心层，在关键层做真实逻辑
```

**案例（航空运营智能管理平台）**：若 **业务层** 仅把“查余票”原样转给 DAO，**无任何** 舱位策略、缓存、风控，即 **污水池**；**只读热点航班** 可由 **BFF/查询服务** **直达缓存或只读库**（层次开放），**写路径** 仍必须经过 **事务与规则层**。

## 三、表现层架构：MVC / MVP / MVVM（★★★★★）

### 3.1 三者对比表

| 维度 | MVC | MVP | MVVM |
|------|-----|-----|------|
| **核心组件** | Model、View、**Controller** | Model、View、**Presenter**（呈现器） | Model、View、**ViewModel**（视图模型） |
| **Model** | 数据存储与业务逻辑 | 同左 | 同左 |
| **View** | 用户界面展示 | 用户界面交互接口 | 绑定数据的 UI |
| **第三元** | 控制器：处理输入，协调 M 与 V | 呈现器：处理逻辑，**控制视图更新** | 视图模型：**数据转换与双向绑定** |
| **数据流** | **单向**：V→C→M→V（经 C 更新） | **双向**：V↔Presenter↔M（Presenter 主动更新 V） | **双向**：V↔ViewModel↔M（**自动同步**，基于绑定） |
| **View 与其它** | 可调控制器，可能访问 Model | 通过**接口**与 Presenter 通信，**不直接访问** Model | 通过**数据绑定**与 ViewModel 交互，**无直接逻辑调用** |

**记忆口诀**：
- **MVC**：单线 C 协调，V 可能碰 M。
- **MVP**：P 控 V，V 不碰 M，通过接口。
- **MVVM**：VM 双向绑定，V 不调逻辑。

**数据流示意**：
```
  MVC:   View ——→ Controller ——→ Model ——→ (Controller 更新) View
  MVP:   View ←→ Presenter ←→ Model   （Presenter 主动更新 View）
  MVVM:  View ←→ ViewModel ←→ Model   （绑定自动同步）
```

**案例（航空运营智能管理平台）**：

- **MVC**：管理端 Web 早期常用 **Controller** 接 **HTTP 请求**，协调 **Model** 与 **JSP/模板**。
- **MVP**：**小程序/原生** 可用 **Presenter** 处理 **购票步骤**，**View** 通过接口回调，**不直接碰领域模型**。
- **MVVM**：**数据可视化看板、航班动态大屏** 适合 **ViewModel** 绑定 **查询结果与加载状态**，减少样板代码（素材：**数据更新延迟≤3 秒** 的展示层）。

## 四、中间层架构（★★★☆☆）

### 4.1 三层 + DomainModel—Service—Control

- **整体**：表示层、**业务层**、数据库层。业务层内部按 **DomainModel — Service — Control** 解耦。
- **作用**：职责分离、服务与服务控制隔离，提高**可重用性**和**灵活性**。

| 组件 | 职责 |
|------|------|
| **DomainModel** | 封装**业务实体及核心逻辑**（如用户、订单的属性和行为），保证业务规则一致 |
| **Service** | **协调跨领域**业务操作（事务、权限等），可调用多个 DomainModel 完成复杂功能 |
| **Control** | **服务控制器**：根据业务状态和参数决定 Service 的**执行顺序与依赖**，实现服务间松耦合 |

**示意**：
```
  表示层
     ↓
  业务层：  Control → Service → DomainModel（多实体）
     ↓
  数据库层
```

**案例（航空运营智能管理平台）**：

- **DomainModel**：`订单`、`客票`、`航班`、`库存行` 等 **领域行为**（如 **不可超售**）。
- **Service**：`购票服务` **协调** 库存、支付、核验、消息 **跨聚合**；管 **分布式事务边界**。
- **Control**：**用例编排**（如 **航变批量退改** 先冻结再批量退款），决定 **服务调用顺序与补偿**。

## 五、数据访问层（★★★★★）

### 5.1 数据访问模式对比

| 模式 | 核心思想 | 特点 |
|------|----------|------|
| **在线访问模式** | 应用与数据库**持续连接**，实时执行数据操作 | 实时性强、直接操作库；依赖网络与连接稳定性 |
| **DAO（数据访问对象）** | 将**数据访问逻辑与业务逻辑分离**，用接口封装数据库操作 | 解耦数据层与业务层，可换库，代码复用 |
| **DTO（数据传输对象）** | 跨进程/网络传输数据的**轻量容器**，**无业务逻辑** | 减少传输次数、可序列化、与数据源结构解耦 |
| **离线数据模式** | 数据从数据源取走后按**预定义结构存本地**，与数据源**断开** | 离线处理，支持 XML/JSON，不依赖持续连接 |
| **ORM（对象/关系映射）** | 在关系库与对象间建**映射**，像操作对象一样操作库 | 面向对象、自动 SQL、减少数据访问层开发 |

**DAO**：把“和数据库打交道”封装在 DAO 类里，业务层只调接口（如 `BookDAO`：addBook、deleteBook、getBookById），实现类里用 JDBC 等做具体 SQL。  
**DTO**：像“快递包裹”，只装数据、不装逻辑；例如后端查库后只把前端需要的字段放进 ProductDTO 传给前端。

**案例（航空运营智能管理平台）**：**OrderDAO / TicketDAO** 封装 **分库分表与 SQL**；**Feign/REST** 返回 **OrderDTO**（脱敏、无内部状态）；**ORM** 映射 **订单实体** 与 **关系表**；**离线/同步** 场景可将 **T+1 分析宽表** 落地本地再算（离线模式思想）。

---

### 5.2 在线访问 vs ORM（简答常考）

| 方式 | 优点 | 缺点 |
|------|------|------|
| **程序在线访问** | 性能较优、能处理复杂查询 | 要求程序员懂 SQL，修改维护相对困难 |
| **ORM** | 降低学习与开发成本、无需手写 SQL、减少代码量与 SQL 质量风险 | 复杂查询处理困难、性能一般不如直接 SQL |

**选题 ORM 的常见理由**：团队缺乏数据库经验、避免 SQL 质量与维护成本、应用相对简单、性能非瓶颈。

**案例（航空运营智能管理平台）**：**核心票务路径** 常 **手写 SQL + 乐观锁/悲观锁** 保 **5000 TPS**；**内部管理模块** 可用 **ORM** 提速；**复杂报表** 多用 **专用 SQL/OLAP 引擎** 而非 ORM 拼接。

---

### 5.3 增加数据访问层的原因（约 100 字）

实现**数据与业务逻辑分离**，提高可维护性、扩展性和灵活性。数据访问层负责数据库操作、封装访问细节，使**业务逻辑层与具体存储技术解耦**，简化开发；便于对数据访问统一管理、优化，提升整体性能与可靠性。

**案例（航空运营智能管理平台）**：**OLTP 库**、**实时数仓**、**对象存储归档** 技术栈不同，**DAL + 抽象** 可避免业务层散落连接细节，便于 **读写分离与容灾切换**。

---

### 5.4 异构数据库架构填空（工厂 + 数据访问层）

题干常给出：表示层 → 业务逻辑层 → 数据访问层（多数据库）。空常位于业务层与 DAL 之间、DAL 内部。

**典型答案**：
- **（1）** 执行业务逻辑 / 业务组件 / 业务构件（位于业务逻辑层）
- **（2）** 数据访问接口层 / DAL 接口
- **（3）** 工厂层 / DAL 工厂 / 数据访问工厂（**工厂模式**产出不同数据库的访问实现）

**易错**：不要把（2）（3）只写成“数据访问接口、数据访问实现”，要体现**工厂**用于支持**多种异构数据库**。

**案例（航空运营智能管理平台）**：同时存在 **MySQL 订单库**、**分析型列存/湖仓** 时，**DAL 工厂** 按配置产出 **不同实现**，业务层只依赖 **接口**。

---

### 5.5 ORM 落地：Hibernate（★★★☆☆）

- **作用**：ORM 框架，完成 **Java 对象 ↔ 数据库表** 的映射。
- **架构图填空**（如 23 年 11 月案例）：常涉及**持久化对象/实体**、**Session / SessionFactory**、**持久化层/数据访问层**等，按题干图填空即可。

**案例（航空运营智能管理平台）**：若技术栈含 **Java + Hibernate/JPA**，**实体类** 对应 **订单/旅客** 表；**SessionFactory** 管理连接；考点仍按 **题干图** 填，不必强行写业务名。

## 六、物联网层次架构（★★★☆☆）

> 案例直接考较少，可能作**论文**背景；各层职责与典型设备需了解。

| 层次 | 介绍 | 典型设备/技术 |
|------|------|----------------|
| **感知层** | 物联网基础，**识别物体、采集信息**，解决物理世界数据获取；传感器采集后经**短距离传输** | 传感器、RFID 标签与读写器、二维码与识读器、摄像头、GPS、M2M 终端、传感器网关等 |
| **网络层** | **连接感知层与应用层**，传递与处理信息，将感知数据安全可靠传到应用层，按需处理 | 通信网、互联网、网络管理中心；接入网（光纤/无线等）、传输网（电信网、广电网）；3G/4G、IPv6；路由器、交换机、网关等 |
| **应用层** | 最顶层，**行业智能化应用**，与行业技术深度融合，对感知数据**计算、处理、知识挖掘**，实现控制、管理与决策 | 智能操控、安防、远程医疗等应用；终端产品；物联网中间件、云计算 |

**记忆**：**感（感知）— 网（网络）— 应（应用）**；感知采数据，网络传数据，应用用数据。

**示意**：
```
  应用层（行业应用、中间件、云）
       ↑
  网络层（通信网、接入/传输、路由/交换/网关）
       ↑
  感知层（传感器、RFID、二维码、GPS 等）
```

**案例（航空运营智能管理平台）**（拓展联想，考点仍以教材为准）：

- **感知层**：**机载/地面传感器**、**RFID 行李标签**、**舱门与设备遥测** 采集 **设备运行与检修相关数据**（与素材 **设备参数异常、检修 AI** 呼应）。
- **网络层**：**4G/5G、专线、机载通信** 将数据送入信息中心。
- **应用层**：本平台 **检修管理、实时告警、调度决策、可视化** 即 **行业应用**。

## 七、速记清单（考前扫一眼）

1. **层次**：每层为上层服务、作下层客户；关注分离；避免**污水池反模式**（透传无加工 → 层次开放、直达核心）。
2. **MVC/MVP/MVVM**：MVC 单向 C 协调；MVP 的 P 控 V、V 不直接访问 M；MVVM 双向绑定、V 不写逻辑。
3. **中间层**：DomainModel（实体与核心逻辑）— Service（协调、事务）— Control（调用顺序与依赖）。
4. **数据访问**：在线访问（性能好、要懂 SQL）vs ORM（易开发、复杂查询弱）；DAO 分离访问逻辑，DTO 只传数据。
5. **加数据访问层**：数据与业务解耦、易维护易扩展、统一管理与优化。
6. **异构库填空**：业务逻辑 → DAL 接口 → **工厂层**（产出多库实现）。
7. **物联网三层**：感知（采集）、网络（传输）、应用（行业应用与决策）。
