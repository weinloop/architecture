## 二、ES 概述（★★★★★）

- **特点**：以**全文检索**为基础；使用**分析器、分词**处理文本，支持词干、停用词、同义词等；建立索引时**分词**，写入**倒排索引**；查询时对查询语句分词，在倒排索引中找文档，再按**相关性评分**（如 BM25、TF-IDF）排序。
- **流程**：文档 → **分词** → 词项 → **倒排索引**；查询 → 分词 → 查倒排索引 → **相关性打分** → 排序返回。

## 三、分词原理（★★★★★）

### 3.1 分析器（Analyzer）三部分

分析器是一条**流水线**，依次为：

| 环节 | 作用 |
|------|------|
| **字符过滤器（Character Filters）** | 对**原始文本**预处理：去/换特殊字符、HTML 转可读等；**可选** |
| **分词器（Tokenizer）** | **核心**：按规则把文本切成**词元（Token）**；如标准分词器按空格、标点切分 |
| **标记过滤器（Token Filters）** | 对词元再加工：**小写**、去**停用词**、**词干提取**等，便于索引与搜索 |

**口诀**：字符过滤 → 分词器切词元 → 标记过滤优化。

**示意**：
```
  原始文本 → [字符过滤器] → [分词器] → [标记过滤器] → 词元序列
```

---

### 3.2 常用 Tokenizer（分词器）

| 类型 | 输入示例 | 输出示例 | 特点 |
|------|----------|----------|------|
| **standard**（标准） | Hello, this is Elasticsearch! | ["hello","this","is","elasticsearch"] | 去标点、转小写，适合自然语言 |
| **whitespace**（空白） | Hello World, 你好 Elasticsearch | ["Hello","World,","你好","Elasticsearch"] | **仅按空格**切，标点保留 |
| **keyword**（关键词） | OrderID-12345 | ["OrderID-12345"] | **不切分**，整段作为一个词元，适合 **ID、手机号**等精确匹配 |
| **uax_url_email** | 请联系 support@example.com 或 https://cheko.cc | 邮箱和 URL 为**整体** | 识别 URL、邮箱，保持完整 |

**记忆**：standard 自然语言，whitespace 只看空格，keyword 不切（精确），uax_url_email 保 URL/邮箱。

## 四、倒排索引（★★★★★）

- **定义**：**反向索引**；先对文档**分词**得到词项，再建立「**词项 → 文档 ID 列表**」的索引；查词即可得包含该词的文档。
- **与正排**：正排是 **key（文档）→ value（内容）**；倒排是 **value（词项）→ key（文档列表）**。
- **优势**：无需扫描全部文档，通过**词典 + 倒排表**快速定位包含某词的文档，**全文检索效率高**。

**示例**：  
文档 1：苹果是一种水果。 文档 2：香蕉是一种热带水果。 文档 3：苹果和香蕉都是水果。  
倒排索引：**苹果 → [1, 3]**；**香蕉 → [2, 3]**。搜「苹果」即得文档 1、3。

**示意**：
```
  正排： 文档ID → 内容
  倒排： 词项 → [文档ID1, 文档ID2, ...]
```

## 五、相关性算法（★★★★★）

- **作用**：**相关性得分**决定排序，得分越高越符合查询意图。
- **算法**：由 **Lucene** 计算；常用 **TF-IDF**、**BM25**（ES 默认）。BM25 为 TF-IDF 改进版，长文档与参数可调性更好。

### 5.1 影响相关性的主要因素

| 因素 | 含义 | 影响 | 举例 |
|------|------|------|------|
| **查询类型** | 不同查询方式对应不同打分规则 | **match** 会算相关性；**term** 偏精确匹配，不参与复杂评分 | match「手机」考虑词频与相关性；term 只匹配完全一致的「手机」 |
| **词频（TF）** | 词在**当前文档**中出现次数 | 出现越多，得分越高 | 文档 A 出现 10 次「手机」比 B 出现 1 次得分高 |
| **逆文档频率（IDF）** | 词在**所有文档**中的稀有程度 | 越稀有，贡献得分越高 | 「iPhone17」比「手机」更稀有，区分度更强 |
| **文档长度归一化（Norms）** | 关键词在**短文档 vs 长文档**中的代表性 | 短文档中出现一次更有代表性 | 标题里出现一次「iPhone」比长文中一次更突出 |
| **字段权重（Boost）** | 手动提高某字段重要性 | 被 Boost 的字段匹配得分更高 | title boost=2，优先于 description |

**口诀**：查询类型定规则，TF 多则分高，IDF 稀则分高，短文档更突出，Boost 抬权重。

## 六、ES 与传统关系型数据库对比（★★★★★）

| 对比维度 | Elasticsearch（ES） | 传统关系型数据库（如 MySQL） |
|----------|---------------------|------------------------------|
| **类型** | **分布式搜索引擎**，**文档型**存储 | **关系型**，**表结构**存储 |
| **核心功能** | **全文搜索**、复杂查询、日志分析、**实时检索** | 结构化存储、**事务**、**数据一致性** |
| **应用场景** | 全文检索（电商搜索）、日志分析、监控查询 | 订单、财务、用户信息等**强事务**场景 |
| **优点** | 搜索性能高、实时搜索、**分布式扩容方便**、全文检索强 | **ACID 事务**、一致性强、生态成熟、SQL 标准 |
| **缺点** | 部署运维复杂、查询学习成本高、**一致性弱** | 实时搜索弱、大数据扩展难、全文能力有限 |
| **数据结构** | **文档（JSON）**，半结构化 | **二维表**，结构化 |
| **分布式** | **原生**分布式集群，**横向扩展**能力强 | 需**分库分表**，架构复杂 |
| **事务** | 仅**文档级**简单操作，**事务弱** | **完整 ACID**，原子性、一致性有保障 |

**口诀**：ES 强搜索与扩展、弱事务一致；关系库强事务一致、弱搜索与扩展。

## 七、速记清单（考前扫一眼）

1. **ES 定位**：分布式全文搜索与分析引擎，基于 Lucene，RESTful；适用全文检索、日志、监控、地理等。
2. **分词流水线**：字符过滤器（可选）→ 分词器（切词元）→ 标记过滤器（小写、停用词、词干等）。
3. **Tokenizer**：standard 自然语言、whitespace 按空格、keyword 不切（精确）、uax_url_email 保 URL/邮箱。
4. **倒排索引**：词项 → 文档 ID 列表；先找词再找文档；正排 key→value，倒排 value→key。
5. **相关性**：TF-IDF、BM25（默认）；影响因素：查询类型（match/term）、TF、IDF、Norms、Boost。
6. **ES vs 关系库**：ES 文档/搜索/分布式/弱事务；关系库 表/事务/一致性/强 ACID、弱全文与扩展。
