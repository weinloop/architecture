## 二、数据类型（★★★★★）

| 类型 | 含义 | 适用场景 |
|------|------|----------|
| **String** | 字符串 | 配置、缓存对象、统计、限流（时间内请求次数） |
| **Hash** | 键值对（field-value） | 用户/商品/文章/购物车等对象信息 |
| **List** | 列表 | 最新文章/动态、简单消息队列 |
| **Set** | 集合，不重复，可交并差、随机取 | UV 统计、点赞、共同好友、抽奖、随机点名 |
| **Zset** | 有序集合，按权重排序 | 排行榜（礼物、步数、段位、热度） |
| **Bitmap** | 位图，0/1 表示状态 | 签到、活跃用户、行为统计 |
| **HyperLogLog** | 基数计数 | 海量计数（百万/千万级 IP、UV 统计） |
| **Geospatial** | 地理空间 | 附近的人、地理位置 |

**String vs Hash 存对象**：  
- **String**：存**序列化后的整对象**，更省内存（约 Hash 一半），适合多层嵌套复杂对象；  
- **Hash**：可**单独存、取、改**部分字段，省网络流量，适合字段常变或需单独查询的场景。

## 三、持久化机制（★★★★★）

| 方式 | 含义 | 特点 |
|------|------|------|
| **RDB** | Redis DataBase，**快照** | 按时间间隔将数据**快照**写入磁盘；**恢复快**；宕机可能丢一段数据；**默认**开启 |
| **AOF** | Append-Only File，**追加日志** | 每个**写命令**追加到文件；可做到**秒级**持久化；**恢复慢**；可重写压缩 |

**AOF 同步策略**：**always**（每次写都刷盘）、**everysec**（每秒）、**no**（由操作系统决定）。  
**AOF 重写**：压缩文件，减小占用；Redis 7.0 引入 Multi-Part AOF 优化。

**口诀**：RDB 快照恢复快易丢，AOF 日志丢得少恢复慢。

## 四、Redis 集群（★★★★★）

### 4.1 三种模式概览

| 模式 | 作用 | 优点 | 缺点 | 适用 |
|------|------|------|------|------|
| **主从复制** | 主节点数据复制到从节点；主写从读 | 配置简单、数据冗余、读写分离 | **不能自动故障转移**，受单机内存限制 | 备份、读写分离、在线升级 |
| **哨兵（Sentinel）** | 在主从基础上加哨兵，**监控+自动故障转移** | 高可用，具备主从优点 | 配置管理较复杂 | 高可用要求高 |
| **Cluster** | **数据分片（slot）+ 故障转移**，多主多从 | 水平扩展、高可用、突破单机容量 | 架构与运维更复杂 | 大数据量、高并发、需分片 |

---

### 4.2 主从复制流程

**全量同步**（从库首次或需全量时）：  
① **握手**：从库发 `PSYNC -1`，主库回 `FULLRESYNC <runid> <offset>`（runid=主库身份，offset=复制进度）；  
② 主库 **BGSAVE** 生成 RDB，期间新写命令进 **replication buffer**；  
③ 主库把 **RDB** 发给从库，从库清空旧数据、加载 RDB；  
④ **补齐**：主库把 replication buffer 中的命令发给从库，从库执行后与主库一致；  
⑤ **常态复制**：后续写命令通过长连接实时同步。

**增量同步（部分重同步）**（断线重连后）：  
① 从库带 `runid + offset` 发 `PSYNC <runid> <offset>`；  
② 主库校验 runid 是否一致、缺口数据是否还在 **repl_backlog_buffer**；在则返回 CONTINUE，执行部分重同步；不在则全量；  
③ 主库从 backlog 中取“>offset”的日志发给从库执行；  
④ 进入常态复制。

---

### 4.3 哨兵（Sentinel）

- **作用**：**监控**主从健康；主节点下线时**自动故障转移**（选新主、通知从库与客户端）；**通知**管理员或其它服务。
- **建议**：至少 **3 个**哨兵节点，奇数便于投票。

**故障转移流程**：  
① **选 Sentinel Leader**：主节点客观下线的哨兵发起选举，得**过半数**票者成为 Leader；  
② **选新 Master**：从存活从节点中按 **slave-priority → 复制 offset → 实例 ID** 选数据最完整者为新主；  
③ **重建主从**：其它从节点复制新主；通过**发布/订阅**把新主地址通知客户端；  
④ **旧主恢复**：旧主上线后执行 `slaveof` 降级为新主的从节点。

---

### 4.4 Cluster 模式

- **目的**：解决单机内存与性能上限、主从+哨兵无法**水平分片**的问题；通过**数据分片（slot）+ 自动故障转移**实现扩展与高可用。
- **槽位**：键空间划分为 **16384** 个槽；每个主节点负责一部分槽；键定位：**CRC16(key) % 16384**；扩容缩容时迁移对应槽数据。
- **结构**：多个 Master（各负责一部分槽），每个 Master 可有多个 Slave；主库故障时从库可被提升为主库。

---

### 4.5 哨兵 vs Cluster

| 对比 | 哨兵 | Cluster |
|------|------|--------|
| **数据分片** | **不分片**，所有数据在一个主库 | **分片**，数据分布到多主 |
| **主要作用** | 监控 + **主从故障转移** | **分片 + 故障转移** |
| **选型** | 单 Redis 高可用、读写分离 | **横向扩展、大数据量、高并发** |

**口诀**：要分片高并发选 Cluster，只要高可用读写分离用主从+哨兵。

## 五、常见生产问题（★★★★★）

| 问题 | 原因 | 解决方案（简） |
|------|------|----------------|
| **缓存穿透** | 请求的 key 在**缓存和库都不存在**，请求直接打到库 | **校验**非法参数；**缓存空 key**（短过期）；**布隆过滤器**先判存在再查 |
| **缓存击穿** | **热点 key 过期瞬间**大量请求打到库 | 热点**永不过期**或长过期；**互斥锁**（只有一个请求查库）；**预热**热点数据 |
| **缓存雪崩** | **大量 key 同时失效**，请求压垮库 | **集群**避免单点；**限流**；**多级缓存**；**错峰过期**；**预热** |

**口诀**：穿透→查/记/滤；击穿→固/锁/预；雪崩→集/限/多/预。

**布隆过滤器**：多位数组 + k 个哈希函数；添加时把对应位标 1；判断时若有一位为 0 则**一定不存在**，全为 1 则**可能存在**（有假阳性，无假阴性）。

## 六、缓存与数据库一致性（★★★★★）

**旁路模式**：读先查缓存，没有则查库并回写缓存；**写先更新库，再删除缓存**，下次读时从库加载到缓存。

**主从延迟导致读旧缓存**：写库后删缓存前，其它请求可能读到旧缓存。常用做法：

| 方式 | 思路 | 一致性 | 备注 |
|------|------|--------|------|
| **延迟双删** | 删缓存 → 更新库 → **延迟后再删一次缓存** | 最终一致 | 延迟时间需根据业务权衡 |
| **先更新库再删缓存** | 更新库 → 删缓存 | 最终一致，概率上偶发不一致 | **最常用、最简单** |
| **分布式锁** | 写时加写锁（更新库+删缓存），读时加读锁，串行化 | **强一致** | 性能差 |
| **MQ 串行化** | 同一 key 的写请求进同一队列，顺序消费更新库与缓存 | 最终一致 | 无乱序覆盖 |
| **订阅 binlog** | 库更新后通过 **Canal** 等把 binlog 发 MQ，消费后更新缓存 | 最终一致 | 顺序消费保证最终一致 |

## 七、过期删除与内存淘汰（★★★★★）

**过期删除策略**：  
- **惰性删除**：**取 key 时**才检查是否过期；对 CPU 友好，可能堆积大量过期 key。  
- **定期删除**：**周期性**抽一批 key 删过期；通过限制执行时长和频率降低对 CPU 影响。

**内存淘汰策略**（达 **maxmemory** 时）：

| 策略 | 描述 | 适用 |
|------|------|------|
| **noeviction**（默认） | 内存满时**拒绝写入** | 要数据完整、对可用性要求不高 |
| **volatile-lru** | 在**设了过期**的 key 中淘汰**最近最少使用** | 缓存，优先淘汰不常用 |
| **allkeys-lru** | **全部** key 中淘汰最近最少使用 | **通用缓存** |
| **volatile-random** | 设了过期的 key 中**随机**淘汰 | 对顺序无要求 |
| **allkeys-random** | 全部 key 中**随机**淘汰 | 均匀淘汰 |
| **volatile-ttl** | 设了过期的 key 中淘汰**剩余 TTL 最小** | 优先淘汰即将过期 |

## 八、大 Key 与热 Key（★★★★★）

### 8.1 大 Key

- **定义**：Key 对应的 **Value 大**（如 String 5MB、Zset 成员上万、Hash 成员 value 总和大）。
- **危害**：命令变慢、易触发 maxmemory 淘汰或阻塞、集群下某分片内存倾斜、删除易阻塞引发主从中断。
- **处理**：**拆分**大 Key（如大 Hash 拆成多个小 Key）；**清理**不适于 Redis 的数据；**定期清理**过期数据防堆积。

### 8.2 热 Key

- **定义**：**QPS/带宽/CPU** 集中在一个或少量 key（如单 key QPS 远高于其它）。
- **危害**：占用大量 CPU、集群下某分片被打满、易引发击穿。
- **处理**：**多级缓存**（本地缓存分担）、**热 key 备份**（多节点备份分散读）、**热 key 拆分**（带后缀分散到多实例）、**业务隔离**（热点业务独立集群）。

## 九、分布式锁（★★★★★）

- **目标**：分布式环境下，保证同一时刻**只有一个进程**能访问共享资源。
- **三种实现**：

| 实现 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **数据库表** | 锁表插入一条记录，释放删记录 | 简单 | 依赖库可用性、无失效时间、非阻塞、非重入 |
| **Redisson** | 基于 Redis，支持单机/主从/Sentinel/Cluster；**看门狗**定时续期防提前过期；**可重入**（hset 记录加锁次数） | 性能好、可重入、续期 | 非阻塞需自旋 |
| **Zookeeper** | 在 /lock 下建**临时有序**子节点，序号最小者获锁；释放删节点；监听前序节点 | 无单点、可重入、阻塞等待、锁可释放 | 高并发下性能不如 Redis（ZAB 协议） |

## 十、Redis 事务（★★★★★）

- **用法**：**MULTI** 开启事务 → 命令入队 → **EXEC** 顺序执行；**DISCARD** 放弃事务；**WATCH** 监视 key，被改则事务失败。
- **特点**：**不保证原子性**—EXEC 后某条失败**不回滚**，后续仍执行；**持久性**依赖 RDB/AOF 配置。适合**批量命令串行执行**，而非严格 ACID 事务。

## 十二、速记清单（考前扫一眼）

1. **类型**：String/Hash/List/Set/Zset/Bitmap/HyperLogLog/Geo；场景对应表；String 省内存整对象，Hash 可改部分字段。
2. **持久化**：RDB 快照恢复快易丢，AOF 日志丢得少恢复慢；AOF 策略 always/everysec/no。
3. **集群**：主从（读写分离无自动故障转移）、哨兵（监控+故障转移）、Cluster（分片+故障转移）；槽 16384，CRC16(key)%16384。
4. **主从**：全量（握手→RDB→补齐→常态）、增量（PSYNC runid offset，backlog 补齐）。
5. **哨兵**：监控、故障转移、通知；选 Leader→选新主（priority/offset/id）→重建主从→旧主降级。
6. **穿透/击穿/雪崩**：原因+方案表；口诀查记滤、固锁预、集限多预；布隆过滤器有假阳性无假阴性。
7. **一致性**：旁路模式；先更新库再删缓存；延迟双删、分布式锁、MQ、binlog 订阅。
8. **过期**：惰性删除、定期删除；**淘汰**：noeviction、volatile-lru、allkeys-lru 等。
9. **大 Key**：Value 大；拆分、清理、定期清过期；**热 Key**：多级缓存、备份、拆分、隔离。
10. **分布式锁**：数据库表/Redisson/Zookeeper；Redisson 续期、可重入；ZK 临时有序节点。
11. **事务**：MULTI/EXEC/DISCARD/WATCH；不保证原子性与持久性。
12. **指令**：SET/GET、ZADD/ZRANGE、HSET/HGET、LPUSH/LRANGE、SADD、MULTI/EXEC、BGSAVE 等。
