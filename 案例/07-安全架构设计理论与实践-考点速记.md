## 二、安全架构三性（★★★☆☆）

| 特性 | 定义 | 典型例子 |
|------|------|----------|
| **可用性** | 确保系统资源**在需要时可正常访问**，防止故障或攻击导致服务中断 | 数据备份与恢复、负载均衡、双机热备 |
| **完整性** | 保护数据**不被未经授权修改或破坏**，确保数据一致性和准确性 | 文件哈希校验（MD5/SHA）、**数字签名**、数据库事务回滚 |
| **机密性** | **防止敏感信息在未授权情况下被泄露或获取** | 数据加密存储、HTTPS、访问控制列表（ACL） |

**辨析**：**完整性** = 防篡改、保一致；**机密性** = 防泄露、保密。二者不同，答题时分别写清概念即可。

**案例（航空运营智能管理平台）**：

- **可用性**：**99.99%**、多副本、限流降级、备份恢复（与 **票务高峰、航变洪峰** 相关）。
- **完整性**：**对账、数字签名回调、订单哈希校验**；防 **错账与报文篡改**。
- **机密性**：**身份证/银行卡加密存储**、**TLS** 传输、**脱敏** 展示与画像分析。

## 三、数据加密技术（★★★★★）

### 3.1 对称加密

- **原理**：**同一密钥**既加密又解密；发送方用密钥把明文变密文，接收方用**同一密钥**还原。
- **优点**：加解密**速度快、效率高**，适合**大量数据**。
- **典型算法**：AES、DES、3DES。
- **数学基础**：替代/置换等运算。

---

### 3.2 非对称加密

- **原理**：**一对密钥**——**公钥**（可公开，用于加密）、**私钥**（用户自持，用于解密）。发送方用**接收方公钥**加密，接收方用**自己私钥**解密。
- **优点**：**密钥管理方便**，无需在通信双方间传递私钥；安全性基于**数学难题**（如 RSA 大整数分解）。
- **典型算法**：RSA、ECC、DH。
- **用途**：数字签名、密钥交换、身份验证等。

---

### 3.3 非对称的两种应用场景

| 场景 | 谁加密 / 谁解密 | 传递内容 | 典型用途 |
|------|-----------------|----------|----------|
| **数字签名（信源验证）** | **私钥加密，公钥解密** | 明文 + 签名（对摘要签名） | 验证来源与完整性，防抵赖 |
| **数据加密** | **公钥加密，私钥解密** | 密文 | 只有拥有私钥的接收方能解密；也可用于**交换对称密钥** |

**记忆**：签名 = 私钥签、公钥验；加密 = 公钥加密、私钥解密。

**密钥交换**：非对称加密慢，适合小数据；实际大量数据用对称加密。双方可用**公钥加密“对称密钥”**传给对方，对方用私钥解密得到密钥，后续用对称密钥通信。

**案例（航空运营智能管理平台）**：**HTTPS** 即典型 **TLS 握手协商对称会话密钥**；与素材 **SSL/TLS 全程加密** 一致。

---

### 3.4 对称 vs 非对称对比

| 对比项 | 对称加密 | 非对称加密 |
|--------|----------|------------|
| **密钥** | **单一密钥** | **密钥对**（公钥 + 私钥） |
| **速度** | **快**，适合大数据 | **慢**，适合小数据 |
| **密钥管理** | 需**安全共享**密钥 | 公钥公开，私钥保密 |
| **主要用途** | 数据加密 | 密钥交换、数字签名、身份验证 |
| **数学基础** | 替代/置换 | 数论难题（分解、离散对数等） |
| **典型算法** | AES、DES、3DES | RSA、ECC、DH |

**口诀**：对称一钥快大数据，非对称一对慢小数据、管钥方便。

**案例（航空运营智能管理平台）**：**库内敏感列** 用 **AES/SM4** 等 **对称加密**；**握手阶段** 用 **RSA/ECC/SM2** 交换 **会话密钥**；**国密合规** 场景优先 **SM 系列**。

## 四、认证技术（★★★★★）

### 4.1 Hash（杂凑）算法

- **特性**：**易压缩**（任意长→固定长）、**单向**（不能由哈希反推原文）、**高灵敏**（输入微变输出大变）、**抗碰撞**（不同输入极难同哈希）。
- **用途**：数据**完整性校验**，是数字签名的重要组成部分。
- **常用算法**：SHA-1（已不推荐）、SHA-256、SHA-384、国产 **SM3**。

---

### 4.2 数字签名原理

**创建签名**：  
1. 对文件做 **Hash** 得到**消息摘要**。  
2. 用**签名者私钥**对摘要加密，得到**数字签名**。  
3. 可加时间戳。  
4. 发送：**原文 + 数字签名**（注意：只对摘要签名，不对整个文件，省资源、提效率）。

**验证签名**：  
1. 接收方对**原文**再做 Hash，得**本地摘要**。  
2. 用**签名者公钥**解密数字签名，得**发送时摘要**。  
3. **两摘要一致** → 文件完整且签名有效；不一致 → 被篡改或签名无效。

**示意**：
```
  发送方：  原文 → Hash → 摘要 → 私钥加密 → 签名  ； 发送 [原文 + 签名]
  接收方：  原文 → Hash → 摘要A ； 签名 → 公钥解密 → 摘要B ； 比较 摘要A 与 摘要B
```

---

### 4.3 数字证书

- **问题**：公钥如何保证属于真正的发送方？若公钥被篡改，验证会失效。
- **解决**：**数字证书**——由 **CA（证书颁发机构）** 签发，对**公钥及其持有者信息**做认证；CA 用**自己的私钥**对“公钥+信息”签名，形成证书。
- **验证**：接收方用 **CA 的公钥**解密证书，验证证书有效性，即可信任其中的公钥。
- **标准**：X.509；CA 维护**证书吊销列表（CRL）**；不同 CA 间通过**证书链**建立信任。
- **典型流程**：客户端请求 → 服务器返回**数字证书** → 客户端用 CA 公钥验证证书 → 用服务器公钥加密后续数据（或交换对称密钥）。

**案例（航空运营智能管理平台）**：**支付/出票回调** 可用 **Hash + 签名** 校验报文完整性、防抵赖；**HTTPS** 依赖 **X.509 证书** 建立对 **购票与 API 网关** 的信任链；国密场景可写 **SM3/SM2** 与 **TLS 国密套件**。

---

### 4.4 身份认证方式简表

| 方式 | 特点 |
|------|------|
| **口令认证** | 用户名/密码，简单但不安全，易泄露、被猜、被截获 |
| **动态口令** | 按时间或次数变化，一次一密，较安全；可能同步问题、输入不便 |
| **生物特征** | 身体/行为特征，分身体特征、行为特征及不同层级 |

**案例（航空运营智能管理平台）**：**RBAC + 权限申请审批** 属于强 **访问控制**；**异地登录、暴力破解告警** 强化 **鉴别**；与 **民航局身份核验** 结合，体现 **多因子/权威源** 思路（答题可类比动态口令、证书认证）。

---

### 4.5 典型例题：公钥体系 vs 口令认证

**题干**：用户身份鉴别与资源授权、网络与存储数据加密、敏感数据按权限访问、单文件可达数百兆、终端用户私有数据、约 100 用户局域网。

**问**：为何采用基于公钥体系的认证而非口令认证？

**答要点**：  
（1）**安全性**：口令易遭猜测、暴力破解；公钥体系私钥不在网络传输，可防认证攻击，更安全。  
（2）**私有数据保护**：口令为用户与服务器共享，无用户独有秘密；公钥体系中可用**用户私钥**对私有数据加密，实现更简便。  
（3）**效率**：公钥计算复杂度高于口令，但用户规模 100 人、局域网环境下可满足效率要求。

**案例（航空运营智能管理平台）**：平台用户规模大、含 **票务与敏感数据**，更宜 **证书/TLS、密钥体系 + RBAC** 组合，而非仅靠口令；与素材 **加密存储、全程 TLS、操作审计** 叙事一致。

## 五、网络安全体系（★★★☆☆）

### 5.1 安全服务与安全机制

| 安全服务 | 对应安全机制 | 作用 |
|----------|----------------|------|
| **鉴别服务** | 认证机制、数字签名机制 | 验证实体身份真实性，防身份伪造 |
| **访问控制服务** | 访问控制机制、路由控制机制 | 管理访问权限，仅授权主体可访问资源 |
| **数据机密服务** | 加密机制、业务流填充机制 | 保护保密性；业务流填充可防流量分析 |
| **数据完整性服务** | 数据完整性机制 | 防篡改、删除、插入，保证准确一致 |
| **抗否认性服务** | 公证机制 | 记录与验证操作，防抵赖，可追溯 |

**关系**：安全服务 = “要实现什么”；安全机制 = “用什么技术实现”。

**案例（航空运营智能管理平台）**：

| 安全服务 | 素材落点 |
|----------|----------|
| **鉴别** | 登录认证、**民航局核验**、异常登录告警 |
| **访问控制** | **RBAC**、角色自定义与审批 |
| **数据机密** | **TLS**、敏感字段加密、**脱敏** |
| **数据完整性** | 对账、回调验签、事务一致性 |
| **抗否认** | **关键操作日志≥1 年**、票务与权限变更可追溯 |

---

### 5.2 典型架构的脆弱性（★★★☆☆）

| 架构 | 脆弱性梳理 |
|------|------------|
| **分层架构** | 层间依赖强，单层故障易波及整体；层间通信易成性能瓶颈 |
| **C/S 架构** | 客户端易藏漏洞；网络开放、易受攻击；数据易遭破坏 |
| **B/S 架构** | 易受病毒、恶意代码攻击，安全性较弱 |
| **事件驱动架构** | 组件控制力不足；数据交互逻辑复杂；易死循环、高并发难题；固定流程存在安全漏洞 |
| **MVC 架构** | 设计复杂；视图与控制器耦合高，影响复用；视图读模型效率低 |
| **微内核结构** | 整体优化难；进程间通信开销大，效率下降 |
| **微服务架构** | 开发逻辑复杂；需额外通信与容错设计；管理维护复杂度高 |

**考法**：给出某种架构，要求分析脆弱性或安全风险，可从表中对应条目展开。

**案例（航空运营智能管理平台）**：本平台为 **云原生微服务 + B/S/多终端**，需写清：**服务间通信与入口暴露** 带来的攻击面、**事件驱动/异步** 下的消息伪造与重放、**高并发** 下的 **DDoS/滥用**；对策对应素材 **WAF、限流、漏洞扫描、SQL 注入/XSS 防护** 及 **弹性扩容**。

## 六、速记清单（考前扫一眼）

1. **安全三性**：可用性（可访问）、完整性（防篡改）、机密性（防泄露）。
2. **对称**：一钥、快、大数据、AES/DES；**非对称**：公钥+私钥、慢、小数据、密钥交换与签名、RSA/ECC。
3. **非对称场景**：私钥加密公钥解密 = 数字签名（信源验证）；公钥加密私钥解密 = 加密/密钥交换。
4. **Hash**：易压缩、单向、高灵敏、抗碰撞；用于完整性校验与数字签名。
5. **数字签名**：原文→Hash→摘要→私钥加密→签名；验证：原文 Hash 与“公钥解密签名”得到的摘要一致。
6. **数字证书**：CA 签发，证明公钥归属；用 CA 公钥验证证书；X.509、证书链、CRL。
7. **公钥体系 vs 口令**：更安全（私钥不传）、便于保护用户私有数据、规模不大时效率可接受。
8. **安全服务**：鉴别、访问控制、数据机密、数据完整性、抗否认；各对应若干安全机制。
9. **架构脆弱性**：分层依赖与瓶颈、C/S 客户端与网络、B/S 病毒与恶意代码、事件驱动控制与死循环、MVC 耦合与效率、微内核 IPC 开销、微服务复杂与运维。
