## 二、CAP 理论（★★★★★）

### 2.1 三性定义

**结论**：分布式系统**最多同时满足两个**，无法三者兼得；实践中**分区容错必然存在**，只能在**一致性 C** 与**可用性 A** 之间权衡。

| 概念 | 定义 |
|------|------|
| **一致性（Consistency）** | 写请求后**各相关节点数据均变化**，读请求能读到**变化后的数据**；多副本一致 |
| **可用性（Availability）** | 节点**收到请求就要处理并返回结果**，即使数据可能有问题（如返回旧数据）也不能拒绝 |
| **分区容错性（Partition Tolerance）** | 节点间**通信出现故障形成分区**时，系统仍能继续运行，**各分区可独立对外服务** |

**一致性**：一次写后，所有节点都更新，读能拿到新数据 → 称“发生了一致性变化”。  
**可用性**：能收请求就必须响应，哪怕返回的是过期数据（如三天前的数据）。  
**分区**：网络不可靠，节点间通信出问题即“分区”；**分区容错** = 出现分区后系统仍运行、分区可独立服务。  
**分区容错 vs 可用性**：分区容错强调“**通信故障导致分区**后各分区独立服务”；可用性强调“**非通信故障下**系统整体对外服务”。

---

### 2.2 CAP 组合与系统实现

| 组合 | 含义 | 典型例子 |
|------|------|----------|
| **CP**（一致性+分区容错） | 分区期间**宁可不可用**，也要保证数据一致 | **Zookeeper**：写经 Leader、多数派确认；Leader 失联或不足多数则**暂停服务**，直到重新选主 |
| **AP**（可用性+分区容错） | 分区期间**允许数据不一致**，保证继续可用 | **Eureka**：心跳丢失时**自我保护**，不剔除实例、不删注册信息，仍可查询，保持可用 |
| **CA**（一致性+可用性） | **分布式环境下无法实现**（无法避免分区） | — |

**口诀**：分区必选 P；选 C 则牺牲 A（CP），选 A 则牺牲 C（AP）；CA 在分布式下不存在。

**示意**：
```
  分区发生时：
  CP → 暂停部分服务，保证一致（如 ZK 停写）
  AP → 继续服务，允许各分区数据不一致（如 Eureka 保护）
```

## 三、BASE 理论（★★★★★）

- **来源**：eBay 架构师 Dan Pritchett，2008 年 ACM；基于 CAP 的**一致性与可用性权衡**，牺牲强一致换可用性，允许**一段时间不一致**，**最终一致**。
- **思路**：通过业务设计，牺牲强一致性获得可用性，允许中间状态，最终达到一致。

| 概念 | 定义 |
|------|------|
| **基本可用（Basically Available）** | 遇不可预知故障时，**允许损失部分可用性**（如响应变慢、功能降级） |
| **软状态（Soft State）** | 允许系统存在**中间状态**，且该状态不影响整体可用性；**不同节点副本同步可有延迟** |
| **最终一致性（Eventually Consistent）** | 所有数据副本**经一段时间同步后**，**最终达到一致**，不要求实时强一致 |

**口诀**：Base 可用、软状态、最终一致。

## 四、分布式事务（★★★★★）

### 4.1 什么是分布式事务

- **定义**：事务的参与者、资源、事务管理器等分布在**不同节点**；一次业务涉及**多个服务/多库**，要么**全成功**，要么**全失败**。
- **典型场景**：下单 = 扣库存 + 扣优惠 + 生成订单，跨订单服务、库存服务、优惠服务等；需保证这些操作的整体一致性。
- **本质**：保证**不同数据库/服务**间的**数据一致性**。

---

### 4.2 刚性事务：2PC（二阶段提交）（★★★★★）

**两阶段**：**投票阶段** → **提交/回滚阶段**。

| 阶段 | 内容 |
|------|------|
| **阶段一（投票/预处理）** | ① 协调者向所有参与者发送事务内容，**询问**能否提交；② 各参与者**执行事务**，写 Undo/Redo 日志；③ 参与者向协调者反馈 **Yes/No** |
| **阶段二（提交/执行）** | 协调者根据反馈决定：**全 Yes** → 发 **Commit**，参与者提交并释放资源，回 ACK，协调者完成事务；**有 No 或超时** → 发 **Rollback**，参与者用 Undo 回滚并释放资源，回 ACK，协调者中断事务 |

**示意**：
```
  协调者          参与者1    参与者2
     | 询问         |          |
     |------------->| 执行+日志 |
     |------------->| 执行+日志 |
     |<--Yes--------|          |
     |<--Yes-------------------|
     | Commit/Rollback         |
     |------------->|          |
     |------------->|          |
     |<--ACK--------|          |
     |<--ACK-------------------|
```

**2PC 优缺点**：

| 优点 | 缺点 |
|------|------|
| 实现原理简单 | **性能**：两阶段内参与者**阻塞**，占资源 |
| | **单点故障**：协调者挂了，参与者无法释放资源或完成事务 |
| | **数据不一致**：发 Commit 后若网络异常或协调者崩溃，可能只有部分参与者提交，导致不一致 |
| | **保守**：依赖超时判断，容错差，难适应动态网络 |

---

### 4.3 柔性事务：TCC（Try-Confirm-Cancel）（★★★★★）

- **来源**：Pat Helland 2007 年论文《Life beyond Distributed Transactions: an Apostate's Opinion》。
- **与 2PC 对比**：① **无单点**，由主业务方发起，业务活动管理器可**集群**；② **同步阻塞**减轻，引入超时与补偿，资源以业务逻辑形式预留，**粒度更小**；③ **补偿机制**由业务活动管理器控制一致性。

| 阶段 | 含义 | 要点 |
|------|------|------|
| **Try** | **尝试执行** | 完成**业务检查**（一致性），**预留**必需业务资源（准隔离）；如库存检查+预留“可用库存” |
| **Confirm** | **确认执行** | **真正执行业务**，不再做检查，只使用 Try 预留的资源；**幂等**，失败需重试 |
| **Cancel** | **取消执行** | **释放** Try 阶段预留的资源；**幂等**；异常处理同 Confirm |

**示例**：订单+库存。Try：检查库存是否够、预留可用库存（如扣减“可用数量”）；Confirm：真正扣减库存、生成订单；Cancel：释放预留库存。

**缺点**：需把原一个接口拆成 **Try、Confirm、Cancel 三个接口**，**开发复杂度高**。

**示意**：
```
  业务  → Try（检查+预留）→ Confirm（确认） 或  Cancel（释放）
```

---

### 4.4 柔性事务：本地消息表（★★★★★）

- **来源**：eBay Dan Pritchett，2008 年 ACM；通过**消息**异步确保分布式任务执行，实现**最终一致性**。
- **核心**：**本地库中消息表**记录发送状态 + **MQ** 异步投递；业务数据与消息表**同一库、同一本地事务**写入。

**发送方**：  
① 本地事务内：写**业务数据** + 写**消息表**；② 投递消息到 **MQ**；③ 若消费方 **ACK**，可删除消息表记录；④ 未 ACK 则**重试**；消费方需**幂等**防重复处理。

**消费方**：  
① 消费 MQ 消息，执行业务逻辑；② 本地事务成功即处理成功；③ 失败则重试；④ 业务失败可向生产方发**补偿消息**（回滚等）。

**优缺点**：

| 优点 | 缺点 |
|------|------|
| 实现简单，开发成本低 | 与业务**高耦合**，占业务库资源，量大可能影响库性能 |
| | 消息**异步**，存在**延迟**；需处理重复消息（幂等） |

**示意**：
```
  发送方： [业务表+消息表] 同库同事务 → 写MQ → 消费方ACK后删消息
  消费方： 消费MQ → 本地事务执行业务 → 成功/重试/补偿
```

## 五、速记清单（考前扫一眼）

1. **CAP**：C 一致、A 可用、P 分区容错；**最多满足两个**；分区必然存在，实际在 C 与 A 间选。
2. **CP**：分区时牺牲可用（如 Zookeeper 停服务保一致）；**AP**：分区时牺牲一致（如 Eureka 保护机制保可用）；**CA** 分布式下不可实现。
3. **BASE**：基本可用、软状态、最终一致；牺牲强一致换可用，最终一致。
4. **分布式事务**：多节点多库/多服务，要么全成功要么全失败；保证跨库/跨服务数据一致。
5. **2PC**：两阶段——投票（询问+执行+Yes/No）→ 提交或回滚；协调者单点、阻塞、可能不一致、依赖超时。
6. **TCC**：Try（检查+预留）→ Confirm（确认，幂等）或 Cancel（释放，幂等）；无单点、粒度小、有补偿；开发复杂（三接口）。
7. **本地消息表**：业务表+消息表同库同事务 → MQ → 消费方处理；最终一致；简单但耦合、有延迟、需幂等。
