## 二、结构化分析

### 2.1 数据流图 DFD（★★★★★）

#### 2.1.1 基本符号（必背）

两种常见记号：**Gane & Sarson** / **Yourdon & Coad**，名称不同、含义一致。

| 符号名称 | 表示内容 | 图形表示 |
|----------|----------|----------|
| **数据流** | 有名字和方向的数据 | 标有名字的**箭头** |
| **加工（数据处理）** | 对数据流的变换 | **圆圈** |
| **数据存储** | 可访问的存储信息 | **直线段**（双线或开口矩形） |
| **外部实体** | 系统外的数据来源/去向 | 标有名字的**方框** |

**记忆口诀**：流箭加圆存直线，外部方框画外边。

**符号示意（Gane & Sarson / Yourdon 两种画法等价）**：
```
数据流：   --------→   （带名字的箭头）
加工：      (  P  )    （圆圈，内写加工名）
数据存储：  ═══════   或  ┌─────┐  （直线段/开口矩形）
外部实体：  ┌─────┐    （方框，如 管理员、外部系统）
```
**重要规则**：数据存储要被外部实体使用，**必须经过加工**。

**案例（航空运营智能管理平台）**：外部实体可画 **旅客、支付渠道、民航局身份核验系统、承运人/机场接口**；加工如 **售票计算、库存扣减、运力校验、消息推送、流式告警**；数据存储如 **订单库、库存表、航班动态缓存、分析主题库**；数据流如 **支付结果、核验令牌、出票回执、设备遥测**。旅客 **不得直连** 订单表，须经过 **购票/同步** 等加工。

---

#### 2.1.2 平衡原则（★★★★★ 必考）

**（1）父图与子图平衡（分层细化）**

- **个数一致**：父子两层数据流个数一致  
- **方向一致**：数据流方向一致  

**（2）子图内部平衡——三种错误**

| 类型 | 含义 | 记忆 |
|------|------|------|
| **黑洞** | 加工只有输入没有输出 | 只进不出 |
| **奇迹** | 加工只有输出没有输入 | 只出不进 |
| **灰洞** | 输入不足以产生应有输出（挂羊头卖狗肉） | 该出 A 却出 B |

**数据存储**：通常既有**读**又有**写**；某张子图可只有读或只有写。

**简答模板**：分层数据流图平衡 = ① 父子图数据流个数、方向一致；② 每张图内加工输入/输出平衡，避免黑洞、奇迹、灰洞。

**案例（航空运营智能管理平台）**：将 **“在线购票”** 逐层展开时，父图进入加工的 **订单请求、支付结果、航班上下文** 等，在子图边界须 **个数与方向一致**。若某加工仅有 **支付成功** 而无 **出票失败/回滚** 输出 → **黑洞**；无输入却输出 **行程单** → **奇迹**；输入仅有旅客 ID 却输出完整 **票价税费客票** 而无 **计价与核验** → **灰洞**。

---

#### 2.1.3 典型题思路（选词填空）

- 看**数据流向**：流向某表/某库 → 对应“录入/写入”；从某表输出 → 对应“分析/读取”。
- 看**角色**：安全员→填报，安全副经理→审核，项目经理→确认。
- 答案示例（2022 架构）：(1)f (2)g (3)h (4)d (5)b (6)e；并简述分层平衡原则（如上）。

**案例（航空运营智能管理平台）**：填空时看数据走向——**写入订单/库存** 对应 **录入、同步、更新**；**读取航班/报表** 对应 **查询、分析**；角色可对应 **运营值班、客服主管、信息中心管理员** 等审批链（按题干）。

### 2.2 状态转换图 / 状态机图（★★★☆☆）

- 即 UML 中的**状态图**，描述**状态 + 事件 → 状态变化**。
- **主要概念**：状态、初态、终态；事件、转移、动作；并发状态机。

#### 基本符号

| 元素 | 表示 |
|------|------|
| **初态** | 黑圆点 |
| **终态** | 黑圆点 + 外圈圆 |
| **中间状态** | 圆角矩形 |
| **状态转换** | 带箭头的线（可标触发事件） |

**符号示意**：
```
初态 ●  →  [ 状态A ] --事件--> [ 状态B ]  →  ◎ 终态
```
**典型题**：给 6 个状态、6 个事件 T1–T6，把事件填到转换线上。技巧：按题干“当…则进入…”逐条对应（如：按下启动→增加动力；开始程序→运行；停止程序→已挂起；恢复程序→运行；结束程序→终止；处理程序已结束→手动；改变程序→手动）。

**案例（航空运营智能管理平台）**：**电子客票** 可建模 **待支付→已锁座→已出票→已改签/已退票→已作废**；**检修工单** 可建模 **计划→执行中→待验收→关闭**；**航变处置** 可建模 **正常→延误→取消→批量退改中**。事件如 **支付成功、核验失败、航变触发、退款完成**。

## 三、面向对象分析 · UML（★★★★★）

### 3.1 UML 常考图（记这 6 类）

| 名称 | 描述 |
|------|------|
| **类图** | 类、接口、协作及关系 |
| **用例图** | 用例、参与者及关系 |
| **顺序图** | 对象间消息，**强调时间顺序** |
| **通信图/协作图** | 对象间消息，**强调结构协作**（UML1 叫协作图） |
| **活动图** | 控制流/数据流，顺序、分支、并行 |
| **状态机图** | 对象状态转移 |

其他：对象图、包图、组合结构图、构件图、部署图、时间图、交互概览图、外廓图——了解即可。

**案例（航空运营智能管理平台）**：**类图/部署图** 对应服务划分与云上部署；**用例图** 定义对外能力；**顺序图** 拆 **购票与支付回调** 时序；**活动图** 画 **航变批量退改**；**状态图** 画 **客票/工单**；**通信图** 强调 **票务—支付—核验—消息** 结构协作。

---

### 3.2 用例图（★★★★★）

#### 基本步骤（顺序背）

1. **识别参与者**  
2. **合并需求获得用例**  
3. **细化用例描述**  
4. **调整用例模型**  

#### 参与者类型

| 类型 | 说明 | 例子 |
|------|------|------|
| **人** | 使用系统的人 | 患者、医生、管理员 |
| **其他系统** | 与本系统交互的系统 | 企业 OA 系统 |
| **硬件设备** | 需交互的设备 | IC 卡读写器 |
| **时钟** | 定时触发 | 定时交卷 |

**注意**：参与者一定在**系统之外**，不是系统一部分。

#### 用例关系（必背：箭头指向 + 构造型）

| 关系 | 含义 | 构造型 | 箭头指向 |
|------|------|--------|----------|
| **包含** | 提取公共行为，原始用例→基本用例，提取出的→抽象用例 | `<<include>>` | 指向**抽象用例** |
| **扩展** | 基本用例 + 多种扩展场景，扩展用例可选 | `<<extend>>` | 指向**基本用例** |
| **泛化** | 父子用例，子继承父 | — | 指向**父用例** |

**口诀**：包含指抽象，扩展指基本，泛化指父用例。

**关系示意**：
- **包含**：基本用例 ----`<<include>>`----→ 抽象用例
- **扩展**：扩展用例 ----`<<extend>>`----→ 基本用例
- **泛化**：子用例 ----泛化----→ 父用例

**例题思路**：先按参与者分类（患者侧 / 医院侧），再按题干功能列表往空里填；账号管理可能归患者也可能归医院，视题目表述而定。

**案例（航空运营智能管理平台）**：

- **参与者**：**旅客、客服、调度、检修工程师、系统管理员**；外部系统 **支付网关、民航局核验、消息推送**；**时钟** 触发 **T+1 批处理、检修计划滚动**。
- **用例**：**在线购票、改签退票、航班查询、实时看板、超售告警、设备故障预警、权限申请审批** 等。
- **包含 `<<include>>`**：**在线购票** 包含 **身份核验**、**库存锁扣**。
- **扩展 `<<extend>>`**：**恶劣天气批量退改签** 扩展 **退票**。
- **泛化**：**App/小程序/官网** 泛化为 **售票终端**。

---

### 3.3 类图 · 六大关系（★★★★★）

**强度顺序（口诀）**：**饭（泛）食（实现）组聚关羽（依）**  
即：**泛化 = 实现 > 组合 > 聚合 > 关联 > 依赖**

| 关系 | 定义 | 记忆 |
|------|------|------|
| **关联** | 明确、稳定的结构联系 | 工人—任务 |
| **依赖** | B 变化可能引起 A 变化，A 依赖 B | 司机依赖汽车 |
| **泛化** | 一般与特殊，父类与子类 | 狗、猫 泛化 动物 |
| **聚合** | 整体与部分，部分可属多整体，**好聚好散** | 学生—班级 |
| **组合** | 整体与部分，部分只属一整体，**生命周期同步** | 汽车—发动机 |
| **实现** | 类实现接口 | 音频播放器 实现 播放器接口 |

**聚合 vs 组合**：聚合=部分能单独存在；组合=部分随整体生灭。

**例题**：University–Student 可换系统仍存在→聚合；University–Department 同生同灭→组合；Student–Course 常答关联（或聚合视题目）。

**案例（航空运营智能管理平台）**：**订单** **组合** **订单明细**；**航线** **聚合** **航班**；**订单** 与 **旅客、航班** **关联**；**报表服务** **依赖** **票务查询接口**；**票价策略** **实现** **计价接口**；**国内票** **泛化** **票**。

---

### 3.4 顺序图（★★★★★）

- **用途**：对象间消息的**时间顺序**，用例分析/设计。
- **元素**：生命线、消息（同步/异步）、端点、门、**组合片段**。

#### 组合片段类型（常考）

- **alt**：条件分支  
- **loop**：循环  
- **par**：并行  
- **opt**：可选  

**填空技巧**：按题干“谁请求→谁处理→谁反馈”顺序对应对象与消息名（如：预约人员、预约挂号请求、显示医生可预约时段、显示预约是否成功）。

**案例（航空运营智能管理平台）**：**旅客 → 网关 → 票务服务 → 支付 → 核验 → 库存 → 消息服务 → 旅客**，消息可标 **同步/异步**；**alt** 分支 **支付失败/核验超时**；**loop** 重试；**par** 并行 **写行程与发推送**（若题干允许）。

---

### 3.5 通信图 / 协作图（★★★★★）

- **元素**：对象、**通信链接**、消息。
- **与顺序图区别**：顺序图强调**时序**；协作图强调**对象结构与协作**。语义等价，可互相转换；关注时序用顺序图，关注结构用协作图。

**案例（航空运营智能管理平台）**：在协作图上突出 **票务、支付、核验、通知** 对象之间的 **链接编号与消息序号**，适合回答 **“谁与谁交互”** 而不强调绝对时间轴。

---

### 3.6 活动图（★★★★★）

- **用途**：控制流/数据流，顺序、分支、**并行**；与流程图最大区别是**分叉/合并**和**信号（发送/接收）**。

#### 主要元素

| 元素 | 含义 |
|------|------|
| 初始/终止节点 | 开始 / 结束 |
| 动作状态 | 原子步骤 |
| 决策节点 | 分支条件 |
| 合并节点 | 多分支汇合 |
| 分叉/合并 | 并行控制流 |
| 泳道（分区） | 按角色/部门划分 |
| 对象节点/对象流 | 对象传递 |
| 信号节点 | 发送/接收事件（异步） |

**活动图与流程图区别**：活动图有**分叉/合并**（并行）和**信号发送/接收**（异步）。
```
     [动作A] → ◇决策 → 分支1 → ...
                    → 分支2 → ╋合并 → [动作B]
     ╋分叉 → [并行1] ╋
          → [并行2] ╋ 合并
```

**典型题**：根据“先…再…若…则…”补活动名、参与者、消息（如：地面综合控制计算机、下传起飞就绪信息、垂直起飞、高度保持、发送目标高度、垂直升降、发送任务结束指令、飞机降落、无线电遥控飞行）。

**案例（航空运营智能管理平台）**：**航变场景** 用 **泳道** 分 **旅客、客服、票务、调度、通知**；**分叉/合并** 表示 **并行查询与批量任务**；**决策节点** 区分 **全额退/改期/候补**；与素材 **突发航班变动、协同处置** 一致。

## 四、常见图对比（★★★★★ 简答常考）

| 图类型 | 主要关注点 | 对比结论 |
|--------|------------|----------|
| **用例图** | 参与者与系统交互 | “做什么”，需求视角，不涉及实现 |
| **类图** | 类、属性、方法、关系 | “怎么做”，实现视角；与用例图形成需求→设计 |
| **数据流图** | 数据加工、流动、存储 | 数据在系统中**如何流动** |
| **活动图** | 工作流、顺序/分支/并行 | **任务执行步骤**，行为与流程 |
| **流程图** | 步骤与判断 | 与活动图类似，更偏程序逻辑 |
| **状态图** | 状态与触发条件 | **对象随事件的状态演化**，非任务流 |
| **顺序图** | 消息时间顺序 | **时序**，请求—响应过程 |
| **通信图** | 对象连接与消息 | **结构与协作**；与顺序图内容同、视角不同 |

**一句话**：用例“做啥”、类图“咋做”、DFD“数据怎么流”、活动图“步骤怎么走”、状态图“状态怎么变”、顺序图“按时间谁先谁后”、通信图“谁和谁连”。

**案例（航空运营智能管理平台）**：向阅卷人说明选型——**对外范围** 先用 **用例图**；**数据治理与批处理** 用 **DFD**；**端到端时序** 用 **顺序图**；**跨部门航变** 用 **活动图**；**客票生命周期** 用 **状态图**。

## 五、用例描述结构（了解）

| 部分 | 说明 |
|------|------|
| 用例名称/编号 | 与用例图一致 |
| 简要说明 | 为参与者带来的价值 |
| 事件流 | 为达目标的一系列活动 |
| 非功能需求 | 可度量、可验证 |
| 前置/后置条件 | 执行前必须满足的状态 / 执行后可能的状态 |
| 扩展点 | 扩展/包含用例及使用情况 |
| 优先级 | 开发先后 |

**案例（航空运营智能管理平台）**：以 **在线购票** 为例——**简要说明** 为旅客完成 **选航班、支付、出票、通知**；**事件流** 含 **正常流、支付失败、核验失败、库存不足**；**非功能** 写清 **≤1 秒、5000 TPS、强一致、TLS**；**前置** 已选座未过期，**后置** 订单为已出票；**扩展点** 挂载 **航变退改**；**优先级** 春运前 **购票链** 高于 **次要报表**。

## 六、速记清单（考前扫一眼）

1. **DFD 四符号**：流箭、加工圆、存直线、外部框。  
2. **平衡**：父子个数方向一致；无黑洞、奇迹、灰洞。  
3. **状态图**：初态黑点、终态圈、状态圆角矩形、转换箭头。  
4. **用例关系**：include→抽象，extend→基本，泛化→父。  
5. **类关系强度**：泛化=实现 > 组合 > 聚合 > 关联 > 依赖；聚合好聚好散，组合同生同死。  
6. **顺序图**：重时序；协作图：重结构；活动图：可分叉合并、信号。  
7. **参与者**：人、其他系统、硬件、时钟，且在系统外。
