## 二、数据库性能优化（★★★★★）

### 2.1 索引建立注意事项

| 准则 | 说明 |
|------|------|
| 选**常查少改**的属性建索引 | 常作为查询条件、不常更新的属性适合建索引；**常更新的属性**建索引会严重影响性能 |
| **索引不宜过多** | 一个关系上索引过多会拉低 UPDATE、INSERT、DELETE 性能，因为更新时所有索引都要调整 |
| **按查询频度优化** | 分析重要查询的使用频度，优先优化使用最多的索引 |
| **小表可不建索引** | 数据量很小的关系不必建索引，全表扫描往往更快、耗资源更少 |

**口诀**：常查少改、别太多、先优化热点、小表不建。

**案例（航空运营智能管理平台）**：**航班号+日期、航线、起降机场** 等高频检索条件可建组合索引；**订单状态、支付流水** 若高频更新则控制索引数量；**配置类小表**（舱位字典等）可全表扫描。

---

### 2.2 数据分片（Shard）

- **目的**：**横向扩展**（Scale Out），突破单节点 I/O 与扩展性限制；把“整库”打碎成多块，每块存到不同物理节点。
- **分片分类**（按拆分维度）：

| 类型 | 原理 | 重构方式 |
|------|------|----------|
| **水平分片** | 按**行**划分，元组分成多个子集，每个子集为一个片段；条件由**关系属性值**表示 | **并**操作重构全局关系 |
| **垂直分片** | 按**列（属性）**划分，满足不相交 | **连接**运算重构 |
| **导出分片** | 水平分片条件**来自其他关系**的属性（如按学生表的性别对选课表分片） | — |
| **混合分片** | 水平 + 垂直混合 | — |

**案例（航空运营智能管理平台）**：**订单/票务流水** 适合 **水平分片**（按用户 ID、订单号哈希分散）；**旅客敏感信息与大字段** 可与订单 **垂直分片**（列拆分），兼顾安全与性能。

---

### 2.3 分片算法

| 算法 | 原理 | 适用数据 |
|------|------|----------|
| **哈希分片** | 对键（如 id）哈希，再对节点数**取模**，余数对应存储节点 | 分布均匀、无强顺序/时间相关性、关联弱、可独立读写 |
| **范围分片** | 按**范围区间**（时间、ID 等）划分，同一范围放一起 | 有明显顺序（时间、编号）、可划分区间 |
| **地理位置分片** | 按**地理位置**（城市、地域）分配 | 数据与地理位置强相关，不同地域访问模式不同 |
| **融合算法** | 组合多种分片算法 | 同时有随机读写、范围查询或多维特征 |

**中间件常用**（如 MyCat）：**取模**（随机均匀）、**范围**（范围查询）、**列表**（枚举值指定分片）、**一致性哈希**（节点动态增删时只影响相邻数据）。

**案例（航空运营智能管理平台）**：**订单 ID/用户 ID 哈希取模** 平滑热点；**按出行日期、账务月** 做 **范围分片** 便于对账与归档；**枢纽机场/航线** 可用 **列表** 做地域化策略（与「地理位置分片」思路相通）。

---

### 2.4 数据分区

- **含义**：单表内按规则（范围、哈希、列表等）划分为多个**逻辑子集**（分区）；查询时优化器只访问相关分区。
- **好处**：**查询**—减少 I/O、加快查询；**管理**—大表拆成小块，便于维护、备份、恢复；**隔离**—按类型或时间分隔，便于权限与生命周期管理。
- **分区算法**：与分片类似——**范围分区**（时间、数值区间）、**列表分区**（离散枚举值）、**哈希分区**（均匀、随机读写）、**复合分区**（先一种再另一种）。

**案例（航空运营智能管理平台）**：单库内 **日志、审计、交易明细** 可按 **日期范围分区**，查询只扫近期分区；与素材 **日志保留≥1 年、备份≥3 年** 的生命周期管理契合。

---

### 2.5 分片 vs 分区（★★★★★）

| 特征 | 分片 | 分区 |
|------|------|------|
| **范围** | **跨多个数据库实例/服务器** | **单实例内** |
| **目的** | 提高**可扩展性、并发** | 优化**查询性能** |
| **实现** | 多个**物理表**（多节点） | **PARTITION BY** 子句（单库内） |
| **管理难度** | 相对复杂 | 相对简单 |

**口诀**：分片跨节点扩规模，分区单机内提查询。

**案例（航空运营智能管理平台）**：**分片** = 订单库多节点扛 **≥5000 TPS**；**分区** = 单库大表按时间切分区，加速 **报表与追溯** 而未必立刻拆实例。

---

### 2.6 分库分表（★★★★★）

- **动机**：单机连接数、I/O、网络瓶颈；单表过大导致 B+ 树变高、I/O 增多；热点集中拖慢整体。
- **目的**：**分库**—分散请求到多台服务器，突破单机能力；**分表**—减小单表数据量，查询在更小范围完成。
- **分库分表是分片的物理实现**：“做了分片”的实现手段 = 分库 + 分表。

| 类型 | 定义 | 举例 |
|------|------|------|
| **水平分表** | 同一张表按**行**划分到多张表 | 订单表按订单 ID 取模拆成 orders_0、orders_1、orders_2 |
| **垂直分表** | 同一张表按**列**拆到多张表，减少单表字段 | 商品表：product_base（基本信息）、product_detail（描述、图片等大字段） |
| **水平分库** | 相同表结构分布到**多个库**，分摊压力 | 消息表按用户 ID 拆到 message_db1、message_db2 |
| **垂直分库** | **不同业务**的数据放到**不同库**，按模块拆 | 用户库 user_db、课程库 course_db、支付库 payment_db |

**案例（航空运营智能管理平台）**：**水平分表**：`ticket_order_0…n` 按用户或订单号；**垂直分表**：订单主表 vs 扩展/票据明细；**垂直分库**：**票务库、旅客库、检修库** 与素材 **七大模块** 对应；**水平分库**：票务库多副本分散连接与 I/O。

## 三、数据库高可用（★★★★★）

### 3.1 读写分离

- **目的**：把**读、写**分散到不同节点；**小幅提升写、大幅提升读**；符合写少读多。
- **常见**：**一主多从**—主库写，从库读；主从之间**数据同步**保证从库数据正确。
- **实现**：应用层如 **Shardingsphere**；数据库层中间件如 **MyCat**。

**示意**：
```
  写请求 → 主库  ←── 同步 ──→  从库  ← 读请求
```

**案例（航空运营智能管理平台）**：**航班查询、报表类读** 走从库；**下单、支付、库存扣减** 写主库；与素材 **分布式缓存、负载均衡** 一起分担 **票务高峰与航变洪峰** 读压力。

---

### 3.2 主从复制

- **手段**：利用**日志**（如 MySQL 的 **binlog**）；主库写操作记入 binlog，推给从库，从库**重放**完成复制。
- **流程（MySQL）**：① 主库把数据变化写入 **binlog**；② 从库连接主库请求复制；③ 从库 **I/O 线程**向主库请求 binlog；④ 主库 **dump 线程**发送 binlog，从库 I/O 线程接收；⑤ 从库 I/O 线程将 binlog 写入**中继日志（relay log）**；⑥ 从库 **SQL 线程**从 relay log 读事件并应用到本地库。

| 优势 | 描述 |
|------|------|
| **数据冗余** | 多副本，主库故障时减少数据丢失风险 |
| **负载均衡** | 读分发到从库，减轻主库负载 |
| **高可用** | 主库故障时可切换到从库 |

**案例（航空运营智能管理平台）**：**多副本 + binlog 复制** 支撑 **99.99%** 与 **数据可靠性≥99.999%**（分析域分布式存储另述）；主从切换与备份策略衔接素材 **实时备份、按季归档**。

---

### 3.3 主从一致性问题（主从延迟）（★★★★★）

- **现象**：主库写入后，同步到从库有**时间差**，读从库可能拿到**旧数据**。
- **解决思路**：减少延迟或延迟窗口内**强制读主**。三种方式对比：

| 方式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **半同步复制** | 主库提交事务后，等**至少一个从库**把事务 binlog 写入 relay log 并**回 ACK**，主库才提交并返回客户端 | 用数据库**原生**能力，实现简单 | 主库**写时延增加**，**吞吐下降** |
| **数据库中间件** | 读写都走中间件；写走主库，读走从库；**记录写库的 key**，在**主从同步时间窗**（如 500ms）内对该 key 的**读请求走主库**，过后再走从库（如 Canal、Otter） | **强一致** | 中间件**成本高** |
| **缓存记录** | 写前把 **key 写入 cache** 并设超时（如 500ms），再写主库；读时**先查 cache**，命中则说明刚写过，**读主库**，未命中则读从库 | 比中间件**成本低** | 多 **cache** 组件，读写多一步缓存操作 |

**key**：能唯一标识写操作的主键或业务键（如主键 ID、用户 ID、订单号），用于在延迟窗口内把“刚写过”的读请求路由到主库。

**示意**：
```
  半同步：  主库 --等从库 ACK--> 提交
  中间件：  写 key 记录 → 时间窗内该 key 读主库
  缓存：    写前 key 入 cache(500ms) → 读时命中则主库
```

**案例（航空运营智能管理平台）**：用户 **刚支付/刚出票** 后立即查 **订单/行程**，若读从库可能 **短暂旧状态**——可用 **半同步 / 中间件路由 / 写后短暂读主**（订单号、用户 ID 作 key）；与 **Redis 热点缓存** 策略可叠加。

## 四、数据库选型（★★★★★）

> 给场景选合适数据库是常规考法；下表按**分类—场景—结构—产品**记忆。

| 分类 | 典型场景 | 结构/特点 | 产品 |
|------|----------|-----------|------|
| **键值数据库** | 内容缓存、高访问负载、日志等 | Key-Value，常用 hashtable | **Redis** |
| **列存储数据库** | 分布式文件、海量存储与分析、大数据 | 列簇存储，同列放一起 | **Cassandra、HBase** |
| **文档型数据库** | Web 应用、灵活结构（如电商商品信息） | Key-Value，Value 为结构化文档 | **MongoDB** |
| **图数据库** | 社交网络、推荐、知识图谱、关系链分析 | 图结构（节点、边、属性） | **Neo4J** |
| **时序数据库** | 物联网监控、设备状态、金融时序分析 | 以**时间戳**为核心，按时间序列组织 | **InfluxDB、TDengine** |

**口诀**：缓存键值 Redis，海量列存 HBase/Cassandra，灵活文档 MongoDB，关系图谱 Neo4J，时序监控 InfluxDB/TDengine。

**案例（航空运营智能管理平台）**：

| 分类 | 素材落点 |
|------|----------|
| **键值（Redis 等）** | **分布式缓存**、会话、热点航班/运价、限流计数；衔接 **≤1 秒** 购票体验 |
| **列存（HBase/Cassandra 等）** | **数据服务** 域 **PB 级、单表 50 亿条+** 明细与离线分析底座 |
| **文档（MongoDB 等）** | 非强事务的 **日志形态、可变结构** 数据（答题可选） |
| **图（Neo4J 等）** | 航线网络、关联推荐等 **图分析**（选填） |
| **时序（InfluxDB/TDengine 等）** | **设备运行、航班动态、指标监控** 类时序数据 |

关系型库仍承担 **票务事务、强一致订单** 核心；与素材 **兼容离线与实时分析引擎、分布式存储** 混合架构一致。

## 五、速记清单（考前扫一眼）

1. **索引**：常查少改、别太多、小表可不建；索引多会拖慢增删改。
2. **分片**：水平（按行、并）、垂直（按列、连接）、导出（按他表属性）、混合；**分片跨节点**，为扩展与并发。
3. **分片算法**：哈希/取模—均匀随机；范围—顺序与区间；列表—枚举；一致性哈希—节点动态变化。
4. **分区**：单机内逻辑划分，**优化查询与管理**；范围/列表/哈希/复合；实现用 PARTITION BY。
5. **分片 vs 分区**：分片跨实例、分区单实例；分片扩规模、分区提查询。
6. **分库分表**：水平分表（按行）、垂直分表（按列）、水平分库（同表多库）、垂直分库（按业务分库）；分片概念，分库分表是实现。
7. **读写分离**：一主多从，写主读从；Shardingsphere、MyCat。
8. **主从复制**：binlog → 从库 I/O → relay log → SQL 线程应用；数据冗余、负载均衡、高可用。
9. **主从延迟**：半同步（等从库 ACK，简单但主库性能降）、中间件（记 key 时间窗读主，强一致成本高）、缓存（写前 key 入 cache，读命中走主，成本适中）。
10. **选型**：键值缓存 Redis、列存 HBase/Cassandra、文档 MongoDB、图 Neo4J、时序 InfluxDB/TDengine。
