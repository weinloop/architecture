## 二、系统架构原则（★★★☆☆）

| 原则 | 要点 | 解释（简） |
|------|------|------------|
| **可扩展** | 分布式环境 | 分布式运行，任务分配、资源管理、负载均衡，**横向扩展**计算与存储 |
| **可扩展** | 模块化设计 | 拆分为独立模块，便于设计、开发、维护与扩展 |
| **可管理** | — | 自动化管理、统一管理平台、可视化管理，提升效率与可靠性 |
| **数据安全** | 加密、访问控制、备份恢复、审计日志、持续监控 | 存储/传输/处理加密；权限控制；备份与恢复；操作审计；监控日志与访问模式 |
| **高性能** | 数据分区、分布式计算、并行处理、内存计算 | 数据分块并行；多节点计算合并；子任务同时执行；内存计算减 I/O 延迟 |
| **高可用** | 数据冗余、自动故障转移、负载均衡 | 多节点/设备复制；ZooKeeper 等自动切换；Haproxy 等负载均衡 |
| **稳定性** | 监控预警、性能调优、故障排除、升级维护 | 实时监控与告警；找瓶颈优化；快速排障；及时升级与维护 |

**考法**：给场景判断属于哪条原则（如“增加节点扩展”→可扩展；“双机热备”→高可用）。

## 三、批处理 vs 流处理 vs 混合（★★★★★）

| 架构类型 | 概念 | 优势 | 局限 | 应用场景 |
|----------|------|------|------|----------|
| **批处理** | 大规模数据**批量**任务，**离线**执行；含采集、存储、处理、输出；按**时间间隔或触发**处理 | 可处理大规模数据，**吞吐高、稳定**，充分利用计算资源 | **实时性差**，无法满足实时分析 | 对实时性要求不高的大规模数据处理 |
| **流处理** | 数据**连续、无界**流式处理，**新数据到达即处理**；由数据流、运算符组成，流处理引擎管理 | **实时**处理，快速响应，适应数据快速变化 | 无法处理历史数据、可能漏数据，需考虑并发与一致性 | 实时分析、监控、推荐等 |
| **混合架构** | 同时使用批处理与流处理，数据在两种模式间按需转换 | 兼具批与流优点，可处理历史+实时，快速响应 | 需权衡架构与数据处理，管理维护成本高 | 既有历史又有实时需求的场景 |

**口诀**：批处理大批量离线高吞吐，流处理实时到达即处理。

## 四、Lambda 架构（★★★★★）

### 4.1 三层结构

| 层次 | 概念 | 支撑技术 |
|------|------|----------|
| **批处理层** | 负责**所有批处理**，存储**全量数据集**并计算**批处理视图** | Hive、Spark-SQL、MapReduce 等 |
| **加速层/速度层** | 用**流式计算**实时处理**当前数据**，弥补批处理高延迟 | Storm、Spark Streaming、Flink 等 |
| **服务层** | 以批处理层与加速层结果为数据源，提供**低延时查询与即席查询** | 关系型库，或 Kylin、Presto、Impala、Druid 等 OLAP |

**示意**：
```
  数据源 → [批处理层] → BatchView ──┐
       ↘ [加速层] → RealtimeView ──┼→ [服务层] → 查询
```

---

### 4.2 批处理层与加速层分开设计的原因

| 原因 | 说明 |
|------|------|
| **容错性** | 加速层数据持续写入批处理层；批处理层**重算全量**后可覆盖加速层数据，**修正加速层错误** |
| **复杂性隔离** | 加速层用**增量算法**处理实时数据，复杂度高于批处理层，需隔离 |
| **横向扩容** | 数据量或负载增大时，分层便于**增加机器**维持性能 |

**二者对比**：加速层处理**最近增量数据流**，不断更新 RealtimeView；批处理层处理**全量数据集**，得到 BatchView。

---

### 4.3 Lambda 实现中各技术角色

| 层次 | 技术 | 功能 |
|------|------|------|
| **批处理层** | Hadoop | 大规模**批量**数据处理；从 Kafka 读数据进行批处理 |
| **加速层** | Spark / Storm | Spark：微批准实时、吞吐与复杂计算（ML、图）；Storm：**纯流**、亚秒级、低延迟与可靠性 |
| **服务层** | HBase / Cassandra | 实时访问与更新；HBase 基于 HDFS 列存；Cassandra 宽列、高吞吐 |
| **服务层** | Hive | 在 Hadoop 上建**可查询视图**，HiveQL→MapReduce，便于离线查询分析 |

**Spark vs Storm**：Spark 偏吞吐与复杂逻辑，Storm 偏**极致实时**。

---

### 4.4 Lambda 优缺点

| 类型 | 描述 |
|------|------|
| **优点** | **容错性好**：可修复算法或重算视图；**查询灵活**：批/实时存不同存储，多接口（SQL、REST 等）；**易伸缩**：分布式、弹性调度；**易扩展**：分层解耦、插件化 |
| **缺点** | 技术栈复杂，**两套系统**开发维护成本高、代码重复；全场景覆盖编码开销大；重部署与迁移成本高 |

## 五、Kappa 架构（★★★★★）

### 5.1 核心思想与组件

- **思想**：**去掉批处理层**，数据通道用**消息队列**替代；以**流处理为主**，数据存数据湖，需离线或重算时从消息队列**重放**数据即可。
- **数据流**：数据视为**持续事件流** → 进入 **Kafka/Pulsar**（高吞吐、持久化、可重放）→ **Flink/Kafka Streams/Spark Structured Streaming** 流处理（窗口、聚合、状态管理）→ 结果写入 **Redis/ClickHouse/ES** 或数据湖（Hudi、Iceberg）。
- **可回放**：算法或程序变更时，**重置消费位点、重放历史流**即可重新出结果，无需单独批处理程序。

| 核心组件 | 内容 |
|----------|------|
| **消息传输层** | Kafka 等，高吞吐、低延迟、**持久化与重放** |
| **流处理层** | Flink、Storm 等，实时计算，低延迟结果输出至服务层 |
| **数据存储** | HBase、Cassandra 等，支持快速查询 |

---

### 5.2 Kappa 优缺点

| 类型 | 描述 |
|------|------|
| **优点** | **实时与离线代码统一**，维护方便，**统一数据口径**，避免两套逻辑不一致 |
| **缺点** | 消息中间件**缓存与回溯**有性能瓶颈；多实时流**关联时顺序**问题易导致数据丢失；**抛弃离线模块**，失去离线计算更稳定的特点 |

---

### 5.3 Lambda vs Kappa（★★★★★ 必背）

| 对比内容 | Lambda | Kappa |
|----------|--------|-------|
| **复杂度与维护成本** | **两套系统**（批+流），复杂度高，开发维护成本高 | **一套系统**（流），复杂度低，成本低 |
| **计算开销** | 批处理和实时**同时常开**，计算开销大 | **必要时**全量计算（重放），计算开销相对小 |
| **实时性** | 满足实时性 | 满足实时性 |
| **历史数据处理能力** | **批式全量**，吞吐大，**历史数据处理能力强** | **流式全量**，吞吐相对低，历史数据处理能力**较弱** |

**口诀**：Lambda 两套批流历史强，Kappa 一套流回溯弱。

## 六、IOTA 架构（★★★☆☆）

- **特点**：**边缘计算** + **统一数据模型**；数据流连续、一致。
- **思路**：设定**标准数据模型**（如“主-谓-宾”“对象-事件”），在**数据产生、计算、查询**各环节分散计算，**统一模型贯穿**，支持即时查询。
- **要点**：Edge SDK 做校验、格式转换、边缘聚合；实时缓存（Kudu/HBase）+ 历史沉淀（Dumper→HDFS）。
- **优点**：简化架构（无批处理层）、低延迟（边缘计算）、统一数据模型。**缺点**：历史分析依赖流式重放，效率瓶颈；乱序、延迟、幂等处理难；全量日志存储成本高。

## 八、速记清单（考前扫一眼）

1. **原则**：可扩展（分布式+模块化）、可管理、数据安全、高性能、高可用、稳定性；能根据场景归到某条原则。
2. **批处理**：大批量离线、高吞吐、实时性差；**流处理**：连续无界、到达即处理、实时。
3. **Lambda 三层**：批处理层（全量 BatchView）→ 加速层（增量 RealtimeView）→ 服务层；分开原因：容错、复杂隔离、横向扩容。
4. **Lambda 技术**：批处理 Hadoop/Hive；加速 Spark/Storm；服务 HBase/Cassandra、Hive。
5. **Kappa**：无批处理层，消息队列+流处理+重放；一套流引擎，维护简单；历史与回溯有瓶颈。
6. **Lambda vs Kappa**：两套 vs 一套、计算开销大 vs 相对小、历史能力强 vs 弱。
7. **奥运填空**：采集 Flume/ETL/Sqoop/Kafka/Zookeeper/Nginx；存储 MemSQL/HBase/HDFS；计算 Spark/SparkStreaming/MapReduce/Hive/Impala；层名 数据采集层、数据存储层。
8. **技术归属**：Flume/Filebeat 日志采集；Kafka 消息队列；HDFS 离线存储；HBase 列存/实时；MemSQL 内存；Hive 离线分析；Spark/SparkStreaming 批/流；Flink 流；ES 日志库；OpenTSDB 时序；Tair 实时缓存；CDH 集群。
