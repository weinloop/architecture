#### 1\.2\.2 用例图（超级重点★★★★★）

**案例考频**：★★★★★

**✅ 标准答案话术：**

- **用例**：系统执行的一系列动作，对参与者产生有价值的结果

- **参与者**：与系统交互的外部实体（人、系统、设备）

- **包含关系 \&lt;\&gt;**：基础用例必定执行包含用例，提取公共行为

- **扩展关系 \&lt;\&gt;**：扩展用例在特定条件下可选执行，增强基础用例

- **泛化关系**：父子用例的继承关系，子用例可特化父用例

**💡 解题思路：**

1. 先找参与者（外部实体）

2. 再找主要用例（核心功能）

3. 最后判断关系（include/extend/ 泛化）

4. **采分点**：每个用例名称、参与者名称、关系类型各 1 分

|原则|核心描述|易混点辨析|

### 4\.2 云原生架构模式（重点★★★★★）

|架构模式|核心定义|优点|缺点|适用场景|
|---|---|---|---|---|
|**服务化架构**|按业务域拆分，通过 API 交互|灵活扩展、独立部署、降低耦合|分布式治理成本高|电商订单、支付、库存|
|**ServiceMesh**|Sidecar 统一处理网络通信、治理|统一治理、解耦复杂逻辑|额外性能开销|Istio、Linkerd|
|**Serverless**|无服务器，事件触发，按调用计费|运维成本极低、按需伸缩|冷启动延迟、厂商锁定|图片处理、日志分析、IoT|
|**存储计算分离**|计算无状态，数据存分布式存储|弹性伸缩、成本优化|计算存储分离带来延迟|Snowflake、TiDB|
|**分布式事务**|跨服务数据一致性保证|见 10\.3 详细对比|见 10\.3 详细对比|跨库 / 跨服务操作|
|**可观测架构**|Logging\+Tracing\+Metrics|全面感知系统状态|存储和计算成本|监控、报警、运维优化|
|**事件驱动 EDA**|以事件为核心交互|服务解耦、异步处理、可扩展|事件顺序难保证|订单完成触发发货|

### 4\.3 容器 vs 虚拟机对比（超级重点★★★★★）

**✅ 案例标准答案话术必背：**

|对比维度|容器（Docker）|虚拟机（VM）|
|---|---|---|
|**运行原理**|OS 层面虚拟化，**共享宿主机内核**|硬件虚拟化，**运行独立 OS**|
|**资源隔离**|cgroups 限制 \+ namespace 隔离，共享内核|完全硬件隔离，独立内核|
|**启动速度**|**秒级**，无需启动完整 OS|数十秒到数分钟，加载完整 OS|
|**资源占用**|极小，仅运行应用所需资源|大，完整 OS \+ 硬件模拟开销|
|**隔离性**|相对较弱，内核漏洞可能影响多容器|强，虚拟机间完全独立|
|**移植性**|极好，镜像打包环境一致|一般，受虚拟硬件和 OS 影响|
|**管理复杂度**|简单，主要管理容器生命周期|复杂，需管理 OS 和虚拟硬件|

**💡 采分点：** 只要写出 \&\#34;共享内核\&\#34;vs\&\#34;独立 OS\&\#34;、\&\#34;秒级启动\&\#34;vs\&\#34;分钟级启动\&\#34; 即得分

### 4\.4 容器 vs 容器编排

|对比维度|容器（Docker）|容器编排（K8s）|
|---|---|---|
|**定义**|应用打包和运行环境单元|自动化部署、管理、扩展容器集群|
|**核心功能**|隔离环境、快速部署迁移|集群管理、资源调度、服务发现、负载均衡、自动伸缩|
|**代表工具**|Docker|Kubernetes|
|**适用场景**|单一简单应用、微服务单个载体|大规模复杂容器化应用集群|

## 第 5 章 面向服务架构 SOA（超级重点★★★★★）

**案例考频**：★★★★★

### 5\.1 SOA vs 微服务对比

|对比维度|SOA|微服务|
|---|---|---|
|**服务粒度**|较粗，按业务领域划分|更细，单一职责|
|**服务通信**|ESB 企业服务总线，重量级|轻量级通信（REST/gRPC）|
|**数据管理**|倾向共享数据库|每个服务独立数据库|
|**部署方式**|整体部署或按服务部署|完全独立部署，容器化|
|**治理方式**|集中式治理，ESB 统一管理|去中心化治理，服务自治|
|**适用场景**|大型企业，遗留系统集成|互联网应用，快速迭代|

### 5\.2 ESB 核心功能

**✅ 案例答题话术：**

1. **协议转换**：不同服务间协议适配（HTTP、SOAP、JMS 等）

2. **消息路由**：根据消息内容动态路由到目标服务

3. **服务编排**：多个服务组合成新的业务流程

4. **数据转换**：不同数据格式间转换（XML、JSON 等）

5. **安全控制**：认证、授权、加密

6. **监控审计**：服务调用监控和日志记录

### 5\.3 微服务反模式

|反模式|问题描述|解决方案|
|---|---|---|
|**庞大单体**|缺乏隔离，代码耦合，接口无治理|适度拆分，明确边界，组织架构匹配|
|**硬拆微服务**|过度拆分，数据依赖复杂，性能下降|合理评估粒度，避免为拆分而拆分|
|**缺乏自动化**|人工运维，交付慢，风险高|建立 CI/CD，自动化测试发布|

## 第 8 章 大数据架构（次重点★★★☆☆）

**案例考频**：★★★☆☆

### 8\.1 Lambda vs Kappa 架构（超级重点★★★★★）

|对比维度|Lambda 架构|Kappa 架构|
|---|---|---|
|**处理层**|**批处理层 \+ 速度层 \+ 服务层**|**仅流处理层**|
|**核心思想**|批处理保证准确，流处理保证实时|用流处理统一批处理和实时处理|
|**数据处理**|批处理全量重算，流处理增量计算|所有数据都作为流处理，重放实现批处理|
|**复杂度**|高，两套代码维护|低，一套代码|
|**存储**|需要批处理存储 \+ 流处理存储|仅需要消息队列持久化|
|**延迟**|批处理小时级，流处理秒级|统一秒级|
|**准确性**|批处理最终准确|依赖流处理准确性|
|**适用场景**|历史数据重算频繁，准确性要求高|实时性要求高，代码维护成本敏感|

**✅ 案例答题话术：**

- Lambda：**双架构**，批处理做全量计算保证准确，流处理做实时计算保证低延迟

- Kappa：**单架构**，以流处理为核心，通过消息重放实现批处理功能，简化架构

### 8\.2 大数据技术栈

|层级|技术|作用|
|---|---|---|
|**数据采集**|Flume、Logstash、Sqoop|日志、数据库数据采集|
|**消息队列**|Kafka、Pulsar|数据缓冲、解耦|
|**存储**|HDFS、HBase、ClickHouse|分布式存储|
|**计算**|Spark、Flink|批处理 / 流处理计算|
|**查询**|Hive、Presto、Impala|OLAP 查询|

## 第 10 章 分布式系统（重点★★★★★）

**案例考频**：★★★★★

### 10\.1 CAP 定理

**✅ 必背结论：**

- **C 一致性**：所有节点数据一致

- **A 可用性**：每个请求都能获得响应

- **P 分区容错性**：网络分区时系统继续运行

- **分布式系统必须选 P，只能在 C 和 A 间权衡**

- **CP**：ZooKeeper、HBase、MongoDB

- **AP**：Cassandra、CouchDB、Eureka

### 10\.2 BASE 理论

- **BA 基本可用**：故障时允许损失部分可用性

- **S 软状态**：允许系统存在中间状态

- **E 最终一致性**：经过一段时间后数据最终一致

### 10\.3 分布式事务对比（超级重点★★★★★）

|方案|原理|一致性|性能|侵入性|适用场景|
|---|---|---|---|---|---|
|**2PC 两阶段提交**|准备阶段 \+ 提交阶段，资源管理器 \+ 事务管理器|强一致|❌ 差，锁时间长|低|单体跨库，短事务|
|**3PC 三阶段提交**|CanCommit\+PreCommit\+DoCommit，引入超时|强一致|❌ 差|低|较少使用|
|**TCC**|Try\-Confirm\-Cancel，应用层实现|最终一致|✅ 好|❌ 高，每个服务写 3 个方法|核心业务，灵活性要求高|
|**SAGA**|长事务拆分为本地事务 \+ 补偿操作|最终一致|✅ 好|中|长事务，业务流程复杂|
|**本地消息表**|本地事务 \+ 消息表 \+ 定时任务|最终一致|✅ 好|中|非核心业务|
|**可靠消息最终一致**|消息队列 \+ ack 机制|最终一致|✅ 好|低|异步场景|
|**Seata AT**|代理 JDBC，自动生成回滚 SQL|最终一致|✅ 较好|极低|支持的数据库|

### 11\.1 数据库范式

|范式|核心要求|解决问题|
|---|---|---|
|**1NF**|属性不可再分|原子性|
|**2NF**|消除部分依赖，非主属性完全依赖主键|插入、删除、更新异常|
|**3NF**|消除传递依赖，非主属性不依赖其他非主属性|数据冗余|
|**BCNF**|消除主属性对候选键的部分和传递依赖|更严格的 3NF|

### 11\.2 数据库优化

**✅ 案例答题要点：**

1. **索引优化**：合适的索引类型，避免索引失效

2. **SQL 优化**：避免 SELECT \*，避免大表 JOIN，分页优化

3. **分库分表**：垂直分库（按业务），水平分表（按范围 / 哈希）

4. **读写分离**：主库写，从库读

5. **缓存**：热点数据缓存，减轻数据库压力
