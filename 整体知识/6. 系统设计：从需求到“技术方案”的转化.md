# 6. 系统设计：从需求到“技术方案”的转化

系统设计是“需求落地的桥梁”，基于《需求规格说明书》，给出“如何实现”的技术方案，覆盖**整体设计、面向对象分析与设计、软件架构设计**。

## 6.1 整体设计流程：定框架、拆模块

### 1. 确定设计目标

满足“高可用（支持 7×24 小时运行）、可扩展（后续新增如智能推荐舱位、辅助排班等能力）、易维护（微服务域独立部署与发布）”。

### 2. 处理流程设计

设计“不正常航班协同处置流程”：运行监控发现航班延误或限载 → 推送告警至运行指挥中心 → 自动触发预案任务（舱位保护、旅客通知模板） → 派发保障工单至地服/座席班组 → 处置完成后审计闭环。

### 3. 流程设计工具

用“活动图”描述上述流程，用“时序图”描述“航班动态下发流程”（外部数据源/运行系统 → 接入网关 → 航班状态服务 → 消息总线 → 订阅方：订座缓存、旅客通知、看板）。

### 4. 输出设计

#### 输入设计：

针对“旅客购票/支付服务”，用户下单需输入包含“旅客证件或会员标识（唯一约束按合规脱敏展示）、航段与舱位、联系人手机/邮箱、支付方式（微信/支付宝/银行卡/企业支付，默认可记住上次方式）”，同时设计“输入校验规则”：证件/会员号格式按规则校验，支付金额与运价试算一致，若输入错误则实时提示（如“舱位已售罄，请重新查询”）。

#### 输出设计：

支付出票成功后，输出内容包含“电子行程单/订单摘要（含订单号、出票时间、旅客掩码信息、金额、票号）、电子凭证下载或邮箱推送、短信/App 推送（如【××航空】出票成功，订单号×××）”；对于“经营与运行数据报表中心”，输出设计需支持“多格式导出（Excel/PDF/CSV）、自定义报表模板（如运行部门可自定义‘月度延误结构与服务补偿报表’的字段与图表类型）、定时自动推送（按周/月推送至指定邮箱或安全文件柜）”。

### 5. 人机交互设计

#### 桌面端（如运行指挥中心/运控工作台）：

采用“左侧功能导航栏 + 中间主可视化区 + 右侧告警与任务栏”布局，主可视化区以“航线—机场态势底图”或关键运行列表为骨架，支持“缩放/平移、图层切换（天气限制、流量管理、保障节点）”；告警与任务栏采用“红（运行红线）、黄（注意）、蓝（提示）”区分级别，点击告警可下钻至关联航班与工单；高频操作符合“短路径”原则，如“启动不正常航班预案”“批量改签通知”置于显著区域，避免多层级翻找。

#### 移动端（如地服/场站保障 App）：

采用“底部 Tab 导航（任务/消息/扫码/我的）”，首页默认展示“待保障航班数（角标）、今日关键节点任务”；保障页设计“一键扫码（登机牌/行李牌）、离线草稿（弱网写入本地队列，恢复后自动同步）、快捷语音备注（可选转文字并打标签）”；适配单手与户外场景（可操作热区≥48×48dp，降低误触）。

## 6.2 结构化分析与设计：以“数据流”为核心

基于项目中“数据驱动”的业务场景（如航班动态汇聚、订座与库存同步），采用结构化分析与设计方法，融合需求工程阶段的成果：

### 1. 数据流图（DFD）：深化

以“订座与库存一致性服务（示意子域）”为例，绘制“三级 DFD”：

#### 0 级 DFD：外部实体

（旅客渠道 / 座席 / 外部订座或库存源）→ 订座与订单域 → 外部实体（支付机构、旅客通知、运行看板）。

#### 1 级 DFD：拆分

拆分“舱位查询与锁舱模块”“价格与规则引擎模块”“订单落库与状态机模块”“对账与补偿模块（异步）”；

#### 2 级 DFD：进一步拆分

“锁舱模块”可拆为“请求预处理（幂等键、重试策略）”“库存扣减（乐观/悲观策略按场景选取）”“冲突检测与回滚事件发布”。

### 2. 数据字典完善

为 DFD 中的“数据源、数据流、数据存储”定义详细属性，如“数据项—航班号+日期”：类型为字符串组合，符合航司编码规范；“数据流—座位锁定事件”：包含“锁单 ID、航段、舱位、数量、超时时间、traceId”；数据存储—订单与票号表：主键为“订单 ID”，外键关联旅客主数据，审计字段满足监管留存周期，支持按时间/航班/状态分页查询。

### 3. E-R 图设计

针对“旅客服务与订单子域”，设计 E-R 图：核心实体为“旅客（旅客主键、姓名掩码、证件类型与掩码、常旅客等级、联系方式）、订单（订单 ID、创建时间、状态、支付流水）、航段明细（订单 ID、航班实例、舱位、票号/PNR 片段）”；实体间关系为“旅客—订单（一对多）、订单—航段明细（一对多）”；后续可转化为物理表并建立外键或分库分表策略。

### 4. 结构化设计原则（高内聚低耦合）

#### 高内聚：

将“统一接入平台”的“协议与报文适配（离港/订座/渠道约定格式）、连接管理（限流、熔断）、数据质量管理（校验、补全）”拆为独立模块，例如“协议适配模块”仅负责解析与映射，不直接写业务订单库；

#### 低耦合：

模块间通过“标准化接口与事件”通信，如“航班动态服务”与“数字孪生/可视化”通过 REST 或订阅主题交换数据（约定查询参数与时间窗）；避免跨模块直连接口数据库，数据交互经服务边界与仓储层，降低变更涟漪。

## 6.3 面向对象分析与设计：以“对象”为核心

针对项目中“交互频繁、业务逻辑复杂”的场景（如不正常航班工单、呼叫中心服务），采用面向对象分析与设计方法，融合 UML 工具与设计原则：

### 1. UML 应用

#### 类图：

以“保障工单管理”为例，设计核心类“工单（工单 ID、类型、状态、创建时间、处理人、优先级、关联航班/登机口）”“操作员（用户 ID、姓名、角色、班组）”“节点资源（登机口/行李转盘 ID、状态）”；类间关系为“操作员—工单（一对多创建/认领）、节点资源—工单（多对多或按规则绑定）”，并定义方法（如工单的“创建、派发、升级、关闭、审计留痕”）。

#### 用例图：

针对“旅客服务中心/在线客服子系统”，绘制用例图——参与者为“客服座席、旅客、系统管理员”。座席用例包括“接听进线、创建服务单、查询订单与行程、回复退改规则咨询、满意度回访”；旅客用例包括“在线/chat 进线、催办/补充材料、查询服务进度、投诉”；管理员用例包括“配置技能组与权限、管理知识库分类、敏感操作审批开关”。

#### 序列图：

以“旅客投诉工单处理”为例，绘制序列图——旅客提交 → 座席受理并建单 → 系统路由至责任班组 → 业务处理并上传证据 → 系统更新状态与 SLA 计时 → 座席回访 → 系统归档与质检抽样；明确每步消息的先后顺序与责任系统。

#### 活动图：

针对“自愿改期（含价差）流程”，绘制活动图——开始 → 旅客或座席发起改期 → 规则与舱位试算 → 通过则支付/确认 → 出票或重打票链 → 触发通知与积分异步任务 → 结束；不通过则返回原因（如“机票使用限制不可改”“目标航班已关闭”），展示分支、人工审核节点与闭环。

### 2. 设计原则与设计模式应用

#### 单一职责原则：

将“运行资源管理（机位/登机口视图）”与“地面保障任务管理”拆为两个服务，前者专注场地资源状态维护，后者专注任务编排与 SLA，避免单服务职责膨胀。

#### 开闭原则：

针对“报表与指标导出”，采用“策略模式”——定义“导出策略接口”，实现 Excel/PDF/CSV 等策略；新增 HTML 或监管专用格式时仅扩展策略类，稳定遵循开闭。

#### 依赖倒置原则：

运行指挥中心依赖“运行数据获取抽象接口”，由“航班动态实现类、库存与订座快照实现类、机场保障节点实现类”等具体类注入；新增第三方气象或流量数据时扩展实现即可，指挥端核心逻辑少改。

#### 观察者模式：

应用于“运行与营销类告警广播”——定义“延误/超售/保障超时”等主题，观察者包括“指挥大屏、座席工作台提醒、App 推送通道、邮件值班列表”；主题触发后批量通知订阅方，避免点对点硬编码依赖。
