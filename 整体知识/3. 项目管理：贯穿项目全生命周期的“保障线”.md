# 3. 项目管理：贯穿项目全生命周期的“保障线”

项目管理是确保“航空运营智能管理平台”按时间、成本、质量落地的关键，覆盖“范围、进度、成本、配置、质量、人力、风险、文档”8 大领域。

## 3.1. 范围管理：避免“需求蔓延”

- **范围定义**：以“项目功能清单”为核心，明确“必做”与“不做”——例如“航班运行与资源可视化”需包含“统一时刻与批复状态的运行视图”，但暂不支持“全机队健康度 AI 预测”（留待二期），形成《工作分解结构（WBS）》，将“统一 API 网关与集成”拆解为“路由与鉴权、流量治理、遗留系统适配、联调验收”等子任务。

- **范围控制**：项目实施中，业务部门提出“增加常旅客积分与异业权益实时同步”的需求，需通过“需求变更流程”评估影响（需对接会员中心接口、增加幂等与补偿任务、扩展审计字段），经甲方审批后纳入范围，避免无序变更。

## 3.2. 进度管理：确保建设周期与上线窗口落地

制定进度计划，用甘特图明确里程碑节点（示例，以批复计划为准）：

- 2024 年 3–5 月：需求分析与总体设计（输出《需求规格说明书》《系统设计方案》）；

- 2024 年 6 月–2025 年 3 月：集成底座与核心业务开发（身份与权限、订座与订单、航班动态与通知链路等上线或达到联调条件）；

- 2025 年 4–6 月：旅客触达、检修协同、数据分析等扩展模块开发与接口收口；

- 2025 年 7 月：生产环境部署、全链路压测与安全测评配合；

- 2025 年 8 月：并行切换、试运行与正式上线。

- **进度控制**：若“与既有离港/订座系统的接口联调”因对方版本窗口延后 2 周，需增加适配层人力或向后续“高峰演练”任务前置了一部分测试，确保关键上线窗口不突破门禁。

## 3.3. 预算成本管理：控制批复总投资

- **成本估算**：按“基础设施与软件授权（含云平台/专线）+ 定制开发 + 第三方测试与测评 + 培训与驻场运维”拆分成本，例如可按预算科目列出占比区间（如基础设施与商用中间件、应用开发、集成与数据迁移、运维与保障等），便于评审与滚动预测。

- **成本控制**：若对象存储或实时计算资源采购超出阶段预算，可采用“冷热分层”（热数据高性能介质、冷数据低成本归档）与“按需弹性配额上限”方式，在保证峰值容量门禁的前提下压降常态成本。

## 3.4. 软件配置管理：避免版本混乱

- **配置项识别**：将“需求规格说明书、系统设计文档、源代码、测试用例、部署清单（含 Helm/编排模板）、接口契约（OpenAPI/Protobuf）”纳入配置管理，例如“航班动态订阅主题配置”“网关路由与灰度规则文件”为关键配置项。

- **版本控制**：使用 Git 管理源代码，例如“旅客服务 App”的 V1.0 实现“行程查询与推送”，V1.1 新增“电子登机牌与异常航班指引”，每个版本标注变更内容与责任人。

## 3.5. 其他管理领域应用

- **质量管理**：制定《质量保证计划》，例如“核心购票链路成功率与一致性满足门禁”“关键 API 在目标并发下 P99 时延达标”，通过功能测试、性能测试、回归与（视需要）混沌演练验证质量指标。

- **人力资源管理**：组建“需求分析师、架构师、开发工程师（按领域分队）、测试工程师、运维/DevOps”等角色，明确“架构师牵头技术路线与非功能指标，各分队负责子系统交付与联调”。

- **风险管理**：识别“外部主数据或离港接口不可用”（风险等级高），制定应对措施（缓存降级、兜底队列、对账补偿）；识别“一线座席对新工作台接受度不足”（风险等级中），应对措施为“新旧系统并行与集训 Sandbox 演练 3–6 个月过渡”。

- **信息文档管理**：整理《项目周报》《会议纪要》《测试报告》，例如每次需求评审后输出《需求评审纪要》，明确修改意见与完成时间。
