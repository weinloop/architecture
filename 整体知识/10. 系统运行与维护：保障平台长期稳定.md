# 10. 系统运行与维护：保障平台“长期稳定、持续迭代”

## 10.1. 运维技术指标与日常维护管理

### 1. 运维技术指标定义与监控：

- **可用性指标**：例如系统整体可用性≥**99.9%**（月化故障时间可换算为分钟级）、核心链路与运行数据接入可用性≥**99.95%**（以合同 SLO 为准）；通过 **Prometheus + Grafana** 统计各服务 UP 时间与黄金探针成功率，低于阈值时短信/电话/IM 告警到值班；

- **性能指标**：例如对外 API **P95/P99** 时延、核心数据库查询 **P95**、主机 **CPU/内存/磁盘**水位；在 Grafana 配置持续高负载、磁盘将满、连接池耗尽等告警，对接企业微信/飞书机器人；

- **数据指标**：离港/运行类报文**接入成功率**、**备份成功**、关键链路**同步延迟**（如动态服务至缓存/看板的秒级延迟）；自定义 Exporter 或脚本生成《数据质量日报》（失败批次、重试队列深度、对账差异）。

### 2. 日常维护管理：

- **系统运行管理**：**7×24** 值班与交接班制度；工作日加强变更窗口值守，节假日与**春运/长假**升级重保；每日**例行巡检**——基础设施、中间件、备份任务、证书与密钥有效期、告警静默规则；

- **故障管理**：分级示例 ——

    - **一级**（整体不可用、无法出票或支付大面积失败）：短时限响应与指挥升级，同步业务与公关口径；

    - **二级**（单域不可用、大面积订阅延迟、核心报表错误）：限定时间内定位与缓解，必要时切只读或降级；

    - **三级**（文案、非核心按钮、低流量页面问题）：按服务目录时限处理；

- **故障处理流程**：“发现（告警/客服工单）→ 登记与影响面 → 排查（日志、链路、变更关联）→ 修复/回滚 → 验证 → **复盘**（时间线、根因、改进项）”。

### 3. 系统维护：

- **硬件维护**：机房巡检、温湿度与电源、网络端口与光模块；存储健康度与预测性更换；

- **软件维护**：OS/DB/中间件**补丁节奏**（先测后产）；微服务小版本合入性能与安全修复；**证书轮转**自动化；

- **数据维护**：全量+日志协议备份，异地与不可变副本；按策略**归档**历史日志与影像附件，**热冷分层**降本；遵循留存与审计要求。

### 4. 评价管理：

按月《运维服务评价》：**可用性达标率、MTTR、变更成功率**及**内部业务方满意度**抽样，纳入运维与供应商考核。

## 10.2. 遗留系统与新旧切换

### 遗留系统评价：

基于“**技术水平**”和“**业务价值**”双维度，航司常见遗留资产可落入四象限（示意）：

- **老旧主机订座/报文前置系统** → 常落在**低水平、高价值**（技术债高，但仍是库存与结算真相源之一）；

- **Excel/电话驱动的地面保障台账** → 常落在**低水平、低价值**（已被移动工单部分替代，维护收益低）；

- **核心旅客/订单主库（如成熟商业数据库集群）** → 常落在**高水平、高价值**（稳定承载账务级数据）；

- **单一部门自建、少集成的统计分析系统** → 常落在**高水平、低价值**（技术尚可但数据孤岛、全局协同弱）。

### 差异化演化策略：

- **低水平、高价值——继承/包裹**：保留核心交易能力，建设**适配层与事件出口**，向新平台实时/准实时同步；硬件与协议分步升级；**并行运行**直至新链路经高峰验证；

- **低水平、低价值——淘汰**：停用入口，**历史数据迁移与只读存档**，校验后方可下电；

- **高水平、高价值——改造**：保留库或核心 schema，通过 **CDC/同步工具** 与新域服务**双向或主从**对齐；渐进**模型标准化**（旅客主数据、订单状态机）；在不破坏账务的前提下叠加标签与画像；

- **高水平、低价值——集成**：经 **ESB/API 网关** 把报表或算法结果嵌入统一流程与门户；**统一身份与权限**，避免多套账号。

### 新旧系统切换策略：

- **并行切换**：适用于**结算、票款核对、大客户协议价**等强一致场景——切换窗内新老双写或双查，财务/营收**日切对账**，差异清零后再关停旧入口；

- **分阶段切换**：适用于**多基地、多航站**广域 rollout——按区域或航线波次灰度，每阶段收集缺陷与容量反馈再扩大；

- **直接切换**：适用于**非资金型的展示、只读大屏、内部培训环境**等——选择低峰窗口，冒烟通过后启用，保留快速回滚包。

### 数据转换与迁移：

- **数据转换**：例如历史导出为**定界文本/专用报文**，经脚本**清洗**（去重、非法舱位、孤儿记录）、**字段映射**（旧旅客键→新主数据 ID、日期与时区规范化）、**校验**（条数、金额汇总与抽样比对）；

- **数据迁移**：**提前全量** + **切换前增量追平** + 切换后**一致性 SQL/对账任务**；必要时支持**回切**演练。

## 10.3. 架构的演化

随航线网络扩张与技术迭代，航空运营平台架构需持续演化：

1. **业务驱动**：新开**子公司航司、枢纽航站、联盟联运、货运子品牌**等——新增对应**限界上下文微服务**，扩展接入协议与多租户隔离，**K8s/中间件容量**与**多活/灾备**策略同步扩容；

2. **技术驱动**：在稳定前提下引入**服务网格**灰度与 mTLS、**无服务器**承载突发报表或批处理任务、**CSI** 与对象存储统一数据面，提升弹性与运维标准化；

3. **性能驱动**：当**年度经营明细、行程宽表**达百亿行级、交互查询变慢——采用**湖仓/离线预计算**（如夜间 Spark 作业写入分析库），热点 Redis/本地缓存；将在线查询从数十秒压到秒级；

4. **安全驱动**：落地**数据安全网关/数据库代理**防拖库与危险 SQL；**动态脱敏**与下载审批；**UEBA/态势感知**识别撞库、批量导出与异常 API 行为并自动阻断。
