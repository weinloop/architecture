# 8. 信息安全：贯穿平台全生命周期的“防护盾”

“航空运营智能管理平台”涉及“旅客隐私数据（身份联系信息、行程轨迹）、关键业务数据（订座库存、运价规则、运行态势）、高敏操作（大额退改、舱位批量调整、对外接口调用）”，需从“安全体系、系统可靠性”两方面构建安全防护，贯穿“设计、实现、部署、管理”全生命周期。

## 8.1. 安全体系：构建“多层防护、全面管控”

### 1. 数据安全和保密

#### 数据分类分级

数据可分为“核心数据”（如旅客证件号、支付标记、服务间高权限密钥、可改变库存或资金的指令）、“重要数据”（航线网络、联盟协议参数、运行保障节点数据）、“一般数据”（已发布的公开时刻表与营销文案）；核心数据采用“传输加密（如 TLS 1.3）+ 存储加密（如 AES-256 或 KMS 托管密钥）”，重要数据采用“存储加密 + 传输加密”，一般数据以访问控制与审计为主。

#### 数据脱敏

在界面、日志与客服工作台展示时脱敏，如手机号显示为 138****5678、证件号部分掩码、地址细化到城市/区县层级，避免敏感信息外泄。

#### 数据备份与恢复

核心库采用“定期全量 + 连续增量/日志”策略，备份介质存放于异地灾备（与主中心满足监管要求的地理隔离），按 RPO/RTO 目标组织恢复演练（如要求核心数据在约定时间内可恢复到一致点）。

#### 数据访问控制

遵循最小权限：例如“场站保障人员仅能查看本航站保障工单与关联航班”；“监管/稽核角色可访问脱敏后的汇总与抽样明细”；通过 **RBAC + 数据域（行/列级）权限** 落地。

### 2. 通信与网络安全

#### 网络分区隔离

通过防火墙与微分段将网络划分为“对外合作与渠道 DMZ、应用服务区、数据与中间件区、运维管理区”等，区域间默认拒绝，仅放行**最小端口与协议**（如 HTTPS、消息队列、数据库专用端口），禁止横向随意探测。

#### 入侵防御

部署 **IPS/下一代防火墙**，识别端口扫描、注入、撞库等异常；对高频失败登录与恶意 API 扫描源执行临时封禁与黑名单联动。

#### VPN 接入

远程运维经企业 **VPN（如 IPsec/SSL VPN）**，结合“账号 + 口令 + MFA/硬件令牌”，并与跳板机、堡垒机审计联动。

#### 安全审计

通过网络与应用层审计，记录关键管理操作与特权行为（如批量改舱、手工调账、字典导出），日志留存满足等保与民航数据留存要求，支持按时间、主体、资源追溯。

### 3. 系统访问控制设计

#### 身份认证

面向人的登录采用 **MFA**（口令 + OTP 等）；服务间调用经 **mTLS / 客户端凭证 / 短期 JWT**，由 API 网关联机校验Scope 与 audience；对外 API 可采用 **OAuth2.0 客户端模式** 与频控。

#### 权限管理

**RBAC + ABAC** 混合：RBAC 定义岗位（座席组长、运控值班、系统管理员等）；ABAC 增加环境属性（所属基地/航站、班组、数据owner），例如“座席仅能处理本航司+授权销售渠道的订单域数据”。

#### 会话管理

会话标识安全随机、服务端集中存储或签名 Cookie；Cookie 设 **HttpOnly、Secure、SameSite**；按角色设置**空闲超时**（普通座席较短、高敏管理员更短），并支持服务端强制下线。

### 4. 安全管理措施

#### 安全制度建设

制定《平台安全管理制度》《数据安全管理办法》《应急响应与重大活动保障预案》，明确巡检、备份、变更窗口与事件分级处置。

#### 安全培训

按季度对开发、运维、业务关键用户开展培训，覆盖 OWASP 常见 Web 风险、密钥与凭据管理、社会工程防范与事件上报通道。

#### 安全合规检查

对照**等保 2.0** 相应级别与行业要求，开展自查与第三方渗透/漏扫；对未授权接口、越权访问、敏感接口无频控等问题闭环整改并形成《安全合规报告》。

## 8.2. 系统可靠性：保障平台“持续稳定运行”

### 1. 可靠性指标应用

#### 平均无故障时间（MTBF）

对**航班动态接入与解析、订座核心路径**等关键软件组件，通过冗余实例、健康检查与故障域隔离提升 MTBF；对汇聚交换机、存储阵列等硬件选用工业级或电信级指标，降低硬件失效率。

#### 平均故障修复时间（MTTR）

建立**故障分级**：例如一级（全网交易阻断）要求最短 MTTR 与指挥长机制；二级（单域服务不可用）；三级（局部延迟或降级）；结合 Runbook、自动化编排与定期演练缩短修复时间。

#### 可用性度量（示意）

对可修复系统，常用 **可用性 ≈ MTBF / (MTBF + MTTR)**（或基于运行面板统计的年度可用分钟数）；需分别定义“旅客查询/购票链路”与“记账类服务”的 SLO，与高可用架构（多副本、跨可用区、自动切换）一致。

#### 系统可用性目标

示例：面向公众的查询类服务 **≥99.95%**（年化允许中断时间可换算为小时级）；核心交易与数据平面可要求更高（如 **99.99%**），通过集群、复制与仅保留只读降级等手段达成（具体数值以合同与监管为准）。

### 2. 系统可靠性分析

#### 单系统（串联）可靠性

针对“旅客查票流程（终端 → API 网关 → 查询服务 → 缓存/数据库）”，串联整体可用性约等于各组件可用性之积；通过提高各跳可用性（双活网关、服务多实例、库主从+RPO 控制）抬高端到端指标。

#### 并联系统

例如“站坪—中心”链路采用**双运营商专线 + VPN 备份**，或双活接入，只要一条可用即可转发，整体可用性高于单链路；数据库**主从/多副本**同理，主不可用时可提升从库为主或切换只读服务。

#### 冗余系统设计

对**高敏计费或超售校验**等模块，可讨论**三模冗余（TMR）**或“双机热备 + 判决”思路：多节点独立计算，多数一致则采纳，异常节点隔离，以降低单点软件缺陷带来的错误输出（实施成本需与业务关键度平衡）。

### 3. 影响可靠性的因素与应对

#### 硬件故障

磁盘阵列与健康巡检、预测性更换；关键节点避免单盘无冗余配置。

#### 软件漏洞与缺陷

静态检查（如 SonarQube）、依赖漏洞扫描、发布前压测与灰度；对内存泄漏、连接池耗尽等建立**基线告警**。

#### 网络中断

在机场或边缘侧**缓存离港/行李等关键报文**，网络恢复后补发与对账，避免数据空洞。

#### 人为操作失误

**Binlog/归档 + 时间点恢复**；高危操作**双人复核**与变更工单；定期恢复演练验证备份有效。

### 4. 冗余技术与软件容错技术应用

#### 冗余技术

##### 结构冗余

硬件双机、电源与链路冗余；软件层**多实例 + 负载均衡**，无单点实例承担全量流量。

##### 信息冗余

在关键报文或账单字段附加**MAC/HMAC 或校验和**，防止传输与落盘后被篡改。

##### 时间冗余

对**传感器类或离港重复报送**，可做滑动窗口聚合或重复报过滤，降低偶发尖刺导致的误策略。

##### 冗余附加

在订单、工单状态迁移处打印**结构化审计日志**（含 traceId），便于排障与监管取证。

#### 软件容错技术

##### 多版本表决（慎用）

对极少数**不可逆高额计算**（如特殊费用试算），可采用 N 版本独立实现与结果比对；需评估开发与运维成本，一般业务优先幂等与对账。

##### 恢复块方法

更新旅客档案或里程等时：**前置校验 → 执行 → 后置一致性检查**，失败则回滚到快照或补偿事务。

##### 防卫式程序设计

API 严格**参数校验、限流、超时与异常分支**，对依赖失败返回**可理解的错误码与降级提示**，避免进程级崩溃与信息泄露。
