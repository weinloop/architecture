# 11. 基础技术：数据库 —— 支撑平台数据存储与管理的核心

## 11.1. 数据库模式与关系模型

### 1. 数据库模式

“航空运营智能管理平台”可采用**三级模式两级映射** ——

- **外模式（用户模式）**：按角色定制视图与授权，如“地服视图”侧重**保障工单、登机口节点、航班实绩**；“营收/财务视图”侧重**票款、结算、报表口径**；“座席视图”在最小可见集内暴露**订座与退改相关字段**；通过视图与列级权限防止越权浏览；

- **模式（概念模式）**：建立企业级概念模型，划分**旅客域、订单与票联域、航班运行域、服务工单域、结算域**等，以 ER 图表达实体关系（如旅客与历史订单**一对多**，订单与航段明细**一对多**）；

- **内模式（存储模式）**：定义物理结构与存取路径，如海量**运行事件或审计流水**表按**日期/航班日**分区；核心交易表在旅客键、订单号、航班键上建**B+ 树/辅助索引**；明细分析表可结合列式压缩与 TTL；

- **两级映射**：外模式—模式映射保证业务视图变更可控制地影响基表；模式—内模式映射使**分区、索引、存介质升级**尽量不破坏应用 SQL 语义。

### 2. 关系模型与关系计算

概念上的关系代数（选择、投影、连接、聚合等）在实现层落地为 **SQL**；航司通常在事务型库中完成订票、支付、状态迁移，在分析型库中完成经营与运行统计。

## 11.2. 数据库设计与优化

### 1. 设计步骤

- **需求分析**：明确需持久化**旅客主数据、订单/票号、航班计划与变更、保障工单、支付与结算明细、审计日志**等；区分**高频点查/短事务**（锁舱、支付回调）与**大批量写入/扫描**（事件流水、日志）、**复杂统计**（客座、收入、延误结构）；

- **概念结构设计**：ER 图中实体可包括旅客、航班实例、舱位库存快照、订单、航段、工单等；厘清基数与生命周期；

- **逻辑结构设计**：化为关系表（如工单含**工单 ID、关联航班、类型、状态、处理人、SLA 时间**），选定主外键、类型（金额用 `DECIMAL`，时间统一含时区或utc_timestamp），避免悬浮列；

- **物理结构设计**：选型示例 —— 核心业务可用**高可用关系型集群**（商业库或 **MySQL / 兼容协议分布式库** 等）；海量时序/明细可用 **ClickHouse** 等引擎；设计**分区键、分片键、覆盖索引、物化视图**与留存策略；

- **数据库实施**：搭建主从/集群，执行 DDL、初始化字典数据（机场、舱位等级、机型），迁移基线；

- **运行维护**：监控 QPS、慢查询、复制延迟、空间增长；按变更流程演进 schema（如新增**常旅客标签**表）。

### 2. 范式设计与反范式设计

- **范式设计**：核心账务向 **3NF** 靠拢，消除传递依赖 —— 旅客基本信息与“常旅客等级历史”分表，避免在订单头重复堆叠可变属性；

- **反范式设计**：列表类查询（如**呼叫中心工单列表需旅客姓氏、航班号**）可在工单表冗余展示列，配合**异步刷新或触发器/事件**与主数据对齐，用一致性成本换读性能。

### 3. 优化手段

- **表设计**：大表按**业务日期或哈希**分区/分表，将扫描范围限制在单次查询涉及的少量分区；

- **索引**：高频条件建组合索引（如 `flight_id + dep_datetime`），避免过多二级索引拖累写入；必要时**覆盖索引**减少回表；

- **查询**：禁止无谓 `SELECT *`，分页用**游标/seek**替代深偏移；重型报表走**离线预聚合表**或 OLAP 引擎；

- **物理层**：SSD、缓冲池调优、并行度与线程池匹配 CPU；分析库调整 parts 合并与 max_threads 等参数（以厂商文档为准）。

## 11.3. 数据库控制功能

### 1. 并发控制

#### 问题解决

针对**丢失更新、不可重复读、幻读**，在 InnoDB/兼容引擎中常用 **MVCC + 行锁/间隙锁** 等机制 ——

- **丢失修改**：两事务改同一座位状态，提交时通过版本或锁冲突检测，失败方回滚或重试；

- **不可重复读**：报表事务内需要**可重复读**隔离级别，避免友方事务中途改账面；

- **幻读**：对在“未关闭订单”区间做统计时，可由引擎在相应范围加**间隙锁**（具体行为依隔离级别与实现而定）；

#### 锁机制应用

读多写少路径可共享读；改票、锁舱、调账走排他写；尽量**缩小锁粒度至行级**，缩短事务跨度。

### 2. 完整性

- **实体完整性**：主键非空唯一（订单 ID、工单 ID、传感器/设备 surrogate key 等）；

- **参照完整性**：订单行的旅客 ID、航班外键存在且策略明确（RESTRICT / CASCADE 按业务）；

- **用户定义完整性**：CHECK（金额≥0、舱位代码在枚举内）、默认值（工单初始状态）、非空（票号、支付流水关键列）。

### 3. 安全性

- **授权**：**RBAC** 到库表/列级 —— 开发账号只读预发、运维只读生产或经由跳库、DBA 全流程审计；

- **加密**：敏感列 **TDE** 或应用层加密；链路 **TLS**；

- **审计**：记录**登录、DDL、高频导出、特权 DML**，留存满足等保与民航审计周期。

### 4. 备份机制与恢复验证

#### 备份机制

- **全量**：低峰窗口物理/逻辑全备；

- **增量/日志**：开启 **binlog / 连续归档**，支持时间点恢复；

- **验证**：定期在隔离环境**挂载备份做 spot-check**（行数、checksum、关键查询）。

#### 恢复验证

- **时间点恢复**：误删批数据时，用全备 + 日志回放到故障前一事务点；

- **故障切换**：主库故障切换从库或选举新主，明确 **RTO/RPO** 指标并在演练中达标。

## 11.4. 常见的数据存储技术与其他相关技术

### 1. 常见数据存储技术

#### 关系型数据库

承载**订单、旅客、结算、工单**等强一致事务与复杂 SQL；选型侧重**ACID、HA、容量线性扩展能力**。

#### 分布式数据库

通过分片/分区把热表分布到多节点，横向扩展连接数与吞吐；订单、行程宽表等可按旅客或订单号哈希分片。

#### 非关系数据库

- **时序/分析型列存（如 ClickHouse）**：承接**运行事件、接入日志、指标明细**，支撑时间窗聚合与大盘；

- **图数据库（如 Neo4j，可选用）**：表达**经停串飞、共享航班、地面保障资源拓扑**，便于“一点故障/限载影响面”图遍历；

- **缓存（Redis）**：**会话、热点航班动态、限流计数、短期舱位快照**，削峰保护核心库；

- **对象存储（MinIO/S3 兼容）**：**电子档案、检修影像、导出报表、训练样本**等大对象，多副本与生命周期策略。

### 2. 其他相关技术

- **数据仓库**：整合 ODS/DWD/DWS，用 **Hive/Spark/SQL-on-lake** 做**航线客座、票价舱、延误成本**等主题分析；

- **数据挖掘**：识别**异常囤票、退改套利、代理人异常模式**；关联**天气流量限制与航班波延误**规则发掘；

- **向量与检索增强（可选）**：把延误前传感器序列、保障节点事件序列编码为向量，做**历史相似日**检索，辅助運力与预案；亦可对接规章问答等场景（与模型训练单独论证）。
