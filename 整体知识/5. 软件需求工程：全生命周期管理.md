# 5. 软件需求工程：全生命周期管理

软件需求工程是“系统规划与分析”的深化，覆盖“需求获取、分析、规格、确认、管理”5 步，确保需求“无遗漏、可落地”。

## 5.1. 需求分类：明确需求层次

按“业务需求、用户需求、系统需求”三级分类：

- **业务需求**：航司管理层“提升航班运行与地面协同效率、保障关键链路高可用与合规、提高旅客全旅程满意度”（高层目标，可绑定可度量指标由甲方定义）；

- **用户需求**：座席人员“在工作台实时查看舱位与运价规则并处理退改签”；地服人员“接收航班变动任务并与登机口资源联动”；旅客“通过 App 查询行程并接收延误/改签通知”（用户视角的功能诉求）；

- **系统需求**：接入层“支持与既有订座/离港或第三方渠道约定的报文或 API 协议，达到约定的峰值吞吐与超时重试策略”；通知服务“支持按渠道模板与频控策略完成批量触达”（技术层面的具体要求）。

## 5.2. 需求获取：多方式收集需求

结合项目实际，采用 5 种获取方式：

1. **用户访谈**：与运控、营销、地服负责人访谈，明确“运行指挥中心看板需实时展示哪些 KPI（如航班正常性、延误结构、关键航路限制）”；

2. **问卷调查**：向常旅客或试点旅客发放问卷，收集“偏好通知渠道（App 推送 / 短信 / 微信）、是否需要行程一键分享与电子凭证”；

3. **现场观察**：到值机柜台与登机口观察高峰保障流程，发现“手工核对名单耗时、需与离港状态自动回填一致”；

4. **原型演示**：制作“航班动态 + 座位图/航班列表”低保真原型（如 Axure / Figma），让业务方直观反馈“是否需要展示共享航班与代码共享标识”；

5. **文档分析**：分析航司《运行手册》《旅客服务标准》《现有系统接口说明》及监管报送要求，提取“不正常航班处置需记录关键时间节点与责任主体”“退改签审计字段必须可追溯”等需求。

**需求记录**：用“任务卡片”记录用户需求，例如“卡片编号：REQ-001；需求描述：地服可对特服旅客发起保障工单并全程跟踪状态；优先级：高；关联用户：机场地服组长”。

## 5.3. 需求分析：用方法梳理需求

采用“结构化分析（SA）+ 面向对象分析（OOA）”结合的方式：

- **SA 方法**：用“数据流图（DFD）”梳理“旅客购票支付流程”（旅客发起下单 → 订座服务锁舱/验价 → 支付网关扣款 → 订单与票号落库 → 触发行程通知与积分异步任务），用“数据字典”定义“订单表（字段：订单 ID、旅客标识、航段、舱位、金额、状态、审计时间戳）”；

- **OOA 方法**：用“用例图”描述“座席角色”的用例（查询航班、预订、变更、退票、处理异常订单），用“类图”定义核心类（如“行程变更请求类”包含属性：请求 ID、原航班、新航班、规则命中结果，方法：校验规则、生成费用试算、提交审批）。

## 5.4. 需求规格、确认与管理：确保需求落地

- **形成需求规格**：输出《系统需求规格说明书》，明确“功能需求（如订座服务在目标峰值 TPS 下保持库存一致性与幂等）、非功能需求（如系统可用性≥99.9%、关键 API P99 时延）、接口需求（如与支付机构、离港系统、企业 IAM 单点登录的接口与安全要求）”。

- **需求确认与验证**：组织“甲方、需求分析师、架构师”召开需求评审会，通过“需求用例测试”验证需求可落地（如“用例：旅客发起退票，验证是否包含‘不可退原因说明与费用明细’”）。

- **需求管理**：用“需求跟踪矩阵”关联“需求 - 设计 - 测试用例”，例如“REQ-001（地服保障工单）”对应“设计文档中的‘工单与状态机设计’”和“测试用例 TC-001（验证工单创建、派工与关闭）”，需求变更时同步更新矩阵。
