# 9. 软件实现与测试：从“代码”到“可用系统”的落地

## 9.1. 软件实现综述：技术选型与开发常识

### 1. 技术选型适配项目场景：

- **Web 系统（如运行指挥中心、统一运营门户）**：前端可采用 **Vue 3 + Element Plus**（组件化与大屏可视化），后端采用 **Spring Boot + Spring Cloud Alibaba**（注册发现、配置、限流熔断），适配多浏览器与高峰访问（订座/查询洪峰由容量规划定义）；

- **移动 APP（旅客服务/地服保障 App）**：可采用 **Flutter** 等跨端方案降低 iOS/Android 双栈成本，接口 **REST/JSON**，支持**弱网离线草稿**（本地队列待网络恢复后同步，如现场保障备注、离线扫码缓存）；

- **高并发接入层（渠道/报文接入）**：可采用 **Netty** 等异步 IO 框架承载长连接与高并发报文接入，协议侧按项目对接**离港/订座报文、HTTPS API、消息中间件**等形态，由适配层完成编解码与背压；

- **大数据与可视化（经营分析、运行分析）**：**Spark** 承担离线汇总（如日线客座、收入口径报表），**Flink** 承担实时指标（如延误结构实时看板）；存储可选用 **ClickHouse**（明细与指标）、**Elasticsearch**（订单/工单/事件全文检索）；可视化采用 **ECharts、DataV** 等呈现曲线、柱状、地图航线等。

### 2. 开发常识落地：

- **Git 版本控制**：采用 **Git Flow** —— `master/main` 存稳定发布，`develop` 集成，`feature/*` 开发功能（如 `feature/boarding-offline-queue`），`hotfix/*` 紧急修复；提交信息规范化（如 `fix: 出票成功后积分异步任务未投递`）；通过 **GitLab CI** 在提交后触发单元测试；

- **构建工具**：前端 **npm/yarn + webpack/vite**；后端 **Maven/Gradle**，镜像 **Docker**（如 `aviation-order-service:1.3.0` 标签可追溯）；

- **开源协议**：选用 **Apache 2.0、MIT** 等兼容商业使用的协议，法律合规审查纳入物料清单（SBOM）；

- **RESTful 规范**：资源导向设计，如查询旅客行程 `GET /api/v1/passengers/{id}/trips`，创建服务工单 `POST /api/v1/service-tickets`；响应体统一信封 `{"code":200,"message":"OK","data":{}}`，错误码按域分段（如订单域 2xxx、库存域 3xxx）。

## 9.2. 软件测试的组织和部署：系统化验证系统质量

### 1. 测试过程：

- **需求分析**：梳理例如“地服保障子系统”的验证点：“保障人员可扫码关联航班节点”“弱网下任务草稿可补传”，输出《测试需求说明》；

- **制定测试计划**：范围（功能、性能、兼容、安全抽检）、资源（人员、设备、环境套数）、节奏（功能 N 天、性能 M 天、回归 K 天）、风险（测试与生产拓扑差异）及应对（对齐中间件版本与数据规模），输出《测试计划》；

- **设计测试文档**：编写用例如“TC-001：扫码绑定登机口任务——前置：已登录且任务下发；步骤：打开任务页→扫码→提交；期望：状态为已签到且后台时间戳一致”；准备测试数据（合法旅客记录、失效二维码、边界舱位）；

- **构建测试环境**：硬件/虚拟机、旅客端真机矩阵、报文仿真工具；部署全量微服务与依赖，数据采用**生产脱敏副本或合成数据**；网络模拟专线延迟与抖动；

- **执行测试**：顺序 **单元→集成→系统→验收**；单元测试由开发用 **JUnit/Pytest** 覆盖状态机、规则试算等核心逻辑；集成测试验证服务间契约（如“订单服务下单成功后通知服务收到事件”）；系统测试跑通端到端（如“旅客支付出票→行程展示→推送触达”）；验收由业务方按标书逐条确认；

- **缺陷管理**：用 **Jira** 记录标题、严重级别、复现步骤、期望/实际、日志与截图；流程**提交→指派→修复→验证→关闭**；周期性《缺陷统计》（模块分布、逃逸率）。

### 2. 测试管理：

- **配置管理**：测试用例与自动化脚本入 Git；**Ansible/Terraform** 等描述测试环境，实现可重复搭建；

- **过程管理**：测试日会同步进度与阻塞；用 **TestRail** 等跟踪用例执行率与阻塞原因；

- **缺陷 SLA**：致命/严重缺陷优先窗口内修复；轻微 UI 文案可排入下一迭代；双周复盘根因（如“未校验空舱位导致缺陷占比高”→ 强化契约测试与静态规则）；

- **风险管理**：数据不足→ **Mock/合成数据**；第三方接口不稳→ **Mock Server** 或录制回放；维护《测试风险清单》。

### 3. 软件部署：

- **环境准备**：生产主备或双活逻辑机房；**K8s** 集群、中间件高可用；出入口 **负载均衡/WAF**；渠道与内部管理区分域；

- **软件安装**：**CI 构建镜像 → Harbor → Helm/K8s 发布**；执行 DDL/DML 基线脚本与配置中心密钥轮换；重要变更前备份；

- **测试验证**：上线后**冒烟**（登录、核心购票/退改链路、关键管理操作）、**容量抽检**（目标并发下 P99 与错误率）、**安全复检**（高危漏洞清零策略）；通过后签发《部署验证报告》。

## 9.3. 软件测试类型与方法：全方位验证系统质量

### 9.3.1. 软件测试类型：

#### 按测试对象划分：

- **功能测试**：验证“不正常航班处置”“自愿改期试算”等是否符合需求，如“能否按规则冻结舱位并生成费用明细”；

- **性能测试**：用 **JMeter/Gatling** 模拟目标并发查询/下单，观测响应时间、TPS、资源水位；对接入层可做报文洪峰或长连接保持测试；

- **安全测试**：**OWASP ZAP** 等对开放 API 做注入、越权、XSS 抽检；验证敏感字段密文存储与密钥轮换；

- **兼容性测试**：旅客 App 在主流 iOS/Android 版本与机型；Web 在 Chrome/Edge/Firefox 指定版本上的关键路径；

- **界面测试**：运控大屏信息层级、告警颜色与可读性、组件对齐与分辨率自适应；

- **易用性测试**：邀请**座席、地服**完成高频任务路径，收集可理解性与步长反馈；

- **稳定性测试**：对消息消费、接入网关等执行**长跑 soak**（如 72 小时），观察内存、FD、线程池与重试积压。

#### 按测试阶段划分：

- **单元测试**：如“订单状态机”从已创建→已支付→已出票 的合法迁移与非法回退被拒；

- **集成测试**：验证“支付回调与订单域”参数与幂等；验证“航班动态服务至缓存与看板”延迟与字段一致性；

- **系统测试**：端到端例如“旅客应用端发起退改→规则与支付→通知→积分异步”，或“呼叫中心建单→班组处理→回访闭环”；

- **验收测试**：由航司方组织运控、营销/座席、财务、旅客代表等，对照《需求规格》执行：**峰值购票、结算报表、行程通知**等用例，**通过率与不严重缺陷门槛**按合同约定判定。

#### 按被测软件划分：

- **APP 测试**：弱网队列、扫码准确性、后台推送到达、电量与唤醒、权限与隐私弹窗；

- **Web 测试**：运控多视图同开是否卡顿、地图/时间轴操作是否流畅、长时间值守是否泄漏；

- **接入与中间件测试**：协议兼容性、报文频率与背压、断线重连与**断点续传/对账**。

#### 其他分类：

- **回归测试**：修复“分配后工单状态未刷新”类缺陷后，重跑订单/工单相关批量用例；新增强报表后回归老报表口径；

- **冒烟**：每版先跑**登录、查票、下单/出票、关键管理入口**等少量高价值用例，失败则停止全量测试并查发布与环境。

### 9.3.2. 软件测试方法：

#### 按被测程序是否可见划分：

- **黑盒测试**：如“常旅客注册/会员升级”仅通过输入输出验证；方法：**等价类**（手机号/证件格式）、**边界值**（姓名长度、乘机日期边界）、**场景法**（首乘、携童、联程）；

- **白盒测试**：如“退改费用规则或舱位保护判断”阅读分支，设计用例覆盖**判定/条件/路径**（满足规则与不满足规则的上下界）；

- **灰盒测试**：已知“航班列表查询需传入日期范围与航段”，不了解内部缓存策略；组合合法/非法参数并核对返回与审计日志是否记录异常访问。

#### 按是否需要执行被测程序划分：

- **静态测试**：评审需求/设计中的歧义（如“不正常航班统计口径”是否唯一）；**SonarQube** 扫描代码异味与安全模式；

- **动态测试**：实际驱动系统，测量延误工单派发时延、通知到达率、全链路数据是否闭环一致。

#### 自动化测试：

- **单元自动化**：JUnit 等接入 CI，阻断破坏核心逻辑的提交；

- **接口自动化**：**Postman/Newman** 或 **RestAssured** 批量回归订单、票号、退改试算接口；

- **UI 自动化**：**Selenium/Playwright** 录制运控或后台“登录→打开航班视图→导出报表”等黄金路径；

- **报文/协议自动化**：用脚本或工具**模拟离港/渠道报文批量入账**，校验解析、入库与监控指标。
