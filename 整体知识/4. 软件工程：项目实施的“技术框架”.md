# 4. 软件工程：项目实施的“技术框架”

软件工程指导“航空运营智能管理平台”的开发管理，核心是“软件生命周期、开发方法与模型、过程管理”的落地。

## 4.1. 软件生命周期：对应项目阶段

按“规划→需求→设计→实现→测试→部署→维护”7 阶段推进，与项目进度深度契合：

- **规划阶段**：对应“系统规划与分析”，确定项目可行性与目标；

- **需求阶段**：对应“系统分析”，输出《需求规格说明书》；

- **设计阶段**：对应“系统设计与架构设计”，输出《系统设计方案》《架构设计文档》；

- **实现阶段**：开发各子系统（如订座与订单、航班运行与通知、旅客 App/小程序、管理端工作台），编写代码与单元测试；

- **测试阶段**：执行功能测试（如“航班变动后的舱位冻结与退改签规则是否一致”）、性能测试（如“高峰时段核心链路在目标并发下是否满足时延与错误率门禁”）；

- **部署阶段**：发布至容器云或虚拟化资源池（含网关、注册配置、可观测组件）、按蓝绿/金丝雀策略滚动发布应用软件、数据迁移（将历史订单、常旅客档案、航班计划主数据导入或同步至新平台）；

- **维护阶段**：对应项目“运维服务”，提供 7×24 小时故障处理、容量与成本优化、版本迭代（如运控大屏指标口径调整、新机型/新航站扩展）。

## 4.2. 开发方法与模型：选择“增量迭代模型”

因项目子系统多（跨订座、运行、旅客、检修、数据等多域）、需求复杂度高，选择“增量迭代模型”（时间以批复为准）：

- **增量 1（示例：2024 年中前）**：建设集成与安全访问、消息与缓存等共性能力，打通“主数据 + 核心订座/订单 + 基础航班动态”最小可用链路；

- **增量 2（示例：2024 年下半年–2025 年中）**：扩展旅客全旅程触达（行程、通知、值机相关能力）、地服协同接口，形成可运营的端到端闭环；

- **增量 3（示例：临近上线窗口）**：完善检修预警与维保工单联动、运营分析与管理驾驶舱，并预留智能化（如策略推荐、异常检测）接入点；

- **迭代优化**：每个增量上线后，收集运控、座席、地服与旅客反馈（如“App 值机步骤在弱网下超时”），在后续迭代中优化体验与鲁棒性。

## 4.3. 软件过程管理：标准化开发流程

- **制定过程规范**：明确“需求评审流程”（需求分析师 → 架构师 → 甲方代表签字确认）、“代码评审流程”（开发工程师 → 技术负责人评审）、“测试流程”（测试工程师编写用例 → 执行测试 → 提交缺陷 → 开发修复 → 回归测试）。

- **过程监控**：使用“项目管理工具（如 Jira）”跟踪任务进度，例如“统一网关与遗留订座系统 OAuth 联调”任务需在约定窗口前完成，逾期需预警并分析依赖与风险。

## 4.4. 软件重用与再工程：降低开发成本

- **软件重用**：复用成熟组件，例如“统一门户的单点登录（SSO）”采用开源组件 Keycloak 或企业 IAM，“运行与经营看板”复用 ECharts 等可视化库，消息与搜索等选用成熟中间件客户端，减少重复开发。

- **再工程**：若航司原有老旧订座或离港周边系统仍有业务价值，通过“逆向工程 + 适配层”梳理其报文/接口与时间窗约束，改造后接入新平台的事件总线与 API 网关，避免推倒重来。
