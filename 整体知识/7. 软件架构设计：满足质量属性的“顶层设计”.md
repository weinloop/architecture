# 7. 软件架构设计：满足质量属性的“顶层设计”

软件架构设计是在“系统设计”基础上，进一步解决“如何满足高可用、高并发、可扩展”等质量属性问题，结合“航空运营智能管理平台”的分布式、多模块特点，落地架构相关知识点。

## 7.1. 软件架构风格选择：适配项目场景

根据项目不同子系统的功能与质量需求，选择合适的架构风格：

1. **数据流架构风格**：应用于“航班与运行数据采集/分发类子系统”—— 数据从“外部离港报文、运行系统、渠道订单事件（数据源）”流入，经“协议解析 → 清洗对齐 → 持久化/仓储 → 订阅分发”等处理步骤，最终流向“运行看板、订座缓存、旅客通知”等下游，采用“管道 - 过滤器”模式；每个处理步骤为“过滤器”，数据通道为“管道”，支持“并行解析多路报文、可替换过滤器（如切换更高效的规整算法）”。

2. **调用返回架构风格**：应用于“统一门户与后台管理类子系统”—— 采用“分层架构”，从下至上分为“数据访问层（仓储与事务边界）、领域/业务层（权限、主数据、策略配置）、应用/接口层（REST/内部 gRPC）、表现层（Web / 移动 BFF）”，层间遵循“单向依赖”，便于分层测试与分层演进（如单独优化 SQL 与索引而不波及表现层）。

3. **以数据为中心架构风格**：应用于“经营分析/运行分析报表子系统”—— 以“数据仓库或湖仓一体存储”为核心，报表、即席查询、导出等模块围绕统一数据模型展开，负责“多源汇聚（订座、运行节点、结算、旅客行为等）、按主题域建模（如‘收入与座公里’‘航班正常性’‘旅客服务’）、面向分析的存储与物化视图”，业务通过“受控 SQL / 指标 API”访问，支持灵活报表定制与口径统一。

4. **面向服务的架构风格（SOA）**：应用于“跨子系统协同场景”—— 将核心能力封装为服务，如“身份与会员服务、订单与票联服务、运行事件服务、告警与工单服务”，通过“企业服务总线（ESB）或现代化 API 网关 + 服务注册中心”完成路由、策略与观测；例如“不正常航班闭环”—— 身份服务完成主体鉴别 → 工单服务创建保障任务 → 告警服务通知相关班组 → 订单服务触发退改规则试算，SOA 强调服务复用与对调用方的稳定契约。

5. **微服务架构风格**：作为 SOA 的延伸，应用于平台整体 —— 将“订座与库存、支付与结算、航班动态、旅客触达、检修协同、分析与报表”等域拆为可独立部署的微服务，按域划分数据所有权（Database per Service 或清晰的分库边界），通过“API 网关（如 Spring Cloud Gateway）”统一入口，结合“服务注册与发现（如 Nacos）、配置中心、限流熔断（如 Sentinel）”等，满足“高可用（单服务故障隔离）、可扩展（购票高峰单独扩容）、易维护（团队按域负责）”。

## 7.2. 分布式系统：支撑平台的“底层架构”

航空运营平台涉及“多基地多系统协同、高峰交易与时序数据、分析与检索”，需基于分布式架构实现：

### 1. 分布式系统优势落地：

- **高可用性**：采用“多副本与选主/编排自愈”—— 关键中间件与有状态组件按生产规范部署主从或集群；核心事务库（如 MySQL）采用“主从复制 + 读写分离”，写主库、报表与分析走从库或只读实例，兼顾可用与读扩展。

- **可扩展性**：采用“水平扩展”—— 春运或促销尖峰时，通过 Kubernetes 自动扩容订座/查询类微服务实例与网关处理能力；对象存储采用分布式集群，按需增加节点扩展容量与安全冗余。

- **高性能**：采用“并行计算与缓存”—— 海量日志、行为轨迹或运行事件的离线汇总可交由分布式计算框架处理，缩短批处理窗口；分布式缓存（如 Redis 集群）承载“热点舱位查询、航班状态摘要、会话与限流计数”，降低核心库压力。

### 2. 分布式系统挑战与解决方案：

- **挑战 1：网络通信故障**：采用“有限重试 + 超时 + 熔断降级”（如客户端重试策略、服务端超时、Sentinel/Resilience4j 熔断），避免级联阻塞。

- **挑战 2：数据一致性**：对“支付成功与出票状态”等强一致路径采用“分布式事务或业务补偿（如 Seata AT/TCC、或幂等号 + 对账）”；对“积分、通知、统计看板”等采用“事件驱动 + 最终一致性 + 重试/死信”，保证可恢复与可审计。

- **挑战 3：分布式锁**：对“同一航段舱位扣减、同一订单重复提交”等场景，采用“Redis/协调服务实现的分布式锁或数据库乐观锁”，设置合理租约与幂等，避免死锁与双花。

- **挑战 4：节点负载不均**：在网关与服务间采用“负载均衡 + 健康检查”，在 K8s 中结合资源请求/限制与亲和性策略，将 Pod 调度到合适节点，并配合 HPA 按 CPU、QPS 或自定义指标伸缩。

### 3. 常见分布式系统组件应用（示例选型，以项目批复为准）：

- **分布式数据库与存储引擎**：关系型数据库承载订单、旅客档案、审计等核心业务；分析型或列式引擎承载指标汇总、明细下钻；时序或宽表存储承载大规模运行事件；按需要采用“分库分表、读写分离、冷热分层”。若有强空间检索需求（如机场范围、航线走廊），可采用支持空间扩展的引擎。

- **分布式文件与对象存储**：存储“机型手册与培训资料、检修影像附件、大屏素材、报表导出物”等大文件，采用分片与多副本，暴露兼容 S3 的 API，便于影像云平台与 BI 调用。

- **分布式缓存**：Redis Cluster 等缓存登录态、热点航班动态、防刷计数与短期票务会话，通过分片提升集群吞吐。

- **分布式消息**：采用 Kafka 或 RocketMQ 等实现异步解耦，例如“支付出票成功事件 → 积分、通知、数据仓库 CDC”多订阅；配置重试、死信与幂等消费，避免丢消息与重复入账。

### 4. 云计算与云原生集成：

- **分布式与云计算的关系**：可采用“专有云 / 行业云 + 公有云弹性算力”的混合形态—— 核心交易与旅客隐私数据落在受控环境，离线分析、模型训练等可弹性使用公有云资源，通过专线/VPN 与统一身份、密钥与审计打通。

- **服务方式与部署方式**：对内提供 PaaS 能力（容器、中间件、可观测），对业务单位提供门户化 SaaS 体验；微服务以镜像交付，由 Kubernetes 滚动/金丝雀发布，降低停机窗口。

- **与云原生集成**：可选服务网格（如 Istio）做流量灰度、mTLS、熔断与指标旁路；GitOps（如 Argo CD）配合 CI 实现构建—部署流水线，并与链路追踪（如 SkyWalking、Jaeger）联动定位慢调用。

## 7.3. 软件架构实现：从“设计”到“落地”的流程

基于平台架构需求，可采用“基于架构的软件开发（ABSD）”思路，通过下列过程落实架构（步骤划分可结合组织裁剪）：

### 1. 体系结构需求：

明确“功能需求”（如峰值订座 TPS、日均消息量、对接系统数量）与“质量属性”（可用性、时延、RTO/RPO、合规、未来 3～5 年演进球），输出《架构需求说明》，并由业务、运维、开发等干系人评审确认。

### 2. 体系结构设计：

结合 7.1 风格与 7.2 分布式方案，产出“逻辑/物理/部署/数据流”等多视图；明确**关键技术选型**（微服务框架、网关、消息、缓存、数据库与观测栈等）及边界。

### 3. 体系结构文档化：

沉淀 ADR（为何选微服务、为何选某种消息中间件）、OpenAPI/Proto 契约、部署拓扑与容量模型，保证实施与运维对架构共识一致。

### 4. 体系结构复审：

从**质量属性达成度、团队技能与供应商风险、演进成本**（如新增联盟运价、国际航段、子公司多租户）等维度评审；对缺口（如跨可用区网络分区下的补偿策略）迭代设计直至闭环。

### 5. 体系结构实现：

- **基础设施**：部署 K8s、消息、缓存、数据库与高可用链路，接入统一监控告警（如 Prometheus + Grafana，对 CPU、内存、错误率、P99 设阈值）。

- **业务微服务**：可按 DDD 划分限界上下文，例如“订单服务”内含聚合、领域服务与防腐层，对接外部订座报文；通过 CI/CD 发布并进行契约测试与集成测试。

### 6. 体系结构演化：

- **业务变化**：新开基地/子公司航司时，以新增服务或租户隔离扩展，尽量不伤及核心交易域。

- **性能瓶颈**：报表类查询改为“离线预汇总 + 物化表/宽表”，将交互式查询从分钟级降到秒级。

- **技术升级**：中间件与框架版本通过灰度与回滚策略滚动推进。

### 7. 其他架构实现方法应用：

- **基于构件的软件工程（CBSE）**：复用 OAuth2/OIDC 认证、校验框架、日志与审计 SDK 等。

- **中间件扩展**：如用 Elasticsearch 做工单/事件全文检索；用调度平台执行换季任务、日终对账与数据质量巡查。

- **特定领域参考**：参考民航客运、销售与运行领域的成熟参考架构与行业标准接口实践，降低领域风险。

## 7.4. 软件架构质量属性：保障平台“好用、可靠、安全”

平台需满足多维度质量属性，通过“质量属性场景”细化需求，通过“架构评估”验证设计有效性：

### 1. 系统质量属性与场景描述：

- **可用性**：例如“航班动态汇聚服务主实例异常时，流量在约定时间内切换至健康副本，短暂抖动期内事件不丢（数据面缓存或消息队列堆积可恢复）”。

- **性能**：例如“购票高峰段支持目标并发与订单链路 P99 时延；API 网关与会话热点通过扩容与缓存达标”。

- **可扩展性**：例如“新增航站、航线波次或新销售渠道时，以配置与服务扩展为主，核心模型无需推倒重来”。

- **安全性**：例如“未授权主体不得调用运行控制或大额退款类接口；旅客敏感字段加密存储与脱敏展示；全链路审计满足监管要求”。

- **可维护性**：例如“线上缺陷可通过调用链在约定时间内定位到服务与版本，并通过独立发布与回滚修复”。

### 2. 软件架构评估：

- **评估方法**：可采用 SAAM/ATAM 等基于场景的方法，组队覆盖架构、开发、运维、甲方代表。

- **评估过程**：针对可用性、性能、安全等设计典型故障与滥用场景（报文积压、依赖超时、密钥泄露、误配置扩缩容），识别风险与权衡点。

- **评估结果**：形成《架构评估报告》：哪些质量目标已覆盖、哪些需加固（如跨集群对账）、改进项纳入变更 backlog。
