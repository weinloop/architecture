# 2. 系统规划与分析：项目的“源头设计”

系统规划与分析是项目启动的核心，决定了项目“为何做”“为谁做”“做什么”，对应“航空运营智能管理平台”的立项背景与需求边界定义。

## 2.1. 项目提出：明确立项价值与动机

结合项目背景，本阶段需回答“为什么要建航空运营智能管理平台”，对应知识点中的“项目立项目标、动机与价值判断”。

### 行业痛点驱动

航司侧长期存在多套业务系统烟囱式建设，“数据与流程割裂”，航班动态、票务、旅客服务、检修维保之间协同依赖人工传递与多次登录，故障影响面大；节假日、恶劣天气导致航班大面积变动时，高峰并发与瞬时通知量陡增，传统架构难以稳定承载，易出现“查询拥塞、通知滞后、座席与地服口径不一致”等问题，亟需通过统一平台、实时数据链路与智能化能力实现“可观测、可协同、可扩展”的运营。

### 政策与城市发展需求

国家《"十四五"民用航空发展规划》《智慧民航建设路线图》等推动民航数字化、智慧化升级；项目需服务“运控与航班运行、营销与座席、机场地服与检修、管理决策与数据分析、旅客与常旅客”等多类核心角色，实现“航线航班与票务交易一体化管控、旅客全旅程便捷服务、运行与检修协同预警”的价值。

## 2.2. 系统规划步骤：落地立项目标

按“初步调查→定系统目标→分析子系统→拟定实施方案→可行性研究→制定建设方案”6步流程推进：

### 1. 初步调查

调研航司运营现状（航线网络与基地分布、日均航班与旅客量、现有离港/订座/结算类系统）、现有系统（老旧接口、批量文件交互、移动端能力）、各部门需求（运控需“统一航班视图与变更下发”，座席需“高并发订座与退改签”，地服需“值机与行李协同”，机务需“检修工单与预警闭环”，旅客需“行程查询与变动通知”）。

### 2. 定系统目标

核心目标为“一平台、全链路、全场景智能运营”——构建统一航空运营智能管理底座，贯通“市场—订座—运行—地面—检修—旅客触达—决策分析”，支撑高峰容量下的稳定交易与分钟级运行响应。

### 3. 分析子系统

拆分出“统一门户与集成、票务与订单、航班运行与资源、旅客服务与行程触达、检修预警与维保协同、数据智能与分析报表、技术中台（消息/缓存/检索等）、运维与可观测”等若干子系统（与批复建设范围及功能清单对齐），明确各子系统边界与协同关系。

### 4. 拟定实施方案

确定“先建统一集成与安全访问、再上线核心业务、再叠加分析与智能化能力、最后组织联调与割接”的实施顺序，例如优先打通主数据与航班/订座主链路，再扩展检修与数据服务平台。

### 5. 可行性研究

从四维度验证项目可行性：

- **经济可行性**：按批复建设投资规模，对比分散系统重复建设、重大故障与业务中断损失、人工对账与协同成本，预计可通过降低运维复杂度、提升运力利用与服务质量，在规划周期内形成可量化的投资回报。

- **技术可行性**：分布式微服务、消息驱动、云原生与成熟中间件技术已在大型民航与互联网业务中验证；高并发交易、海量日志与可观测体系有成熟参考架构可复用。

- **法律可行性**：符合《网络安全法》《数据安全法》《个人信息保护法》及民航领域数据与审计相关要求；旅客身份信息、行程与支付数据需分类分级保护与全链路审计，跨主体数据共享需满足合规约定。

- **用户使用可行性**：运控、座席、地服、机务等角色可通过岗位培训分级授权使用 Web 与移动工作台；旅客通过 App/小程序等渠道使用，学习成本低。

### 6. 制定建设方案

输出《系统设计任务书》，明确建设范围（如覆盖主运营基地与协作机场网络、对接既有离港/订座或适配接口）、交付物（应用软件、接口规范、部署与运维手册、培训材料）、里程碑节点（示例：2024 年 3 月启动建设并完成现状调研与集成规划，2025 年 8 月核心系统正式上线，上线后组织并行切换与稳定性观察；以项目批复为准）。

## 2.3. 系统分析：定义需求边界

通过“问题分析、业务流程分析、数据流程分析”，输出《系统需求规格说明书》：

### 问题分析

识别核心问题——“航班变动与库存状态无法多端一致同步”“检修与运行信息依赖电话/表格、难以追溯”“高峰时段购票与查询易拥塞”，并转化为需求（如“航班动态与订座需支持订阅推送与多渠道通知”“检修预警需与工单、航材与排班联动”“核心业务需支持弹性容量与降级策略”）。

### 业务流程分析

梳理关键业务流程，例如“航班大面积延误处置流程”：运行监控触发告警 → 运控确认调整方案 → 系统冻结/释放舱位与触发退改签规则 → 座席与自助渠道联动 → 向旅客与机场地服下发一致口径的通知与服务指引 → 事后复盘与报表，为后续数据流程设计提供依据。

### 数据流程分析

绘制数据流程图（DFD），例如“航班动态与订座数据汇聚流程”：外部与内部数据源（运行态势、订座记录、机场保障节点）→ 接入与协议适配层 → 数据清洗与主数据对齐 → 实时计算与缓存/检索服务 → 运控看板与旅客触达 → 异常告警与审计流水，为数据库设计（如“航班计划与实绩表”“订单与变更日志表”“消息投递状态表”）奠定基础。

## 2.4. 与论文结合：项目背景的核心素材

论文中“项目背景”部分可直接复用本阶段内容，重点突出：

- “为什么有”：即传统烟囱式系统与峰值场景下的运营痛点，以及智慧民航政策与服务质量提升要求（立项价值）；

- “为谁提供什么服务”：为运控与决策层提供“统一运行视图与预案联动”，为营销与座席提供“高并发订座与一致库存”，为地服与机务提供“任务协同与预警闭环”，为旅客提供“行程查询、值机与变动通知”（系统分析阶段确定的用户与服务）。
