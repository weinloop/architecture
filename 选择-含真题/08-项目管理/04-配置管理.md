# 配置管理

**知识点编号**：8.4  
**权重**：次重点 ★★★☆☆

## 核心要点

**配置**：软件产品中受控的**功能/物理属性**及相关文档、版本、变更、支撑数据等，保证一致性。

| 活动 | 要点 |
|------|------|
| **配置管理计划** | 项目计划组成部分 |
| **配置标识** | 确定**配置项**、命名与描述 |
| **变更管理与配置控制** | 变更受控，避免混乱 |
| **配置状态说明/报告** | 记录与通报配置项状态 |
| **配置审核** | 验证与标识一致，保证 CM 有效 |

### （2021年11月·第17题）

产品配置是指一个产品在其生命周期各个阶段所产生的各种形式（机器可读或人工可读）和各种版
本的（）的集合
问题（1）
（A）需求规格说明、设计说明、测试报告
（B）需求规格说明、设计说明、计算机程序
（C）设计说明、用户手册、计算机程序
（D）文档、计算机程序、部件及数据

**正确答案：** D

**解析**

本题考察的是项目管理中产品配置的概念。
A选项 需求规格说明、设计说明、测试报告：这些属于文档类型，但缺少计算机程序、部件和数据
，范围不完整，错误。
B选项 需求规格说明、设计说明、计算机程序：缺少部件和数据，范围不完整，错误。
C选项 设计说明、用户手册、计算机程序：同样缺少部件和数据，错误。
D选项 文档、计算机程序、部件及数据：完整涵盖了产品在生命周期中可能产生的全部配置项，符
合定义，正确。
 因此选择 D。

### （2018年11月·第19题）

下面关于变更控制的描述中，（）是不正确的。
问题（1）
（A）变更控制委员会只可以由一个小组担任
（B）控制需求变更与项目的其他配置管理决策有着密切的联系
（C）变更控制过程中可以使用相应的自动辅助工具
（D）变更的过程中，允许拒绝变更

**正确答案：** A

**解析**

正确答案：A. 变更控制委员会只可以由一个小组担任
 本题考察的是配置管理中变更控制的基本原则和组织形式。
A选项：“变更控制委员会只可以由一个小组担任”是错误的。变更控制委员会（Change Control
Board，CCB）可以根据项目的需要设立多个，或者根据不同类型的变更设立不同的变更控制小组。
其组成应具有代表性，通常包括项目经理、技术负责人、质量保证等相关干系人，不限于某一个固
定小组。
B选项是正确的。配置管理中，变更控制与其他配置管理活动（如配置标识、配置状态记录、配置审
计）密切相关，变更可能会影响配置项的基线，从而需要审慎管理。
C选项是正确的。现代项目管理中广泛使用自动化工具来支持变更请求的提交、审批流程、影响分析
、状态跟踪等过程，提高效率和规范性。
D选项是正确的。在变更控制过程中，对于不合理或风险过大的变更请求，变更控制委员会有权拒绝
变更，以保障项目目标和范围的稳定性。
 因此，选项 A 是不正确的，故本题正确答案为 A。

### （2017年11月·第17题）

项目配置管理中，产品配置是指一个产品在其生命周期各个阶段所产生的各种形式和各种版本的文
档、计算机程序、部件及数据的集合。该集合中的每一个元素称为该产品配置中的一个配置项
，（）不属于产品组成部分工作成果的配置项。

问题（1）
（A）需求文档
（B）设计文档
（C）工作计划
（D）源代码

**正确答案：** C

**解析**

本题考察的是配置管理中配置项的范围。
 配置项是指在产品生命周期中需要纳入配置管理的对象，它通常是产品形成过程中的工作成果，典
型的有各种文档、源代码、可执行文件、数据库、接口描述以及其他与产品直接相关的成果。配置
项必须是产品构成的一部分，并且需要通过配置管理来进行控制。
A选项需求文档：是定义产品功能与性能需求的文档，直接决定产品开发内容，属于典型的配置项。
B选项设计文档：描述产品系统结构和详细设计的文档，是指导开发和维护的重要依据，属于典型的
配置项。
C选项工作计划：是项目管理的计划性文件，用于指导项目实施进度和资源安排，属于项目管理成果
，而不是产品组成部分，因此不属于产品配置项。
D选项源代码：是产品开发的重要成果，是最终产品的一部分，必须纳入配置管理，属于配置项。
 因此，选择 C。

### （2015年11月·第19题）

项目配置管理中，配置项的状态通常包括（）。
问题（1）
（A）草稿、正式发布和正在修改
（B）草稿、技术评审和正式发布
（C）草稿、评审或审批、正式发布
（D）草稿、正式发布和版本变更

**正确答案：** A

**解析**

本题考察的是项目配置管理中配置项的状态管理。
 配置管理是项目管理中的关键组成部分，主要用于控制和记录项目产品（如文档、软件模块、硬件
部件等）在生命周期中的变更情况。配置项在不同阶段的状态反映了其开发进度和变更控制情况。
A选项 草稿、正式发布和正在修改：正确。这是配置项的典型三种状态。草稿表示尚未定型；正式
发布表示已经评审批准，可作为工作或交付基线；正在修改表示该项处于修改阶段，将产生新版本
。
B选项 草稿、技术评审和正式发布：错误。技术评审是过程活动而非配置项的状态，不适合作为状
态划分。
C选项 草稿、评审或审批、正式发布：错误。"评审或审批"同样是过程行为，不属于配置项的固有
状态。
D选项 草稿、正式发布和版本变更：错误。"版本变更"是动作或事件，不是状态。
 所以选择 A。

### （2023年11月·第11题）

软件生存周期各个阶段活动的产物经审批后即可称之为（）。
问题（1）
（A）产品
（B）软件配置项
（C）版本
（D）里程碑

**正确答案：** B

**解析**

本题考察的是需求变更管理和软件配置管理的相关知识。
A选项产品：产品是最终交付给客户的系统或软件，通常是在多个阶段的开发过程中逐步形成的，但
并不专指经过审批的阶段产物。它更强调最终交付成果，而不是开发过程中每个阶段的产物。
B选项软件配置项：软件配置项是指在软件生存周期各个阶段（如需求、设计、开发等）产出的成果
，经过正式审批和确认后，成为软件配置项。软件配置项是配置管理的基础，经过变更控制和版本
管理，确保产品开发的一致性和稳定性。
C选项版本：版本是指软件的特定发布或发布状态，通常包括多个软件配置项。在开发过程中，版本
是由不同的配置项组成的，并非直接代表各阶段的产物。版本发布需要经过配置管理，但并不等同
于阶段产物的审批结果。
D选项里程碑：里程碑是项目进度中的关键节点，用于标识项目的重要阶段和成就。它并不指代具体
的产物，而是项目管理中的时间标记。
 因此，选项 B 正确。

### （2024年11月·第57题）

下列标准中不包括配置管理的是（）。
问题（1）
（A）UML
（B）ISO9000
（C）PMBOK
（D）CMMI

**正确答案：** A

**解析**

本题考察的是配置管理在不同标准体系中的体现。
 配置管理是一种对产品及其过程进行标识、控制、状态记录和审计的管理活动，目的是保障产品的
一致性、完整性和可追溯性，广泛应用于软件工程与项目管理等领域。
A选项UML：统一建模语言（Unified Modeling Language）是一种通用建模语言，用于对软件系统进
行可视化建模，主要用于需求分析、系统设计等阶段，不涉及软件配置管理的内容，因此是正确答
案。
B选项ISO9000：是国际标准化组织发布的一系列质量管理体系标准，包含了对产品实现过程中过程
控制与配置管理的基本要求，强调通过配置管理实现产品的一致性与可追溯性，因此包含配置管理
。
C选项PMBOK：即《项目管理知识体系指南》，它将配置管理作为项目整体变更控制的一部分，提出
应管理项目的产品和文档配置，属于其内容范畴。
D选项CMMI：能力成熟度模型集成（Capability Maturity Model Integration）中，将配置管理作
为关键过程域之一，要求建立和维护配置项的完整性和可控性，明确包含配置管理内容。

### （2025年5月·第60题）

影响软件质量的因素，应从用户的角度来看，主要包括以下哪些方面（）。
问题（1）
（A）产品运行、产品修改、产品转移
（B）产品设计、产品实现、产品测试
（C）产品重构、产品集成、产品部署
（D）产品效率、产品可靠性、产品可维护性

**正确答案：** A

**解析**

本题考查的是软件质量模型中的三维度模型（McCall 模型），它从用户角度划分软件质量
的三个主要方面：
产品运行（Product Operation）：强调软件在运行时的质量特性，如正确性、可靠性、效率、完整
性等。

产品修改（Product Revision）：关注软件维护和修改的难易程度，如可维护性、灵活性、测试性
。
产品转移（Product Transition）：强调软件在不同平台和环境中的适应能力，如可移植性、可重
用性、互操作性。
 选择选项 A。
