# 成本管理

**知识点编号**：8.3  
**权重**：次重点 ★★★☆☆

## 核心要点

在批准预算下，对成本过程进行管理与控制。

| 过程 | 要点 |
|------|------|
| **成本估算** | 估计总成本；方法如**自顶向下**、**自底向上**、**差别估算** |
| **成本预算** | 将总成本分配到工作包，形成**成本基准** |
| **成本控制** | 各项工作在预算内执行，监控偏差 |

### （2016年11月·第16题）

项目的成本管理中，（）将总的成本估算分配到各项活动和工作包上，来建立一个成本的基线。
问题（1）
（A）成本估算
（B）成本预算

（C）成本跟踪
（D）成本控制

**正确答案：** B

**解析**

本题考察的是项目成本管理流程中的概念。
A选项 成本估算：是对完成项目活动所需资源成本进行近似计算的过程，结果是总成本估算，但不
涉及将成本分配到各工作包，错误。
B选项 成本预算：是在已有成本估算的基础上，将总成本分配到各具体活动和工作包上，形成成本
基线，用于后续成本控制与绩效测量，正确。
C选项 成本跟踪：指在项目执行过程中监控成本支出与预算的差异，不是建立基线的过程，错误。
D选项 成本控制：是根据成本基线监控和调整项目成本的过程，错误。
 因此，选项 B 正确。

### （2017年11月·第19题）

一个好的变更控制过程，给项目风险承担者提供了正式的建议变更机制。如下图所示的需求变更管
理过程中，①②③处对应的内容应分别是（）。
问题（1）
（A）问题分析与变更描述、变更分析与成本计算、变更实现
（B）变更描述与成本计算、变更分析、变更实现
（C）问题分析与变更分析、成本计算、变更实现
（D）变更描述、变更分析与变更实现、成本计算

**正确答案：** A

**解析**

本题考察的是需求变更管理流程的基本内容与顺序。
 需求变更管理是需求工程中的关键过程之一，强调从识别问题到实施变更的全过程控制，确保需求
修改的有效性、合理性和一致性。
①问题分析与变更描述：在识别出问题后，需要分析问题的来源、范围和影响，并正式记录变更提
议，形成变更描述文档。
②变更分析与成本计算：在明确变更意图后，项目团队需要评估变更带来的影响，包括对现有系统
结构、功能、接口的影响，并进行成本估算，决定是否采纳。
③变更实现：一旦决定执行变更，则进入实施阶段，修改需求文档、设计文档及相关系统实现部分
，确保系统与文档的一致性。
B选项错误：将变更描述和成本计算放在一起，忽略了问题分析的前置作用，流程顺序不当。
C选项错误：将问题分析和变更分析混为一体，没有体现成本计算在变更评估中的关键作用。
D选项错误：将变更分析与实现合并，忽视了成本计算在决策前的独立重要步骤。
 因此，选项 A 正确。

### （2015年11月·第21题）

一个大型软件系统的需求总是有变化的。为了降低项目开发的风险，需要一个好的变更控制过程。
如下图所示的需求变更管理过程中，①②③处对应的内容应是（）；自动化工具能够帮助变更控制
过程更有效地运作，（）是这类工具应具有的特性之一。
问题（1）
（A）问题分析与变更描述，变更分析与成本计算，变更实现
（B）变更描述与变更分析，成本计算，变更实现
（C）问题分析与变更描述，变更分析，变更实现
（D）变更描述，变更分析，变更实现
问题（2）
（A）自动维护系统的不同版本
（B）支持系统文档的自动更新

（C）自动判定变更是否能够实施
（D）记录每一个状态变更的日期及变更者

**正确答案：** A,D

**解析**

本题考察的是需求变更管理流程以及自动化工具的基本功能要求。
问题 1
A选项：问题分析与变更描述，变更分析与成本计算，变更实现
正确。 该选项完整准确地反映了需求变更管理的标准流程：
①问题分析与变更描述：识别出当前需求存在的问题或新增需求，并对变更请求进行初步描述。
②变更分析与成本计算：分析该变更对系统、文档、人员、资源的影响，并评估所需的工作量与
成本。
③变更实现：根据批准的变更内容修改系统需求、设计、实现和测试等相关内容。流程闭环，确
保变更有效实施。
B选项：变更描述与变更分析，成本计算，变更实现
 错误。该选项未将“问题分析”单独列出，并将“变更描述”与“变更分析”混为一体，逻辑结构
不清晰，也不符合标准的需求变更流程分工。
C选项：问题分析与变更描述，变更分析，变更实现
 错误。此选项省略了“成本计算”这一关键环节。在项目管理中，进行成本评估是变更决策的重要
依据，因此缺失不当。
D选项：变更描述，变更分析，变更实现
 错误。未包含“问题分析”过程，且忽略了“成本计算”，结构过于简化，不符合变更控制流程的
实际步骤。
 选择选项呢 A。
问题 2：
A选项：自动维护系统的不同版本
 错误。该功能属于配置管理工具的基本功能，与变更控制工具的核心功能要求不同，虽然相关，但
不是重点。
B选项：支持系统文档的自动更新
 错误。此功能属于文档管理或文档生成工具的职责范畴，而非变更控制工具的核心职责。
C选项：自动判定变更是否能够实施
 错误。变更是否能够实施通常由变更控制委员会（CCB）或项目经理依据分析结果人工判断，工具
不应自动替代人做决策，否则容易带来风险。
D选项：记录每一个状态变更的日期及变更者
正确。 这项功能属于可追溯性管理范畴，是变更控制系统最基本的要求之一。通过日志记录每一个
变更的时间、人员、状态，有助于项目审计、问题追溯、质量控制等工作。
 选择选项呢 D。

### （2023年11月·第45题）

配置项控制即对配置项和基线的变更控制，下列变更管理顺序正确的是（）。
问题（1）
（A）提出变更申请->进行风险评估->进行变更批准->开展变更实施->变更验证->存档
（B）进行风险评估->提出变更申请->进行变更批准->开展变更实施->变更验证->存档
（C）提出变更申请->进行变更批准->进行风险评估->开展变更实施->变更验证->存档
（D）进行变更批准->提出变更申请->变更验证>开展变更实施->进行风险评估->存档

**正确答案：** A

**解析**

本题考察的是配置项控制的变更管理流程。
A选项：提出变更申请 -> 进行风险评估 -> 进行变更批准 -> 开展变更实施 -> 变更验证 -> 存档
。
 这个选项符合变更管理的标准流程。首先提出变更申请，接着进行风险评估，分析变更的影响与可

行性，随后由变更控制委员会（CCB）进行变更批准，接着实施变更，对修改后的配置项进行验证
，最后进行存档和发布。
B选项：进行风险评估 -> 提出变更申请 -> 进行变更批准 -> 开展变更实施 -> 变更验证 -> 存档
。
 这个顺序不正确。风险评估应该在变更申请提出之后进行，而不是在变更申请之前。
C选项：提出变更申请 -> 进行变更批准 -> 进行风险评估 -> 开展变更实施 -> 变更验证 -> 存档
。
 这个顺序不正确。变更批准应在风险评估之后进行，而不是提前进行。
D选项：进行变更批准 -> 提出变更申请 -> 变更验证 -> 开展变更实施 -> 进行风险评估 -> 存档
。
 这个顺序完全不符合常规流程，变更批准不应该先于变更申请，且变更验证和风险评估应该在实施
后进行。
 因此，选项 A 正确。
