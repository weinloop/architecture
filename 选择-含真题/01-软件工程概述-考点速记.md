## 二、软件工程定义与 PDCA（★★★☆☆）

- **软件工程**：由**方法、工具和过程**三部分组成。
- **PDCA 循环**（选择题考过）：
  - **P（Plan）**：软件规格说明，规定功能及运行限制。
  - **D（Do）**：软件开发，开发出满足规格的软件。
  - **C（Check）**：软件确认，确认满足用户需求。
  - **A（Action）**：软件演进，运行中不断改进。

**口诀**：计划—开发—确认—演进。

**案例（航空运营智能管理平台）**：

- **P（Plan）**：立项阶段将政策与市场目标固化为可验证规格，例如“运营效率提升 30% 以上”“核心数据强一致与安全”“全年可用性≥99.99%”，并约束高并发票务、PB 级数据分析、对接民航局旅客身份核验等边界。
- **D（Do）**：按“方法、工具、过程”组织建设——微服务与分布式事务支撑购票/改签；大数据与 AI 支撑航线与检修智能分析；多终端适配与加密传输支撑旅客端与管理端。
- **C（Check）**：通过系统测试验证 5000 TPS、响应≤1 秒等指标；通过验收确认满足信息中心与业务部门用户需求；安全侧验证 RBAC、审计日志、脱敏与合规。
- **A（Action）**：上线后针对节假日峰值、突发航班变动等场景持续弹性扩容、优化模型阈值与运营策略，体现运行期演进。

## 三、软件开发方法分类（★★★★★）

| 分类方法 | 具体方法 | 描述 |
|----------|----------|------|
| 开发风格 | 自顶向下 | 先定义整体结构，再逐步细化模块与功能 |
| 开发风格 | 自底向上 | 从具体模块开始逐步构建整个系统 |
| 性质 | 形式化方法 | 坚实数学基础，严格规格与验证 |
| 性质 | 非形式化方法 | 不把严格性作为主要着眼点 |
| 适应范围 | 整体性方法 | 适用于软件开发全过程 |
| 适应范围 | 局部性方法 | 适用于某个具体阶段 |

**案例（航空运营智能管理平台）**：

- **自顶向下**：先确定“基础支撑—航信—旅客—票务—检修—数据服务—辅助管理”的整体分层与跨模块联动链路（购票→数据同步→运力校验→行程与通知→检修阈值→分析），再细化各子服务与接口。
- **自底向上**：先落地成熟能力如分布式存储、数据同步、多终端适配、消息与缓存，再拼装出完整购票与可视化等高阶业务能力。
- **形式化 vs 非形式化**：票务库存扣减、支付与出票一致性、身份核验对接等强约束环节更强调规格与可验证性；旅客营销文案、部分运营看板迭代可采用非形式化、快速沟通。
- **整体性 vs 局部性**：全生命周期上采用统一微服务与数据治理贯穿各模块；局部上可仅对“数据服务/实时分析”或“检修 AI 模型”等单域采用专门方法学与工具链。

## 四、形式化方法与净室（★★★☆☆）

### 4.1 形式化方法

- **含义**：用数学模型与形式化语言描述需求/设计/行为，通过逻辑推理或模型检测获得**可证明的正确性**；常用于安全攸关、高可靠性系统。
- **特点**：① 正确性验证难且耗时；② 仍需与传统测试结合（形式化提供理论保证，测试发现运行时错误）。

**案例（航空运营智能管理平台）**：对“强一致性票务交易、敏感旅客数据加密存储、对接民航局身份核验、防 SQL 注入与审计追溯”等安全攸关需求，可用形式化思路强化需求/接口规约与关键路径验证；同时仍依赖峰值压测、混沌演练与线上监控发现运行时与基础设施问题，二者互补而非替代。

### 4.2 净室软件工程（CSE）

- **定位**：一种**形式化方法**，不是测试方法、管理模型或单纯过程模型；强调在**设计与规约阶段**用数学与统计理论消除错误。
- **理论基础**：**函数理论**（程序=输入到输出的严格映射；完备性、一致性、正确性——口诀：**玩一阵**）、**抽样理论**（抽样测试推断系统可靠性）。
- **局限**：验证依赖严格数学，门槛高；不依赖传统单元测试但仍需考虑底层环境；仍属传统软件工程体系。

**案例（航空运营智能管理平台）**：可将“选座→计价→锁库存→支付回调→出票→同步行程与通知”抽象为输入到输出的函数式链条，在规约阶段尽量消除歧义；对海量交易可靠性则结合**抽样与统计**思路，用生产级压测与线上采样评估 TPS、错误率与尾延迟，而不是仅靠单次手工测试。

## 五、逆向工程与再工程（★★★☆☆）

| 概念 | 说人话 | 定义 |
|------|--------|------|
| **逆向工程** | 从低层（代码）推回高层设计 | 分析程序，在比源代码更高抽象层次建立程序表示 |
| **重构** | 同一抽象层改结构 | 同一抽象级别上转换系统描述形式 |
| **设计恢复** | 借助工具抽取设计信息 | 从已有程序抽象出数据设计、总体结构、过程设计等 |
| **再工程** | 逆向+新需求+正向工程 | 在逆向信息基础上修改/重构产生新版本 |
| **正向工程** | 从设计到代码 | 利用逆向得到的信息改变或重构系统 |

**抽象层次（低→高）**：实现级（语法树、符号表）→ 结构级（调用图、结构图）→ 功能级（程序段功能及关系）→ 领域级（与应用领域概念对应）。

**口诀**：**世仇结分工龄**——世（实现）仇（抽象信息）结（结构）分（程序分量）工（功能级）龄（领域级）。

**案例（航空运营智能管理平台）**：

- **逆向工程**：对接既有离散的航线/航班/旅客子系统或外部民航接口时，往往只有可调用 API 与日志，需要通过逆向分析在更高抽象层还原“数据模型、调用关系、时序约束”，再映射到本平台微服务划分。
- **重构**：在不改变对外业务能力的前提下，调整某服务内部结构（如拆分出票与库存、优化缓存键设计），属于同一抽象层上的结构优化。
- **设计恢复**：从遗留库表与批处理脚本中抽取领域对象（航班、舱位、检修工单），补全概要/详细设计资产，便于治理与测试。
- **再工程 / 正向工程**：在逆向得到的旧系统理解之上，结合新目标（统一入口、PB 级分析、AI 检修预警）重新设计并编码实现新平台，形成可替换旧烟囱系统的新版本。

## 六、基于构件的软件工程 CBSE（★★★★★）

### 6.1 构件的定义（萨默维尔）

| 构件特性 | 描述 |
|----------|------|
| 可组装性 | 外部交互通过公开接口；提供对自身信息的外部访问 |
| 可部署性 | 自包含，独立实体在构件平台上运行；通常二进制、无须部署前编译 |
| 文档化 | 完全文档化，接口语法甚至语义需定义 |
| 独立性 | 可无其他特殊构件下组装部署；需外部服务时在“请求”接口显式声明 |
| 标准化 | 符合标准化构件模型（接口、元数据、文档、组成、部署） |
| 可见性 | 无外部可见状态（特殊情况：容器管理状态、不影响功能的属性可可见） |

**口诀**：**竹鼠挡毒标**——竹（可组装）鼠（可部署）档（文档化）毒（独立性）镖（标准化）。

**案例（航空运营智能管理平台）**：各业务子服务通过 **REST/gRPC + API 网关**对外暴露稳定接口，独立打包部署在容器平台（**可部署性**）；票务、旅客、检修等团队可并行开发，通过契约与 Mock 联调（**可组装性**）；接口文档与错误码语义对多渠道售票、运维排障必不可少（**文档化**）；单个服务可单独扩缩容，依赖外部身份核验时在接口显式声明（**独立性**）；技术栈与发布规范统一（**标准化**）；业务状态尽量外置到缓存/数据库，避免服务内隐式全局可变状态（**可见性**）。

### 6.2 构件分类（按外部形态）

| 类型 | 内容 | 例子 |
|------|------|------|
| 独立而成熟的构件 | 成熟度高，完全隐藏接口，规定命令使用 | 数据库管理系统、操作系统 |
| 有限制的构件 | 提供接口与使用条件；可能资源冲突，需测试 | 基础类库 |
| 适应性构件 | 处理不兼容、资源冲突，可直接多环境使用 | ActiveX |
| 装配的构件 | 已装配在 OS/DBMS/信息系统层次，胶水代码连接 | 成品软件产品 |
| 可修改的构件 | 可改错、增功能，包装或写接口实现版本替换 | 功能模块 |

**口诀**：**独显时装秀**——独立、限制、适应、装配、修改。

**案例（航空运营智能管理平台）**：

- **独立而成熟**：分布式数据库、消息中间件、Kubernetes、负载均衡等以黑盒方式使用，按厂商能力运维。
- **有限制**：某些 SDK/基础类库要求固定线程模型或连接池上限，集成时需压测避免资源争用。
- **适应性**：多终端适配组件、跨云厂商的弹性伸缩适配层，屏蔽环境差异。
- **装配**：在成熟 OS/DB 之上用少量“胶水代码”串联票务、数据同步与通知服务。
- **可修改**：自研的票价策略、库存扣减、检修预警规则等模块需随业务快速改版，可通过特性开关与灰度替换实现。

### 6.3 构件组装类型

| 组装类型 | 描述 | 特点 |
|----------|------|------|
| **顺序组装** | 按顺序调用已有构件创造新构件；先 A 再 B | 构件间不相互调用，需胶水代码保证接口兼容 |
| **层次组装** | 一构件直接调用另一构件服务；“提供”与“请求”接口需兼容 | 接口匹配无需额外代码，不匹配需转换；构件为网络服务时可能不适用 |
| **叠加组装** | 多构件叠加，新构件接口为原接口组合，分别调用 | 原构件互不依赖、不互相调用 |

**接口不兼容**：参数不兼容（操作名同、参数类型/数目不同）、操作不兼容（操作名不同）、操作不完备（提供/请求接口为对方子集）。

**案例（航空运营智能管理平台）**：

- **顺序组装**：链路“在线购票 → 数据服务同步订单与库存 → 航信模块运力校验 → 旅客行程与通知”按阶段依次触发，中间以编排或消息保证顺序与补偿。
- **层次组装**：票务服务调用库存服务、支付网关、身份核验；数据服务调用计算引擎与存储，形成上下层服务调用关系，接口需严格匹配或通过适配器转换。
- **叠加组装**：报表统计、日志审计、监控告警等横切能力叠加到多个业务服务之上，各原服务互不调用，由独立组件分别订阅事件或旁路采集。

## 七、软件过程/生命周期模型（★★★★★）

- **软件过程 = 软件开发生命周期模型**（同义词）。
- **前提与模型**：需求完全确定→瀑布；仅基本需求→喷泉、螺旋、RUP、敏捷等；形式化为基础→变换模型。

**案例（航空运营智能管理平台）**：行业政策、旅客行为与峰值模式持续变化，**需求难以一次性冻结**，因此单纯瀑布不适合作为唯一模型；更适合“主线迭代 + 分域增量”，在票务一致性、安全合规等基线上用工程化约束，在可视化与运营策略上用快速反馈。

### 7.1 瀑布模型

- **阶段**：需求分析与定义→系统与软件设计→实现与单元测试→集成与系统测试→运行维护。
- **缺点**：依赖早期需求调查，**不能适应需求变化**；需求完整性、正确性难确定。

**案例（航空运营智能管理平台）**：若把“对接民航局核验、票务强一致、年度可用性指标”等在合同期初就写得极其稳定，瀑布可用于**子系统规格冻结后的实现阶段**；但整体业务与数据分析需求仍在演进，故瀑布通常只作为局部阶段模型而非端到端唯一模型。

### 7.2 原型/演化/快速原型模型

- 软考中**不做区分**，当做一个；适用于需求难完整定义，基于快速原型、用户反馈迭代直至最终产品。
- **特点**：迭代；功能开发后可及时验证需求；过早接触不稳定功能可能带来负面影响。

**案例（航空运营智能管理平台）**：可对**管理端数据可视化、旅客出行服务流程**先上线可点击原型或 MVP，收集信息中心与一线运营反馈后再固化功能范围，降低“一次性写全需求”的风险；需注意过早向旅客开放未经验证的票务逻辑可能带来品牌与资金风险，原型边界要划清。

### 7.3 螺旋模型

- **螺旋模型 = 快速原型 + 风险分析**（选择题常考）。
- 适用于庞大、复杂、高风险系统；支持需求动态变化；迭代过多会增加成本、延迟提交。

**案例（航空运营智能管理平台）**：项目同时具备**高并发票务、PB 级数据、实时流式分析、AI 检修预警、多系统对接**等高风险点，适合螺旋式分轮识别风险（如峰值压测失败、分布式事务死锁、模型准确率不足），每轮建立原型或试验环境验证再扩大范围。

### 7.4 喷泉模型

- 以**用户需求为动力、对象为驱动**；各阶段**相互重叠、多次反复**，可交互进行，可随时补充遗漏。

**案例（航空运营智能管理平台）**：航线/航班/旅客/票务/检修/数据服务多团队并行，**分析与设计、实现、联调、数据治理**可交叉推进；例如数据服务在采集规范未完全终稿时即可先建管道，后续迭代补字段与质量规则，体现喷泉式重叠迭代。

### 7.5 变换模型

- 基于**形式化规格说明语言和程序变换**；需严格数学理论和整套开发环境支持。

**案例（航空运营智能管理平台）**：在**支付回调幂等、库存扣减与超卖防护、分布式事务隔离级别**等关键规则上，可采用接近形式化的规格推导与代码生成或严格评审；需配套工具链与团队能力，一般与其他模型混用。

### 7.6 V 模型与测试依据

| 测试类型 | 测试对象 | 依据（文档） |
|----------|----------|--------------|
| 单元测试 | 模块/类 | **详细设计说明书** |
| 集成测试 | 模块间接口、已集成软件 | **概要设计文档** |
| 系统测试 | 完整集成系统 | **需求分析文档/开发合同** |
| 验收测试 | 准备交付的完整系统 | **用户需求** |

**注意**：概要设计是**集成测试**的依据，不是系统测试；系统测试依据是需求分析/合同；验收测试依据是用户需求。

**案例（航空运营智能管理平台）**：

- **单元测试**：例如售票计算、退改签规则引擎对照**详细设计**中的分支与边界。
- **集成测试**：票务与数据服务、航信运力校验、通知模块之间的接口与时序，对照**概要设计**中的模块边界与交互。
- **系统测试**：整体验证 10 万并发、≤1 秒响应、5000 TPS、数据延迟等指标，对照**需求规格/合同**中的系统级指标。
- **验收测试**：信息中心与业务方按**真实用户场景**（节假日购票、航班大面积延误）进行 UAT。

### 7.7 增量模型 vs 原型/迭代

- **增量**：按模块完成顺序逐步实现，每次增量可独立可操作；**模块叠加、线性扩展**。
- **迭代**：分阶段循环（需求→设计→开发→测试）**逐步完善同一功能**，每次交付完整版本；**功能深度优化、螺旋上升**。
- 增量更早让客户使用并获得价值。

**案例（航空运营智能管理平台）**：

- **增量**：可先交付“基础支撑 + 票务 + 航信”形成可售票闭环，再叠加旅客全旅程、检修 AI、PB 级离线分析等，每一增量对外可独立产生业务价值。
- **迭代**：同一模块（如动态票价、实时风控、可视化看板）在多轮 Sprint 中**逐步加深**准确率与体验，每轮交付可用版本而非只做纸面规划。

---

## 八、统一过程 RUP（★★★★★）

- **RUP**：用例驱动、以**体系结构为中心**、**迭代与增量**的软件开发过程；采用“4+1”视图描述体系结构（与后续架构 4+1 视图区分）。
- **九大核心工作流**（系分考过，必须背）：
  - **建需分食测**：建（业务建模）需（需求）分（分析与设计）食（实现）测（测试）。
  - **部配变项环**：部（部署）配变（配置与变更管理）项（项目管理）环（环境）。

**案例（航空运营智能管理平台）**：

- **用例驱动**：以业务用例组织需求，如“多渠道购票”“恶劣天气下批量退改签”“检修预警阈值随载客量调整”“实时超售风险告警”等，用例贯穿分析、设计与测试用例。
- **以体系结构为中心**：云原生微服务、分布式事务、缓存与消息、读写分离与实时/离线双引擎构成骨架，迭代中优先保护架构不变量（一致性、可用性、可观测性）。
- **迭代与增量**：与 2025 上线、2026 稳定运行节奏一致，分版本持续交付。
- **九大工作流（映射举例）**：业务建模（航司运营域与合规流程）；需求（指标与功能表）；分析与设计（服务划分与数据流）；实现（各子服务）；测试（压测与专项）；部署（灰度与弹性）；配置与变更（特性开关、版本回滚）；项目与环境（多团队协作、CI/CD 与监控环境）。

## 九、敏捷方法（★★★★★）

### 9.1 敏捷宣言

- 个体和交互 **胜过** 过程和工具；可工作的软件 **胜过** 大量文档；客户合作 **胜过** 合同谈判；**响应变化** **胜过** 遵循计划。

**案例（航空运营智能管理平台）**：信息中心与业务、运维、客服建立**高频协同**而非仅依赖工具工单；以**可上线**的购票链路与监控大盘为沟通媒介，减少纯文档扯皮；与监管核验、承运人规则变更等场景下强调**合作解释需求**而非僵化合同条款；航班大面积延误时优先**调整发布与降级策略**而非死守原计划排期。

### 9.2 适用场景

- 规模较小的项目；需求初步萌芽、尚未明确、**快速改变**。高关键性/可靠性/安全性要求可能不完全适合。
- **口诀**：**小变**——小（规模小）变（快速改变）。

**案例（航空运营智能管理平台）**：子域（如报表、运营策略试验）团队规模可控、需求变化快，适合“小变”；**核心票务账务、旅客敏感数据、监管对接**仍需强工程化、严格回归与审计。敏捷宣言中的“胜过”不是取消文档与计划，而是**价值优先**；若题干强调“极高安全性单一关键路径”，应承认敏捷需与质量体系结合。

### 9.3 主要敏捷方法

| 方法 | 简介 |
|------|------|
| **极限编程 XP** | 轻量、短周期、频繁发布；交流、朴素、反馈、勇气；YAGNI、DRY |
| **水晶系列** | “机动性”方法，以人为中心 |
| **Scrum** | 侧重项目管理的迭代增量框架；团队协作、响应变化、商业价值 |
| **FDD** | 迭代、人/过程/技术三要素 |

**案例（航空运营智能管理平台）**：可设**极限编程**式小团队负责高频变更的定价与规则引擎，强调结对评审与自动化测试；**Scrum** 组织跨职能团队交付 Sprint 目标；**FDD** 按领域对象（航班、票、旅客、工单）拆分特性；**水晶**强调驻场业务代表与透明沟通，适合信息中心与厂商协作。

### 9.4 Scrum（★★★☆☆）

- **角色**：Product Owner（业务价值、Backlog 优先级）、Scrum Master（流程推动、清障、赋能）、Development Team（跨职能、自组织、交付 Sprint 目标）。
- **活动**：Sprint Planning（Sprint Backlog、目标）、Daily Scrum（昨日/今日/障碍）、Sprint Review（演示、反馈）。
- **工件**：Product Backlog（产品待办）、Sprint Backlog（冲刺待办）、Increment（可交付增量）。

**案例（航空运营智能管理平台）**：

- **Product Owner**：信息中心或业务方维护待办优先级，例如“国庆前完成票务峰值压测与扩容预案”优先于“新版可视化皮肤”。
- **Scrum Master**：协调跨团队依赖（数据同步延迟、外部核验接口限流），清除障碍。
- **Development Team**：含后端、前端、大数据、测试、运维技能，共同对 Sprint 增量负责。
- **活动**：节前 **Sprint Planning** 锁定购票链路与降级方案；峰值期间 **Daily Scrum** 同步故障与容量；节后 **Review** 演示真实监控数据与复盘。
- **工件**：Backlog 中的史诗可对应“票务高并发”“实时分析≤2 秒”“检修 AI 准确率”等；Increment 必须是可部署、可验证的版本，而非文档。

## 十、软件过程管理：CMM / CMMI（★★★☆☆）

### CMM 五级

| 级别 | 含义 |
|------|------|
| 初始级 | 过程缺乏结构化，依赖个人经验 |
| 可重复级 | 能定义和文档化过程，确保项目成功 |
| 已定义级 | 过程标准化、文档化 |
| 已管理级 | 监控、量化过程，持续改进 |
| 优化级 | 动态调整过程适应不同项目 |

**口诀**：**初重定管优**——除虫顶管用。

**案例（航空运营智能管理平台）**：从**初始**依赖少数核心骨干救火，到**可重复**沉淀压测脚本与发布 checklist，再到**已定义**统一需求变更与代码评审流程，**已管理级**对 TPS、错误率、P99 延迟等建立度量与阈值告警，**优化级**根据节假日与突发航变持续调整容量模型与演练频率——可写成过程改进叙事线。

### CMMI 五级与知识体系

- **等级**：初始级、已管理级、已定义级、**量化管理级**、优化级。**口诀**：储罐定粮油。
- **四大知识体系**：系统工程（SE）、软件工程（SW）、集成产品与流程开发（IPPD）、供应商采购（SS）。**口诀**：细软开工。

**案例（航空运营智能管理平台）**：**已管理级/量化管理级**与项目指标天然对应：票务 5000 TPS、响应≤1 秒、实时分析延迟≤2 秒、批量成功率≥99.9%、年度 10PB+ 离线数据任务 SLA 等；**系统工程**视角覆盖终端—边缘—云端链路；**软件工程**覆盖微服务与数据平台；**IPPD** 体现航信、票务、旅客、检修、数据多产品线协同发布；**供应商采购**对应云资源、第三方核验与安全服务选型与 SLA 管理。

## 十一、速记清单（考前扫一眼）

1. **PDCA**：Plan 规格说明、Do 开发、Check 确认、Action 演进。
2. **开发方法**：自顶向下/自底向上；形式化/非形式化；整体性/局部性。
3. **净室**：形式化方法；函数理论（玩一阵）+ 抽样理论；非银弹。
4. **逆向**：逆向（低→高）、重构（同层）、设计恢复、再工程（逆+需求+正）、正向；抽象层次：世仇结分工龄。
5. **构件**：竹鼠挡毒标；分类独显时装秀；组装：顺序/层次/叠加；不兼容：参数/操作/不完备。
6. **生命周期**：瀑布（需求确定）、原型/螺旋（需求难定）、螺旋=原型+风险、喷泉（阶段交互）、V 模型测试依据（详设→单元、概设→集成、需求→系统、用户需求→验收）。
7. **RUP**：用例驱动、体系结构为中心、迭代增量；九流：建需分食测、部配变项环。
8. **敏捷**：宣言四胜过；小变；XP/水晶/Scrum/FDD；Scrum 角色 PO/SM/Team，活动 Planning/Daily/Review，工件 Backlog/Sprint Backlog/Increment。
9. **CMM**：初重定管优；**CMMI**：储罐定粮油；知识体系细软开工。
