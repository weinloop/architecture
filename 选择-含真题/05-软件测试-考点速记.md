## 二、测试分类

### 2.1 按程序是否可见

| 类型 | 要点 |
|------|------|
| **黑盒** | 功能测试；不看内部，基于输入输出；等价类划分等；系统/确认测试常用 |
| **白盒** | 结构测试；了解内部逻辑，覆盖所有路径；逻辑/循环/基本路径覆盖；军工、航天等高可靠领域 |
| **灰盒** | 介于两者；多用于集成测试；既看输入输出也看内部表征 |

**案例（航空运营智能管理平台）**：**黑盒**——按需求与合同验证“选航班→支付→出票→通知”对外行为与异常提示，不关心服务内部实现；**白盒**——对 **锁库存、幂等键、补偿事务** 等关键代码做分支与路径覆盖；**灰盒**——集成阶段结合 **API 契约 + 消息轨迹** 验证票务—数据服务—航信—旅客—通知的衔接。

### 2.2 按是否执行程序

| 类型 | 要点 |
|------|------|
| **静态测试** | 不运行程序；分析语法、结构、接口等；代码检查、静态结构分析、质量度量；发现缺陷早，耗时长 |
| **动态测试** | 运行程序；比较结果与预期；构造用例、执行、分析输出 |

**案例（航空运营智能管理平台）**：**静态**——代码评审、接口规范检查、SQL 与依赖扫描、敏感信息硬编码检查；**动态**——功能用例执行、全链路压测、混沌注入、渗透与漏洞扫描（动静态结合）。

### 2.3 按测试对象

功能测试、性能测试、安全测试、兼容性测试、界面测试、易用性测试、稳定性测试等。

**案例（航空运营智能管理平台）**：**功能**——七大模块与联动链路；**性能**——票务峰值、可视化并发、流式延迟；**安全**——RBAC、TLS、脱敏、审计、防注入/XSS/DDoS；**兼容**——多终端与浏览器；**界面/易用**——旅客端与运营看板；**稳定**——长稳运行与泄漏观察。

### 2.4 按测试阶段

| 阶段 | 对象 | 方法 | 要点 |
|------|------|------|------|
| **单元测试** | 最小可测单元 | 白盒为主、黑盒为辅 | 模块接口、局部数据；开发人员在组长监督下 |
| **集成测试** | 模块/子系统 | — | 模块间数据传输、高层设计；单元测试之后 |
| **系统测试** | 整个系统 | 黑盒 | 功能、性能、恢复、安全、压力等；集成测试之后 |

**案例（航空运营智能管理平台）**：

- **单元**：**售票计算、退改签规则、库存扣减函数** 等，白盒覆盖 + 接口契约。
- **集成**：**票务↔支付↔核验↔消息** 模块间数据与事务边界；对应概要设计中的接口与时序。
- **系统**：端到端验证 **5000 TPS、≤1 秒、99.99% 可用、数据延迟≤3 秒（看板）** 等系统级指标与安全合规。

## 三、白盒覆盖标准（★★★★★）

**强度从弱到强**：语句 < 判定 ≈ 条件 < 判定条件 < 条件组合 < 路径。

**口诀**：**路条组判｜条条判语**——路（路径）条组（条件组合）判（判定条件）｜条（条件）条（条件）判（判定）语（语句）。

| 覆盖类型 | 描述 | 发现错误能力 |
|----------|------|--------------|
| 语句覆盖 | 每条语句至少执行一次 | 最弱 |
| 判定覆盖 | 每个判定的每个分支至少一次 | 2.5 |
| 条件覆盖 | 每个条件取到各种可能值，**不考虑组合** | 2.5 |
| 判定条件覆盖 | 同时满足判定覆盖与条件覆盖 | 4 |
| 条件组合覆盖 | 每个判定中各条件每一种组合至少一次 | 5 |
| 路径覆盖 | 所有可能路径都执行过 | 最强 |

**案例（航空运营智能管理平台）**：对 **出票服务** 中 `if (库存不足) / else if (核验失败) / else if (支付超时)` 等判定，需从 **语句** 逐级加强到 **判定/条件/组合**，关键路径（支付成功→写订单→发消息）争取 **路径覆盖**；分布式场景下还需结合 **并发交错** 的集成级测试，单靠单元路径覆盖不足以发现竞态。

## 四、功能测试与性能测试细分

### 4.1 功能测试

链接测试、表单测试、数据校验、Cookies 测试、数据库测试等；概念辨识即可。

**案例（航空运营智能管理平台）**：**表单/校验**——证件号、手机号、航班日期、舱位；**链接与路由**——深链购票与航变公告；**数据库**——订单与库存一致性、分库分表下的唯一约束；**Cookies/Session**——多端登录与异地登录风控联动。

### 4.2 性能测试辨析（★★★★★）

| 类型 | 要点 |
|------|------|
| **负载测试** | 在**已达一项或几项性能瓶颈**时，让系统继续运行一段时间，观察各项指标 |
| **压力测试** | **不断加压**（增加负载），直到某些指标达瓶颈（如 CPU 100%）；**过程性、渐进加压** |
| **强度测试** | 制造**极限状态**（超负荷或最差环境），看系统抵抗能力与性能下降幅度；侧重**极限状态**，非渐进过程 |
| **并发测试** | 负载+压力结合；逐渐增加负载至瓶颈或不可接受点，综合分析交易与资源指标 |
| **容量测试** | 预先分析某指标极限值（如最大并发用户数），在极限状态下测试是否故障或仍保持主要功能 |
| **可靠性测试** | 为满足可靠性要求的设计、分析、测试；系统测试阶段提高可靠性、发现对可靠性影响大的缺陷 |
| **配置测试** | 找系统资源最优分配，为调优提供依据；选典型配置测试 |
| **速度测试** | 网络连接速度、业务处理速度 |

**压力 vs 强度**：压力=慢慢加压、过程性；强度=制造极限状态（超负荷或环境）。

**案例（航空运营智能管理平台）**：

- **负载测试**：CPU/连接数已达瓶颈时，系统 **持续运行** 观察 GC、连接泄漏、票务账务是否漂移。
- **压力测试**：从低并发 **阶梯加压** 至 **5000 TPS** 或 CPU 打满，观察拐点与降级是否按预案触发。
- **强度测试**：模拟 **单机故障、网络分区、磁盘满、核验接口突然超时** 等 **极限/恶劣环境**，看是否丢单、错账。
- **并发测试**：综合加压与业务混合（查+订+退），看 **分布式事务与缓存** 是否一致。
- **容量测试**：验证宣称的 **≥10 万并发核心访问**、**1000+ 运营用户**、**50 万+ 实时接入** 是否在极限下仍满足 SLA。
- **可靠性/长稳**：7×24 或节前 **Soak**，贴合 **99.99%** 可用目标。
- **配置测试**：不同 **副本数、线程池、缓存 TTL、JVM 参数** 组合寻优。
- **速度测试**：购票 **≤1 秒**、看板 **PB 查询 ≤5～10 秒**、流式 **≤2 秒** 等端到端耗时。

## 五、安全测试类型

数据加密、用户身份验证、日志文件、Session、备份与恢复、访问控制、安全漏洞（XSS、SQL 注入等）、TCP 端口、服务器端脚本、防火墙等；选择题概念辨识。

**案例（航空运营智能管理平台）**：验证 **TLS** 全链路、敏感字段 **加密存储**、**民航局身份核验** 对接与失败降级；**RBAC** 与权限审批、**异地登录/暴力破解** 告警；**票务与权限变更审计**、日志 **≥1 年** 可追溯；**备份恢复与季度归档**（备份 **≥3 年**）；**SQL 注入/XSS/DDoS** 扫描与 WAF 策略；用户画像 **脱敏** 合规。

## 六、测试模型 W/H/X（★★★☆☆）

| 模型 | 特点 | 与开发关系 |
|------|------|------------|
| **W 模型** | 在 V 模型基础上发展；测试与开发**同步** | 同步 |
| **H 模型** | 测试活动**独立**，贯穿整个生命周期 | 独立 |
| **X 模型** | 针对**单独程序片段**编码与测试，**频繁交接**集成 | 交错 |

**案例（航空运营智能管理平台）**：

- **W 模型**：需求/设计评审与 **测试设计、用例** 同步（如春运峰值场景与性能指标同时定稿）；与 V 模型一致但强调 **左移**。
- **H 模型**：**测试团队独立**，全程介入单元策略、集成计划、压测环境、节前演练，不依附于某一开发阶段尾部。
- **X 模型**：各微服务 **独立开发自测** 后 **频繁集成** 到联调环境，适合多团队并行与 **CI 流水线** 中的小步提交。

## 七、速记清单（考前扫一眼）

1. **黑盒**：功能、不看内部；**白盒**：结构、覆盖路径；**灰盒**：集成、两者结合。
2. **静态**：不运行、分析代码；**动态**：运行、比较结果。
3. **白盒覆盖**：路条组判｜条条判语（弱→强）；路径最强，语句最弱。
4. **负载**：已达瓶颈后继续运行；**压力**：不断加压至瓶颈；**强度**：极限状态（超负荷/环境）；**并发**：负载+压力结合。
5. **W/H/X**：W 同步、H 独立、X 交错。
