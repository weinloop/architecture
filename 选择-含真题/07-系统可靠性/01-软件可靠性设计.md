## 核心要点

| 类别 | 技术 | 要点 |
|------|------|------|
| **容错** | 恢复块 | 多版本实现同功能，故障时切换备用块 |
| **容错** | **N 版本程序** | 多版本独立设计 + **表决**；降低共模故障 |
| **容错** | 冗余 | 主备模块切换，提高可靠、增加成本 |
| **检错** | 检错技术 | 检测点、超时/结果监测，故障停跑与告警 |
| **降复杂度** | 简化结构/数据流 | 降缺陷密度 |
| **配置** | 双机热备 | **心跳**、主备切换 |
| **配置** | 服务器集群 | 多机一体，故障由其他节点接管 |

**N 版本补充（相对常规开发）**：常考多出的阶段如 **相异成分规范评审、相异性确认、背对背测试** 等（以教材/真题表述为准）。

**可靠性管理**：需求阶段可定目标、因素分析、验收标准等；**可靠性建模**等多在测试/数据积累阶段开展。

### （2024年5月·第15题）

在软件可靠性管理过程中，以下工作不属于需求分析阶段应完成的是（）。
问题（1）
（A）分析可能影响可靠性的因素
（B）确定软件的可靠性目标
（C）可靠性建模
（D）确定可靠性的验收标准

**正确答案：** C

**解析**

本题考察的是软件可靠性管理各阶段的关键活动。
 根据可靠性管理流程：需求分析阶段的主要任务包括：
分析可能影响软件可靠性的因素；
确定软件的可靠性目标；
明确可靠性的验收标准；
确定管理机制与数据收集规范；
制定初步的可靠性管理计划。
可靠性建模是测试阶段的重要任务之一，通过收集测试数据并结合实际运行环境建立数学模型，以
评估和预测软件的可靠性。
A选项分析可能影响可靠性的因素：正确，属于需求阶段的初期分析工作。
B选项确定软件的可靠性目标：正确，属于需求阶段的核心任务之一。
C选项可靠性建模：错误，建模工作依赖测试数据，通常在测试阶段完成。
D选项确定可靠性的验收标准：正确，是需求阶段确定评估标准的重要内容。
 因此，正确答案为 C。

### （2024年11月·第51题）

与通常软件开发过程不同的是，N版本程序设计增加了三个新的阶段，分别是（）。
问题（1）
（A）相同成分规范评审、面对面测试和背对背测试
（B）相同成分规范评审、相异性确认和背对背测试
（C）相异成分规范评审、相异性确认和背对背测试
（D）相异成分规范评审、面对面测试和背对背测试

**正确答案：** C

**解析**

本题考察的是N版本程序设计（N-Version Programming, NVP）中为实现容错与可靠性所引
入的新开发阶段。
 与普通软件开发过程相比，N版本设计通过多版本冗余提高容错能力，增加了三个特有阶段：
相异成分规范评审：多个开发小组在不了解其他组设计的前提下独立开发，确保设计“相异性
”，通过独立理解同一份 SRS，并通过问题单反馈与协调。
相异性确认：在详细设计之后，通过分析或评估确保不同版本设计的实现机制、方法等具有相异
性，从而降低公共故障的概率。
背对背测试（Back-to-Back Testing）：对各个版本使用相同的测试数据，并行执行，对比输出
结果，检测是否存在一致性问题，从而发现潜在的软件缺陷。
因此，正确答案是：C. 相异成分规范评审、相异性确认和背对背测试。
