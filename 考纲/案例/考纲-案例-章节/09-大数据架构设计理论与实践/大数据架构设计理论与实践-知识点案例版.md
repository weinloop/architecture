## 9.1 大数据处理系统概述

### 考纲要点

- 传统数据处理系统存在的问题
- 大数据处理系统面临的挑战
- 大数据处理系统的定义、范围和特征

### 考试详细内容

#### 传统系统问题

- **Scale-up 瓶颈**：单机 CPU/内存/磁盘上限；难以应对 **PB 级**与突发流量。
- **架构耦合**：OLTP 与重分析混跑相互拖累。
- **时效性**：批处理 T+1 难以支撑**实时运营**。

#### 大数据挑战

- **Volume/Velocity/Variety/Veracity**；**成本**与**运维复杂度**；**安全与隐私**；**数据质量**。

#### 定义、范围与特征

- **定义**：面向海量、多源、高速数据，提供**采集、存储、计算、分析与服务**的一体化能力。
- **范围**：批处理、流处理、交互式分析、机器学习平台、数据治理与元数据。
- **特征**：分布式、可扩展、容错、最终一致性常见、与云原生结合。

### 航空运营智能管理平台案例

- **痛点对齐**：原有多系统分散、决策靠经验、峰值系统不稳（素材 1.3）；需整合 **10PB+ 历史** 与 **日均约 800GB 实时**（素材 6.3）。
- **目标对齐**：**AI + 大数据** 优化航线、检修、旅客服务（素材 1.4）；数据服务模块承担**采集、计算、挖掘、可视化**（素材 3）。

## 9.2 典型的大数据架构

### 9.2.1 Lambda 架构

#### 考纲要点

- Lambda 的来源、定义、应用场景和优缺点
- Lambda 的体系结构（Batch Layer（批处理层）、Speed Layer（加速层）、Serving Layer（服务层））

#### 考试详细内容

- **定义**：同时保留**不可变主数据集（批）**与**实时增量视图（速）**，在**服务层**合并对外查询，兼顾**准确性与低延迟**。
- **三层**：
  - **Batch Layer**：全量/增量批处理，生成**预计算视图**（准确、高吞吐）。
  - **Speed Layer**：流处理补偿**最新增量**（低延迟）。
  - **Serving Layer**：对外提供低延迟查询接口（API/OLAP/缓存）。
- **优点**：实时 + 离线互补，成熟可解释。
- **缺点**：**双栈维护**成本高；逻辑重复与一致性校验复杂。

#### 案例映射（强贴合素材）

| Lambda 层 | 本项目素材依据 |
|-------------|------------------|
| **Batch Layer** | **T+1** 处理每日 **800GB** 增量、**10PB+** 历史；模型训练 **≤12 小时**；批量成功率 **≥99.9%** |
| **Speed Layer** | **秒级采集**、流式延迟 **≤2 秒**、告警 **≤3 秒**；峰值 **15MB/s+** 写入；**50 万+** 并发实时接入 |
| **Serving Layer** | **数据可视化与接口服务**；**PB 级查询 ≤5s/10s**；运营侧 **≤3 秒** 更新延迟 |

### 9.2.2 Kappa 架构

#### 考纲要点

- Kappa 的来源、定义、应用场景和优缺点
- Kappa 的体系结构
- 常见 Kappa 架构的变形架构

#### 考试详细内容

- **定义**：**以流为唯一事实来源**，通过**日志保留与重放**完成批处理等价任务，减少双栈。
- **结构**：流处理 + 消息日志（如 Kafka）+ 状态存储 + 查询服务。
- **优点**：架构简化、实时一致性强（在流语义内）。
- **缺点**：对**消息系统、状态管理、回放成本**要求高；复杂离线报表仍需**流批一体引擎**或变形。

#### 变形

- **Kappa+**：流为主，必要时**周期性 compaction** 或 **Spark 结构化流** 统一 API。

#### 案例

素材同时强调 **T+1 批** 与 **实时流**，工程上常见 **Lambda** 或 **Lambda 向 Kappa/流批一体演进**。答题可写：当前以 **“批处理保证全量准确 + 流处理保证低延迟”** 为主，与 **Lambda** 高度同构；若未来统一日志重放与流批一体，可向 **Kappa/变形** 演进（论述题常用）。

## 9.3 案例分析

### 考纲要点

- 某网在奥运转播中的大数据应用
- 某网广告平台的应用与演进
- 某证券公司大数据系统
- 某电商智能决策大数据系统

### 考试详细内容

共性结构：**高并发采集 → 实时处理（指标/推荐/风控）→ 离线数仓建模 → 机器学习 → 大屏/运营产品**；演进上从**烟囱**走向**中台/数据平台**。

### 航空运营智能管理平台案例（与四类案例对照）

| 考纲案例 | 与本项目的对应写法 |
|----------|----------------------|
| **奥运转播** | **突发流量 + 实时指标 + 低延迟可视化**；对应春运/航变峰值、**≤3 秒** 大屏更新 |
| **广告平台** | **实时竞价/策略** ↔ **动态票价**、实时销售数据驱动策略（素材 6.2） |
| **证券大数据** | **强合规 + 精确账务** ↔ **票务交易强一致** + **审计日志**；**风控** ↔ **超售/异常退票/欺诈检测** |
| **电商智能决策** | **推荐/补货** ↔ **航线预测、航材采购、旅客偏好模型**（素材 6.2 离线应用） |

**一句话收束**：本项目 **数据服务模块** 承担“**采集—批流计算—挖掘—可视化/接口**”闭环，数据量级与 SLA 已满足**典型互联网级大数据平台**特征。
