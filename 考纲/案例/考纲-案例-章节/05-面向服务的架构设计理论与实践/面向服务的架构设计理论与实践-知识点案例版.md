## 5.1 面向服务的架构概述

### 考纲要点

- SOA 的定义和发展现状
- 面向 Web 服务的业务流程执行语言（BPEL）
- SOA 的参考架构（IBM 的 Websphere 业务集成参考架构）
- SOA 主要遵守的协议和规范（UDDI 协议、WSDL 规范、SOAP 协议和 REST 规范）
- SOA 的作用

### 考试详细内容

#### 定义与发展

- **SOA（Service-Oriented Architecture）**：通过**可复用的服务**与**明确契约**组织系统，强调**松耦合、可组合、可治理**。
- **发展**：从 **Web Services（SOAP/WSDL）+ ESB** 的企业集成，演进到 **REST/HTTP + 微服务 + API 网关 + 事件驱动**；理念延续，技术栈更新。

#### BPEL

- **BPEL（WS-BPEL）**：编排**多个 Web 服务**完成长事务业务流程（顺序、分支、补偿）；适合早期 SOA 编排，现代场景常被 **BPMN 引擎、编排服务、Saga** 等替代或并存。

#### 参考架构（IBM WebSphere BI 等）

- 典型分层：**接入/交互、流程、服务、集成、信息、基础设施**（教材图示为准）；体现**集成中介、流程编排、服务治理**。

#### 协议与规范

| 名称 | 作用（答题要点） |
|------|------------------|
| **WSDL** | 描述服务接口：操作、消息、绑定、端点 |
| **SOAP** | 基于 XML 的报文协议，常与 HTTP 绑定 |
| **REST** | 资源风格，常用 HTTP + JSON；**无状态、统一接口** |
| **UDDI** | 服务**注册与发现**（历史上服务目录；现多由注册中心替代） |

#### SOA 的作用

- **重用与解耦**、**异构集成**、**流程编排**、**统一治理**（安全、监控、版本、SLA）。

### 航空运营智能管理平台案例

- **服务化**：七大模块以**业务能力服务**形式提供（票务、旅客、航信、检修、数据服务等），对外 **API** 与内部 **事件** 组合，符合 SOA“**标准契约 + 松耦合**”核心思想。
- **协议现实**：对外旅客与第三方集成多用 **REST/HTTPS**；监管类对接可能是 **SOAP/XML** 或专用规范（素材未写细节，答题写“按对方接口规范”）。
- **注册发现**：云原生环境下由 **服务注册中心** 承担 UDDI 的“目录”职能。
- **编排**：第 4 节**多模块联动**即业务流程；实现上可用**编排服务 + 事件 + 分布式事务**，与 BPEL 时代“长流程”目标一致、技术不同。
- **ESB 角色**：可由 **API 网关 + 消息总线** 分工替代（路由、协议转换、鉴权、审计）。

## 5.2 面向服务的架构设计

### 考纲要点

- SOA 的设计原则
- SOA 的设计模式（服务注册表模式、企业服务总线模式、微服务模式）
- 设计 SOA 架构应注意的问题

### 考试详细内容

#### 常见设计原则（教材归纳）

- **服务契约明确**（接口稳定、版本策略）
- **松耦合、高内聚**
- **可发现、可组合**
- **无状态/外部化状态**（利于伸缩）
- **标准化**（数据模型、错误码、安全）

#### 三种模式（考纲）

| 模式 | 要点 |
|------|------|
| **服务注册表** | 服务元数据注册、发现、健康检查；对应注册中心 |
| **企业服务总线（ESB）** | 集中路由、转换、中介、策略执行 |
| **微服务** | 细粒度服务、独立部署、去中心化治理；**SOA 理念在运行时的极端化** |

#### 应注意的问题

- **分布式复杂性**：事务、一致性、调试、运维。
- **服务粒度**：过细导致调用链过长；过粗失去复用。
- **治理**：版本、弃用、兼容性、SLA、配额。
- **安全**：服务间认证鉴权、数据最小暴露。

### 案例

- **原则落地**：票务与库存服务**契约**固定，支撑 **≤1s** 与 **强一致**；**无状态实例 + 分布式会话/令牌** 支撑弹性。
- **注册表**：各微服务注册，网关与调用方发现实例。
- **ESB 与网关**：南北向 **TLS、鉴权、限流**；东西向可考虑 **Mesh**（与云原生章衔接）。
- **微服务**：素材明确 **云原生微服务**；按域拆分避免“**分布式单体**”。

## 5.3 面向服务的架构实施过程

### 考纲要点

- 选择 SOA 解决方案
- 业务流程分析（建立服务模型和建立业务流程）

### 考试详细内容

#### 选择 SOA 解决方案

- **维度**：业务适配度、集成能力、协议支持、治理与监控、团队技能、TCO、供应商锁定风险。
- **路线**：偏 **ESB 集中治理** vs **轻量 API + 微服务去中心化**；可混合。

#### 业务流程分析

- **建立业务流程**：从价值链梳理端到端流程（购票、航变处置、检修联动等），识别**编排点**与**人工审批点**。
- **建立服务模型**：**自顶向下**（流程拆服务）或**自底向上**（能力沉淀）；产出**服务目录**、**数据主权**、**接口清单**。

### 案例

- **解决方案**：选用 **云原生微服务 + API 网关 + 消息中间件 + 数据服务平台**，与 **大数据与 AI** 一体建设（素材 1.5）。
- **流程**：以第 4 节链路为**候选编排**：下单→同步→运力校验→核验→行程→通知→检修阈值→分析；识别可**同步调用**与应**异步解耦**的步骤（通知、分析）。
- **服务模型**：例如 **TicketingService、FlightCapacityService、PassengerIdentityAdapter、NotificationService、AnalyticsIngestService**（名称示意）；强调**身份核验**为外部 **Service** 适配器模式。

## 附：与“微服务 / 云原生”关系（论述题常用）

- **微服务可视为 SOA 的一种实现风格**：更小粒度、去中心化、持续交付；**SOA 目标**（重用、集成、治理）仍然成立。
- 本项目同时体现：**SOA 思想**（服务化与集成）+ **云原生工程**（容器、弹性、可观测）（见第 4 章案例考纲）。
