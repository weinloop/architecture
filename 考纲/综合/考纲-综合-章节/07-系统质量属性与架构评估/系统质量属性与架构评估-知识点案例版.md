## 7.1 软件系统质量属性

### 7.1.1 质量属性概念

#### 考纲要点

- 软件质量的定义、范围和具体内涵
- 影响软件质量的主要因素
- 开发期质量属性和运行期质量属性
- GB/T 16260.1 标准

#### 考试详细内容

- **质量**：实体满足明确与隐含需求的程度（产品、过程均可度量）。
- **内涵**：功能性、性能效率、兼容性、可用性、可靠性、安全性、可维护性、可移植性等（**以 GB/T 16260 / ISO 25010 体系为准，教材版本略有差异**）。
- **主要因素**：需求清晰度、过程成熟度、人员能力、架构与技术选型、测试与运维。
- **开发期**：可分析性、可修改性、可测试性等（便于构建与演进）。
- **运行期**：性能、可用性、可靠性、安全性等（用户与运维直接感知）。
- **GB/T 16260.1**：规定质量模型框架与术语，答题写“质量特性分类与度量基础”即可。

#### 航空运营智能管理平台案例

素材同时提出 **效率（效率提升 30%+）**、**可用性（≥99.99%）**、**一致性**、**安全合规**、**多终端体验**，属于**多质量属性并发约束**；分析侧还有 **延迟、吞吐、准确率** 等运行期指标（如流式延迟 ≤2 秒、异常识别准确率 ≥98%）。

### 7.1.2 面向架构评估的质量属性

#### 考纲要点

- 评估方法普遍关注的质量属性（性能、可靠性、可用性、安全性、可修改性、功能性、可变性和互操作性）

#### 考试详细内容

- **架构评估**关注**对架构敏感**的属性，常用：**性能、可靠性、可用性、安全、可修改性**；另：**功能性**（架构能否支撑能力）、**可变性**（适应业务与策略变化）、**互操作性**（与民航局核验、旧系统、多渠道对接）。
- **区分**：可靠性强调**持续正确运行**；可用性强调**可访问比例/停机时间**（常与冗余、容错相关）。

#### 案例速配

| 属性 | 素材中的直接线索 |
|------|------------------|
| 性能 | ≤1s 购票响应、≥5000 TPS、PB 级查询 ≤5s/10s、50 万+实时接入 |
| 可用性 | 系统可用性 ≥99.99%；高并发下稳定 |
| 可靠性 | 数据可靠性 ≥99.999%；强一致性；备份与归档 |
| 安全 | RBAC、TLS、加密、审计、防注入/XSS/DDoS、脱敏 |
| 可修改性 | 微服务拆分；动态定价/策略；模型迭代（预测、预警） |
| 功能性 | 七大模块、全流程数字化、AI+大数据 |
| 可变性 | 旺季峰值、航变突发、业务规则调整 |
| 互操作性 | 身份核验对接、数据同步、跨系统联动分析 |

### 7.1.3 质量属性场景描述

#### 考纲要点

- 质量属性场景的定义、组成和重点关注的属性
- 重点关注的 6 个属性场景的内涵

#### 考试详细内容

- **质量属性场景**：以**可验证**方式描述“在什么条件下，系统应如何响应以体现某质量属性”。
- **常见组成**：**刺激（Stimulus）**、**环境（Environment）**、**响应（Response）**、**度量（Measure）**（不同教材用词略异）。
- **架构评估中常重点练习的六类场景**（与多数教材一致，具体表述以教材为准）：
  1. **性能**
  2. **可用性**
  3. **安全性**
  4. **可修改性**
  5. **易用性**（或使用性）
  6. **可测试性**（部分教材将**互操作性**纳入或替换其一）

#### 航空运营智能管理平台：六类场景示例（刺激—环境—响应—度量）

1. **性能**  
   - **刺激**：票务高峰，交易请求骤增。  
   - **环境**：云原生微服务已启用弹性与缓存。  
   - **响应**：核心购票链路在限制时间内完成。  
   - **度量**：≥5000 TPS；响应 ≤1 秒（素材）。

2. **可用性**  
   - **刺激**：单可用区故障或局部服务异常。  
   - **环境**：多副本、负载均衡、无单点（概念）。  
   - **响应**：业务切换或降级仍对外可用。  
   - **度量**：全年可用性 ≥99.99%（素材）。

3. **安全性**  
   - **刺激**：撞库、SQL 注入、越权访问敏感行程与支付信息。  
   - **环境**：公网多终端接入。  
   - **响应**：阻断攻击、告警、审计可追溯。  
   - **度量**：敏感字段加密、TLS、RBAC、日志保留 ≥1 年（素材）。

4. **可修改性**  
   - **刺激**：退改签规则或票价策略变更；新增分析模型。  
   - **环境**：微服务与数据分层。  
   - **响应**：局部变更、灰度发布。  
   - **度量**：变更影响范围可控、回归成本可接受（定性+发布策略）。

5. **易用性**  
   - **刺激**：旅客与运营人员在多终端完成查询与处置。  
   - **环境**：航变导致信息量激增。  
   - **响应**：信息清晰、流程可完成。  
   - **度量**：多终端适配；可视化更新延迟 ≤3 秒（素材）。

6. **可测试性 / 互操作性**（二选一或同时准备）  
   - **可测试性**：对购票链路、分布式事务、流式告警建立可重复压测与对账测试。  
   - **互操作性**：与民航局身份核验、外部数据源对接成功率和接口稳定性。

## 7.2 系统架构评估

### 7.2.1 系统架构评估中的重要概念

#### 考纲要点

- 架构评估的基本概念
- 核心概念（敏感点和权衡点、风险承担者（利益相关人）和场景）
- 架构评估方法的分类（基于调查问卷或检查表的方式、基于场景的方式和基于度量的方式）

#### 考试详细内容

- **架构评估**：在实现前/早期，对架构决策能否满足质量目标进行**系统化分析**，降低风险。
- **敏感点（Sensitivity Point）**：对某质量属性影响显著的架构决策点（如缓存层对性能敏感）。
- **权衡点（Tradeoff Point）**：改善一个属性可能损害另一属性的决策点（如强一致对延迟；安全审计对性能）。
- **风险承担者 / 干系人**：信息中心、业务部门、旅客、监管方等，关注点不同。
- **场景**：驱动评估的基本单位（见 7.1.3）。
- **方法分类**：
  - **问卷/检查表**：快速、主观依赖经验。
  - **基于场景**：SAAM、ATAM 等，主流。
  - **基于度量**：原型测试、基准数据、监控指标。

#### 案例

- **敏感点**：票务服务的**库存与事务模型**对一致性与性能均敏感；**流式处理链**对实时性敏感。  
- **权衡点**：**PB 级即席查询** vs **在线事务低延迟**——存储模型与资源分配需权衡。  
- **干系人**：旅客重响应与资金安全；管理层重效率与决策数据；监管重合规核验。  
- **度量评估**：上线前压测（TPS、P99）、故障演练（可用性）、渗透测试（安全）。

### 7.2.2 系统架构评估方法

#### 考纲要点

- 非功能质量属性的架构分析方法（SAAM）
- 架构权衡分析方法（ATAM）
- 成本效益分析法（CBAM）
- 其他评估方法（如 SAEM、SAABNet 和 SACMM 等）

#### 考试详细内容

- **SAAM（Software Architecture Analysis Method）**  
  - **目的**：评**可修改性**为主，也可扩展场景到其他属性。  
  - **特点**：场景优先，比较候选架构。

- **ATAM（Architecture Tradeoff Analysis Method）**  
  - **目的**：识别**质量目标—架构决策—风险/非风险点—权衡点**。  
  - **核心输出**：效用树、风险、敏感点、权衡点。

- **CBAM（Cost Benefit Analysis Method）**  
  - **目的**：在 ATAM 等基础上，对架构策略做**经济权衡**（收益 vs 成本）。

- **其他**：了解名称即可（SAEM、SACMM 等），表明架构评估方法族在演进。

#### 案例（用于论述题）

- 用 **SAAM**：对比“单体 + 垂直扩展”与“**微服务 + 弹性**”在**航变突发流量**下的可修改性与扩展成本。  
- 用 **ATAM**：梳理**强一致票务**相关的敏感点（分布式事务、缓存一致性）与权衡点（延迟、复杂度）。  
- 用 **CBAM**：评估增加**专用缓存集群、抗 DDoS 清洗**等的成本与对可用性/安全性的效用提升。

## 7.3 ATAM 方法架构评估实践

#### 考纲要点

- 评估实践的四个基本阶段：演示、调查和分析、测试和报告 ATAM

#### 考试详细内容

ATAM 完整流程在教材中常写为多步骤；考纲归纳为**四阶段**时，建议按下列**意涵**作答（与“演示—调查分析—测试—报告”对齐）：

| 阶段（考纲） | 典型活动内容 |
|--------------|----------------|
| **演示（Presentation）** | 向干系人介绍 ATAM 流程与目标；架构团队讲解**业务驱动因素与质量目标**（效用树雏形）。 |
| **调查和分析（Investigation & Analysis）** | 收集场景；走查架构；用**效用树**细化优先级；识别**架构方法**、**敏感点/权衡点**、**风险与非风险**。 |
| **测试（Testing）** | 对关键假设做**验证**（原型、试验、度量、走查测试设计）——部分教材将“分析中的验证环节”独立强调。 |
| **报告（Report）** | 汇总风险、权衡、建议；形成改进行动清单。 |

#### 航空运营智能管理平台：ATAM 四阶段可写内容示例

1. **演示**：明确驱动目标——效率提升 30%+、99.99% 可用、票务强一致、合规核验与隐私。  
2. **调查分析**：选取场景——春运高峰购票、恶劣天气航变、实时超售风险、检修告警；分析微服务拆分、分布式事务、缓存、流式计算、备份策略；标出权衡（一致性与延迟、安全审计开销）。  
3. **测试**：全链路压测、混沌演练（单点故障）、渗透与漏洞扫描、对账与一致性验证。  
4. **报告**：列出剩余风险（如极端峰值下的第三方依赖）、改进项（限流、降级预案、容量基线）。
