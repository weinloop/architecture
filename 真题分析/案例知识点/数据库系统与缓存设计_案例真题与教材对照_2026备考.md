### 1. 数据库逻辑设计与建模（2020 包裹单等）

| 案例得分点 | 教程依据 |
|------------|----------|
| 逻辑数据模型设计任务：**确定数据模型、E-R→关系模式、完整性约束、用户视图** | **第6章 §6.3** 数据库设计步骤与 **§6.3.2** E-R 转关系 |
| **超类实体** 抽取收件人/寄件人公共属性 | 概念建模常见做法，与 **泛化/特化** 思想一致 |
| **派生属性**（总计由分项费用计算） | **§6.2** 属性分类；是否存储属性能效权衡（同 **反规范化中派生列** 思想） |

**教材原文线索**：逻辑设计阶段需 **确定完整性约束**（实体/参照/用户自定义）（**§6.3.2** 脉络）。

**【平台案例】**  
- **逻辑模型任务**：将「航班—航段—机位—机组—旅客订座—行李—机务工单」等抽象为 E-R，再转为关系模式；为运控、地服、财务分别定义 **用户视图**（同底表不同字段与脱敏策略）。  
- **超类/子类**：将 **参与方** 抽象为超类，子类可为旅客、货主、地面代理、航司内部部门等，共享证件/联系方式类属性，子类扩展舱位等级、合约编号等。  
- **派生属性**：如 **延误时长**（实际起飞/到达与计划值之差）、**轮档时间**、某航季 **准点率** 等可由事件与时间字段计算；答题时可写「是否落库」取决于报表频率与口径审计要求（与派生列/反规范化衔接）。

### 2. 范式、反规范化与冗余一致性（2021 医药等）

| 案例得分点 | 教程依据 |
|------------|----------|
| 反规范化手段：**冗余列、派生列、表重组、表分割（水平/垂直）** | **第6章 §6.3.4 第5点「反规范化」**（约 **239 页** 起） |
| 反规范化目的：**提高读性能**，代价是冗余与更新异常 | 同节 |
| 解决冗余不一致：**应用程序同步、批量处理同步、触发器同步** | 同节明确列举 |
| 业务上「高频更新 + 强一致」倾向 **触发器 / 应用层事务双写** | 在教材三种方法上的**场景化选用** |

**【平台案例】**  
- **反规范化**：在「航班动态宽表」或 **运行快照表** 中 **冗余** 航站中文名、机型名称、登机口名称，减少高峰联表；**派生列** 可存「当前状态摘要码」「是否跨日」等供列表页直接读。  
- **表分割**：按 **航季/日期** 水平分表存放历史航班执行记录；按 **业务域** 垂直拆分「计划与定价」与「执行与事件流」以控制锁竞争。  
- **冗余一致性**：登机口变更时，应用事务内更新主表并刷新冗余字段；或 **夜间批量** 对齐历史统计表；对强一致口径字段可用 **触发器** 防止应用漏更。高频冲突场景（如同时改机位与滑行道）倾向 **应用层协调 + 事务**。

### 3. 缓存架构、一致性与常见问题（多年重复）

| 案例得分点 | 教材与工程常识 |
|------------|------------------|
| **异步准实时** 更新缓存：性能与可伸缩，适合「秒级展示」类业务 | 与 **最终一致性** 实践一致；教程无单独小节，可关联 **§6.3** 反规范化后的同步策略 |
| **Cache-Aside（旁路）**：读 **先缓存→miss 读库→回填**；写 **先写数据库→删除/失效缓存** | 案例标准答案；避免简单「双写」 |
| **双写** 为何仍不一致：任一侧失败、并发读写乱序 | 2019、2024 案例解析 |
| **缓存穿透**：无效 key 打穿；**布隆过滤器** 前置拦截 | 教程 **绪论能力** 中提及缓存穿透等（**§1.2** 能力表述）；方法以真题为准 |
| **缓存雪崩**：大量 key 同时失效；**TTL 随机化**、**多级缓存**、**互斥重建/队列** | 同上 |
| **空值缓存** 风险：恶意随机 key **撑爆内存**；需布隆或规则过滤 | 2019 案例 |

**【平台案例】**  
- **异步准实时**：面向旅客端的 **预计起飞/登机口** 可先写库再 **异步刷新** Redis/推送渠道，大屏展示允许 **秒级～十秒级** 延迟，与 **最终一致性** 一致。  
- **Cache-Aside**：查航班详情 **先读缓存**；miss 则读 **运行库/主数据** 并回填；写操作（如人工改航）**先提交数据库事务，再删除或失效** 对应航班缓存键。  
- **双写与乱序**：并发下「改时刻」与「读缓存」交错可能导致旧值回填，需结合 **版本号、先 DB 后删缓存、延迟双删** 等（与第 5 节 Redis 同步呼应）。  
- **穿透 / 布隆**：用随机航班号、无效行李牌号恶意查询时，**布隆过滤器** 拦截明显不存在的键；合法前缀规则（如航司二字码+数字位数）也可前置校验。  
- **雪崩**：早高峰 **批量航班状态批量过期** 时，对 TTL **加随机抖动**，配合 **多级缓存**（本地 Caffeine + Redis）或 **互斥重建**，避免数据库瞬时尖峰。  
- **空值缓存**：对「经核实不存在」的查询可短 TTL 缓存否定结果，但需 **容量与黑名单** 防护，避免扫描型攻击撑满内存。

### 4. 分布式缓存实现要点（2022/2018）

| 案例得分点 | 说明 |
|------------|------|
| **普通哈希取模**：节点数变化 → **大量迁移** | 与 **§6.3** 物理分布变更需回溯逻辑设计类比：架构层「迁移成本」 |
| **一致性哈希 + 虚拟节点**：扩缩容只影响环上邻域，负载更均衡 | 工程常考点 |
| **MemCache vs Redis**：数据结构丰富度、**持久化**、事务、集群方案 | 2018 填表题 |

**【平台案例】**  
- **普通哈希取模**：若按 **航班号哈希** 分片缓存集群，节点扩缩会导致大量 key 迁移，**高峰换机位/大面积延误** 时易放大抖动。  
- **一致性哈希 + 虚拟节点**：适合 **全国多机场、多活缓存** 场景，扩缩容仅影响环上邻域，**热点航站**（如枢纽机场）可用虚拟节点摊平负载。  
- **MemCache vs Redis**：纯展示型、无复杂结构的 **只读热点** 可考虑 Memcached；平台若需 **ZSet 排行、Hash 结构化航班对象、持久化容灾**，更常选 Redis（与真题对比表一致）。

### 5. Redis 专项（2020/2021/2018）

| 主题 | 案例要点 |
|------|----------|
| **数据类型选型** | 计数 **String(INCR)**；结构化 **Hash**；时序列表 **List**；去重集合 **Set**；排行 **ZSet** |
| **持久化** | **RDB** 快照：恢复快、文件小、可能丢间隔内数据；**AOF** 日志：更安全、文件大、重放慢 |
| **过期与淘汰** | **惰性 + 定期** 无法清理「永不访问的过期键」→ 需 **maxmemory-policy**（LRU/LFU/随机等） |
| **同步到 MySQL** | 应用双写、**Canal+Binlog**、**MQ**、延迟双删；按一致性与吞吐取舍 |

**【平台案例】**  
- **String**：某航季 **航班执行架次计数**、**登机口变更次数** 等 INCR 统计。  
- **Hash**：单架航班的 **结构化快照**（状态、计划/实际时间、机位、跑道、延误原因码）。  
- **List**：**事件时间线**（上轮挡、撤轮挡、推出、除冰等按序追加）。  
- **Set**：某日 **已登机旅客集合**（去重）、**已到位飞机集合**（与业务去重需求结合）。  
- **ZSet**：**机场延误时长排行**、**航线准点率榜**、**机位占用紧张度** 等排行榜场景。  
- **RDB / AOF**：若缓存承担 **限流计数、会话** 等，需权衡 **恢复快** 与 **少丢数据**；纯动态展示可更偏 RDB，带审计倾向可 AOF 或缩短 fsync 策略（具体以运维规范为准）。  
- **过期与淘汰**：海量 **按航班号+日期** 的 key，若仅依赖惰性/定期删除，冷航班可能长期占内存，需配置 **maxmemory-policy**（如 allkeys-lru）与合理 TTL。  
- **同步 MySQL**：**ACARS/地面系统** 写入运行库后，经 **Canal 订阅 Binlog** 或 **MQ** 刷新 Redis；写缓存后删缓存的 **延迟双删** 可减轻与读请求的竞争。

### 6. 数据架构选型（2015 / 2024）

| 主题 | 教材依据 |
|------|----------|
| 文件系统 vs RDBMS：设计难度、冗余、以谁为中心、扩展性 | **§6.1** 数据库系统优势（共享、冗余控制、独立性） |
| 内存数据库 vs 关系库：模型、性能、容量、可靠性 | **§6.3.5 物理设计**「高频数据放高速设备」**（§9488-9490）** 与案例混合架构呼应 |
| 地理空间分层：接口 / 处理 / 存储；**MySQL + HDFS + HBase + ES** | **第6章** 传统 RDBMS；**第11章** 大数据/NoSQL 扩展；教程作体系补充 |
| **MongoDB**：BSON 文档、模式灵活；**向量字段 + 相似检索** | **§6.5 NoSQL** 思路；向量题为 **2024 新形态** |

**【平台案例】**  
- **文件系统 vs RDBMS**：**QAR/FDR 原始轨迹、舱音索引、PDF 航行通告、气象格点文件** 等适合 **对象存储/文件系统**；**航班计划、票务与结算、适航与维修记录** 等强一致业务以 **RDBMS** 为中心，享事务与参照完整性。  
- **内存库 + 关系库**：**当日运行态势、热点机场列表** 进 Redis/内存分析引擎；**账套与审计日志** 持久化在关系库，符合教材「高频数据放高速设备」的物理设计思想。  
- **分层（接口 / 处理 / 存储）**：API 网关对接 App/大屏；处理层做 **聚合、规则引擎、流处理**；存储层 **MySQL** 存结构化主数据，**HDFS/HBase** 存海量轨迹与日志，**ES** 做全文检索（机号、航班号、事件描述）。  
- **MongoDB + 向量**：可将 **机务案例、故障现象描述、排故报告** 作文档库；**向量检索** 用于「相似故障/相似天气下历史处置」辅助（与 2024 向量新题型同构，领域换成航空）。

### 7. 其他（2025 医疗图谱 / Scrapy）

- **图数据库** 存知识图谱：三元组、多跳查询、Cypher 等——与 **关系模型** 对比优势在「关系遍历」；教程 **§6.1～6.2** 讲关系模型，图库作 **扩展架构选型**。
- **Scrapy**：Engine / **Scheduler** / **Downloader** / Spider / **Item Pipeline**；**异步 I/O**：非阻塞、回调/事件驱动、高并发网络。

**【平台案例】**  
- **图数据库**：将 **机场—航线—机型—适航限制** 建成图，便于 **多跳查询**（如某故障件影响哪些航线/机队、备降场可达性、枢纽中转链），对比纯关系库在 **深度遍历** 上的代价；智能问答可围绕 **运行规章与排故知识图谱**（类比医疗图谱流水线，领域替换为航空运行与机务）。  
- **Scrapy 与异步 I/O**：可用于 **抓取公开航情、气象产品、NOTAM 文本** 等外源页面，经 **Pipeline** 清洗入队；Downloader 侧 **异步** 提高多源并发拉取效率（注意合规与源站 robots/授权，答题侧重架构组件与 IO 模型）。

## 三、答题与复习建议

1. **背「流程填空」**：**旁路缓存** 读写步骤、**先 DB 后删缓存**、miss 后 **回源 + 回填**。  
2. **背「对比类」**：RDB vs AOF；哈希 vs 一致性哈希；MemCache vs Redis；文件 vs RDBMS。  
3. **与上午题联动**：反规范化三种手段与三种同步方式（**§6.3.4 反规范化**）与案例 **2021** 完全同构。  
4. **新题型**：**2024–2025** 出现 **分层填图、图数据库、Scrapy、异步 IO、向量**，需在教材之外补充 **大数据与工程常识**，但答题仍要 **分点、短句、扣题**。  
5. **案例素材**：默写套路时可用 **航空运营智能管理平台** 串起「计划—执行—事件—分析」数据链，把 **E-R/反规范化/旁路缓存/一致性哈希/Redis 类型/冷热分层** 落到 **航班、机位、机组、旅客、机务** 等具体对象上，避免空泛。
