## 二、问题（1）：身份鉴别——口令 vs 公钥体系（PKI）

### 案例得分逻辑

- **口令认证**：实现简单，但易受弱口令、字典攻击、嗅探、重放等；口令常与服务器**共享**，难以天然绑定「用户独有长期秘密」。  
- **公钥体系（PKI）**：**私钥不出网**、可配合证书完成强认证；私钥可用于**签名**或保护数据，与「终端私有敏感文件」类需求结合更顺。  
- **规模与场景**：题干 **100 人以内、局域网**，公钥运算开销可接受。

### 教材对照

| 教材位置 | 内容摘要 |
|----------|----------|
| **§4.4.3 非对称密钥加密算法** | 公钥加密、私钥解密实现保密；私钥签名、公钥验证实现**数字签名**；RSA 等典型算法 |
| **§4.5.2 公钥加密体制的密钥管理** | **公钥证书（CA）** 交换公钥，解决公钥可信问题 |
| **§4.6.1 访问控制** | 访问控制含 **认证**、策略、审计；认证是主体/客体互信的基础 |

**答题关键词**：抗攻击能力、私钥不传输、与用户私有数据保护结合、用户规模小故性能可接受。

**【平台案例】**  
- **口令认证的局限**：地服子系统若仅用 **共享账号+静态口令**，易出现 **弱口令、撞库、屏幕旁窥**；内网嗅探或木马可截获 **重放凭证**。旅客值机柜台、货运柜台等 **多人共用终端** 时，口令更难绑定到「个人长期秘密」。  
- **PKI 的契合点**：**签派、放行、关键运行变更** 要求主体可审计——**个人证书 / 智能卡** 使 **私钥驻留终端或密码模块、不出网**，与 **电子签名、责任追溯** 一致；与 **机组排班、执照校验** 等「终端侧敏感文件」保护同一套信任根。  
- **规模与性能**：若题干类似 **「某航司运行中心 + 关联航站」百人级内网**，公钥运算（登录、握手）相对 **全国旅客并发** 很小，**性能可接受** 的理由可直接写进答案。  
- **对比表述**：口令适合 **低风险内部查询**；**PKI** 适合 **高敏操作入口 + 需与数字签名/加密邮件联动** 的平台模块（与案例「为何选公钥认证」同构）。

## 三、问题（2）：大文件加密——流加密 + 数字信封

### 案例结论

- **数百兆级单文件** → 优先 **流密码（Stream Cipher）**：连续处理、适合大流量吞吐（案例解析相对分组密码强调**效率**）。  
- **数字信封**：用 **公钥加密保护「对称会话密钥」**，用 **对称密钥加密正文**；兼顾非对称的安全性（密钥分发）与对称的高性能（ bulk data）。

### 教材对照

| 概念 | 教材依据 |
|------|----------|
| 对称加密（单钥）用于大量数据、公钥用于密钥保护 | **§4.5.3 公钥加密分配单钥密码体制的密钥**（图4-5 会话密钥建立思想：**用公钥体制保护会话密钥**，再用于后续保密通信） |
| DES/AES 等为分组密码（块） | **§4.4.2 对称密钥加密算法**（DES 按块处理等） |
| 流密码与分组密码的对比 | 教程以 **分组密码** 展开较多；**流密码** 可在案例中按「长文件、吞吐」作答，并注明与 **§4.4.2** 的**对称加密体系**同属一大类 |

**标准过程表述（与案例答案一致）**：

1. **加密**：生成随机对称密钥 → 用用户**公钥**加密对称密钥置于文件头 → 用对称密钥对数据做 **流加密**（正文）。  
2. **解密**：用户用**私钥**解开文件头得到对称密钥 → 对称密钥解密正文。

**【平台案例】**  
- **为何是大文件**：**QAR 原始记录、FDR 子帧导出、长时段舱音、气象再分析格点、场面监控视频归档** 等常达 **数百 MB 级**，适合强调 **流密码连续处理、高吞吐**（相对分组密码在答案里突出 **效率与实时性**）。  
- **数字信封在业务中的含义**：将 **随机对称会话密钥** 用 **接收方公钥** 封装在包头，**正文** 用对称密钥做 **流加密**——等价于「把高敏运行包安全交给 **维修工程部 / 监管报送通道**」，兼顾 **RSA/ECC 解决密钥分发** 与 **对称算法扛 bulk data**。  
- **流程复述**：签派员或机务导出 **加密运行数据包** → 文件头为 **公钥加密的会话密钥** → 正文流加密；接收方 **私钥解开头** → 再解正文（与上节标准表述一致，仅把「用户」换成 **航站/监管/协作单位** 接收方）。

## 四、问题（3）：数据库敏感字段——加解密 API vs 透明加密（TDE）

### 对比要点（答题骨架）

| 方式 | 含义 | 优点 | 缺点 |
|------|------|------|------|
| **加解密 API** | 应用读写敏感字段时显式调用 DB API 做加解密 | **灵活**，可按字段选算法与密钥强度 | 改应用代码、**安全职责分散**、维护成本高 |
| **透明加密 TDE** | DBMS 内部按策略自动加解密，应用仍见明文逻辑 | **应用无改**、策略集中、**安全管理员**配置密钥与强度 | 灵活性略弱于全自定义 API |

### 案例结论

- 需求：**不同敏感级别不同密钥强度** + **尽量减少安全与开发负担** → 选 **透明加密（TDE）**。

### 教材对照

| 教材位置 | 说明 |
|----------|------|
| **§4.1.2 信息存储安全** | 标识与鉴别、存取权限、监控、加密等**存储侧**安全措施 |
| **第6章** 数据库完整性、安全与访问控制 | 字段级保护与实现策略在工程上对应 **TDE / 列加密** 等；教程以原理为主，**TDE 名词**以案例与厂商文档为准 |

**【平台案例】**  
- **加解密 API**：若仅 **少数字段**（如某报表里的 **机组联系方式**）需与国密算法、外部 HSM 深度耦合，可在应用内 **显式调用** 加解密接口；代价是 **各业务线重复实现、密钥使用分散、代码审计面大**——适合 **强定制** 而非「全库统一策略」。  
- **透明加密 TDE**：**旅客证件号、常旅客联系方式、结算银行卡号、合同金额** 等大量敏感列若要求 **算法与密钥轮换由 DBA/安全岗统一配置**、应用仍按明文逻辑开发（权限仍由库与应用层控制），则与案例结论一致：选 **TDE** 以 **减少开发与运维的安全职责碎片化**。  
- **综合叙述**：题干若同时出现 **「不同敏感级别不同密钥强度」+「减轻开发与安全管理负担」**，可答：策略在 **数据库层集中**，避免每个微服务各自封装加解密；与 **§4.1.2 存储安全**、**第6章** 字段保护 **交叉呼应**。

## 五、备考提示

1. **架构师案例** 常考「**为何不用 A 用 B**」：先写 **威胁与弱点**，再写 **机制如何对症**，最后 **规模/性能是否可接受**。  
2. **数字信封** 与教材 **§4.5.3「公钥加密分配单钥密码体制的密钥」** 同一思想，可互证。  
3. **流 vs 分组** 若遇辨析题：大文件、吞吐优先时倾向 **流加密**；块加密强调 **块填充、模式（教材未展开流密码细节时以案例表述为准）**。  
4. 本汇编仅 **1 题**，建议与「**信息安全**」上午真题、**第4章** 教材同步复习，避免知识孤岛。  
5. **案例素材**：默写「为何 A 不选、为何选 B」时，可用 **航空运营智能管理平台** 串起 **主体（运控/地服/机务/旅客数据）—传输（大文件包）—存储（库内敏感字段）**， threats 写 **嗅探、重放、弱口令、职责分散**，机制写 **PKI、数字信封、TDE**，落点具体，避免空泛。
