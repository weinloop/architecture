**【题型 · 与航空运营智能管理平台类比（备考「带场景」）】**  
| 题型 | 平台中可写的素材 |
|------|------------------|
| **A 效用树** | **性能**：航班动态/App 查询 **P95 时延**、行李分拣 **吞吐**；**可用性**：运控双活、**RTO/RPO**、大屏不断流；**安全**：舱单/证照 **审计**、渗透与权限；**可修改性**：航季 **改界面/改规则** 人天；**易用性**：柜台 **少步完成** 值机 |
| **B 风险 / 敏感点 / 权衡点** | **风险**：需求未收敛就定「全国一张图」、紧耦合遗留与新建；**敏感点**：心跳间隔影响 **误报与切换**；**权衡**：**链路加密** ↑ 与 **席位响应** ↓ |
| **C 架构风格** | **管道-过滤器**：**ACARS/报文解析→规则校验→入库** 流水线；**仓库**：**运行态势/资源池** 中心模型多子系统读写；**解释器**：**运价/除冰/限重** 规则外置；**规则 vs OO**：促销级 **动态规则** vs **机型机队** 稳定对象模型 |
| **D 非功能四类** | **瘦客户端**：航站 **浏览器席位**；**胖客户端**：**本地缓存图形气象、离线预案** 的运控终端 |
| **E Lambda / Kappa** | **批**：历史航班、准点统计、**QAR 离线分析**；**流**：**ADS-B/场面雷达** 实时；**服务层**：API 给 App/大屏 |
| **F 微服务 + 质量** | **单体**：早期 **本地航站** 一体化；**微服务**：**订座、运行、货运、结算** 独立演进；效用树与 **6 要素** 套 **航班改签高峰** |
| **G 归类 + Ping/Echo vs 心跳** | **Ping/Echo**：对 **关键服务实例** 主动探测；**心跳**：**机位设备、车辆、网关** 周期上报；场景映射到 **除冰完成、登机开始** 等 |
| **H 大模型平台** | **解释器**：**训练/推理流水线** 节点可配置；业务上 **排故脚本、签派辅助** 与「代码即任务」类比 |
| **I AES 长数据** | **地空数据链长报文**、**批量旅客名单加密导出** 等需 **分块 + 填充 + 模式**（题型与充电案例同构，领域换航空） |
| **J DDD + 弱网一致** | **限界上下文**：**运行控制 / 旅客服务 / 机务 / 结算**；弱网航站：**本地队列、幂等、对账**（行李牌/餐食订单与真题餐饮同构） |

**【平台案例总览 · 航空运营智能管理平台】**  
平台横跨 **计划—执行—监控—分析**：前端有 **App/大屏/柜台/机务终端**，后端有 **实时流（位置/事件）** 与 **批处理（统计/合规/报表）**，架构评估常落在 **能否兑现 SLA**（时延、可用）、**能否安全演进**（可修改性、服务拆分）、以及 **批流架构选型**（Lambda/Kappa）。下文各节 **【平台案例】** 均以此统一场景展开。

## 二、教材章节精确定位

### 1. 第8章——质量属性与架构评估（核心）

| 真题考点 | 教程位置 |
|----------|----------|
| **质量属性**（性能、可用性、安全性、可修改性、易用性、可测试性、互操作性等） | **§8.1** |
| **质量属性场景六要素**：刺激源、刺激、环境、制品、响应、响应度量 | **§8.1**（架构评估常用） |
| **效用树 Utility Tree** | **ATAM / 架构评估实践 §8.3** 配套工件 |
| **敏感点**：影响某一质量属性的关键构件特性 | **§8.2** 架构评估重要概念 |
| **权衡点**：影响多个质量属性，是多个敏感点的交汇 | **同上**（教材例：**加密级别** 影响安全与性能） |
| **架构风险**：错误决策带来的隐患 | **§8.2** |
| **ATAM** | **§8.3** |

**答题提示**：风险多选 **需求未收敛、紧耦合、业务逻辑未达成共识**；敏感点多选 **“某指标影响协议/结构设计”**；权衡点多选 **“安全↑ 性能↓”** 类表述。

**【平台案例】**  
- **效用树**：根结点下列 **性能、可用性、安全、可修改性** 等；叶结点用编号需求，例如 **(a)**「全国航班动态 **1 秒内** 刷新」→ **性能**；**(b)**「运控主备 **自动切换**、业务不断」→ **可用性**；**(c)**「舱单与证照操作 **可追溯审计**」→ **安全**；**(d)**「航季切换时 **两周内** 改完界面」→ **可修改性**。  
- **质量场景 6 要素**：例：**刺激源** 旅客；**刺激** 并发查询；**环境** 早高峰；**制品** 航班查询服务；**响应** 返回列表；**响应度量** **P95 &lt; 500ms**（数字按题干）。  
- **敏感点**：如 **会话保持策略**、**缓存一致性协议** 直接影响 **可用性与性能** 的数值。  
- **权衡点**：如 **全链路 TLS** 提升 **安全**，增加 **CPU 与握手时延**（§8.2 加密级别例的同构）。  
- **架构风险**：如 **未统一「延误原因」口径** 就开发多系统报表 → 后期 **集成风险**；**ATAM** 用于重大版本上线前 **干系人共识**（运控、地服、信息部）。

### 2. 第7章——软件架构风格（风格对比题）

| 风格 | 适用场景（与真题一致处） |
|------|---------------------------|
| **管道-过滤器** | 固定阶段数据流、编译流水线；**不适合**频繁改流程的交互式 IDE |
| **仓库 / Repository（数据仓储）** | **中心数据结构**，多工具围绕同一模型（语法树/AST、IDE 集成） |
| **解释器 Interpreter** | **规则/脚本/流程外置**，促销规则、ML 流程拖拽、**代码即任务** |
| **隐式调用 / 事件** | 事件驱动、解耦；全局流程可控性弱于解释器 |
| **面向对象** | 直观、性能好；规则多变时需频繁改类层次 |
| **基于规则** | 规则引擎、动态折扣；与「解释器」在案例中常同属「规则外置」家族 |

教材依据：**§7.3** 各风格定义与典型应用。

**【平台案例】**  
- **管道-过滤器**：**报文接入 → 解析校验 → 规则引擎 → 持久化 → 推送** 等 **固定阶段流水线**，适合 **批式、可重复** 的处理；若某 IDE 式「拖拽任意改流程」则 **不适合** 纯管道（与真题 IDE 类比：换成 **交互式排班/除冰沙盘** 若要求极度灵活，需配合 **解释器/编排**）。  
- **仓库（Repository）**：**运行态势库**（航班、机位、跑道、除冰位）作为 **中心数据结构**，**离港、场面引导、配载** 等子系统 **围绕同一模型** 读写，类似 **语法树/AST** 地位。  
- **解释器**：**运价规则、限载、除冰程序、除冰等待时间** 等 **外置为脚本/配置**，换航季 **少改代码**；**大模型训练平台** 类比为 **「DAG/任务即配置」** 的 **解释执行**。  
- **隐式调用 / 事件**：**航班状态变更** 发布事件，**登机口屏、行李、配餐** 订阅；**解耦** 强，但 **全局编排可视性** 弱于中心化解释器，需 **流程监控** 补强。  
- **OO vs 规则**：**机型、航段、舱位** 等相对稳定用 **对象模型**；**促销、临时航路限制、动态费率** 多变可用 **规则引擎 / 解释器** 减少类爆炸。

### 3. 第19章——大数据架构（2023 案例）

| 考点 | 教程位置 |
|------|----------|
| **Lambda**：批处理层 + 速度层 + 服务层；批 + 流双路径 | **§19.3、§19.5** |
| **Kappa**：以流处理**单链路**统一批与实时 | **§19.4、§19.5** |
| **某网奥运** 案例与架构图 | **§19.6.1**（与真题同源） |
| 技术栈：**Spark / MapReduce、Spark Streaming、Kafka、Flume、HDFS、HBase** 等在图中的层次 | 结合 **§19.3.4～19.4.3** 实现与图示 |

**【平台案例】**  
- **Lambda**：**批处理层** 做 **历史航班、准点率、燃油统计、旅客流量预测** 的 T+1 报表；**速度层** 用 **Spark Streaming / Flink** 消费 **Kafka**（**位置、事件、日志**）；**服务层** 合并 **批视图 + 实时视图** 对外 **REST**，供 **大屏、BI、监管报送**。题干若同时强调 **离线准确 + 实时近似**，倾向 **Lambda**。  
- **Kappa**：若业务接受 **一切以流为真**、用 **流回放** 重算历史，可 **单链路** 简化运维——适合 **事件溯源成熟** 的团队；平台侧可表述为 **「全事件流驱动运行日志」** 的演进方向。  
- **填图记忆**：**Flume** 采集 **日志/传感器**；**Kafka** 缓冲；**HDFS** 存 **原始轨迹与冷数据**；**HBase** 存 **海量键值查询**（机号—事件）；**Spark** 批、**Spark Streaming** 流，与教材 **§19.6** 奥运图 **同构换领域**。

### 4. 扩展专题（教程分散）

| 考点 | 说明 |
|------|------|
| **微服务 vs 单体** | 下篇云原生/微服务案例与 **§7章** 分布式架构思想；**2024-05** 默写优缺点 |
| **Ping/Echo vs 心跳** | 可用性设计模式，教材 **§8.1 可用性** 与场景战术可互证 |
| **DDD、限界上下文** | 教程仅 **零星提及 DDD**；案例 **2025-11** 以 **统一语言、边界、独立演进** 作答 |
| **AES 分组长数据** | **第4章 §4.4.2** 对称分组加密；超长明文需 **分块 + 填充（如 PKCS#7）+ 工作模式（CBC/GCM 等）** |

**【平台案例】**  
- **微服务 vs 单体**：**单体** 适合 **初期单航站或单一航司闭环**、部署简单；**微服务** 适合 **多租户、多业务线**（**旅客、货运、运行、结算**）独立发布，代价是 **分布式事务、链路治理、测试复杂度**——效用树常考 **可修改性 vs 性能/可用** 的权衡。  
- **Ping/Echo vs 心跳**：**Ping/Echo** 如调度服务 **主动探测**「配载服务是否存活」；**心跳** 如 **廊桥、地面保障车辆终端、场面雷达前置机** **周期上报**，被动检测 **静默故障**。  
- **DDD、限界上下文**：**运行控制**（改航、限重）、**旅客服务**（订座、改签）、**机务**（工卡、航材）、**结算与常旅客** 各自 **统一语言** 与 **聚合根**，跨边界用 **防腐层** 与 **领域事件**。  
- **弱网 + 最终一致**：**航站/机上 Wi‑Fi 不稳** 时，**值机/零售/行李** 先 **本地队列 + 幂等键**（行李牌号、订单号），后台 **对账** 与 **补偿**——与 **2025-11 餐饮** 题 **同套路换场景**。  
- **AES 长数据**：**旅客名单批量导出、长报文地空链路封装** 等，答题写 **AES 块长、PKCS#7 填充、CBC/GCM 模式与 IV**（与 **2025-11 充电** 子题 **同构**）。

## 三、高频默写清单

1. **效用树**：根 → 质量属性（2～4 个）→ **具体场景编号**；能区分 **性能(时延/吞吐)**、**可用性(切换/RTO)**、**安全(攻防/审计/加密)**、**可修改性(人天/人月改完)**。  
2. **风险 / 敏感点 / 权衡点** 三定义 + 各举一例。  
3. **Lambda 三层** + 与 **Kappa**「一条流」差异；能结合题干判定 **Lambda**。  
4. **仓库 vs 管道-过滤器**：数据为中心 vs 线性流水线；可扩展性与交互式场景。  
5. **解释器**：规则与代码分离、灵活性高、性能开销；适用于 **促销规则 / ML 流程 / 训练平台**。  
6. **质量场景 6 要素** 与 **可用性** 刺激-响应（**2024-05** 曾考原文级背诵）。  
7. **弱网 + 最终一致**：本地队列/事务、**全局唯一订单号**、**幂等**、**对账**（**2025-11 餐饮**）。

**【平台案例 · 与上表逐条对应】**  
1. 效用树：用 **航班动态、运控切换、审计、航季改界面** 填叶结点。  
2. 三概念：各举 **集成风险 / 心跳间隔 / TLS 权衡** 之一。  
3. Lambda：**批报表 + Kafka 实时 + 服务层合并**；Kappa：**单流重算**。  
4. **场面流水线** vs **态势仓库**。  
5. **运价/除冰规则外置**、大模型 **DAG 解释器**。  
6. 六要素套 **高峰查询**；可用性原文背 **2024-05**。  
7. **航站弱网行李/零售** 与餐饮题同构。
