|------|----------------|----------------|

## 二、分模块常考知识点详解

### 1. 评价方法（圈复杂度、监视、基准、TPC、Web）

**教材依据：第2章 2.9.4；第10章（CCN）**

- **环路数 / 圈复杂度**  
  - 用于反映控制流与决策结构带来的**结构复杂度**；教程在架构可维护性度量中给出 **CCN（圈复杂度）** 及与独立路径、测试复杂度相关的结论（如实践中常以 CCN≤10 为经验参考）。  
  - 真题若问「原代码结构复杂度」，优先对应 **环路数 / CCN**，而非模块数、对象数等规模指标。

- **系统监视三种方式**（与教材表述一致）  
  1. **系统命令**（如 UNIX/Linux：`ps`、`last`；Windows 亦常举 `netstat` 等）；  
  2. **系统记录文件**（日志）；  
  3. **集成命令、日志与可视化** 的监控工具（如 Windows **Perfmon**）。  
  - 区分：`top` 等多为**交互式命令行工具**，与「集成可视化」类题目选项需按真题解析择优。

- **评价程序准确度**（必背排序）  
  - **真实程序** 准确度最高；**合成基准** 多题为**最低**。  
  - 「用得最多、最频繁的核心部分」作为标准程序 → **基准测试程序**。

- **TPC-C**  
  - 面向 **OLTP** 的典型基准；**OLAP/决策支持** 常与 TPC-H、TPC-DS 等另一类基准对应，勿混选。

- **Web 服务端性能指标辨析**  
  - **丢包率** 偏**网络传输质量**，不宜作为 Web 服务器自身性能的主指标（真题多次以「不属于」考查）。  
  - **链接正确跳转** 属**功能/正确性**范畴，非性能指标。

**【平台案例】**  
- **CCN**：**「不正常航班规则引擎」** 若 **多层 if/分支**，**环路数** 高（**维护成本↑**）。  
- **监视**：**离港队列积压** 用 **日志 + 指标**；**Perfmon** 类题扣 **集成工具** 选项。  
- **基准排序**：**生产脱敏真实交易回放** 优于 **实验室合成脚本**（**真实 > … > 合成**）。  
- **TPC-C**：**值机购票事务** 与 **OLTP** 同一类；**准点率大屏** **不是** TPC-C 场景。

### 2. 性能指标（Web、数据库、应用、DBMS）

**教材依据：第2章 2.9.3、2.9.4**

- **Web 测试**  
  - 性能指标：教材明确列举 **最大并发连接数、响应延迟、吞吐量** 等。  
  - 评测方法：教材明确 **基准性能测试、压力测试、可靠性测试**。

- **数据库 vs 应用系统调优**  
  - 数据库侧：CPU/内存、**查询语句性能**、进程/线程、日志等（教材 **2.9.3**）。  
  - 应用侧：可用性、响应时间、**并发用户数**、资源占用等（同节）。

- **计算机 vs DBMS**  
  - **数据处理速率**、内存等属计算机整体能力表述；**最大连接数**、并发事务、索引数量等属 **DBMS** 典型评价维度。

**【平台案例】**  
- **Web**：**早高峰** **最大并发、P95 延迟、TPS**（与教材列举一致）。  
- **评测方法**：**基准、压力、可靠性** —— **「登机口压力测试」** 仅作情景联想（**方法名背原文**）。  
- **DB 调优**：**热点航班查询** **慢 SQL**；**应用**：**并发值机** **线程池**。

### 3. 遗留系统（2×2 演化策略）

**教材依据：第10章（演化策略与维护）；真题图形与选项为纲**

- 按 **业务价值** 与 **技术水平** 两维划分四类策略（真题解析常见归纳）：  
  - **淘汰**：价值低、技术低；**继承**：价值高、技术低；**改造**：价值高、技术高；**集成**：价值低、技术高（易成**信息孤岛**时，通过集成打通）。  
- 注意：同一策略在不同年份选项 **字母位置可能变化**，答题以**策略名称**为准，勿死记字母。

**【平台案例】**  
- **淘汰**：**已下线航站**、**无业务** 的 **陈旧模块**。  
- **继承**：**老离港** **业务仍重**、**技术落后** → **先继续用**。  
- **改造**：**核心运行系统** **高价值高要求** → **重构/升级**。  
- **集成**：**多套货运/离港** **高技术孤岛** → **ESB/中台** 打通（题干 **「信息孤岛」** 常指 **集成**）。

### 4. 系统维护（四类维护、版本控制、可行性）

**教材依据：第3章 3.1.4；信息化生命周期语境**

- **四类维护**（教材与软件工程一致）  
  - **完善性**：用户提出**新功能、性能提升、扩展**等改进需求。  
  - **改正性**：纠错；**适应性**：环境/规则变化；**预防性**：为未来可维护性做预防性调整。

- **版本控制**  
  - 管理多版本存储、回溯与协作（Git/SVN 等工具属此类）。

- **可行性分析**  
  - **经济可行性**：成本效益、无形收益等；**技术可行性**：现有技术能否支撑目标。  
  - **对现有系统运行问题的罗列与分析** 多属**现状调查/问题识别**，真题曾判为**不属可行性分析范畴**（选项需读清「不属于」）。

**【平台案例】**  
- **完善性**：**旅客提出的** **新改签规则、新报表**（**改进需求**）。  
- **适应性**：**局方数据格式/接口变更**、**新航季字段**。  
- **改正性**：**舱单与离港不一致** **缺陷修复**。  
- **预防性**：**重构高 CCN 模块**、**补监控** 降低 **未来故障**。  
- **版本控制**：**航季脚本、接口包** **Git 分支与回滚**。  
- **可行性**：**「新建运控平台」** 才做 **经济/技术可行性**；**仅列现状** **≠** 可行性分析。

## 三、备考提示与复习建议

**【平台案例】**  
- **航空运营智能管理平台** 与 **`案例知识点/Web应用设计_案例真题与教材对照_2026备考.md`** 中 **CDN/压测/连接池** 可 **对照**；本 PDF 侧重 **运维侧指标、维护类型、遗留策略**。
