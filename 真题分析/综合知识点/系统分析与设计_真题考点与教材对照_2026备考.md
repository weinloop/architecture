## 二、高频考点清单（按主题）

### 1. 需求工程

- **需求抽取（获取）**：迭代活动含需求发现、分类组织、协商、文档化。  
- **变更管理**：典型链路含**变更分析与成本计算**；**CCB** 审批；变更**有利有弊**。  
- **好的需求**：应能区分**优先级**，不宜「所有需求同等重要」。  
- **系统建议方案**（可行性层面）：含问题陈述、范围、候选方案与可行性；**不含**详细设计方案。  
- **JRP**：侧重**联合获取**需求；深度分析/验证多在后续阶段。  
- **需求→架构**：强调**追溯/映射**（矩阵、用例映射）。

**【平台案例】**  
- **变更**：**航季切换** 引发 **接口与数据库字段** 变更 → **影响分析与工作量** → **CCB** 决策。  
- **优先级**：**「航班安全相关」** 与 **「报表样式」** 分层（非「同等重要」）。  
- **建议方案**：论证 **「新建运控平台 vs 升级现有套件」**，**不含** **类图级详细设计**（边界题）。  
- **追溯**：**规章条款** ↔ **用例** ↔ **架构构件** **映射表**。

### 2. 结构化分析（SA）

- **DFD**：加工、外部实体、数据流、数据存储；描述**数据流动与加工**，**不强调**加工**执行顺序**；子图与父图**输入输出一致**。  
- **判定表/判定树**：细化 **DFD 中加工**的内部逻辑。  
- **结构化分析特征**：**面向数据流**、**自顶向下**分解（与「自底向上」区分）。  
- **Petri 网**：适合并发、同步、资源竞争等**流程/系统**分析（真题强调数学化与流程视角）。

**【平台案例】**  
- **DFD**：**旅客** 提交 **订座记录** → **订座服务** → **航班库存**；强调 **数据去向**，**不**在 DFD 上画 **「先付款后出票」** 的 **时序**（与真题辨析一致）。  
- **判定表/树**：**「天气/限重/超售」** 组合决定 **是否允许值机**（细化 **加工**）。  
- **Petri 网**：**机位、除冰位、跑道** **令牌** 与 **并发占用**（资源竞争）。

### 3. 结构化设计（SD）

- **耦合**（由弱到强，教材 **表 5-1**）：非直接、**数据**、标记、控制、外部、公共、内容等；设计原则：**多用数据耦合，避免内容耦合**。  
- **内聚**（由强到弱，教材 **表 5-2**）：功能、顺序、通信、过程、时间、逻辑、偶然等；**高内聚、低耦合**。  
- **扇入 / 扇出**：扇出 = 模块直接调用的下层模块数。  
- **概要设计**常用：**模块结构图、层次图、HIPO** 等表达软件结构。  
- **详细设计**工具：**程序流程图、PAD、N-S 盒图**等；**顺序图属 UML**，非传统结构化工具。  
- **McCabe**：\(V(G)=P+1\)（\(P\) 为判定结点数，常用公式之一）。

**【平台案例】**  
- **耦合**：**离港适配层** 与 **核心域** 之间优先 **数据耦合**；避免 **内容耦合**（直接改对方内部）。  
- **内聚**：**「计算配载平衡」** 模块 **功能内聚**（表 5-2 题）。  
- **概要/详细**：**模块结构图** 划 **运控/旅客/集成**；详细设计用 **PAD** 写 **改签算法**（**顺序图不属传统 SD 工具** 类判断）。

### 4. UML 与建模

- **UML2**：基础结构对应 **元模型**；顺序图中 **loop/alt/par** 等用 **交互片段（Combined Fragment）** 表达。  
- **用例图**：用例间 **include、extend、generalization**；**聚合**属类图结构关系。  
- **参与者之间**关系多为 **泛化（继承）**。  
- **视图**（经典表述）：**用例视图**支撑需求；**进程视图**描述并发与同步（逻辑视图的运行实例）。  
- **部署图**：**物理**结点与制品部署。  
- **活动图**：**分支（decision）** vs **分叉（fork）**——条件选择 vs 并行起点。  
- **SysML**：相对 UML 常强调**需求图**等扩展。

**【平台案例】**  
- **用例图**：**参与者** **旅客、地服、签派**；**包含（include）**：**值机** **必须先** **身份校验**；**参与者泛化** **个人/团体旅客**。  
- **顺序图片段**：**alt**「有座/无座」；**par** **并行查证件与行李额度**。  
- **部署图**：**Web/App**、**应用集群**、**数据库/消息中间件** 在 **同城双机房** 的 **物理节点**。  
- **活动图**：**分支** 选 **是否超售**；**分叉** **并行** **装机供品与加油**（**条件 vs 并行** 辨析）。  
- **4+1**：**用例视图** 绑定 **航班正常性需求**；**进程视图** **离港峰值并发**。

### 5. 面向对象分析与设计

- **OOA**（教材）：主题层、对象类层、结构层、属性层、服务层；原则含抽象、封装、继承等。  
- **OOD**：**实体类 / 控制类 / 边界类** 划分。  
- **DDD**：以**领域模型**承载领域知识与业务规则。  
- **COM 重用**：**包含（转发）** vs **聚集（暴露接口引用）**。

**【平台案例】**  
- **实体/控制/边界**：**Flight** **实体**；**RebookingSession** **控制**；**CheckInTerminal** **边界**（真题类题）。  
- **DDD**：**「限重、超售、退改规则」** 放入 **领域模型** 与 **聚合根**，**非** 全堆在 **Controller**。

### 6. 设计模式（GoF）

- **三大类**：创建型、结构型、行为型；**范围**：类模式 vs **对象模式**。  
- 典型配对：**抽象工厂**（产品族）、**建造者**（构建与表示分离）、**原型**（克隆）、**单例**；**桥接**（抽象与实现分离，结构型）、**命令**（请求封装，行为型）、**装饰器**（动态附加职责）。

**【平台案例】**  
- **抽象工厂**：**不同航司离港厂商** 一族 **解析器+写入器**（**产品族**）。  
- **命令**：**「撤销托运」** **请求封装**，可 **日志/重试**（行为型）。  
- **装饰器**：**动态叠加** **保险、选座、行李额** **责任链式** 计费（**意图** 扣题，**勿硬套类名**）。
