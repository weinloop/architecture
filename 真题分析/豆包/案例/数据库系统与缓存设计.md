# 数据库系统与缓存设计（系统架构案例题）核心总结
本资料为**架构师必考案例核心模块**，无复杂计算，全为**场景问答、方案选型、问题解决**，覆盖数据库设计、Redis缓存、分布式存储、缓存异常四大核心，精简背诵版如下：

## 一、数据库设计核心考点
1. **逻辑数据模型任务**
确定数据模型、E-R图转关系模式、定义完整性约束、设计用户视图。
2. **核心概念**
- 超类实体：抽取多个实体**共性属性**（如寄件人/收件人→用户）；
- 派生属性：由其他字段计算得出（如包裹总计=资费+保价费），可按需冗余存储。
3. **反规范化设计（必考）**
- 四种方法：**增加冗余列、增加派生列、重新组表、分割表**；
- 高频场景：多表关联查询慢 → **增加冗余列**（最优解）；
- 一致性解决：**触发器（实时）、应用程序同步、定时同步**，高频更新选前两者。

## 二、缓存一致性（核心问答模板）
1. **两种同步方案对比**
- 实时同步：更新库立即更缓存，**强一致、高并发阻塞、性能差**；
- 异步准实时：日志/消息队列异步刷新，**最终一致、低延迟、高并发首选**。
2. **标准读写策略（旁路缓存Cache-Aside）**
- 读：查缓存→命中返回→未命中查库→回写缓存；
- 写：**先更新数据库 → 再删除缓存**（杜绝双写不一致，行业标准）。
3. **主流同步方案**
应用双写、Binlog同步（Canal）、消息队列异步、延迟双删。

## 三、分布式缓存分片算法
1. **普通哈希**：`hash(key)%节点数`，**节点变动→全量数据迁移**，扩展性差；
2. **一致性哈希**：哈希环映射，节点变动仅**局部迁移**，**虚拟节点解决数据倾斜**；
3. 选型结论：分布式缓存集群**必选一致性哈希**。

## 四、缓存三大异常（100%案例考点）
### 1. 缓存穿透
- 现象：请求**不存在的key**，直接击穿缓存打满数据库；
- 最优解：**布隆过滤器**
  原理：位数组+多哈希函数，判存在/不存在；
  优缺点：**空间小、查询快**；**有小概率误判、不支持删除**。

### 2. 缓存雪崩
- 现象：大量key同时失效/缓存宕机，数据库瞬时压力暴增；
- 解决方案：过期时间**随机化**、加锁串行化、多级缓存、Redis集群高可用。

### 3. 缓存击穿
- 现象：**热点key失效**，高并发直击数据库；
- 解决方案：分布式互斥锁、热点数据**永不过期**。

## 五、Redis 全栈核心考点
1. **数据类型场景（必考匹配）**
- String：计数器（评论数）；List：粉丝列表、发布订阅；
- Set：标签、共同好友；Hash：用户结构化信息；
- **ZSet（有序集合）：排行榜（热销榜/热搜榜）**。
2. **持久化对比**
- RDB：快照存储，**重启恢复快、少量数据丢失**；
- AOF：日志追加，**数据安全、重启慢**；
- 选型：快速恢复业务**选RDB**。
3. **过期&淘汰策略**
- 默认：定期删除+惰性删除；
- 内存淘汰：LRU（最近最少使用）、LFU（最低频率）、随机淘汰。
4. **分布式部署**
- 三种模式：主从、哨兵、Cluster；
- 分片方式：客户端分片、中间件分片、服务端Slot分片。

## 六、存储架构选型
1. **文件系统 vs 关系型数据库**
- 文件系统：开发简单、冗余高、扩展性差；
- 关系库：规范低冗余、支持事务、扩展性强，**企业级首选**。
2. **混合存储架构**
- 高频读写热点数据 → **内存数据库（Redis）**；
- 静态低频数据 → **关系型数据库（MySQL）**。
3. **大数据/特殊场景选型**
- HDFS：海量大文件；HBase：海量随机读写；ES：全文检索；
- MongoDB：非结构化/向量存储；**Neo4j（图数据库）：知识图谱**。
4. **冷热分层存储**
热数据（SSD）、温数据（HDD）、冷数据（低成本归档），**兼顾性能与成本**。

## 七、工具对比：Memcached vs Redis
- Memcached：仅支持KV、无持久化、无事务，纯轻量缓存；
- Redis：多数据结构、持久化、集群、事务，**可靠性/功能优先必选**。

# 备考极简结论
1. 案例题**无计算，纯话术**，核心记：**原理+优缺点+适用场景**；
2. 高频固定答案：异步准实时同步、一致性哈希、布隆过滤器防穿透、ZSet做排行榜、先更库再删缓存；
3. 选型原则：**性能优先选最终一致，数据安全选强一致，高并发必分布式缓存**。
