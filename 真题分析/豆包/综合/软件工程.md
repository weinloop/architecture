# 系统架构设计师-软件工程 核心考点极简总结
**必考分值：8~12分**，纯概念题，无计算，核心围绕**模型、方法、构件、成熟度、重用**五大方向，以下为背诵版速记。

## 一、软件开发模型（高频核心）
1. **瀑布模型**：线性顺序，阶段因果强依赖，**无风险评估**，需求变更成本高
2. **增量模型**：拆分可交付增量，**客户更早用软件、变更成本低**
3. **原型模型**：原型开发+目标开发，**需求不明确首选**
4. **螺旋模型**：**快速原型为基础**，**风险驱动**，每轮迭代含风险分析
5. **喷泉模型**：**面向对象专用**，开发活动并发迭代
6. **V模型**：开发与测试一一对应，测试后置
7. **W模型**：**开发与测试并行**，测试贯穿全生命周期
8. **RAD快速应用开发**：模块化高、构件复用，快速交付

## 二、MDA模型驱动架构（固定考点）
三层抽象模型，**必考区分**
1. **CIM计算无关模型**：业务领域抽象，与实现无关
2. **PIM平台无关模型**：**系统建模，无平台细节**（题干答案）
3. **PSM平台相关模型**：含平台实现，可生成代码

## 三、业务流程建模（秒杀题）
✅ 可描述工作流：**活动图、BPMN、Petri网、IDEF3**
❌ 不可描述：**用例图**（仅描述功能与参与者，无流程）

## 四、软件生命周期（标准划分）
1. 传统三阶段：**软件定义 → 软件开发 → 软件运行与维护**
2. 四大核心活动：**软件描述、软件开发、有效性验证、软件演化**
3. 阶段任务：**可行性研究∈定义阶段**

## 五、敏捷开发（必考判断题）
### 核心思想（背结论）
1. **以人为本≠以过程为本**；**适应型≠可预测型**
2. 迭代增量、短周期交付、轻文档、高协作
3. **不适合大型团队**，适合需求多变/不明确项目

### 敏捷方法区分
1. **Scrum**：固定Sprint周期、**三角色**、每日站会，需求排序**以商业价值为核心**
2. **XP极限编程**：结对编程、测试驱动
3. **开放式源码**：**地理分布广的团队**首选
4. **FDD特性驱动**：分**首席程序员+类程序员**
5. 易错点：**以用例为中心是RUP，不是敏捷**

## 六、逆向工程（4个抽象层级，必考匹配）
1. **实现级**：抽象语法树、代码、数据结构
2. **结构级**：**模块依赖/调用关系**
3. **功能级**：**程序段功能及功能间关系**
4. **领域级**：业务概念映射
5. 核心结论：**逆向工程→抽象提取；再工程→重构生成新版本**
6. 方法匹配：用户指导搜索→**实现+结构级**；基于领域方法→**功能+领域级**

## 七、开发环境与工具
1. 软件开发环境核心：**环境信息库**（存开发信息、共享）；**过程控制+消息服务器**（集成基础）
2. 工具分类：需求→设计→编码→测试工具
3. 需求工具：自然语言/图形、**形式化需求定义语言**
4. 配置管理工具：**版本控制、变更管理、权限控制**（版本控制是子集）
5. SCCS：**经典版本控制工具**；版本控制∈**软件维护工具**

## 八、基于构件软件工程（CBSE，高频）
### 1. 构件核心特性（必考判断）
✅ 独立部署、第三方组装、**无外部可见状态**、**部署单元不可拆分**
❌ 多拷贝共存、可拆分、外部暴露状态

### 2. 构件分类
- 适应性构件：包装适配，解决兼容冲突
- 装配的构件：预装在系统层级，胶水代码连接
- 可修改构件：版本替换、新增功能

### 3. 构件组装
- 三种方式：**层次、顺序、叠加**（无循环）
- 三层层次：**定制→集成→扩展**
- 接口失配：构件失配（基础设施/模型）；**连接子失配**（协议/格式）
- 操作不完备：提供接口是请求接口的**子集**

### 4. EJB构件（固定匹配）
- 会话Bean：**客户端交互**
- 实体Bean：**数据持久化**
- 消息驱动Bean：**并发/异步处理**

### 5. 构件检索
关键字（树形）、**刻面Facet**（特征描述）、超文本（联想跳转）

## 九、软件开发方法
1. 自顶向下：**快速出演示原型**，高层优先
2. 自底向上：底层模块先开发
3. **形式化方法**：严格数学基础，净室软件工程基于此
4. 净室软件工程：理论**函数+抽样理论**，正确性验证，高成本高质量

## 十、统一过程RUP（三大特征）
1. 核心：**用例驱动、以架构为中心、迭代增量**
2. 四阶段：**初始（立项）→细化（建架构、定计划）→构建（编码）→移交（交付）**
3. 9大工作流：**无成本工作流**
4. 优势：早期处理高风险

## 十一、成熟度模型（CMMI/DCMM）
### 1. CMMI 5级成熟度（必考）
1初始→2已管理→3**已定义**（组织级标准化流程）→4量化管理→5优化
### 2. CMMI体系文件层级（自顶向下）
**顶层方针→过程文件→规程文件→模板文件**
### 3. DCMM数据管理
8大能力域：**数据战略、治理、架构、标准、质量、安全、应用、生存周期**

## 十二、软件重用
1. 分类：**水平重用（通用，标准函数库）**、垂直重用（行业专用）
2. 双生命周期模型：**领域工程（生产复用资产）+应用工程（使用资产）**
3. 最佳复用实践：**使用已验证开源组件**，拒绝复制粘贴
4. 重用元素：需求文档、设计文档、**程序代码**、测试用例

## 终极易错秒杀清单
1. 螺旋模型基础=**快速原型**，核心=**风险分析**
2. 敏捷核心=**以人为本**，RUP核心=**用例驱动**
3. 构件无外部状态、不可拆分；**多线程引发数据竞争/死锁**
4. 逆向工程结构级=依赖关系，功能级=功能关系
5. CMMI已定义级=**组织标准化流程**，量化级=数据度量
6. 水平重用=通用功能，垂直重用=行业专属
