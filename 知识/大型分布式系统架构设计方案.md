# 大型分布式系统架构设计方案 (航空运营智能管理平台案例)

## 一、 项目概述与挑战

### 1.1 背景
随着全球民航业数字化转型的深入，航空运营平台已不再仅仅是简单的业务处理系统，而是一个集海量数据接入、高并发交易处理、智能决策分析于一体的综合性底座。

### 1.2 核心挑战
- **数据规模**：日均处理 800GB+ 实时数据流，历史订单数据达 50 亿级。
- **并发压力**：高峰期（9-11 点）面临 10 万级并发访问，核心票务接口 TPS 需达到 5000+。
- **可用性要求**：系统需具备 99.99% 的全年可用性，任何核心链路的故障都可能导致巨大的经济损失。
- **业务复杂性**：涉及多航司差异化策略、多渠道支付接入、全球化航网调度及复杂的票价计算。

---

## 二、 架构设计原则与思维 (Trade-off)

### 2.1 权衡的艺术 (CAP/BASE)
- **强一致性 (CP)**：在“座控库存扣减”等核心金融级链路，优先保证数据的一致性，宁可牺牲部分可用性（如：在高并发写竞争下进行排队或短暂限流），严防“超卖”。
- **最终一致性 (AP)**：在“旅客积分增加”、“行程单生成”等辅助业务中，追求高可用与快速响应，允许秒级数据延迟，通过可靠消息投递实现最终一致。

### 2.2 抽象与建模
- **统一领域模型**：提炼出通用的“航班 (Flight)”、“座位 (Seat)”、“旅客 (Passenger)”实体，屏蔽各航司底层的异构数据差异。
- **无状态化设计**：服务节点不保存本地 Session 或关键状态，统一存储于 Redis 集群，支撑 K8s 的秒级弹性扩缩容。

## 三、 技术栈全景

| 维度 | 选型 | 核心理由 |
| :--- | :--- | :--- |
| **底层语言** | Java (JDK 17+) | 强大的生态、成熟的 JMM 并发模型、丰富的工程化工具。 |
| **核心数据库** | MySQL (InnoDB) | 支撑 ACID 事务、聚簇索引优化、Binlog 审计与同步。 |
| **高性能缓存** | Redis | Hash/ZSet 支持复杂业务场景、Redlock 保障分布式锁安全。 |
| **消息总线** | Kafka | 零拷贝技术支撑 PB 级流转、解耦异构系统、削峰填谷。 |
| **服务治理** | Nacos + gRPC | 高效的注册发现、二进制协议提升通信性能。 |
| **高可用保障** | Sentinel + Seata | 流量熔断限流、AT 模式简化分布式事务复杂度。 |
| **云原生基础设施** | Docker + K8s | 容器化部署、HPA 自动扩容、CI/CD 自动化流水线。 |
| **可观测性** | SkyWalking + Prometheus | 全链路 Trace 追踪、多维度指标监控、5 分钟快速定界。 |

## 四、 核心层级详细设计

### 4.1 底层基石：性能与稳定性的源头
- **并发控制策略**：
    - **JMM 可见性**：全局配置（如：检修阈值）使用 `volatile` 修饰，确保多核 CPU 下的瞬时同步。
    - **读写分离锁**：在座控查询/扣减场景应用 `ReentrantReadWriteLock`，极大提升读并发能力。
- **网络与 IO 优化**：
    - **Netty NIO**：利用 Linux `epoll` 机制维持海量长连接，解决 C10K 甚至 C100K 问题。
    - **HTTP/3 (QUIC)**：预研 0-RTT 建连特性，优化全球弱网环境下的旅客交互体验。

### 4.2 存储层设计：PB 级数据的高效管理
- **MySQL 优化**：
    - **精密索引设计**：主键聚簇索引直达订单详情；针对身份证号建立联合索引，通过覆盖索引（Covering Index）减少回表，控制 RT 在 5s 内。
    - **数据分片**：针对 50 亿级历史订单，按“年份+月份”进行物理分片存储，解决单表性能瓶颈。
- **Redis 状态中心**：
    - **多级缓存**：结合 Guava 本地缓存与 Redis 远程缓存，应对热点航线的极端流量。
    - **穿透防御**：利用布隆过滤器（Bloom Filter）拦截非法请求，保护底层数据库。

### 4.3 消息层设计：系统的“血液”流转
- **零拷贝与吞吐**：利用 Kafka 的 `sendfile` 与顺序写磁盘特性，支撑日均 800GB 的全量业务审计与实时分析流。
- **幂等性保障**：在消费端通过“全局唯一 ID + 数据库唯一索引”实现业务幂等，确保购票成功后的后续链路（积分、短信）不重不漏。

## 五、 分布式与云原生落地

### 5.1 服务治理与熔断
- **流量卫兵 (Sentinel)**：针对核心购票接口设置 QPS 阈值，当外部核验接口异常时自动触发熔断，保障核心链路“有损但可用”。
- **分布式事务 (Seata)**：在跨“票务”与“营销”服务的交易中，采用 AT 模式自动化处理回滚，降低开发者的心智负担。

### 5.2 K8s 弹性伸缩与灰度
- **HPA 自动扩容**：根据 CPU/内存负载，在 9-11 点高峰期自动将微服务实例从 10 个扩容至 50 个。
- **GitOps 交付**：利用 ArgoCD 实现配置即代码，支撑多航司差异化策略的平滑灰度上线与快速回滚。

## 六、 软件工程化与代码质量

### 6.1 领域驱动设计 (DDD)
- **限界上下文**：明确划分“订单、库存、旅客”等聚合根，通过领域事件（Domain Event）驱动业务流转，实现逻辑的物理解耦。
- **核心逻辑下沉**：将计费引擎、航路校验等共性逻辑下沉为独立组件，支撑多端（App, Web, 柜台）一致性。

### 6.2 设计模式的应用
- **策略模式**：动态适配支付宝、微信、各航司私有支付渠道。
- **模板模式**：标准化各航司差异化的“值机 -> 登机”流程。
- **责任链模式**：在购票风控环节串联“实名校验 -> 黑名单拦截 -> 库存检查”。

---

## 七、 可观测性与质量保障

### 7.1 五分钟定位机制
- **TraceID 链路追踪**：基于 SkyWalking 实现全链路染色，旅客的一次点击可追溯至数据库的每一行 SQL 执行。
- **全量日志审计**：保留 1 年以上的 Binlog 与业务日志，支撑安全合规与纠纷回溯。

### 7.2 测试工程化
- **行覆盖率红线**：核心计费与交易逻辑强制要求 85% 以上的单元测试覆盖率。
- **环境隔离**：利用 Testcontainers 模拟真实数据库与中间件环境，保障集成测试的可靠性。
