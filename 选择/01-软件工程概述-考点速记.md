## 二、软件工程定义与 PDCA（★★★☆☆）

- **软件工程**：由**方法、工具和过程**三部分组成。
- **PDCA 循环**（选择题考过）：
  - **P（Plan）**：软件规格说明，规定功能及运行限制。
  - **D（Do）**：软件开发，开发出满足规格的软件。
  - **C（Check）**：软件确认，确认满足用户需求。
  - **A（Action）**：软件演进，运行中不断改进。

**口诀**：计划—开发—确认—演进。

## 三、软件开发方法分类（★★★★★）

| 分类方法 | 具体方法 | 描述 |
|----------|----------|------|
| 开发风格 | 自顶向下 | 先定义整体结构，再逐步细化模块与功能 |
| 开发风格 | 自底向上 | 从具体模块开始逐步构建整个系统 |
| 性质 | 形式化方法 | 坚实数学基础，严格规格与验证 |
| 性质 | 非形式化方法 | 不把严格性作为主要着眼点 |
| 适应范围 | 整体性方法 | 适用于软件开发全过程 |
| 适应范围 | 局部性方法 | 适用于某个具体阶段 |

## 四、形式化方法与净室（★★★☆☆）

### 4.1 形式化方法

- **含义**：用数学模型与形式化语言描述需求/设计/行为，通过逻辑推理或模型检测获得**可证明的正确性**；常用于安全攸关、高可靠性系统。
- **特点**：① 正确性验证难且耗时；② 仍需与传统测试结合（形式化提供理论保证，测试发现运行时错误）。

### 4.2 净室软件工程（CSE）

- **定位**：一种**形式化方法**，不是测试方法、管理模型或单纯过程模型；强调在**设计与规约阶段**用数学与统计理论消除错误。
- **理论基础**：**函数理论**（程序=输入到输出的严格映射；完备性、一致性、正确性——口诀：**玩一阵**）、**抽样理论**（抽样测试推断系统可靠性）。
- **局限**：验证依赖严格数学，门槛高；不依赖传统单元测试但仍需考虑底层环境；仍属传统软件工程体系。

## 五、逆向工程与再工程（★★★☆☆）

| 概念 | 说人话 | 定义 |
|------|--------|------|
| **逆向工程** | 从低层（代码）推回高层设计 | 分析程序，在比源代码更高抽象层次建立程序表示 |
| **重构** | 同一抽象层改结构 | 同一抽象级别上转换系统描述形式 |
| **设计恢复** | 借助工具抽取设计信息 | 从已有程序抽象出数据设计、总体结构、过程设计等 |
| **再工程** | 逆向+新需求+正向工程 | 在逆向信息基础上修改/重构产生新版本 |
| **正向工程** | 从设计到代码 | 利用逆向得到的信息改变或重构系统 |

**抽象层次（低→高）**：实现级（语法树、符号表）→ 结构级（调用图、结构图）→ 功能级（程序段功能及关系）→ 领域级（与应用领域概念对应）。

**口诀**：**世仇结分工龄**——世（实现）仇（抽象信息）结（结构）分（程序分量）工（功能级）龄（领域级）。

## 六、基于构件的软件工程 CBSE（★★★★★）

### 6.1 构件的定义（萨默维尔）

| 构件特性 | 描述 |
|----------|------|
| 可组装性 | 外部交互通过公开接口；提供对自身信息的外部访问 |
| 可部署性 | 自包含，独立实体在构件平台上运行；通常二进制、无须部署前编译 |
| 文档化 | 完全文档化，接口语法甚至语义需定义 |
| 独立性 | 可无其他特殊构件下组装部署；需外部服务时在“请求”接口显式声明 |
| 标准化 | 符合标准化构件模型（接口、元数据、文档、组成、部署） |
| 可见性 | 无外部可见状态（特殊情况：容器管理状态、不影响功能的属性可可见） |

**口诀**：**竹鼠挡毒标**——竹（可组装）鼠（可部署）档（文档化）毒（独立性）镖（标准化）。

### 6.2 构件分类（按外部形态）

| 类型 | 内容 | 例子 |
|------|------|------|
| 独立而成熟的构件 | 成熟度高，完全隐藏接口，规定命令使用 | 数据库管理系统、操作系统 |
| 有限制的构件 | 提供接口与使用条件；可能资源冲突，需测试 | 基础类库 |
| 适应性构件 | 处理不兼容、资源冲突，可直接多环境使用 | ActiveX |
| 装配的构件 | 已装配在 OS/DBMS/信息系统层次，胶水代码连接 | 成品软件产品 |
| 可修改的构件 | 可改错、增功能，包装或写接口实现版本替换 | 功能模块 |

**口诀**：**独显时装秀**——独立、限制、适应、装配、修改。

### 6.3 构件组装类型

| 组装类型 | 描述 | 特点 |
|----------|------|------|
| **顺序组装** | 按顺序调用已有构件创造新构件；先 A 再 B | 构件间不相互调用，需胶水代码保证接口兼容 |
| **层次组装** | 一构件直接调用另一构件服务；“提供”与“请求”接口需兼容 | 接口匹配无需额外代码，不匹配需转换；构件为网络服务时可能不适用 |
| **叠加组装** | 多构件叠加，新构件接口为原接口组合，分别调用 | 原构件互不依赖、不互相调用 |

**接口不兼容**：参数不兼容（操作名同、参数类型/数目不同）、操作不兼容（操作名不同）、操作不完备（提供/请求接口为对方子集）。

## 七、软件过程/生命周期模型（★★★★★）

- **软件过程 = 软件开发生命周期模型**（同义词）。
- **前提与模型**：需求完全确定→瀑布；仅基本需求→喷泉、螺旋、RUP、敏捷等；形式化为基础→变换模型。

### 7.1 瀑布模型

- **阶段**：需求分析与定义→系统与软件设计→实现与单元测试→集成与系统测试→运行维护。
- **缺点**：依赖早期需求调查，**不能适应需求变化**；需求完整性、正确性难确定。

### 7.2 原型/演化/快速原型模型

- 软考中**不做区分**，当做一个；适用于需求难完整定义，基于快速原型、用户反馈迭代直至最终产品。
- **特点**：迭代；功能开发后可及时验证需求；过早接触不稳定功能可能带来负面影响。

### 7.3 螺旋模型

- **螺旋模型 = 快速原型 + 风险分析**（选择题常考）。
- 适用于庞大、复杂、高风险系统；支持需求动态变化；迭代过多会增加成本、延迟提交。

### 7.4 喷泉模型

- 以**用户需求为动力、对象为驱动**；各阶段**相互重叠、多次反复**，可交互进行，可随时补充遗漏。

### 7.5 变换模型

- 基于**形式化规格说明语言和程序变换**；需严格数学理论和整套开发环境支持。

### 7.6 V 模型与测试依据

| 测试类型 | 测试对象 | 依据（文档） |
|----------|----------|--------------|
| 单元测试 | 模块/类 | **详细设计说明书** |
| 集成测试 | 模块间接口、已集成软件 | **概要设计文档** |
| 系统测试 | 完整集成系统 | **需求分析文档/开发合同** |
| 验收测试 | 准备交付的完整系统 | **用户需求** |

**注意**：概要设计是**集成测试**的依据，不是系统测试；系统测试依据是需求分析/合同；验收测试依据是用户需求。

### 7.7 增量模型 vs 原型/迭代

- **增量**：按模块完成顺序逐步实现，每次增量可独立可操作；**模块叠加、线性扩展**。
- **迭代**：分阶段循环（需求→设计→开发→测试）**逐步完善同一功能**，每次交付完整版本；**功能深度优化、螺旋上升**。
- 增量更早让客户使用并获得价值。

## 八、统一过程 RUP（★★★★★）

- **RUP**：用例驱动、以**体系结构为中心**、**迭代与增量**的软件开发过程；采用“4+1”视图描述体系结构（与后续架构 4+1 视图区分）。
- **九大核心工作流**（系分考过，必须背）：
  - **建需分食测**：建（业务建模）需（需求）分（分析与设计）食（实现）测（测试）。
  - **部配变项环**：部（部署）配变（配置与变更管理）项（项目管理）环（环境）。

## 九、敏捷方法（★★★★★）

### 9.1 敏捷宣言

- 个体和交互 **胜过** 过程和工具；可工作的软件 **胜过** 大量文档；客户合作 **胜过** 合同谈判；**响应变化** **胜过** 遵循计划。

### 9.2 适用场景

- 规模较小的项目；需求初步萌芽、尚未明确、**快速改变**。高关键性/可靠性/安全性要求可能不完全适合。
- **口诀**：**小变**——小（规模小）变（快速改变）。

### 9.3 主要敏捷方法

| 方法 | 简介 |
|------|------|
| **极限编程 XP** | 轻量、短周期、频繁发布；交流、朴素、反馈、勇气；YAGNI、DRY |
| **水晶系列** | “机动性”方法，以人为中心 |
| **Scrum** | 侧重项目管理的迭代增量框架；团队协作、响应变化、商业价值 |
| **FDD** | 迭代、人/过程/技术三要素 |

### 9.4 Scrum（★★★☆☆）

- **角色**：Product Owner（业务价值、Backlog 优先级）、Scrum Master（流程推动、清障、赋能）、Development Team（跨职能、自组织、交付 Sprint 目标）。
- **活动**：Sprint Planning（Sprint Backlog、目标）、Daily Scrum（昨日/今日/障碍）、Sprint Review（演示、反馈）。
- **工件**：Product Backlog（产品待办）、Sprint Backlog（冲刺待办）、Increment（可交付增量）。

## 十、软件过程管理：CMM / CMMI（★★★☆☆）

### CMM 五级

| 级别 | 含义 |
|------|------|
| 初始级 | 过程缺乏结构化，依赖个人经验 |
| 可重复级 | 能定义和文档化过程，确保项目成功 |
| 已定义级 | 过程标准化、文档化 |
| 已管理级 | 监控、量化过程，持续改进 |
| 优化级 | 动态调整过程适应不同项目 |

**口诀**：**初重定管优**——除虫顶管用。

### CMMI 五级与知识体系

- **等级**：初始级、已管理级、已定义级、**量化管理级**、优化级。**口诀**：储罐定粮油。
- **四大知识体系**：系统工程（SE）、软件工程（SW）、集成产品与流程开发（IPPD）、供应商采购（SS）。**口诀**：细软开工。

## 十一、速记清单（考前扫一眼）

1. **PDCA**：Plan 规格说明、Do 开发、Check 确认、Action 演进。
2. **开发方法**：自顶向下/自底向上；形式化/非形式化；整体性/局部性。
3. **净室**：形式化方法；函数理论（玩一阵）+ 抽样理论；非银弹。
4. **逆向**：逆向（低→高）、重构（同层）、设计恢复、再工程（逆+需求+正）、正向；抽象层次：世仇结分工龄。
5. **构件**：竹鼠挡毒标；分类独显时装秀；组装：顺序/层次/叠加；不兼容：参数/操作/不完备。
6. **生命周期**：瀑布（需求确定）、原型/螺旋（需求难定）、螺旋=原型+风险、喷泉（阶段交互）、V 模型测试依据（详设→单元、概设→集成、需求→系统、用户需求→验收）。
7. **RUP**：用例驱动、体系结构为中心、迭代增量；九流：建需分食测、部配变项环。
8. **敏捷**：宣言四胜过；小变；XP/水晶/Scrum/FDD；Scrum 角色 PO/SM/Team，活动 Planning/Daily/Review，工件 Backlog/Sprint Backlog/Increment。
9. **CMM**：初重定管优；**CMMI**：储罐定粮油；知识体系细软开工。
