## 二、系统设计概述

- **狭义**：概要设计/整体设计/蓝图设计。
- **广义**（软考）：概要设计 + 详细设计方案与模型。
- **概要设计**：高层次规划，确定系统基本结构、组成要素及关系，解决“如何组织、如何实现”。
- **详细设计（模块设计）**：在概要设计基础上确定各模块实现方式、接口、算法等，解决“如何完成具体任务”。

### 设计类型（7 类，了解）

| 类型 | 主要任务 |
|------|----------|
| 网络设计 | 网络结构、设备分布、布线、节点权限、管理软件 |
| **代码设计** | 面向对象设计、结构化设计；唯一化、规范化、系统化 |
| 输入/输出设计 | 确保数据完整性、正确性、一致性 |
| **处理流程设计** | 模块内部结构（局部数据、控制流、加工过程） |
| 数据存储设计 | 存储方式/介质/组织、容量估算；概念/逻辑/物理设计 |
| 用户界面设计 | 黄金三原则：用户控制、减轻记忆负担、界面一致 |
| 安全性与可靠性设计 | 运行环境与数据处理控制 |

**案例（航空运营智能管理平台）**：

- **概要设计**：划分 **七大功能域**（基础支撑、航信、旅客、票务、检修、数据服务、辅助管理），定义跨模块链路（购票→同步→运力校验→行程→通知→检修阈值→分析），选定 **微服务边界、网关、消息、缓存、读写分离、实时/离线双引擎**，并给出非功能指标（并发、延迟、一致性、备份与日志保留）。
- **详细设计**：如 **票务服务** 内部分解为锁库存、计价、支付回调、对账；**数据服务** 分解采集、流式计算、批处理、可视化查询；明确接口、幂等键、超时与补偿、表结构与索引策略。
- **七类映射**：**网络设计**（负载均衡、多 AZ、专线与 TLS）；**代码设计**（领域模型与编码规范）；**I/O 设计**（多终端适配、API 版本、文件与报表导出）；**处理流程**（购票与航变处置 BPMN/活动图）；**数据存储**（分布式存储、冷热分层、归档）；**界面设计**（旅客端/运营看板一致交互）；**安全可靠性**（RBAC、加密、审计、异地登录检测）。

## 三、处理流程设计（★★★☆☆）

### 3.1 流程六要素（ISO 9000）

- **输入资源、活动、活动的相互作用（结构）、输出结果、用户、价值**。
- **口诀**：**输入输出活动（相互作用）用户价值**。

**案例（航空运营智能管理平台）**：以 **在线购票** 为例——**输入资源**含旅客请求、航班与舱位上下文、支付渠道与核验结果；**活动**含查询、锁座、计价、支付、出票、同步库存与行程；**相互作用**体现分布式事务与消息顺序；**输出**为电子客票与通知；**用户**含旅客与后台运营；**价值**为成交效率与账务一致。航变批量退改签流程同样可按六要素拆解。

### 3.2 流程设计工具

| 大类 | 工具 | 描述 |
|------|------|------|
| 图形 | 程序流程图 PFD | 图框表示操作，独立于语言；不用于描述模块输入/输出与数据加工（真题考过） |
| 图形 | N-S 图（盒图） | 顺序/选择/WHILE/UNTIL/多分支；强结构化 |
| 图形 | IPO 图 | 模块输入、输出、数据加工 |
| 图形 | PAD | 五种基本控制结构，适合嵌套与层次 |
| 图形 | 判定树 | 树形表示逻辑判断 |
| 表格 | 判定表 | 条件与行动一目了然，多条件多结果 |
| 语言 | PDL（伪代码） | 自然语言+结构化语法，逐步求精 |

**案例（航空运营智能管理平台）**：用 **IPO 图** 描述“售票计算”模块的票价规则输入、税费与舱位输出；用 **判定表/判定树** 整理多航司、多舱位、多退改规则的 **条件—行动**；用 **PAD/N-S** 保证改签分支结构化、避免死锁路径；**PFD** 仅作辅助理解控制流，**不替代**模块 I/O 与数据加工说明（与真题考点一致）。

### 3.3 工作流 WFMS 与 WRM

- **WFMS**：定义、创建并管理工作流执行；功能**简直过分**——建（建模）直（执行）过（过程管理）分（分析）。
- **WRM 六模块**：**指引丁克调管**——指（执行服务）引（引擎）丁（流程定义）克（客户端）调（调用应用）管（管理监控）。

**案例（航空运营智能管理平台）**：**WFMS** 可支撑 **检修工单** 从计划、派工、执行到关闭的审批与超时升级，以及 **权限申请审批** 流程；**建模** 对应流程定义（丁），**执行** 对应引擎与执行服务（引/指），**分析** 对应 SLA 与瓶颈监控（分）。WRM 六模块可映射到：旅客/运维在 **客户端** 发起；**引擎** 驱动；**调用应用** 对接票务、通知、检修服务；**管理监控** 看大屏与告警。

## 四、面向对象设计 OOD（★★★★★）

### 4.1 设计类分类（实体/控制/边界）

| 维度 | 实体类 | 控制类 | 边界类 |
|------|--------|--------|--------|
| **定义** | 映射需求实体，存需永久存储的信息 | 控制用例工作，对用例特有控制行为建模 | 封装用例内外数据流，系统与外界交界 |
| **命名** | 业务领域名词 | 动宾短语转化名词 | 与系统接口相关，体现外部交互 |
| **作用** | 代表业务对象，对用户有意义 | 协调用例中对象交互，推进用例 | 与外界交互，隔离外部变更 |
| **生命周期** | 通常永久 | 用例执行时产生、结束后消亡 | 随系统运行存在，交互时发挥作用 |
| **属性/方法** | 一定有属性，操作按需 | 通常无属性，一定有方法 | 可有属性也有方法 |
| **示例** | 学员、课程 | 身份验证器 | 窗口、通信协议、打印机接口、报表 |

**判定顺序**：先看是否窗口/协议/打印机/报表→边界类；否则名词→实体类，动名词→控制类。

**案例（航空运营智能管理平台）**：

- **实体类**：`航班`、`订单`、`客票`、`旅客档案`、`检修工单`、`设备` 等承载持久化业务状态。
- **控制类**：`出票协调器`、`航变处置控制器`、`分布式事务协调（可映射为领域服务）`，编排用例步骤与异常分支。
- **边界类**：`移动端购票界面`、`开放 API 网关适配器`、`民航局核验客户端`、`短信/推送网关`、`报表导出接口`，隔离外部协议与 UI 变更。

### 4.2 面向对象设计原则（SOLID 等）

| 原则 | 描述 |
|------|------|
| **单一职责 S** | 类只有一个设计目的 |
| **开闭 O** | 对扩展开放、对修改封闭；通过抽象层扩展，是其他原则基础 |
| **里氏替换 L** | 基类对象处可用子类替换；变化类设计为抽象类或接口 |
| **接口隔离 I** | 接口提供客户端需要的行为；大接口拆成小接口 |
| **依赖倒置 D** | 抽象不依赖细节，针对接口编程；实现开闭的主要机制 |
| **组合/聚合复用** | 用组合或聚合复用，比继承更灵活、耦合低 |
| **最少知识（迪米特）** | 实体尽量少与其他实体交互，降低耦合 |

**案例（航空运营智能管理平台）**：

- **S**：`售票计算` 与 `库存` 分离，避免一个类既管规则又管存储。
- **O**：新增 **渠道** 或 **退改规则** 时通过 **策略/插件** 扩展，少改核心出票路径。
- **L**：`国内票`/`国际票` 替换抽象 `票` 时不破坏计价与退改契约。
- **I**：拆分 **查询接口** 与 **交易接口**，避免可视化大屏强依赖购票写接口。
- **D**：业务层依赖 **支付、核验** 的抽象接口，具体银行/民航局实现可替换。
- **组合/聚合复用**：用 **消息+仓储组合** 实现同步，而非深继承链。
- **迪米特**：`旅客服务` 通过 **门面/应用服务** 访问票务与通知，减少跨域对象互相引用。

## 五、结构化设计（★★★★★）

### 5.1 结构化设计四活动

- **数据设计、软件结构设计、人机界面设计（接口设计）、过程设计**。

**案例（航空运营智能管理平台）**：**数据设计** 含订单与库存分库分表、旅客敏感字段加密、实时数仓与离线湖分层；**软件结构** 即微服务拓扑与领域边界；**人机界面** 含旅客 App/小程序与运营可视化；**过程设计** 含购票与航变 SOP 对应的程序流程与补偿。

### 5.2 模块划分原则

- 模块**大小适中**（约 1～2 页或 50～200 行）；**扇入扇出合理**（平均 3 或 4，不超过 7）；**深度与宽度适当**；**信息隐蔽**（黑盒、仅通过接口交互）。

**案例（航空运营智能管理平台）**：**票务、旅客、数据服务** 等按限界上下文拆分，控制单服务代码规模；网关 **扇出** 多个下游时配熔断与超时，避免 **扇入** 过度集中成上帝服务；库存与支付细节 **黑盒化**，对外仅暴露契约与事件。

### 5.3 耦合（7 种，低→高）

| 类型 | 描述 |
|------|------|
| **非直接耦合** | 无直接关系，通过主模块控制与调用联系 |
| **数据耦合** | 参数表传递简单数据值 |
| **标记耦合** | 参数表传递记录/数据结构 |
| **控制耦合** | 传递的信息用于控制对方内部逻辑 |
| **外部耦合** | 多模块访问同一全局简单变量，且非经参数表传递 |
| **公共耦合** | 多模块访问同一公共数据环境（全局结构、共享区等） |
| **内容耦合** | 直接访问对方内部数据/非正常入口/代码重叠/多入口 |

**口诀**：**非鼠标控制外公嘞**——非（非直接）鼠（数据）标（标记）控（控制）外（外部）公（公共）嘞（内容）。

**案例（航空运营智能管理平台）**：

- **非直接/数据耦合**：服务间仅通过 **API 参数** 传递订单号、金额等简单值，理想状态。
- **标记耦合**：传递 **订单 DTO/上下文对象**（含多字段），需控制版本与字段膨胀。
- **控制耦合**：用 **布尔/枚举** 指挥对方内部分支（如 `forceRefund=true` 深入对方逻辑），应收敛到明确命令或策略。
- **外部/公共耦合**：多服务读写 **全局配置或共享缓存键** 无规范，易踩坑；应改为配置中心与显式契约。
- **内容耦合**：绕过接口直接改他库表或内部状态，在微服务中属反模式，应禁止。

### 5.4 内聚（7 种，高→低）

| 类型 | 描述 |
|------|------|
| **功能内聚** | 完成单一功能，各部分协同、缺一不可 |
| **顺序内聚** | 处理元素相关且**必须顺序执行**，**有数据传递与依赖** |
| **通信内聚** | 处理元素集中在一个数据结构区域上 |
| **过程内聚** | 按特定次序执行，**不一定有数据传递与依赖** |
| **瞬时（时间）内聚** | 任务在同一时间间隔内执行 |
| **逻辑内聚** | 完成逻辑相关的一组任务 |
| **偶然内聚** | 完成无关系或松散关系的任务 |

**口诀**：**公孙铜锅涮罗欧**——公（功能）孙（顺序）铜（通信）锅（过程）涮（瞬时）罗（逻辑）欧（偶然）。

**顺序 vs 过程**：顺序—成分与同一功能密切相关且存在数据传递与依赖；过程—按次序执行但可无数据依赖。

**案例（航空运营智能管理平台）**：

- **功能内聚**：`票务库存服务` 专注库存扣减与回补，职责单一。
- **顺序内聚**：`出票流水线` 中 **锁座→支付→写订单** 步步依赖前一步输出，数据顺传。
- **通信内聚**：`实时告警` 模块内各步骤围绕 **同一流式事件结构** 处理。
- **过程内聚**：`系统启动自检` 按固定顺序检查 DB、缓存、消息，步骤间未必传递业务数据。
- **瞬时内聚**：**日终对账** 与 **T+1 批处理** 因时间窗口聚在一起（耦合时间）。
- **逻辑内聚**：一个模块内用 `if` 堆叠“报表导出/日志清理/配置刷新”等弱相关任务，应拆分。
- **偶然内聚**：无关函数硬凑在同一文件，应避免。

## 六、设计模式（★★★☆☆）

### 6.1 创建型

- 工厂方法、抽象工厂、生成器、原型、单例。**口诀**：工厂抽象建造者，单例原型巧构成。

### 6.2 结构型

- 适配器、桥接、组合、装饰、外观、享元、代理。**口诀**：试桥组装外享代。

### 6.3 行为型

- 责任链、命令、迭代器、中介者、备忘录、观察者、状态、策略、模板方法、访问者、解释器。**口诀**：观察命令责任链，迭代模板状态变；中介策略备忘录，解释访问都包含。

**案例（航空运营智能管理平台）**（写论文时择 2～3 个展开即可）：

- **创建型**：**单例** 配置中心客户端；**工厂方法/抽象工厂** 对接不同支付渠道、不同航司退改规则适配；**生成器** 组装复杂 `行程单/报表` 对象；**原型** 复制营销活动页模板。
- **结构型**：**适配器** 包装旧航信接口与民航局核验协议差异；**外观** 对外统一 `购票门面` 简化客户端；**代理** 网关鉴权、缓存与限流；**装饰** 为查询接口动态加脱敏与审计；**享元** 复用舱位/机场元数据。
- **行为型**：**策略** 动态定价与退改费率；**观察者/发布订阅** 航班变动推送各模块；**责任链** 风控规则逐级校验；**命令** 异步出票与撤销；**状态** 订单/工单状态机；**模板方法** 批处理 ETL 固定骨架；**中介者** 协调票务—消息—通知避免网状依赖。

## 七、速记清单（考前扫一眼）

1. **概要/详细**：概要=高层结构与组织，详细=模块实现与接口算法。
2. **流程六要素**：输入输出活动（相互作用）用户价值。
3. **流程工具**：PFD 不用于描述模块输入/输出与数据加工；IPO、判定表、PDL。
4. **WFMS**：简直过分；**WRM**：指引丁克调管。
5. **三类设计类**：边界（窗口/协议/报表等）→实体（名词）→控制（动名词）；实体有属性，控制有方法，边界可都有。
6. **SOLID**：单职、开闭、里氏、接口隔离、依赖倒置；组合聚合复用、最少知识。
7. **耦合**：非鼠标控制外公嘞（低→高）。
8. **内聚**：公孙铜锅涮罗欧（高→低）；顺序有数据依赖，过程可无。
9. **设计模式**：创建 5、结构 7、行为 11；口诀分类速记。
