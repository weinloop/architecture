## 三、基于架构的软件开发 ABSD（★★★★★）

### 3.1 三个基础

1. **功能分解**：使用基于模块的内聚与耦合技术。
2. **选择体系结构风格**：实现质量与商业需求。
3. **软件模板**：利用已有系统结构。

### 3.2 特点

- 由**商业、质量、功能需求**组合驱动；**递归、迭代**；自顶向下细化到构件与类。
- **用例**捕获功能需求；**质量场景**捕获质量需求（变更、性能、可靠性、交互性）；含预期与非预期场景。

### 3.3 六阶段

**口诀**：**许设温敷实验**——许（需求）设（设计）温（文档化）敷（复审）实（实现）验（演化）。

| 阶段 | 要点 |
|------|------|
| 体系结构需求 | 需求来源：质量目标、商业目标；需求过程、需求评审；标识构件 |
| 体系结构设计 | 需求激发设计，多视图表达，迭代 |
| 体系结构文档化 | 输出：体系结构规格说明、测试质量设计说明书；完整、高质量、从使用者角度、**保证最新** |
| 体系结构复审 | 外部人员（用户代表、领域专家）；最小化可运行系统；标识风险、发现缺陷 |
| 体系结构实现 | 按文档组装构件，连接合成，构件与整体测试 |
| 体系结构演化 | 需求变化归类→演化计划→修改/增删构件→更新交互→组装测试→技术评审 |

## 四、软件架构风格（★★★★★）

### 4.1 数据流

- **批处理**：每步独立程序，前一步**整体**结束并**整体传递**数据后下一步才开始；**可并行**弱。
- **管道-过滤器**：前一步**输出即后一步输入**，数据流连接；过滤器+管道；**可并行**。

**区别**：批处理整体传递、前后不一定数据直接关联；管道-过滤器流式、前后输入输出直接相连。

### 4.2 调用/返回

- **主程序/子程序**：单线程、层次调用，过程调用为连接件。
- **面向对象**：数据与操作封装在对象中。
- **层次型**：每层为上层服务、作下层客户；**相邻层**交互；接口一致可替换。
- **客户端/服务器**：资源不对等；二层=客户应用+数据库服务器+网络；三层=表示层+**应用服务器**（功能层）+数据层。

### 4.3 以数据为中心

- **仓库**：中央数据结构 + 对之操作的独立构件；如 IDE、企业数据管理。
- **黑板**：解空间**分级结构**，多知识源协同；如语音识别、模式识别。

### 4.4 虚拟机

- **解释器**：解释引擎 + 被解释代码存储区 + 当前状态 + 执行进度；灵活、效率较低；适合用户自定义规则、业务灵活组合。
- **规则系统**：规则集、规则解释器、选择器、**工作内存**；专家系统/决策支持。

### 4.5 独立构件

- **进程通信**：构件为独立过程，连接件为**消息传递**（点对点、异步/同步、RPC）；分布式、微服务。
- **事件系统**：**隐式调用**，构件注册到事件，事件触发时自动调用；工具集成、一致性约束、语法检查等。

### 4.6 其他

- **C2**：构件与连接件均有顶/底；构件顶连连接件底、构件底连连接件顶；构件间不直连；连接件可多连。
- **过程控制（闭环）**：软件与硬件呈反馈循环，输入→输出→环境新状态；空调、定速巡航。

## 五、架构复用与 DSSA（★★★☆☆）

### 5.1 软件产品线

- 一组系统**共享公共可管理特性集**，用**公共核心资产**按规定方式集成开发；核心资产库含架构、可剪裁元素、文档、测试计划等。
- **机会复用**：开发中发现可复用即复用；**系统复用**：开发前规划哪些复用。

### 5.2 复用基本过程

- **获取**可复用资产（可靠、适用、易理解、易修改）→**管理**（构件库：存储、分类、检索、浏览、维护）→**使用**（选择、定制、组装集成）。

### 5.3 DSSA

- **垂直域**：特定系统族，通用软件体系结构；适用于成熟稳定领域。
- **水平域**：多系统/族的功能共有部分；不限于成熟领域。
- **三活动**：领域分析（领域模型）→领域设计（DSSA）→领域实现（可重用信息）。
- **四角色**：领域专家、领域分析人员、领域设计人员、领域实现人员。
- **五阶段**：定义领域范围→定义领域特定元素（词典、共性与差异）→定义设计/实现约束→定义领域模型与体系结构→产生/搜集可重用产品单元。

### 5.4 基于构件的软件工程 CBSE（掌握）

- **定义**：基于分布对象技术、强调通过**可复用构件**设计与构造软件系统的软件复用途径。
- **构件（组件）**：自包容、可复用的程序集（源程序或二进制），对外**统一接口**；外部只能通过接口访问，不可直接操作内部。核心特性：**自包容**、**可重用**。

**构件的 5 大特征**

| 特征 | 说明 |
|------|------|
| **可组装性** | 所有外部交互经**公开定义接口**进行，且需支持自身信息的外部访问。 |
| **可部署性** | 自包含，能作为**独立实体**在构件平台运行；**二进制形式**，部署前无需编译。 |
| **文档化** | 完全文档化，用户依文档判断是否满足需求。 |
| **独立性** | 可独立组装、部署；若依赖其他构件需**显式声明**。 |
| **标准化** | 符合某种**标准化构件模型**，确保在 CBSE 过程中规范使用。 |

**构件的 3 种组装方式**

| 方式 | 说明 |
|------|------|
| **顺序组装** | 按顺序调用已有构件，用两个已有构件创造新构件。 |
| **层次组装** | 一个构件直接调用另一构件服务，被调用者为调用者提供所需服务。 |
| **叠加组装** | 两个及以上构件组合创建新构件，合并原功能并提供**新接口**。 |

**接口不兼容的 3 种情况**

| 情况 | 说明 |
|------|------|
| **参数不兼容** | 操作名相同，但**参数类型/个数**不同。 |
| **操作不兼容** | 提供接口与请求接口**操作名不同**。 |
| **操作不完备** | 一构件的提供接口是另一构件请求接口的**子集**，或相反。 |

## 六、质量属性与架构评估（★★★★★）

### 6.1 九大质量属性（面向架构评估）

| 属性 | 描述 | 对策（简） |
|------|------|------------|
| **性能** | 响应能力、处理时间、吞吐量 | 优先级队列、增加资源、减少开销、并发、资源调度 |
| **可靠性** | 错误/异常下维持功能；平均失效时间 | 冗余、错误检测与恢复、异常处理、备份与恢复、负载均衡 |
| **可用性** | 正常运行时间比例；故障间隔/恢复速度 | 心跳、冗余、快速恢复、监控与预警、故障转移 |
| **安全性** | 合法服务、阻止非授权；机密性、完整性、不可否认、可控 | 认证、授权、加密、防火墙、入侵检测、审计日志 |
| **可修改性** | 快速、高性价比变更；含可维护、可扩展、重组、可移植 | 模块化、接口标准化、抽象封装、配置与版本管理 |
| **互操作性** | 与其他系统/环境交互 | 标准协议与接口、数据格式转换、中间件、服务注册发现 |
| **可移植性** | 适应不同运行环境 | 标准 API、抽象层/虚拟机、兼容性测试 |
| **可测试性** | 易于发现缺陷、验证功能与性能 | 分层模块化、测试接口、白盒支持 |
| **易用性** | 好学、好找、好操作、少犯错 | 统一交互与视觉、信息架构、反馈、无障碍、引导与帮助 |

### 6.2 质量属性场景六要素

**口诀**：**原词换纸想读**——原（刺激源）词（刺激）换（环境）纸（制品）想（响应）读（响应度量）。

| 组成部分 | 含义 |
|----------|------|
| 刺激源 | 生成刺激的实体 |
| 刺激 | 到达系统时需考虑的条件 |
| 环境 | 刺激发生的条件 |
| 制品 | 被激励的对象 |
| 响应 | 激励后采取的行动 |
| 响应度量 | 对响应度量的方式 |

### 6.3 敏感点、权衡点、风险点

| 概念 | 定义 | 句式/示例 |
|------|------|-----------|
| **敏感点** | 一个或多个构件（及关系）的**特性** | “某设计/协议**将影响**某质量属性” |
| **权衡点** | **影响多个质量属性**的特性；多个敏感点的综合 | 加密级别**同时影响**安全性与性能 |
| **风险点** | 可能导致架构问题或无法满足质量属性的特性/决策 | “如果…，可能…，影响…” |
| **非风险点** | 对架构无负面影响或可接受 | 使用成熟中间件在当前架构下无明显风险 |

### 6.4 ATAM（★★★★★）

- **目标**：对**性能、可用性、安全性、可修改性**在开发前评价与折中。
- **四阶段**：① 场景和需求收集；② 体系结构视图和场景实现；③ 属性模型构造和分析；④ 折中（识别敏感点、权衡点）。
- **实践四阶段**：演示→调查和分析（含上述四步）→测试→报告。
- **效用树**：树根→**质量属性**→属性分类→**质量属性场景**（叶子）；对场景按重要度与实现难易度标 H/M/L，形成优先级对（如 H,L）= 重要且易实现。
- **ATAM 关注**：**需求**（出发点与终点）。

### 6.5 SAAM 与 CBAM

- **SAAM**：场景技术；主要关注**可修改性**；场景开发、架构描述、单场景评估、场景交互、总体评估。
- **CBAM**：按**投资回报率 ROI** 选择架构策略；质量属性变化带来效用。

## 七、速记清单（考前扫一眼）

1. **4+1**：罗凯进五常 / 落实进步用 / 逻实进配。
2. **ABSD**：三基础（功能分解、风格选择、软件模板）；六阶段许设温敷实验；质量场景=变更/性能/可靠性/交互性。
3. **风格**：批处理整体传递 vs 管道-过滤器流式；仓库中央数据+独立构件 vs 黑板分级解空间；解释器=引擎+存储+状态+进度；进程通信=显式消息 vs 事件=隐式调用；C2 顶底连接规则。
4. **复用**：产品线=核心资产+规定方式；三阶段获取→管理→使用；DSSA 垂直/水平、三活动四角色五阶段。
5. **九大属性**：性能/可靠性/可用性/安全性/可修改性/互操作性/可移植性/可测试性/易用性及对策。
6. **场景六要素**：原词换纸想读。
7. **敏感点**：构件特性→影响某属性；**权衡点**：影响多属性；**风险点**：可能…影响…。
8. **ATAM**：四属性（性能、可用性、安全性、可修改性）；四阶段（场景需求→视图与实现→属性模型与分析→折中）；效用树=质量属性→分类→场景；关注需求。
