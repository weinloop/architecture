# 综合知识核心知识点总结

本文档知识点覆盖软件工程、软件架构、数据库、中间件、分布式技术、网络协议、信息安全、AI、项目管理等核心领域，按模块分类梳理如下，重点标注核心考点与关键定义。

## 一、软件工程与开发方法论 {#zonghe-gongcheng}

### 1\. 软件生命周期与核心过程

- 软件四大核心活动：**软件描述、软件开发、软件有效性验证、软件演化**。

- 软件生命周期三阶段：软件定义阶段、软件开发阶段、软件运行与维护阶段。

- 软件工程核心循环：计划（Plan）、执行（Do）、检查（Check）、处理（Action）。

### 2\. 主流开发方法

- **敏捷开发**：核心是**适应性而非预设性**，强调灵活应对变化、以人为中心的协作、快速交付可用软件。

- **RUP 统一过程**

    - 三大核心特征：**用例驱动、以体系结构为中心、迭代和增量开发**。

    - 九大核心流程：业务建模、需求、分析与设计、实现、测试、部署、配置与变更管理、项目管理、环境。

    - 四大项目阶段：初始阶段（可行性验证）、细化阶段（架构与核心需求验证）、构建阶段（功能实现）、交付阶段（系统落地验收）。

### 3\. 需求管理

- 核心任务：在软件全生命周期中**跟踪、控制和管理需求的变更与状态**。

- 需求核心活动：需求抽取（获取）、需求确认、需求规格说明、需求管理。

### 4\. 软件复用

- 复用类型：

    - **横向重用**：跨不同应用领域，对通用软件元素（算法、通用组件）的复用。

    - **纵向重用**：同一应用领域内，不同系统层级 / 生命周期阶段的垂直方向复用。

- 双生命周期模型：**领域工程（生产可复用资产）\+ 应用工程（使用资产构建应用）**。

- COM 对象重用形式：**包含（Containment）**（外部对象转发请求）、**聚集（Aggregation）**（外部对象直接暴露内部对象接口）。

- DSSA（领域特定软件架构）必备特征：严格定义的问题域 / 解域、领域内普遍性、恰当的构件组织抽象、固定可重用的开发元素。

### 5\. 逆向工程

可恢复两类核心信息：**实现级**（函数、类、模块等代码级信息）、**结构级**（组件调用关系、模块划分、层次结构等架构信息）。

## 二、软件架构设计

### 1\. 架构设计核心方法

- **ABSD（基于架构的软件开发方法）**

    - 核心驱动：商业、质量和功能需求的组合驱动。

    - 核心特征：**自顶向下、递归细化**，从整体架构逐步分解到软件构件 / 模块。

    - 描述方式：采用**视角和视图**描述架构，用例和质量场景描述需求。

    - 核心视角：静态视角、动态视角、配置视图、逻辑视图。

### 2\. 架构设计与演化流程

- 架构设计过程：提出体系结构模型→映射构件→分析构件相互作用→产生体系结构→设计评审。

- 架构实现过程：复审后的文档化架构→分析与设计→构件实现→构件库→构件组装→系统测试→体系结构演化。

- **体系结构演化 6 步骤**：需求变化归类→制订体系结构演化计划→修改 / 增加 / 删除构件→更新构件的相互作用→构件组装与测试→技术评审。

### 3\. 架构视图模型

**4\+1 视图模型**：以场景为中心，包含逻辑视图、开发 / 实现视图、进程 / 过程视图、物理 / 部署视图。

### 4\. 主流架构风格

|架构风格大类|核心子类|核心特征|典型场景|
|---|---|---|---|
|数据流风格|批处理、管道 \- 过滤器|前一环节输出为后一环节输入，整体 / 流式处理|编译器、ETL 工具、图像处理管线|
|调用 / 返回风格|主程序 / 子程序、面向对象、层次型|显式调用、分层控制、封装性|结构化程序、GUI 框架、网络协议栈|
|独立构件风格|进程通信、事件驱动|独立构件间消息传递、事件触发隐式调用|分布式系统、消息队列、JS 事件监听|
|虚拟机风格|解释器、规则系统|执行自定义规则 / 脚本，灵活适配场景|脚本引擎、专家系统、规则引擎|
|以数据为中心风格|仓库体系、黑板体系|中央共享数据源 \+ 独立处理构件，黑板核心为松耦合|电商系统、CRM、语音识别、自动驾驶决策|
|其他|闭环控制、C2 风格、SOA|反馈调节、松耦合构件协作、粗粒度服务化|工业自动化、插件化架构、企业级应用集成|

### 5\. 架构评估

- **SAAM 方法**：核心输入为问题描述、需求说明、架构描述文档；场景通过**刺激、环境、响应**描述；**主要评估系统可维护性**。

- **ATAM 方法**：**主要评估性能、可用性、安全性、可修改性**；通过**效用树**对质量属性分类和优先级排序；场景核心要素为刺激源、刺激、环境、制品、响应、响应度量。

- 架构评估核心概念：

    - 敏感点：实现特定质量属性需关注的构件 / 关联特性。

    - 权衡点：影响多个质量属性的特性，是多个属性的敏感点。

    - 风险点：可能对质量属性造成负面影响的设计隐患；非风险点：可接受、无负面影响的设计。

### 6\. 核心设计原则与质量属性

- 可靠性核心维度：**无故障性、可用性、容错性、可恢复性**。

- 耦合设计原则：优先采用**数据耦合**（耦合性最低），限制控制耦合、特征耦合，**避免使用内容耦合**。

- 云原生架构原则：**弹性**（资源动态适配业务需求）、**韧性**（故障场景下的系统保护）。

### 7\. 遗留系统与集成架构

- 遗留系统 4 象限处理策略：改造（高水平高价值）、集成（高水平低价值）、继承（高价值低水平）、淘汰（低水平低价值）。

- EAI（企业应用集成）4 层架构（从下至上）：通讯服务、信息传递与转化服务、应用连接服务、流程控制服务。

## 三、UML 与系统建模

### 1\. UML 核心图分类

- **结构图**：类图、对象图、包图、组合结构图、构件图、部署图、外廓图。

- **行为图**：顺序图（序列图）、通信图、时间图、交互概览图、活动图、状态机图、用例图。

- 扩展差异：**SysML 新增了 UML 没有的需求图**，用于捕获和跟踪系统需求。

### 2\. 核心图核心知识点

- **用例图**：用例之间的三大关系为**包含、扩展、泛化**。

- **顺序图**

    - 标准消息类型：同步消息、异步消息、返回消息、创建消息、销毁消息。

    - 组合片段：alt（互斥选择 if\-else）、opt（可选 if）、loop（循环）、break（跳出）、par（并行）、region（临界区）。

- 分析类划分：**实体类**（名词，对应系统交互的人 / 物 / 系统）、**控制类**（动名词，对应功能 / 处理流程）、**边界类**（系统与外界的交互接口、人机界面）。

### 3\. 主流建模工具

- **Petri 网**：严谨的数学化建模工具，由位置、变迁、弧构成，适合并发、同步、冲突等复杂业务流程的建模与分析。

- **DFD 数据流图**：4 种基本元素为数据流、处理、数据存储、外部实体；核心平衡原则为**父图与子图平衡、子图内部平衡**。

- 其他：IDEF（IDEF0 功能建模、IDEF3 过程描述）、DEMO（组织建模，侧重企业沟通协作）、BPM（业务流程管理理念，非具体建模工具）。

## 四、数据库技术 {#zonghe-shuju}

### 1\. 数据库设计全流程

四大核心阶段：**需求分析→概念设计（E\-R 模型）→逻辑设计（关系模式 \+ 规范化）→物理设计（存储结构 \+ 索引）**。

### 2\. 数据建模核心

- **ER 图**：信息建模核心工具，抽象现实世界的实体、属性、实体间联系，是数据库模型构建的基础。

- 逻辑设计核心：将 E\-R 图转换为关系模式，确定主键、外键、完整性约束，定义用户视图。

### 3\. 关系模型与函数依赖

- 关系代数：以 \\*\\* 关系（元组的集合）\\*\\* 为操作对象，核心操作包括选择、投影、连接、并、交、差。

- 函数依赖类型：完全函数依赖、部分函数依赖、传递函数依赖。

- **Armstrong 公理系统**：核心规则为自反律、增广律、传递律；推导规则为合并规则、伪传递规则、分解规则。

- 范式核心要求：

    - 2NF：消除非主属性对主键的**部分函数依赖**。

    - 3NF：消除非主属性对主键的**传递函数依赖**。

    - BCNF：消除主属性之间的部分和传递函数依赖。

    - 4NF：消除**多值依赖**。

### 4\. 数据库三级模式结构

- **外模式（视图）**：面向用户，用户可见的数据逻辑结构，一个数据库可对应多个外模式。

- **概念模式（库表）**：全体数据的逻辑结构与公共视图，**一个数据库仅一个概念模式**。

- **内模式**：数据的物理存储结构与存储组织，**一个数据库仅一个内模式**。

### 5\. 数据完整性约束

三大核心约束：**实体完整性、参照完整性、用户自定义完整性**。

### 6\. 数据库分类与选型

|数据库类型|核心适用场景|代表产品|
|---|---|---|
|关系型数据库|强事务、结构化数据、复杂关联查询（电商订单、金融转账）|MySQL、Oracle、PostgreSQL|
|文档型数据库|半结构化数据、需求频繁变更、无需复杂关联（内容管理、商品详情）|MongoDB|
|键值对数据库|高并发、简单键值映射、缓存 / 会话存储 / 分布式锁|Redis、etcd|
|列式数据库|大数据 OLAP 分析、批量处理、统计报表 / 日志分析|HBase、ClickHouse|
|图数据库|复杂关系分析、社交网络、知识图谱、欺诈检测|Neo4J|
|向量数据库|AI 相似性查询、推荐系统、语义搜索、RAG 场景|Milvus、Pinecone|

### 7\. 集中式 vs 分布式数据库

核心差异体现在：分布式数据库支持水平扩展、多副本高可用，通过 2PC、Raft 等协议保障一致性，适配海量数据与高并发场景；集中式数据库垂直扩展有瓶颈，一致性保障简单，适配中小规模、强事务场景。

### 8\. 数据访问模式

核心模式包括：在线访问模式、DAO（数据访问对象）、DTO（数据传输对象）、离线数据模式、ORM（对象 / 关系映射）。

## 五、Redis 缓存核心技术

### 1\. 核心数据类型与适用场景

- String：配置存储、对象缓存、数据统计、请求限流。

- Hash：用户 / 商品 / 购物车等多字段信息存储，支持单字段操作。

- List：最新内容记录、简单消息队列。

- Set：UV 统计、交集运算、随机抽奖场景。

- Zset：各类排行榜、热度排序。

- Bitmap：用户签到、活跃统计、行为记录。

- HyperLogLog：海量数据基数计数（百万级 IP/UV 统计）。

- Geospatial：地理相关场景（附近的人）。

### 2\. 过期数据与内存淘汰策略

- 过期数据删除策略：

    - **惰性删除**：取出 key 时才检查过期，对 CPU 友好，易造成过期 key 堆积。

    - **定期删除**：每隔一段时间抽取一批 key 删除过期数据，平衡 CPU 与内存占用。

- 内存淘汰核心策略：volatile\-lru（最近最少使用）、volatile\-ttl（即将过期优先）、volatile\-random、allkeys\-lru、allkeys\-random。

### 3\. 事务与高可用架构

- Redis 事务：核心是命令打包顺序执行，流程为 MULTI（开启）→命令入队→EXEC（提交）/DISCARD（放弃）；**不严格满足原子性**（命令执行失败不回滚），持久性依赖 RDB/AOF 持久化配置。

- 高可用架构：

    - 主从架构：读写分离，主节点写、从节点读，手动故障转移。

    - Sentinel 哨兵：核心功能为**监控、自动故障转移、状态通知**，实现主从架构自动化容灾。

    - Cluster 集群：多主多从架构，通过 16384 个哈希槽分片存储数据，支持水平扩展与自动故障转移。

### 4\. 核心问题与解决方案

- **缓存与数据库一致性**：推荐 Cache\-Aside 模式（读：缓存未命中则读库并更新缓存；写：先写数据库，再删除缓存），避免双写不一致。

- **缓存穿透**：恶意请求不存在的 key 导致数据库压力过大，解决方案为布隆过滤器、空值缓存。

- **大 Key 处理**：拆分大 Key、清理非适配数据、定期清理过期数据。

- **热 Key 处理**：多级本地缓存、热 key 多节点备份、热 key 拆分、核心业务隔离。

### 5\. 补充特性

- 分布式锁选型：简单场景用 Redis\+Lua 脚本；强一致需求用 ZooKeeper/Etcd；高并发弱一致场景用 Redis 集群 \+ RedLock。

- String 与 Hash 对比：String 内存占用更低，适合整体对象存储；Hash 支持单字段操作，适合字段频繁变更的场景。

## 六、分布式与云原生技术 {#zonghe-fenbushi}

### 1\. 分布式核心理论

- 分布式系统核心优势：高可扩展性、高可用性、高性能。

- **BASE 理论**：Basically Available（基本可用）、Soft State（软状态）、Eventually Consistent（最终一致性），是 CAP 定理中 AP 方案的扩展，适配分布式高可用场景。

- ACID vs BASE：ACID 保障强一致性，适配金融转账等核心事务场景；BASE 保障最终一致性，优先保障高可用，适配社交动态、日志处理等场景。

### 2\. 分布式一致性算法

- **Raft 算法**：崩溃容错型，分为 Leader 选举、日志复制两阶段，易实现、效率高，广泛用于分布式存储 / 数据库（TiKV、etcd），不抗拜占庭故障。

- **PoW 工作量证明**：拜占庭容错型，去中心化、安全性高，用于比特币等公链，能耗高、处理效率低。

### 3\. 微服务与云计算

- RESTful API 核心优势：标准 HTTP 通信、统一资源交互、无状态、支持缓存、易扩展；核心 HTTP 方法语义：**POST 创建资源、PUT 整体替换资源、PATCH 局部更新资源**。

- 云计算分类：

    - 区块链分类：私有链、公有链、联盟链。

    - 云部署模式：公有云、私有云、混合云，核心差异在所有权、安全性、扩展性。

- 边缘计算：核心优势为低延迟、低带宽消耗、高数据安全，通过边 \- 云协同缓解云端压力，适配物联网、工业控制、实时性要求高的场景。

- 物联网三层架构：**感知层、网络传输层、应用层**。

### 4\. 流量管控与消息队列

- 限流组件：网关级（Nginx、APISIX、Spring Cloud Gateway）、应用级（Guava RateLimiter、Sentinel）、中间件级（Redis），核心算法包括漏桶、令牌桶、滑动窗口。

- 消息队列核心能力：解耦、异步化、削峰填谷、可靠传输、消息路由、数据管道；主流产品包括 Kafka、RabbitMQ、RocketMQ。

### 5\. 大数据技术体系

- Lambda 架构核心分层：数据采集层→数据存储层→批处理层 \+ 实时加速层→服务应用层。

- 数仓核心分层：ods（原始数据层）→dwd（明细数据层）→dws（汇总数据层）→ads（应用数据层）。

- 核心组件：Sqoop（离线数据采集）、Kafka（实时数据采集 / 消息）、HDFS（分布式存储）、MapReduce/Spark（批处理）、Spark Streaming/Flink（流处理）、Hive（数据仓库）、HBase/TiDB（数据存储）、ELK（日志分析）。

- Elasticsearch：分布式搜索引擎，核心优势为近实时搜索、高扩展性、高可用、多维度查询，适配网站搜索、日志分析、实时监控等场景。

## 七、Web 服务与网络协议

### 1\. Web 服务核心协议栈（从下至上）

网络层（HTTP/HTTPS、SMTP、FTP）→消息层（SOAP）→服务描述层（WSDL）→服务发现层（UDDI）。

### 2\. 核心 Web 服务标准

- **SOAP**：简单对象访问协议，基于 XML 的消息传输协议，定义消息格式、编码规则与 RPC 约定，用于 Web 服务间通信。

- **WSDL**：Web 服务描述语言，XML 格式，核心描述服务三大核心信息：**服务做什么、如何访问、服务位于何处**。

- **UDDI**：通用描述、发现和集成服务，用于描述服务元数据、发现服务提供者、集成现有服务。

### 3\. 核心网络协议与端口

- 常用端口：FTP \(20/21\)、SSH \(22\)、Telnet \(23\)、SMTP \(25\)、DNS \(53\)、TFTP \(69\)、HTTP \(80\)、POP3 \(110\)、HTTPS \(443\)。

- 协议分层：TCP/UDP（传输层）、IP/ICMP/IGMP/ARP/RARP（网络层）、以太网 / 令牌环等（数据链路层）。

### 4\. Web 服务组件对比

- Nginx：Web 服务 / 反向代理，高并发能力强，适配静态服务、负载均衡、反向代理。

- Tomcat：Servlet 容器，中等并发，适配 Java Web 应用、Spring 生态集成。

- Netty：高性能异步通信框架，适配实时通信、微服务底层框架（Dubbo、gRPC）。

## 八、信息安全与知识产权

### 1\. 核心安全协议

- **SSL**：传输层安全协议，由**握手协议、记录协议、警报协议**组成，用于互联网机密文件传输。
