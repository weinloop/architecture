## 核心业务场景一：跨域数据资产检索与共享申请
**场景特点**：涉及总部与5大区域中心、23家分公司的跨域网络协同；属于平台最核心的主业务链路；具有明显的高并发读特征（如节假日 5000 并发）；高度依赖外部业务系统（如 IAM、工单系统）的集成。

| 架构技术主题 | 该场景下面临的具体冲突与问题 | 核心技术解决方案与落地效果 |
| :--- | :--- | :--- |
| **1. 大模型及 AI Coding** | 数据管家频繁使用自然语言检索资产，但公网模型极易导致旅客隐私与企业核心资产外泄，触碰数据出境红线。 | 部署私有化 LLM 并结合 RAG（检索增强生成），对敏感字段进行哈希脱敏与权限分级审批，实现智能问答的内网闭环。 |
| **2. 可靠性设计** | 核心资产目录库或分布式图数据库一旦发生单点硬件宕机，将导致全集团跨域检索与申请流程瘫痪数小时。 | 实施硬件+软件的同城双活冗余设计。采用 Kubernetes StatefulSet 多副本部署及 TiDB 强一致性集群，消除单点故障，保障 99.99% 高可用。 |
| **3. 服务网格 (Istio)** | 跨域申请流程涉及十几个微服务跨网段交互，任何网络抖动都会导致超时，传统排障需数小时。 | 在 Envoy 数据面开启无侵入的全链路追踪（Trace ID 透传）。通过 Kiali 直观呈现跨域调用拓扑，将故障定位时间（MTTR）从小时级降至分钟级。 |
| **4. 大数据 Kappa 架构** | 总部与区域中心之间的资产目录、共享申请状态和授权结果变更频繁，若采用多链路分散同步，极易出现跨域检索口径不一致与状态滞后。 | 将目录变更、申请流转和授权结果统一写入 Kafka，由 Flink 以同一套流式逻辑实时处理并按位点重放校正，实现跨域状态秒级同步与统一口径。 |
| **5. 缓存技术** | 节假日高峰期海量跨域检索请求直打底层数据库，导致连接池耗尽和磁盘 I/O 瓶颈，引发响应急剧劣化。 | 构建“Caffeine 本地缓存 + Redis 分布式缓存”的多级架构，配合夜间低谷期的数据预热。将跨域检索响应压缩至 100ms 内，主库压力锐减 70%。 |
| **6. 智能运维 (AIOps)** | 长链路涉及 K8s、TiDB、网关等多组件，监控数据极度分散，页面加载缓慢时难以排查。 | 构建运维数据中台统一汇聚 Metrics/Logs/Traces。利用动态基线算法进行时序异常检测，并提前 72 小时预测网关和存储的容量瓶颈。 |
| **7. 自动化测试** | 跨域共享申请高度依赖 IAM 等外部系统，联调环境接口契约变更频发，导致申请流断裂。 | 实施消费者驱动的契约测试（Pact）并纳入 CI/CD 发布门禁。外部系统不兼容变更在合入主干前被拦截，消除了线上接口不一致窘境。 |

## 核心业务场景二：月末全量数据质量检测与血缘解析
**场景特点**：属于后台海量数据密集型计算任务；月底跑批时会瞬间占用大量计算资源，与前端用户交互产生严重资源竞争；核心算法规则迭代极其频繁。

| 架构技术主题 | 该场景下面临的具体冲突与问题 | 核心技术解决方案与落地效果 |
| :--- | :--- | :--- |
| **1. 大模型及 AI Coding** | 血缘解析等核心后台模块迭代极快，需 AI 辅助编码，但这类“安全底座”代码绝不能“生成即上线”。 | 将 AI 定位为“副驾驶”，在 IDE 辅助生成骨架，在 CI/CD 叠加 AI Code Review。对核心模块建立严格的“人机协同门禁”与双人交叉复核机制。 |
| **2. 可靠性设计** | 月末跑批任务并发调用底层组件，抢占资源导致目录服务 CPU 飙升甚至 OOM，引发全平台雪崩。 | 在 K8s 层面精细化配置 Requests/Limits 隔离资源；引入 Sentinel 对耗时接口配置熔断降级。将跑批任务对前端交互的影响降至最低。 |
| **3. 服务网格 (Istio)** | 血缘解析等核心算法模块频繁升级，若采用传统全量发布，一旦新版本有逻辑缺陷将导致全量血缘断裂。 | 利用 Istio 构建灰度发布与流量镜像体系。先通过“影子测试”在真实负载下验证，再按特定区域平滑灰度放量，实现“零停机、零故障”升级。 |
| **4. 大数据 Kappa 架构** | 若实时血缘解析（增量日志）与月末全量血缘重构（全量日志）采用独立逻辑，极易导致定义不一致和口径漂移。 | 统一采用 Kafka 作为可重放日志总线，Flink 作为流批一体计算引擎。通过调整消费位点实现历史重放，真正做到“一条管道一套语义”，运维成本减半。 |
| **5. 缓存技术** | 跑批瞬间涌入的海量并发请求读取元数据，若此时支撑的 Redis 集群宕机，将直接击穿至底层数据库导致瘫痪。 | 采用 Redis 主从哨兵模式确保秒级主从切换；结合 K8s 的 HPA 机制根据并发连接数自动扩缩容无状态微服务节点，彻底杜绝缓存层瓶颈。 |
| **6. 智能运维 (AIOps)** | 跑批导致复合型宕机故障时，跨部门联合排查耗时过长，恢复慢，严重影响全集团治理报表产出。 | 将智能根因分析与自动化恢复剧本（Playbook）深度联动。如遇 OOM 故障，AIOps 系统自动触发 K8s 异常 Pod 驱逐与异地重建，MTTR 降至 20 分钟。 |
| **7. 自动化测试** | 数据大屏、质量规则配置等 UI 界面高频改版，传统硬编码脚本大面积失效，回归测试耗时且易遗漏。 | 在前端全面引入页面对象模式（POM），将“血缘节点展开”等封装为独立组件。建立多终端兼容性矩阵并行回归，极大提升了 UI 脚本的健壮性。 |

## 核心业务场景三：敏感数据脱敏策略与访问控制
**场景特点**：涉及民航局等保三级的数据出境与隐私保护红线；脱敏规则和鉴权策略复杂多变，受旅客身份、业务域、网络环境等多重条件组合影响；必须满足极高的合规审计要求。

| 架构技术主题 | 该场景下面临的具体冲突与问题 | 核心技术解决方案与落地效果 |
| :--- | :--- | :--- |
| **1. 大模型及 AI Coding** | 大模型在生成脱敏建议或辅助编写数据处理 SQL 时，出现前后不一致或编造不存在字段的“幻觉”。 | 构建知识图谱并与硬性规则引擎联动，“模型负责建议，规则负责决定”。对模型输出实行三级采纳策略，并在发布前进行对抗样例的回归测试。 |
| **2. 可靠性设计** | 脱敏服务在面对海量调用时，传统人工巡检无法及时发现“脱敏算法内存泄漏”或“跨域鉴权网络隐性瓶颈”。 | 基于 Prometheus 采集全链路指标，并接入大模型分析引擎识别基线异常。在服务彻底卡死前提前数小时推送预警，隐性故障发现率提升 80%。 |
| **3. 服务网格 (Istio)** | Kubernetes 内网扁平缺乏隔离，一旦边缘应用被攻陷，明文传输的脱敏规则极易被网络嗅探窃听。 | 开启“STRICT”模式的双向 TLS（mTLS），Istiod 自动管理证书轮换。结合 AuthorizationPolicy 细粒度控制，构建严格的“零信任”加密内网。 |
| **4. 大数据 Kappa 架构** | 法务部门要求保留所有资产的访问日志与脱敏规则变更轨迹以备长期审查，但 Kappa 原始流长期保留成本极高。 | 实施“冷热分层”存储：Kafka 仅保留 7 天热数据支撑秒级异常告警；历史日志异步归档至低成本 OSS。Flink 无缝读取冷数据，满足任意时刻精准审计回溯。 |
| **5. 缓存技术** | 安全管理员更新脱敏策略后，若各区域网关缓存未能及时失效（脏读），将导致普通用户越权查看到敏感明文。 | 采用 Cache-Aside 模式结合“延迟双删”策略，利用 RocketMQ 广播失效消息同步清除各节点本地缓存。结合 TTL 抖动和布隆过滤器防范雪崩与穿透。 |
| **6. 智能运维 (AIOps)** | 区域专线网络抖动时，系统瞬间抛出数百条关于“脱敏失败、鉴权超时”的相似冗余告警，形成“告警风暴”。 | 利用 NLP 日志聚类与运维知识图谱，将离散告警沿着调用链收敛为单一的“网络抖动”根因事件。针对已知异常配置分级自愈动作，大幅减少重复排障劳动。 |
| **7. 自动化测试** | 脱敏规则生效条件错综复杂，纯人工编写测试用例极易遗漏安全边界，每轮全量安全回归测试耗时冗长。 | 利用 AI 分析 PRD 自动生成脱敏边界异常组合用例，并在 UI 层引入脚本自愈机制实现“测试左移”。全量安全回归测试周期由近 3 天缩短至 4 小时以内。 |
