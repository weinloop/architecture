# 模版和技巧

**整体论文字数建议：** 正文 2500～2800。

## 段落结构与内容

| 段落 | 内容 | 字数 |
|------|------|------|
| **项目背景** | 为什么有这个项目（政策/市场/业务痛点需求）；项目情况和功能介绍（项目为谁，提供什么功能，解决什么业务场景，项目有哪些核心模块）；项目技术介绍（**给出具体且合理的数据量情况**，介绍通用架构或者推导出选题相关的技术）。 | 500 |
| **技术方法说明** | 回应问题 2，说明相关的技术理论，注意过渡语句。 | 400 |
| **论点展开** | 3 个分论点，简化版公式：**结合业务场景**提出问题 → 分析问题 → **关键技术落地**解决问题 → 解决效果展示。 | 1200～1500 |
| **论文结尾** | 总结技术运用成果（贴合题干）、项目效益，站在架构的角度去总结项目或技术架构 1～2 点的不足，并提出解决方案。 | 400 |

## 特别强调

1. 请将论文中大量、很多、海量等形容词替换为**具体且合理的数据量级**，例如：50 个微服务、20 台服务器、50 万用户、单表 10 亿级数据、核心接口 RPS 为 1000。
2. 分论点论证过程：请先把项目中对应的业务场景和问题点讲清楚，再写技术解决方案。
3. **分论点首句**必须是当前的核心论点，约 60 字，句式：**用 xx 技术、解决了 xx 问题，保障/支撑了 xx 业务。**

# 2.2. 分论点示例

> 基于 Kubernetes 对云计算进行资源管理和配置，实现各业务应用、中间件等的资源隔离和动态扩容，保障了平台在业务高峰的稳定性，提升了平台的资源利用率。在集体资产经营管理平台的建设和运维中，因为其业务体系的庞大，涉及资产清查、资金、资产、合同等 30+ 业务应用微服务，为各市区提供的高可用的 5 个数据接入服务应用，为 2000 万组织成员提供的 3 个组织经营数据公开查询服务，同时包括分布式数据库 TiDB、分布式缓存 Redis、消息中间件 RabbitMQ、搜索引擎 Elasticsearch 等基础组件，和 Hadoop、Spark、ClickHouse、QuickBI 等大数据组件，传统的对云计算资源的运维方式管理难度极大，同时难以应对突发的业务高峰，存在稳定性不足、资源利用率低下等问题。于是我们引入 Kubernetes（k8s）对云计算进行资源管理和配置，将所有的云计算资源均作为 node 纳入 k8s 进行统一管理，根据不同云计算资源的配置特性，使用 k8s 的标签（Label）对 Node 进行分类并结合业务特征进行匹配使用，比如将 SSD 盘和高性能 CPU 标记为 `db-node`，用于数据库 TiDB 的部署节点，将机械硬盘标记为 `data-node`，用于 Hadoop 组件存储历史冷数据，充分利用各服务器的性能；基于 Namespace 划分不同的命名空间，合理化各业务应用和中间件的管理，包括业务应用空间、接口服务空间、公开查询空间、基础组件空间、大数据组件空间，做到应用服务的部署和管理隔离。得益于 k8s 的资源管理配置、以及动态水平扩容和垂直扩容的能力，平台建设运行后，成功应对了 2024 年年末的财务上报高峰和 2025 年年初的资产清查报表上报高峰，并支撑日常数十亿数据的计算分析。
