# 2. 云原生可靠性设计——技术拆解

## 1. 项目背景分析与决策
- **核心矛盾**：业务存在跨域长链路、月末全量跑批与在线访问高峰叠加等特征，对高可用拓扑与故障隔离有极高要求。
- **风险分析**：若沿用单机房、弱冗余或依赖人工切换的传统模式，将导致业务连续性难以保障、故障扩散难以抑制、隐性异常难以发现。
- **技术决策**：局部修补难以保障高并发下的稳定性。相比传统架构，决定基于**云原生可靠性设计（多副本编排、精细化流控、全栈可观测）**建设平台。

## 2. 问题 2 回应与过渡
- **技术定义**：保障微服务系统在复杂环境下持续可用的核心技术体系，涵盖基础设施高可用、服务治理容错与全链路可观测。
- **项目背景**：资产目录、质量检测等服务承载总部至分公司协同主干流量，涉及 20+ 微服务，任一抖动易引发级联故障。
- **技术要点**：
    - **Kubernetes** 多副本 + **Pod 反亲和** + **TiDB** 图库高可用集群（冗余与切换）。
    - **命名空间**配额 + **Sentinel** 限流熔断 + 业务**有界降级**（保护关键路径）。
    - **Prometheus** 黄金指标 + **分布式追踪** + **分级告警**（识别异常、快速处置）。
- **过渡总结**：通过高可用拓扑构建、故障隔离与流控、全链路主动预警完成建设。

## 3. 正文论点一：跨域协同/检索
- **标题**：构建多副本与同城双活拓扑，解决跨域协同场景中的单点故障难题
- **业务痛点**：审批流、血缘等核心服务承载跨域主干流量，任一环节副本不可用或链路抖动均可导致业务中断。
- **风险分析**：单机房、单库弱冗余模式易引发业务长时不可用，RTO/RPO 不可控。
- **方案落地**：
    - 实例冗余：同城双活部署，Kubernetes 多副本 + Pod 反亲和 + 健康探针。
    - 存储高可用：对接 TiDB、图数据库集群，统一连接超时与重试策略。
    - 演练验证：定期开展桌面与生产级容灾演练。
- **价值体现**：主链路可用性稳态维持在 99.99%，秒级切换流量，单点/机房级事件影响压缩至最小。

## 4. 正文论点二：后台计算/跑批
- **标题**：实施服务隔离与限流熔断策略，解决资源争抢场景中的故障级联难题
- **业务场景/痛点**：月末跑批与在线查询混部，批处理易耗尽连接池/线程池，局部异常易拖垮全链路。
- **风险分析**：粗放的资源管理会导致关键路径被挤占、P99 时延恶化、全站性服务不可用。
- **方案落地**：
    - 资源隔离：划分命名空间与资源配额，结合 HPA 按画像伸缩。
    - 流控熔断：网关与 Sentinel 配置滑动窗口熔断与排队上限。
    - 业务降级：对非关键子任务实施异步补偿或缓存回退。
- **价值体现**：高峰窗内前台核心接口错误率与 P99 受控，未再因局部异常导致全站阻塞，保障了在线体验。

## 5. 正文论点三：安全合规/审计
- **标题**：落实全链路主动预警机制，解决隐性异常场景中的发现滞后难题
- **业务场景/痛点**：异常多为错慢比缓升或慢查询前兆，若发现不及时，待用户投诉后再处理往往已造成严重后果。
- **风险分析**：传统运维模式识别滞后、根因难还原、审计留痕缺失。
- **方案落地**：
    - 智能识别：Prometheus 全栈采集指标，通过基线分析识别缓升与离群。
    - 同源审计：分布式 Trace 与审计事件串联，支持下钻慢 SQL 根因。
    - 闭闭响应：告警接入值班平台，强化安全红线响应时效。
- **价值体现**：多起隐性异常在大面积故障前被成功拦截，处置过程具备完整审计依据。
