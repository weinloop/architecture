# 论大模型及 AI Coding 技术在数据资产智能管理平台中的应用

## 1. 摘要
2024年3月，我参与了某航空集团数据资产智能管理平台建设。该项目面向集团总部、5大区域中心和23个分公司，服务数据治理委员会、数据管家及运维人员等角色，提供数据资产目录、数据质量、数据安全、血缘分析、数据共享和智能问答等核心功能。在项目中，我担任系统架构师，负责平台总体架构设计和落地。 **本文围绕大模型及 AI Coding 技术应用展开论述，通过私有化大模型与 RAG 知识工程实现数据治理问答闭环与脱敏访问控制，基于 IDE 与 CI/CD 集成的 AI Coding 及人机协同门禁保障安全关键代码可审计交付，结合领域知识图谱与分级采纳机制抑制模型幻觉并落实安全合规审查。** 系统于 2025 年 8 月正式上线，截至目前稳定运行，平台运行平稳、核心能力达到建设预期，系统整体可用性保持在99.99%以上，有力保障集团数据治理业务平稳开展。

## 2. 项目背景
某航空集团业务涵盖航空客运、旅客服务、机务维修、地面服务等领域，总部数据中心、5大区域中心及23家分公司长期积累了大量分散在数据库、数据仓库中的多模态数据资产。依据民航智慧民航数据管理相关政策标准，集团启动数据资产智能管理平台建设，亟需构建统一的数据资产目录、质量规则库、安全策略库、血缘图谱及智能知识库，实现资产盘点、共享申请、质量治理、安全管控、智能运营一体化管理。平台需适配集团统一治理、区域分级运营、分公司轻量接入架构，支撑10万+资产目录、5000条质量规则、年10亿次以上数据服务调用，同时满足5000并发用户、页面查询2秒内响应、系统可用性99.99%以上等高并发、高一致性技术指标。

本人作为中标方系统架构师，负责平台总体技术路线设计。 **经分析，平台在智能问答与研发提效中既要释放大模型与 AI Coding 的效率价值，又要严格守住高敏数据不出域与安全关键代码可审计的底线。若继续沿用公网模型直连、插件各自为政与“生成即合入”的做法，将导致越权召回、幻觉误导与交付风险难以收敛，工程边界不可回滚、不可签核。** 在年十亿次量级的服务调用、多区域协同与强合规等总体约束下，若上述矛盾仅通过局部修补与点状工具堆叠来应对，而未在平台级形成成体系的架构与工程能力，将难以同时保障高并发业务体验、数据与操作可审计、以及平台级可持续演进。

**相比依赖外部通用 SaaS 与散落插件，将私有化大模型与可编排的 RAG 知识工程、可策略化的脱敏与授权，以及嵌入流水线与代码评审的 AI Coding 收束到统一内网与工程规范之下，更契合多区域、多角色、多语种研发并存且须满足民航数据治理安全基线的运行环境。所以我们团队决定基于大模型及 AI Coding 技术建设该平台。** 平台 2025 年 8 月顺利上线，稳定支撑总部-区域多级协同访问，高峰期运行平稳，全面达成建设目标。

## 3. 问题2回应 + 过渡
**大语言模型是面向自然语言理解与生成的基座智能技术；检索增强生成（RAG）将制度、资产目录、质量规则等知识切片、向量化后，由本地推理服务结合权限过滤与引用追溯完成问答。AI Coding 指在 IDE 与 CI/CD 中集成的代码补全、生成与质量门禁，经人机协同后进入版本库。二者分别对应“可治理的语义服务”和“可签核的交付链”，在不大规模改写业务核心代码的前提下，为自然语言访问数据资产与研发提效提供统一工程载体。**

**本项目数据资产目录、质量检测、血缘解析、智能问答等服务采用 Java、Python、Go 多技术栈，若各团队自接公网 API、自管提示词与向量化，将出现模型与知识版本分裂、多语言服务间访问策略与审计口径不一致、敏感片段误入向量库等隐患，问题排查与合规模型收敛成本极高。**

**为此，项目引入大模型及 AI Coding 作为平台智能化与研发工程化底座。第一，通过私有化 LLM 与 RAG、提示工程及租户/数据域级脱敏与引用水印，将智能问答与申请辅助锁在内网，统一权限过滤与可解释引用，降低越权与出境风险。第二，在内部 IDE 集成经安全评估的辅助能力，并与流水线中的静态检查、AI 辅助 Code Review 及“安全关键文件清单”联动，对高风险模块强制双人复核与签核。第三，以领域知识图谱与规则引擎约束模型输出，实行分级采纳、极端样例回归与专家抽检，以“规则裁决、模型建议”的方式抑制幻觉并落实安全合规审查。**

在本项目实施中，我们正是通过私有化知识工程与内网智能闭环、AI Coding 与流水线人机协同门禁、规则与分级采纳下的可信输出，完成了大模型及 AI Coding 技术的建设与应用，具体实践如下。

## 4. 正文部分

### 4.1 基于私有化大模型与 RAG、提示工程及脱敏访问控制，解决智能问答与共享申请中的内网闭环和合规边界问题
在「跨域数据资产检索与共享申请」业务场景中，平台存在的核心痛点为：**用户希望通过自然语言理解目录口径、质量规则与血缘影响，但上下文常关联旅客、航班、机组及维修等高敏信息；若将原始文本无差别送入通用模型，极易在召回与回答环节泄露可识别细节或产生越权建议**。若持续采用 **公网大模型直连、公共知识库或“全文粘贴式”问答** 的传统模式，极易引发 **数据出境、违规留痕、答案不可审计** 等各类风险，进而制约 **在强监管环境下推广智能服务** 的发展目标。
为破解上述难题，我们将对该场景进行 **私有化大模型与 RAG 知识工程** 升级，同时在体系架构层面重点推进 **内网部署、分域切片、脱敏后入库、检索前授权过滤与回答引用可回溯** 能力建设落地。在实际落地执行中，我们先聚焦 **知识治理与向量化质量** 核心环节，依托 **制度与目录元数据等按敏感级别脱敏或抽象化后写入向量库**、**租户/数据域维度统一认证与检索及生成双侧访问校验** 开展工作；再联动 **引用锚点与前端的可解释水印**，形成闭环治理。
经此优化设计，**自然语言查询与共享前检查可达毫秒至秒级响应，问答路径全程闭环在集团内网与策略边界内，在不暴露明细的前提下得到可审计、可解释的解读**。这一改造降低了跨团队隐性协作成本，为集团数据资产跨域共享的稳定、高效运行提供了保障。

### 4.2 通过 AI Coding 与 CI/CD、代码评审联动，解决核心后台模块研发提效与关键代码可审计问题
在「月末全量数据质量检测与血缘解析」场景下，平台面临的核心矛盾是：**跑批与解析链路逻辑重、需求变更多，借助 AI 可压缩样板与用例耗时，但脱敏、权限与血缘计算属于高敏感区；任一生成物若绕过评审进入主干，都可能在长周期大批量任务中被放大**。如果继续沿用 **“全靠手写”与“生成物直接合入”的两极做法**，就容易出现 **效率瓶颈与审核失守并存、主干不可验证** 等问题，从而影响 **后台核心计算链路的稳定演进**。
针对上述问题，我们将 **与 CI/CD、代码评审联动的 AI Coding 能力** 引入该场景，并在架构上重点落实了 **分层授权、模板化生成、人机协同签核与合入可追溯** 等机制。在具体实施过程中，我们 **在开发侧集成通过安全评估的企业插件，限定仓范围与产出类型，优先服务接口骨架、单测、Mock 与说明文档类产出**；再结合 **流水线静态扫描、AI 辅助评审与「安全关键文件」双签，合入统一打标操作者与模型版本** 推进观测与止血。
通过上述落地，**试点模块样板类效率较传统方式提升约四成，主干变更保持可审、可回滚、可归因，未出现关键路径失守类事故**，验证了 **大模型及 AI Coding 工程体系** 在复杂后台链路中的风险控制价值，为平台治理计算链路稳定演进提供支撑。

### 4.3 采用领域知识图谱与规则约束、分级采纳和专家抽检，解决模型幻觉与数据安全合规审查问题
在「敏感数据脱敏策略、动态授权与安全审计」应用场景，平台当前亟待解决的核心问题是：**大模型可加速 SQL 与策略草案产出，但若出现字段臆造、口径偏差或越权路径，将直接冲击脱敏与审批两大合规闸门**。若持续采取 **“模型直接给结论、业务凭经验采纳”的轻治理** 的运行模式，极易滋生 **建议与元数据不一致、误执行脱敏建议、审计证据链断裂** 等各类问题，直接影响平台合规防护与安全治理的核心发展效能。
围绕上述痛点，我们将对该场景实施 **“图谱约束 + 规则裁决 + 分级采纳”** 重塑，并在底层架构层面重点夯实 **主数据与术语对齐、业务规则可执行与变更留痕** 等能力建设工作。在具体落地推进过程中，我们先行布局 **资产字段、域关系与审批动作接入领域知识图谱** 环节，以 **生成侧约束解码与后校验、规则引擎对脱敏/共享/导出强裁决、模型仅输出可拒绝草案** 为核心实施路径；同步融合 **专家抽检、对抗样例与回归集及提示版本灰度回滚**，强化整体落地成效。
凭借该套优化设计方案，**试点周期内将直接影响安全策略的错误采纳压降至零事故目标，显著收敛幻觉带来的合规模糊地带；敏感域内实现“有限度、可负责”使用大模型的可复核路径**，满足业务方对安全红线与监管审计的预期，为平台在敏感域内的持续运营提供可复核依据。

## 5. 总结
在某航空集团数据资产智能管理平台建设中， **我们以内网知识工程与可审计智能问答、AI Coding 与流水线协同门禁、规则与分级采纳下的可信智能为核心，完成大模型及 AI Coding 技术架构落地。** 平台 2025 年 8 月上线后持续稳定运行，承载超 10 万项数据资产目录管理、5000 余条质量规则落地，年支撑超 10 亿次数据服务调用，核心热点查询响应稳定维持在 300 毫秒级，系统整体可用性达 99.99% 以上。依托大模型与工程化能力，平台在受控域内提供自然语言检索与治理辅助，研发侧在试点模块实现可观效率提升且关键路径签核不缺失，技术管控与业务效能得到兼顾。

项目复盘发现架构存在不足：一是平台架构扩展性不足，新增业务模块与第三方系统集成时，适配周期较长，难以快速响应业务迭代需求；二是自动化运维能力有待完善，部分配置部署与版本更新依赖人工操作，存在人为操作风险。后续将针对性优化：推行插件化、低耦合的架构演进，标准化接口与扩展协议，缩短新功能集成落地周期；同时完善全流程自动化运维体系，实现配置自动化、部署流水线化，降低人工干预成本与操作风险，持续探索提升架构灵活性与运维效率，助力该航空集团数字化高质量发展。
