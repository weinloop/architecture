# 10. 云原生可观测性设计实践——技术拆解

## 1. 项目背景分析与决策
- **核心矛盾**：跨总部—区域链路长、微服务多，需解决日志/指标/追踪分散采集、口径不一以及业务语义脱节导致的排障难问题。
- **风险分析**：若继续沿用分散采集与割裂看板，将导致难以在同一请求上关联互证，排障动辄多人天，监管举证也难以拉出同源证据链。
- **技术决策**：局部监控难以保障端到端可追溯性。决定**建设统一的日志—指标—链路（Telemetry）观测平面**，通过整合数据平面应对容量拐点与告警风暴。

## 2. 问题 2 回应与过渡
- **技术定义**：保障复杂分布式系统稳定运行与合规运营的核心技术支撑，涵盖日志聚合、多维指标监控与全链路追踪。
- **项目背景**：跨域共享与跑批涉及 20+ 微服务，缺乏统一视图易在排障时陷入僵局。
- **技术要点**：
    - **EFK** 集中日志体系 + **Kafka** 缓冲（解决日志散落与审计成本问题）。
    - **Prometheus** + **Grafana** 分层大盘（业务语义监控、容量预判）。
    - **SkyWalking** 全链路追踪 + 差异化采样（慢点下钻、成本控制）。
- **过渡总结**：通过集中日志治理、业务语义监控、全链路追踪下钻完成建设。

## 3. 正文论点一：跨域协同/检索
- **标题**：构建 EFK 集中日志体系，解决跨域协同场景中的审计举证难题
- **业务痛点**：业务穿透多环节，日志散落且规范不一，定位单次超时或监管取证需要多团队协同。
- **风险分析**：分散采集、口板对账模式效率低，敏感字段暴露策略不一存在合规风险。
- **方案落地**：
    - 标准化采集：强制输出 JSON 格式并关联 TraceID。
    - 缓冲写入：联动 Kafka 削峰填谷，写入 Elasticsearch 索引。
    - 安全检索：Kibana 构建多维视图，对敏感索引实施脱敏访问控制。
- **价值体现**：典型业务争议秒级拉出上下文，审计举证实现结构化归档，运维协同效率提升。

## 4. 正文论点二：后台计算/跑批
- **标题**：通过黄金指标与分级告警机制，解决跑批场景中的容量预判难题
- **业务场景/痛点**：传统监控缺乏业务语义，无法反映“跑批未完成”等风险，固定阈值易产生告警风暴。
- **风险分析**：业务风险感知滞后，扩容决策缺乏依据，关键任务延期。
- **方案落地**：
    - 黄金指标：Prometheus 抓取成功率、队列堆积量等业务指标。
    - 分层大盘：Grafana 构建多维视图映射 SLO。
    - 分级告警：基于 SLO 配置告警并联动 Webhook，预置扩容校验清单。
- **价值体现**：容量风险前置收敛，调度决策从“凭经验”转向“看数据”，保障了极端负载下的任务韧性。

## 5. 正文论点三：安全合规/审计
- **标题**：落实 SkyWalking 链路追踪方案，解决敏感路径场景中的慢点下钻难题
- **业务场景/痛点**：长链路调用中若无统一追踪上下文，慢点定位效率极低，且全量采样存储开销巨大。
- **风险分析**：盲目排查与隐私数据过度暴露会冲击安全运营复盘与合规要求。
- **方案落地**：
    - 拓扑分析：TraceID 注入各微服务，生成可视化调用拓扑。
    - 差异化采样：对跑批子链实施低采样或尾采样控制成本。
    - 隐私保护：对敏感路径 Span 属性按访问角色进行脱敏打码展示。
- **价值体现**：复杂场景慢点精准下压，安全运营复盘效率改善，平衡了观测成本与隐私合规。
