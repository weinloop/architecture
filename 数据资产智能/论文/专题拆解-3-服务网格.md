# 3. 云原生服务网格——技术拆解

## 1. 项目背景分析与决策
- **核心矛盾**：在多区域、多语言微服务互联场景下，既要统一流量治理与可观测，又要将身份认证与加密通信下沉为基础安全基线。
- **风险分析**：若由各团队自选 SDK、自管证书与策略，将导致治理能力重复实现、策略漂移与故障难定位，安全边界难以统一收敛。
- **技术决策**：局部修补无法形成体系化架构能力。相比侵入式方案，决定基于**云原生服务网格架构**，将流量控制、可观测性和安全通信下沉到基础设施层。

## 2. 问题 2 回应与过渡
- **技术定义**：面向微服务通信治理的基础设施技术，由控制平面与数据平面构成。依托 Sidecar 代理接管通信，无需改造代码。
- **项目背景**：多技术栈（Java/Python/Go）开发，侵入式治理方案（SDK）成本高、标准不统一。
- **技术要点**：
    - **Envoy** 自动透传链路标识（全链路标准化观测）。
    - **VirtualService** / **DestinationRule**（灰度发布、流量镜像、熔断限流）。
    - **mTLS** + 认证授权策略（零信任体系、通信加密）。
- **过渡总结**：通过全链路追踪与统一观测、灰度发布与流量镜像、mTLS 零信任安全完成建设。

## 3. 正文论点一：跨域协同/检索
- **标题**：基于全链路追踪与可观测体系，解决跨域调用链路中的故障定位难题
- **业务痛点**：共享申请需穿透十余类微服务，链路冗长、层级繁杂，任一环节抖动均导致超时卡顿。
- **风险分析**：逐台翻阅日志、人工排查模式定位滞后，责任边界模糊，故障恢复周期长。
- **方案落地**：
    - 链路串联：入口网关生成 TraceID，Sidecar 自动透传链路标识。
    - 统一采集：Istio/Envoy 自动透出运行时指标。
    - 闭环治理：联动 Kiali 拓扑、Jaeger 下钻定位慢 SQL 与根因。
- **价值体现**：P95 时延从 1.8 秒降至 650 毫秒，故障定位时长由 2 小时压缩至 10 分钟内。

## 4. 正文论点二：后台计算/跑批
- **标题**：通过灰度发布与流量镜像策略，解决核心跑批模块升级中的版本风险问题
- **业务场景/痛点**：血缘解析与质量规则升级频繁，直接影响治理结果，一次性替换易将缺陷瞬时放大。
- **风险分析**：粗放发布易出现血缘断裂、质量失真、回滚不及时，影响计算链路稳定。
- **方案落地**：
    - 版本子集：DestinationRule 定义版本子集，VirtualService 控制路由权重。
    - 影子验证：流量镜像将生产请求复制至新版本做影子比对。
    - 快速回滚：结合指标监控实现异常时路由级分钟级回滚。
- **价值体现**：镜像测试前置暴露缺陷，异常时约 3 分钟内完成回滚，跑批链路无业务中断，验证了风险控制价值。

## 5. 正文论点三：安全合规/审计
- **标题**：采用 mTLS 零信任安全网络，解决敏感数据传输中的窃听与越权访问问题
- **业务场景/痛点**：容器网络偏扁平，敏感数据在服务间频繁流转，存在窃听或身份伪造风险。
- **风险分析**：明文通信会导致证书管理混乱、身份校验缺失、内部横向渗透。
- **方案落地**：
    - 加密传输：PeerAuthentication STRICT 模式，Istiod 自动签发与轮换证书。
    - 细粒度授权：AuthorizationPolicy 定义敏感接口白名单。
    - 审计集成：审计日志接入安全运营平台，实现安全访问可追溯。
- **价值体现**：有效阻断非法 Pod 横向访问，证书运维工单降至 1 单以内，满足了安全红线与监管审计预期。
