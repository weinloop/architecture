# 4. 大数据 Kappa 架构——技术拆解

## 1. 项目背景分析与决策
- **核心矛盾**：在血缘解析与质量检测场景下，既要实现低时延实时增量更新，又要在规则变更后对历史事件按同一语义可重放。
- **风险分析**：若沿用“流批双栈”架构，将导致口径漂移、回放路径分叉与运维成本持续上升，审计与对账难以自洽。
- **技术决策**：局部修补难以保障计算语义一致性。相比 Lambda 双栈架构，决定基于 **Kappa 架构通过单一流处理管道实现流批语义统一**。

## 2. 问题 2 回应与过渡
- **技术定义**：面向流处理语义统一的基础设施技术，弃用离线批处理层，实现“一套代码、两种模式”。
- **项目背景**：数据接入、血缘解析涉及 20+ 微服务，传统架构易出现跨域结果矛盾与协同争议。
- **技术要点**：
    - **Kafka** 统一接入 + **Flink** 读模型维护（状态同步与审计）。
    - 消费位点重置 + 历史窗口并行双跑（版本化算子、口径对齐）。
    - **Parquet/OSS** 冷归档 + 统一流式回放（成本可控、合规取证）。
- **过渡总结**：通过单流式处理体系、消费位点回放、冷热分层存储完成建设。

## 3. 正文论点一：跨域协同/检索
- **标题**：构建 Kafka 与 Flink 单流式处理体系，解决跨域协同场景中的口径不一致难题
- **业务痛点**：数据接入与业务变更来源多、频度高，传统批同步易出现总部与区域状态不同步问题。
- **风险分析**：分散管道与多套落表架构引发跨域结果矛盾、审批协同争议及定位分散。
- **方案落地**：
    - 唯一事实：业务变更以结构化事件写入 Kafka，统一分区键与契约规范。
    - 模型维护：Flink 消费事件并维护读侧模型，实时驱动视图更新。
    - 位点纠偏：支持按需位点回放，确保最终一致性。
- **价值体现**：主数据与流程状态趋于一致，跨域协同投诉量显著下降，新区域接入适配周期缩短。

## 4. 正文论点二：后台计算/跑批
- **标题**：通过消费位点回放与版本化算子机制，解决核心计算场景中的流批口径漂移难题
- **业务场景/痛点**：实时增量与全量校验算子分裂，易出现“线上正常、报表对不上”的口径漂移。
- **风险分析**：Lambda 双栈维护成本高，规则定义分叉，公信力下降。
- **方案落地**：
    - 逻辑复用：封装版本化算子，Flink 复用同一套代码支撑实时与重算需求。
    - 并行双跑：切换前按历史位点启动双跑测试，对齐关键指标。
    - 无感切换：对齐后切换消费组并收口旧任务。
- **价值体现**：维护成本显著降低，月末校验与实时处理实现口径对齐，验证了 Kappa 架构在处理复杂计算口径统一的核心价值。

## 5. 正文论点三：安全合规/审计
- **标题**：采用冷热分层存储与统一流式回放方案，解决审计取证场景中的成本与一致性难题
- **业务场景/痛点**：合规要求长周期可追溯，但全量保留 Kafka 热数据会导致存储成本失控。
- **风险分析**：离线审计链路若与在线语义割裂，易造成追溯断裂与取证口径不一致。
- **方案落地**：
    - 冷热分层：Kafka 保留热数据，历史事件异步 Parquet 归档至 OSS。
    - 统一回放：Flink 授权查询服务从冷层回放数据，复用在线解析语义。
    - 证据关联：回放过程与 Trace 链路、审计事件深度绑定。
- **价值体现**：总体存储成本降低约 60%，长周期定点回放具备完整证据链，满足了高敏感域的合规运营要求。
