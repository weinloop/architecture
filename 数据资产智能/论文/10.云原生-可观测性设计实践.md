# 论云原生可观测性在数据资产智能管理平台中的设计实践

## 1. 摘要
2024年3月，我参与了某航空集团数据资产智能管理平台建设。该项目面向集团总部、5大区域中心和23个分公司，服务数据治理委员会、数据管家及运维人员等角色，提供数据资产目录、数据质量、数据安全、血缘分析、数据共享和智能问答等核心功能。在项目中，我担任系统架构师，负责平台总体架构设计和落地。 **本文围绕云原生可观测性设计展开论述，通过基于 EFK 与消息缓冲的集中日志体系实现跨微服务事后追溯与合规审计留痕，基于 Prometheus 与 Grafana 及业务黄金指标构建多维监控与分级告警保障实时健康与容量可见，结合 SkyWalking 分布式链路追踪实现跨域长链路的 Trace 串联、拓扑与慢点下钻。** 系统于2025年8月正式上线，截至目前稳定运行，平台运行平稳、核心能力达到建设预期，系统整体可用性保持在99.99%以上，有力保障集团数据治理业务平稳开展。

## 2. 项目背景
某航空集团业务涵盖航空客运、旅客服务、机务维修、地面服务等领域，总部数据中心、5大区域中心及23家分公司长期积累了大量分散在数据库、数据仓库中的多模态数据资产。依据民航智慧民航数据管理相关政策标准，集团启动数据资产智能管理平台建设，亟需构建统一的数据资产目录、质量规则库、安全策略库、血缘图谱及智能知识库，实现资产盘点、共享申请、质量治理、安全管控、智能运营一体化管理。平台需适配集团统一治理、区域分级运营、分公司轻量接入架构，支撑10万+资产目录、5000条质量规则、年10亿次以上数据服务调用，同时满足5000并发用户、页面查询2秒内响应、系统可用性99.99%以上等高并发、高一致性技术指标。

本人作为中标方系统架构师，负责平台总体技术路线设计。 **经分析，平台在跨域长链路治理场景下既要实现日志、指标、追踪在同一请求上的关联互证，又要满足审计取证与容量预警的可解释、可复盘要求。若继续沿用分散采集、各自格式与看板割裂的方式，将导致根因定位与证据链长期碎片化，故障处置与审计响应成本难以下降。** 在年十亿次量级的服务调用、多区域协同与强合规等总体约束下，若上述矛盾仅通过局部修补与点状工具堆叠来应对，而未在平台级形成成体系的架构与工程能力，将难以同时保障高并发业务体验、数据与操作可审计、以及平台级可持续演进。

**相比只在少数核心服务上堆监控、其余靠人肉 SSH，以 Logging、Metrics、Tracing 为支柱的云原生可观测性体系，将离散的机群行为收敛为可检索、可聚合、可关联的观测平面，并与告警、Runbook 联动，更适合多区域、多服务、多语言与长链路并存的集团数据环境。所以我们团队决定基于云原生可观测性设计建设该平台。** 平台2025年8月顺利上线，稳定支撑总部-区域多级协同访问，高峰期运行平稳，全面达成建设目标。

## 3. 问题2回应 + 过渡
**云原生可观测性（Observability）以日志（Logging）、指标（Metrics）与追踪（Tracing）为三大支柱：日志承载离散事件与业务语义，供事后精查与审计；指标以时间序列方式刻画饱和度、时延、错误与吞吐，支撑 SLO/告警与容量；追踪为单次请求附全局 TraceID 与 Span 树，还原跨服务路径与慢点。三者通过 TraceID、标签与服务名关联，可形成从告警到根因的闭环。在 Kubernetes 中常配合 EFK/Fluentd、Prometheus+Grafana、OpenTelemetry/SkyWalking 等栈落地，并与 Ingress、服务网格等协同传递上下文。**

**本项目数据资产目录、质量检测、血缘解析、共享申请等服务采用 Java、Python、Go 多技术栈，若各服务自定义日志格式、自埋点指标、无统一采样与无传播 TraceID，将导致观测数据异构、难以全局检索与横向对比，AIOps 与自动化处置也缺乏统一数据底座。为此，将可观测性作为平台级基础设施，而非各团队自选插件。**

**为此，项目引入云原生可观测性作为核心运维与治理底座。第一，建设以 Fluentd/Fluent Bit、Kafka 缓冲与 Elasticsearch、Kibana 为主的统一日志与检索，对敏感字段与留存周期按合规策略管理。第二，以 Prometheus 抓取节点、容器、JVM 与业务指标，在 Grafana 配置大盘与多层级告警，对接值班与 Runbook。第三，以 SkyWalking 为主实现自动埋点、拓扑与慢 Span 下钻。** 在本项目实施中，我们正是通过集中日志、多维指标与链路追踪，完成了云原生可观测性设计的建设与应用，具体实践如下。

## 4. 正文部分

### 4.1 基于 EFK 与统一日志流，解决跨域共享链路排障与安全审计可追溯问题
在「跨域数据资产检索与共享申请」业务场景中，平台存在的核心痛点为：**长链路在门户、网关、IAM、审批与目录多服务产出大量离散日志；按主机分散存放时定位一次故障与监管取证的人天成本高**。若持续采用 **半结构化或非统一留存策略** 的传统模式，极易引发 **单次请求难以关联、越权共享留痕难还原与审计不匹配** 等各类风险，进而制约 **强合规下“可说明、可举证”运维目标** 的发展目标。
为破解上述难题，我们将对该场景进行 **集中日志与索引治理** 升级，同时在体系架构层面重点推进 **JSON 字段规范、租户/业务键分区、Kafka 缓冲与写入背压** 能力建设落地。在实际落地执行中，我们先聚焦 **Fluentd/Bit 采集与 TraceID+业务流水强约束** 核心环节，依托 **Elasticsearch/Kibana 汇聚与共享申请多维视图**、**脱敏/拒绝独立索引** 开展工作；再联动 **秒级多维检索回放投诉场景**，形成闭环治理。
经此优化设计，**典型超时纠纷可在秒级拉出跨服务上下文，压缩跨团队推诿**。这一改造降低了跨团队隐性协作成本，为集团数据资产跨域共享的稳定、高效运行提供了保障。

### 4.2 通过 Prometheus、Grafana 与业务黄金指标，解决月末跑批与健康状态不透明及告警质量问题
在「月末全量数据质量检测与血缘解析」场景下，平台面临的核心矛盾是：**关账窗内队列、算力与中间件易饱和，但仅盯 CPU 或单接口难判业务语义上的“不可用”**。如果继续沿用 **固定阈值与无序电话告警**，就容易出现 **误报洪峰或漏掉真实容量风险** 等问题，从而影响 **治理关账时点与前台体验并存的处置时效及后台计算链路的稳定演进**。
针对上述问题，我们将 **分层指标与黄金业务指标接入** 引入该场景，并在架构上重点落实了 **从基础设施至“共享成功率/批处理完成率/堆积深度”的贯通大盘** 等机制。在具体实施过程中，我们 **以 Prometheus 抓取并 Grafana 多维下钻告警经 Webhook 入值班手册**；再结合 **高峰期提前根据堆积与饱和完成限流/扩容/调度** 推进观测与止血。
通过上述落地，**多轮高峰可将风险前置于用户报障之前，批在线对比可见**，验证了 **云原生可观测性** 在复杂后台链路中的风险控制价值，为平台治理计算链路稳定演进提供支撑。

### 4.3 采用 SkyWalking 全链路追踪与拓扑分析，解决敏感长链路与鉴权交织下的慢点定位问题
在「敏感数据脱敏策略、动态授权与安全审计」应用场景，平台当前亟待解决的核心问题是：**多跳重试与缓存叠加掩盖真实慢点；无 Trace 时只能横向盲调**。若持续采取 **不统一 TraceID 或无拓扑视角** 的运行模式，极易滋生 **并行调参与同因复发难复盘** 等各类问题，直接影响平台合规防护与安全治理的核心发展效能。
围绕上述痛点，我们将对该场景实施 **SkyWalking 自动埋点与场景化采样** 重塑，并在底层架构层面重点夯实 **敏感 Span 脱敏展示与安全授权** 等能力建设工作。在具体落地推进过程中，我们先行布局 **网关与 SDK 上下文透传与拓扑环依赖巡检** 环节，以 **共享主链/跑批子链差异化采样** 为核心实施路径；同步融合 **与后续 AIOps 数据关联的慢 SQL 下钻路径**，强化整体落地成效。
凭借该套优化设计方案，**十余跳复杂检索可把慢 SQL 与子路径有据下压**，满足业务方对安全红线与监管审计的预期，为平台在敏感域内的持续运营提供可复核依据。

## 5. 总结
在某航空集团数据资产智能管理平台建设中， **我们以集中日志、多维指标与链路追踪为核心，完成云原生可观测性设计落地。** 平台 2025 年 8 月上线后持续稳定运行，承载超 10 万项数据资产目录管理、5000 余条质量规则落地，年支撑超 10 亿次数据服务调用。平台在统一观测平面下，跨域与批处理类问题的发现与恢复效率相较建设前可量化提升，系统整体可用性达 99.99% 以上。可观测性为容量、发布与事故复盘提供了可共享的事实语言，降低了长链路排障的协作成本。

项目复盘发现架构存在不足：一是平台架构扩展性不足，新增业务模块与第三方系统集成时，适配周期较长，难以快速响应业务迭代需求；二是自动化运维能力有待完善，部分配置部署与版本更新依赖人工操作，存在人为操作风险。后续将针对性优化：推行插件化、低耦合的架构演进，标准化接口与扩展协议，缩短新功能集成落地周期；同时完善全流程自动化运维体系，实现配置自动化、部署流水线化，降低人工干预成本与操作风险，持续探索提升架构灵活性与运维效率，助力该航空集团数字化高质量发展。
