# 论云原生可观测性技术在数据资产智能管理平台中的应用
2024年3月，我参与了某航空集团数据资产智能管理平台的建设。该平台面向集团总部、5大区域中心和23个分公司，服务数据治理委员会、数据管家及运维人员等角色，提供数据资产目录、数据质量、数据安全、血缘分析、数据共享和智能问答等全流程管理功能。在该项目中，我担任系统架构师，负责平台的总体架构设计和落地实施。本文围绕**云原生可观测性设计**展开论述，**通过基于 Fluentd 或 Fluent Bit、Kafka 缓冲与 Elasticsearch、Kibana 的集中日志体系**实现跨微服务事后追溯与合规审计留痕，**基于 Prometheus、Grafana 与业务黄金指标**构建多维监控与分级告警保障实时健康与容量可见，**结合 SkyWalking 分布式链路追踪**实现跨域长链路的 **Trace** 串联、服务拓扑与慢 **Span** 及慢 **SQL** 下钻；平台于2025年4月上线，稳定运行超 1 年，年支撑超10亿次数据服务调用，系统可用性保持在99.99%以上，获得客户高度认可。

---

## 项目背景
随着智慧民航建设战略的深入推进，航空集团数据治理相关政策标准逐步落地，集团总部数据中心、5大区域中心及23家分公司长期积累的多模态数据资产亟需规范化管理，同时对数据资产的共享、质量、安全管控提出了标准化要求。在此背景下，某航空集团于2024年3月启动数据资产智能管理平台的建设，旨在构建覆盖集团全层级的数字化管理平台，实现资产盘点、共享申请、质量治理、安全管控、智能运营一体化管理，支撑10万+资产目录、5000条质量规则、年10亿次以上数据服务调用，同时满足5000并发用户、页面查询2秒内响应、系统可用性99.99%以上的高要求。

我司中标后，我作为系统架构师负责平台的总体架构设计和落地实施，因为平台在跨域长链路治理场景下既要实现日志、指标、追踪在同一请求上的**关联互证**，又要满足审计取证与容量预警的可解释、可复盘要求，若继续沿用分散采集、格式不一与看板割裂的方式，将导致根因定位与证据链长期碎片化，故障处置与审计响应成本难以下降，均对 **Logging**、**Metrics**、**Tracing** **三支柱**统一的**观测平面**提出了极高要求，所以我们团队决定**基于云原生可观测性设计建设该平台**。平台于2025年4月上线，成功应对了跨域数据共享高峰、月末核心跑批与监管取证协同等业务场景，为集团数据治理业务平稳开展提供了有力支撑，得到了用户和客户的一致好评。

## 问题2回应
在该平台的建设落地中，我们重点建设了**云原生可观测性技术体系**，主要包括**基于 EFK 与消息缓冲的统一日志与检索治理**、**基于 Prometheus 与 Grafana 的多维指标与黄金业务指标及分级告警**、**基于 SkyWalking 的分布式链路追踪与拓扑及差异化采样** 3 个部分。基于 **Fluentd** 或 **Fluent Bit** 将容器与主机日志标准化为 **JSON**，经 **Kafka** 缓冲削峰后写入 **Elasticsearch**，在 **Kibana** 侧按租户与业务键分区检索，强制业务流水与 **TraceID** 同域输出，对敏感字段按策略脱敏或独立索引并配置留存周期；基于 **Prometheus** 抓取节点、容器、**JVM** 与自定义业务指标，在 **Grafana** 贯通从基础设施到共享成功率、批处理完成率与队列堆积深度的大盘，经 **Webhook** 对接值班平台与 **Runbook**；基于 **SkyWalking** 在各语言服务侧自动或轻量化**埋点**，由网关与服务间透传上下文，形成**调用拓扑**与慢 Span 下钻，并对共享主链与跑批子链配置**差异化采样**与安全域展示策略。

得益于云原生可观测性技术体系的建设，平台上线运行后，成功应对了长链路排障、关账窗口容量前置预警与敏感路径慢点定位等业务与运维场景，下面我将重点论述该技术体系在该平台的建设实施过程。

## 正文
**基于 EFK 与统一日志流及索引治理**，解决跨域数据资产检索与共享申请长链路中**日志分散**、单次请求难以关联与安全审计**举证成本高**的问题，提升了事后追溯与合规留痕效率。在数据资产智能管理平台的建设和运维中，门户、网关、统一认证、审批与目录等环节涉及数据资产目录、质量检测、血缘解析等**20+微服务**（涵盖 **Java**、**Python**、**Go** 多技术栈），长链路会产生大量离散日志；若按主机分散存放或缺乏统一字段规范，定位一次超时纠纷与监管取证往往消耗多人天，且难以把越权或异常共享还原为可说明的证据链。

于是我们引入**集中日志**与**缓冲写入**能力：首先以 Fluentd 或 Fluent Bit 统一采集并规范化为结构化字段，经 Kafka 缓冲平滑尖峰再入 Elasticsearch，避免写入**背压**拖垮业务侧；其次在日志输出中强约束业务流水号与 **TraceID** 共现，在 Kibana 配置共享申请、审批与回写多维检索视图；同时对高敏内容**脱敏**或写入独立索引，按合规要求设定留存与访问角色。得益于该体系的建设，典型超时与协同争议可在**秒级**拉出跨服务上下文，压缩跨团队推诿，有效保障了集团数据资产跨域共享在运维与审计两侧的可举证性。

经过调研，我们将 **Prometheus**、**Grafana** 与业务**黄金指标**接入月末全量数据质量检测与血缘解析及关账窗口场景，解决仅观察 CPU 或单接口难以表达业务语义上「不可用」、**固定阈值告警**易出现误报洪峰或漏报容量风险的问题。关账窗内队列、算力与中间件易饱和，若缺少从基础设施贯通到批处理完成率、堆积深度与共享成功率的**统一大盘**，运维难以在用户报障前完成限流、扩容或调度决策。

我们决定在工程上落实**分层指标模型**：首先以 Prometheus 抓取机器、容器与运行时指标，并暴露业务自定义的黄金指标；其次在 Grafana 构建自底向上的多级大盘与告警规则，对堆积、错误率与饱和度设置动态或 **SLO** 相关阈值，告警经 Webhook 进入值班手册与处置**剧本**；同时在高峰前根据历史**基线**预置扩容与调度观测项。得益于该机制的落地，多轮高峰将部分风险前置于用户报障之前，批处理与在线健康对比一目了然，验证了**云原生可观测性**在复杂后台链路中的风险控制价值。

针对以上痛点，我们落实 **SkyWalking** **全链路追踪**与**拓扑分析**及**场景化采样**，解决敏感数据脱敏、动态授权与安全审计场景中多跳重试与缓存叠加掩盖真实慢点、无统一 TraceID 时只能横向盲调的问题。缺乏拓扑与跨服务耗时分解时，并行参与方难以对同因复发复盘，慢 SQL 与下游抖动难以下钻定位。

我们首先在入口网关与各类语言 **SDK** 侧统一注入与传播 **Trace** 上下文，由 SkyWalking 汇聚为全局 Trace 与**服务拓扑**；其次对共享主链与跑批子链设置差异化**采样率**，在成本与能见度之间平衡；同时对敏感 Span 在展示层脱敏并限定查询角色，将慢 SQL 与异常 Span 关联路径固化为 **AIOps** 与后续自动化的数据输入。得益于该方案的建设，**十余跳**复杂检索可把慢 SQL 与子路径有据下压，满足业务方对安全红线与监管审计的预期，为平台在敏感域内的持续运营提供了可复核依据。

## 3. 总结
在本平台的建设落地中，通过构建完整的**云原生可观测性技术体系**，我们成功将数据资产智能管理平台打造成了**日志可检索**、**指标可预警**、**链路可下钻**的统一观测平台，平台于2025年4月上线以来，成功应对了跨域数据共享高峰、月末核心跑批与事故复盘等业务场景，保障了年10亿次以上数据服务调用的稳定运行，跨域与批处理类问题的发现与恢复效率相较建设前可量化提升，有效支撑了集团数据资产的全生命周期管理，受到了用户和客户的一致认可。

在项目的复盘会上，我们也发现了不足之处，比如平台架构扩展性不足，新增业务模块与第三方系统集成时适配周期较长，难以快速响应业务迭代需求；同时自动化运维能力有待完善，部分配置部署与版本更新依赖人工操作，存在人为操作风险。后续，我们拟推行插件化、低耦合的架构演进，标准化接口与扩展协议，缩短新功能集成落地周期；同时完善全流程自动化运维体系，实现配置自动化、部署流水线化，降低人工干预成本与操作风险，持续提升架构灵活性与运维效率，助力航空集团数字化高质量发展。
