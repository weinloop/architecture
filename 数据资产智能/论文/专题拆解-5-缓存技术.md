# 5. 云原生缓存技术应用——技术拆解

## 1. 项目背景分析与决策
- **核心矛盾**：在热点读与跨域访问场景下，既要降低主库压力，又要将关键数据的一致性窗口控制在严苛、可审计的范围内。
- **风险分析**：若沿用粗放 TTL、各服务自建缓存且缺乏统一治理，将导致脏读、越权展示、缓存击穿与雪崩等问题反复出现。
- **技术决策**：点状工具堆叠难以保障性能与合规可见性。相比传统单级缓存，决定基于**云原生多级缓存架构**，提供贴近用户的访问能力与灵活的失效控制。

## 2. 问题 2 回应与过渡
- **技术定义**：提升微服务系统读性能与保护后端存储的核心手段，涵盖多级缓存协同、数据一致性保障与高可用支撑。
- **项目背景**：目录、标签等高重叠读数据承载极高并发压力，缺乏统一体系易拖垮主库。
- **技术要点**：
    - **Caffeine** 进程内副本 + **Redis** 共享热层 + 异步预热（防击穿、主库保护）。
    - **Cache-Aside** + 延迟双删 + **Outbox** 广播失效（收敛脏读窗口、策略对齐）。
    - **Redis** 集群高可用 + **HPA** 弹性扩缩 + 客户端熔断退避（洪峰韧性）。
- **过渡总结**：通过多级缓存治理、一致性策略控制、弹性容错保障完成建设。

## 3. 正文论点一：跨域协同/检索
- **标题**：实施多级缓存与热点预热机制，解决跨域检索场景中的主库访问瓶颈难题
- **业务痛点**：用户高频拉取目录、标签等高重叠键，穿透主库导致连接压力大、I/O 负载及往返时延高。
- **风险分析**：无缓存模式易引发主库抖动、P99 时延抬高，难以实现秒级体验。
- **方案落地**：
    - 统一规范：制定统一键前缀规范。
    - 多级缓存：Caffeine 进程内副本 + Redis 共享热层。
    - 治理优化：异步预热高频键，合并回源防止击穿，大 Key 摘要化处理。
- **价值体现**：核心读路径平均时延由 800 毫秒降至百毫秒级，主库重复读请求显著下降。

## 4. 正文论点二：后台计算/跑批
- **标题**：通过 Redis 集群弹性与熔断降级策略，解决跑批场景中的缓存雪崩难题
- **业务场景/痛点**：跑批引发大量元数据伴生读洪峰，若缓存拓扑脆弱，极易出现连接耗尽、节点故障。
- **风险分析**：单点故障或击穿连锁回源会拖垮库层，影响治理报告准时产出。
- **方案落地**：
    - 弹性扩缩：依托 Kubernetes StatefulSet 与 HPA 动态提升副本数量与规格。
    - 熔断保护：客户端配置退避策略与单飞机制。
    - 路径隔离：明确非关键读降级路径，将其与鉴权等强一致读物理隔离。
- **价值体现**：多轮洪峰演练中故障切换秒级完成，未触发全链路雪崩，保障了前台服务容量空间。

## 5. 正文论点三：安全合规/审计
- **标题**：落实一致性治理与广播失效方案，解决敏感数据变更场景中的脏读难题
- **业务场景/痛点**：脱敏与授权策略变更需各区域近似同步可见，长窗口脏读会导致严重的越权展示风险。
- **风险分析**：弱一致性模式会导致授权失效滞后、审计记录与缓存状态不匹配。
- **方案落地**：
    - 失效策略：Cache-Aside 结合数据库落库后的延迟双删。
    - 主动驱逐：Outbox 模式通过消息总线广播失效事件。
    - 分层防护：对鉴权键设短 TTL，配合布隆过滤器拦截异常试探。
- **价值体现**：策略变更在分钟级实现全局对齐，多轮攻防测试未现越权情形，满足了监管复核要求。
