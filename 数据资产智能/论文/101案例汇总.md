## 一、结构化分析

### 1.1 数据流图 DFD（★★★★★）

#### 1.1.1 基本符号（必背）

两种常见记号：**Gane & Sarson** / **Yourdon & Coad**，名称不同、含义一致。

| 符号名称 | 表示内容 | 图形表示 |
|----------|----------|----------|
| **数据流** | 有名字和方向的数据 | 标有名字的**箭头** |
| **加工（数据处理）** | 对数据流的变换 | **圆圈** |
| **数据存储** | 可访问的存储信息 | **直线段**（双线或开口矩形） |
| **外部实体** | 系统外的数据来源/去向 | 标有名字的**方框** |

**符号示意（Gane & Sarson / Yourdon 两种画法等价）**：
```
数据流：   --------→   （带名字的箭头）
加工：      (  P  )    （圆圈，内写加工名）
数据存储：  ═══════   或  ┌─────┐  （直线段/开口矩形）
外部实体：  ┌─────┐    （方框，如 管理员、外部系统）
```
**重要规则**：数据存储要被外部实体使用，**必须经过加工**。

#### 1.1.2 平衡原则（★★★★★ 必考）

**（1）父图与子图平衡（分层细化）**

- **个数一致**：父子两层数据流个数一致  
- **方向一致**：数据流方向一致  

**（2）子图内部平衡——三种错误**

| 类型 | 含义 | 记忆 |
|------|------|------|
| **黑洞** | 加工只有输入没有输出 | 只进不出 |
| **奇迹** | 加工只有输出没有输入 | 只出不进 |
| **灰洞** | 输入不足以产生应有输出（挂羊头卖狗肉） | 该出 A 却出 B |

**数据存储**：通常既有**读**又有**写**；某张子图可只有读或只有写。

**简答模板**：分层数据流图平衡 = ① 父子图数据流个数、方向一致；② 每张图内加工输入/输出平衡，避免黑洞、奇迹、灰洞。

### 1.2 状态转换图 / 状态机图（★★★☆☆）

- 即 UML 中的**状态图**，描述**状态 + 事件 → 状态变化**。
- **主要概念**：状态、初态、终态；事件、转移、动作；并发状态机。

#### 1.2.1 基本符号

| 元素 | 表示 |
|------|------|
| **初态** | 黑圆点 |
| **终态** | 黑圆点 + 外圈圆 |
| **中间状态** | 圆角矩形 |
| **状态转换** | 带箭头的线（可标触发事件） |

**符号示意**：
```
初态 ●  →  [ 状态A ] --事件--> [ 状态B ]  →  ◎ 终态
```

## 二、面向对象分析 · UML（★★★★★）

### 2.1 UML 常考图（记这 6 类）

| 名称 | 描述 |
|------|------|
| **类图** | 类、接口、协作及关系 |
| **用例图** | 用例、参与者及关系 |
| **顺序图** | 对象间消息，**强调时间顺序** |
| **通信图/协作图** | 对象间消息，**强调结构协作**（UML1 叫协作图） |

其他：对象图、包图、组合结构图、构件图、部署图、时间图、交互概览图、外廓图——了解即可。

### 2.2 用例图（★★★★★）

#### 2.2.1 基本步骤（顺序背）

1. **识别参与者**  
2. **合并需求获得用例**  
3. **细化用例描述**  
4. **调整用例模型**  

#### 2.2.2 参与者类型

| 类型 | 说明 | 例子 |
|------|------|------|
| **人** | 使用系统的人 | 患者、医生、管理员 |
| **其他系统** | 与本系统交互的系统 | 企业 OA 系统 |
| **硬件设备** | 需交互的设备 | IC 卡读写器 |
| **时钟** | 定时触发 | 定时交卷 |

**注意**：参与者一定在**系统之外**，不是系统一部分。

#### 2.2.3 用例关系（必背：箭头指向 + 构造型）

| 关系 | 含义 | 构造型 | 箭头指向 |
|------|------|--------|----------|
| **包含** | 提取公共行为，原始用例→基本用例，提取出的→抽象用例 | `<<include>>` | 指向**抽象用例** |
| **扩展** | 基本用例 + 多种扩展场景，扩展用例可选 | `<<extend>>` | 指向**基本用例** |
| **泛化** | 父子用例，子继承父 | — | 指向**父用例** |

**关系示意**：
- **包含**：基本用例 ----`<<include>>`----→ 抽象用例
- **扩展**：扩展用例 ----`<<extend>>`----→ 基本用例
- **泛化**：子用例 ----泛化----→ 父用例

### 2.3 类图 · 六大关系（★★★★★）

**强度顺序（口诀）**：**饭（泛）食（实现）组聚关羽（依）**  
即：**泛化 = 实现 > 组合 > 聚合 > 关联 > 依赖**

| 关系 | 定义 | 记忆 |
|------|------|------|
| **关联** | 明确、稳定的结构联系 | 工人—任务 |
| **依赖** | B 变化可能引起 A 变化，A 依赖 B | 司机依赖汽车 |
| **泛化** | 一般与特殊，父类与子类 | 狗、猫 泛化 动物 |
| **聚合** | 整体与部分，部分可属多整体，**好聚好散** | 学生—班级 |
| **组合** | 整体与部分，部分只属一整体，**生命周期同步** | 汽车—发动机 |
| **实现** | 类实现接口 | 音频播放器 实现 播放器接口 |

**聚合 vs 组合**：聚合=部分能单独存在；组合=部分随整体生灭。

### 2.4 顺序图（★★★★★）

- **用途**：对象间消息的**时间顺序**，用例分析/设计。
- **元素**：生命线、消息（同步/异步）、端点、门、**组合片段**。

#### 2.4.1 组合片段类型（常考）

- **alt**：条件分支  
- **loop**：循环  
- **par**：并行  
- **opt**：可选  

### 2.5 通信图 / 协作图（★★★★★）

- **元素**：对象、**通信链接**、消息。
- **与顺序图区别**：顺序图强调**时序**；协作图强调**对象结构与协作**。语义等价，可互相转换；关注时序用顺序图，关注结构用协作图。

### 2.6 活动图（★★★★★）

- **用途**：控制流/数据流，顺序、分支、**并行**；与流程图最大区别是**分叉/合并**和**信号（发送/接收）**。

#### 2.6.1 主要元素

| 元素 | 含义 |
|------|------|
| 初始/终止节点 | 开始 / 结束 |
| 动作状态 | 原子步骤 |
| 决策节点 | 分支条件 |
| 合并节点 | 多分支汇合 |
| 分叉/合并 | 并行控制流 |
| 泳道（分区） | 按角色/部门划分 |
| 对象节点/对象流 | 对象传递 |
| 信号节点 | 发送/接收事件（异步） |

**活动图与流程图区别**：活动图有**分叉/合并**（并行）和**信号发送/接收**（异步）。
```
     [动作A] → ◇决策 → 分支1 → ...
                    → 分支2 → ╋合并 → [动作B]
     ╋分叉 → [并行1] ╋
          → [并行2] ╋ 合并
```

## 一、类识别（★★★☆☆）

### 1.1 三类与识别方法（词性法）

设计得好的类至少有一种职责：**属性封装数据，方法封装行为**。类按职责分为三类：

| 类型 | 识别方法（词性） | 作用 | 示例 |
|------|------------------|------|------|
| **实体类** | **通常是名词** | 保存需永久存储的信息，表达属性和关系 | 学员类、课程类、账户、图书、借阅记录、罚金 |
| **控制类** | **动宾短语**（动词+名词 / 名词+动词）或**纯动词** | 控制用例工作，对用例特有的控制行为建模，具有协调性 | 身份验证器、借阅、归还、预订、续借 |
| **边界类** | **通常是名词**（多为固定几类） | 系统接口与外部交互，将系统与外部环境隔离 | 窗口、通信协议、打印机接口、报表、读者证等“连接外部与控制类”的东东 |

- **实体**：名词语义存数据（名词 → 实体）
- **控制**：动宾/动词管流程（动宾、动词 → 控制）
- **边界**：窗口协议报表面，连外连控是边界

**边界类特殊记忆**：窗口、通信协议、打印机接口、报表 + 其他能“连接外部和控制类”的都可视为边界类；其余不易判断时从“是否沟通外部与内部控制”来判。

### 1.2 三类关系示意

```
  [外部参与者]  ←——→  边界类  ←——→  控制类  ←——→  实体类
     (人/系统)         (界面/协议)    (用例控制)    (持久化数据)
```

- **边界类**：与外部交互的“界面层”。
- **控制类**：协调用例流程，一般一个用例一个（可复用）。
- **实体类**：存数据、表关系，其他类一般不向控制类发消息，而由控制类发出消息。

---

### 1.3 边界类 · 实体类 · 控制类（概念要点）

- **边界类**：描述外部参与者与系统的交互；对系统外部环境与内部运作之间的交互建模，包括转换事件、记录系统表示方式（如接口）的变更。
- **实体类**：数据管理与业务逻辑层面；主要职责是存储和管理系统内部信息，可有与实体对象密切相关的行为。
- **控制类**：描述一个用例的事件流控制行为，控制用例中的事件顺序；控制其他类工作，通常每用例一个控制类，也可多用例共用；其他类一般不向控制类发消息，由控制类发出消息。

## 二、面向对象设计原则（★★★★★）

### 2.1 七大原则一览

| 原则 | 英文/缩写 | 核心含义 |
|------|-----------|----------|
| **单一职责原则** | SRP (S) | 类应该**只有一个**设计目的 |
| **开闭原则** | OCP (O) | 对**扩展开放**，对**修改关闭**；通过抽象层实现，是其他原则的基础 |
| **里氏替换原则** | LSP (L) | 使用基类对象的地方可替换为子类对象，反之不一定；变化类设计为抽象类或接口 |
| **接口隔离原则** | ISP (I) | 接口提供客户端**需要的行为**，大接口拆成小接口；或把接口视为“角色” |
| **依赖倒置原则** | DIP (D) | **抽象不依赖细节**，针对接口编程；实现开闭的主要机制 |
| **组合/聚合复用原则** | — | 通过**组合或聚合**复用已有对象，比继承更灵活、耦合度低 |
| **最少知识原则** | 迪米特法则 | 软件实体应**尽量少**与其他实体相互作用 |

### 2.2 原则关系示意

```
开闭原则(O) ←—— 依赖倒置(D) 是实现 O 的主要机制
     ↑
单一职责(S)、里氏替换(L)、接口隔离(I) 等共同支撑可扩展、可维护
组合/聚合复用：优先组合/聚合，少用继承，降低耦合
最少知识：少交互，降低耦合
```

---

### 2.3 开闭原则（要点）

**答**：面向对象设计的开闭原则（Open-Closed Principle，OCP）指软件实体（如类、模块、函数等）应对**扩展开放**、对**修改关闭**。即：在不修改原有源代码的前提下，通过扩展来增加新行为；能够在不改动原有代码的情况下改变模块行为。实践中常通过**抽象层**（抽象类、接口）实现，依赖倒置原则是实现开闭原则的主要机制。

### 2.4 各原则一句话记忆

| 原则 | 一句话 |
|------|--------|
| S 单一职责 | 一个类只干一类事 |
| O 开闭 | 能扩展不能改（加抽象层） |
| L 里氏替换 | 基类能用的地方子类也能用，变化放抽象/接口 |
| I 接口隔离 | 接口只给客户需要的，大接口拆小 |
| D 依赖倒置 | 面向接口编程，抽象不依赖细节 |
| 组合/聚合复用 | 用组合/聚合复用，少用继承 |
| 最少知识 | 少跟别的实体打交道（迪米特） |

## 一、层次架构简介与反模式

### 1.1 层次架构要点

- **定义**：系统组成**层次结构**，每一层为**上层服务**，并作为**下层客户**；内部层接口可只对**相邻层**可见。
- **特性**：**关注分离**——本层组件只负责本层逻辑，角色与职责清晰，便于开发、测试、管理与维护。
- **常见三层**：**表现层** → **中间层（业务层）** → **数据访问层**。

**层次关系示意**：
```
  表现层（上层）  —— 使用 ——→  中间层  —— 使用 ——→  数据访问层（下层）
       ↑ 服务对象                    ↑ 服务对象
```

### 1.2 污水池反模式（Sinkhole Anti-Pattern）

- **现象**：请求**只是穿过**各层，没有在每层做实质处理。例如：界面层把“获得数据”交给业务层，业务层再交给持久层，持久层简单 SQL 查库，数据原路返回，**没有任何二次处理**。
- **后果**：层次隔离失效、结构不清晰、业务规则变更时易遗漏或不一致、可维护性降低。
- **解决**：**层次开放**——部分数据处理可**直达核心层次**，不必层层穿越。

**示意**：
```
  污水池反模式：  [界面] → 透传 → [业务] → 透传 → [持久] → SQL → 原路返回（无加工）
  改进：          部分请求可跨层直达核心层，在关键层做真实逻辑
```

## 二、表现层架构：MVC / MVP / MVVM（★★★★★）

### 2.1 三者对比表

| 维度 | MVC | MVP | MVVM |
|------|-----|-----|------|
| **核心组件** | Model、View、**Controller** | Model、View、**Presenter**（呈现器） | Model、View、**ViewModel**（视图模型） |
| **Model** | 数据存储与业务逻辑 | 同左 | 同左 |
| **View** | 用户界面展示 | 用户界面交互接口 | 绑定数据的 UI |
| **第三元** | 控制器：处理输入，协调 M 与 V | 呈现器：处理逻辑，**控制视图更新** | 视图模型：**数据转换与双向绑定** |
| **数据流** | **单向**：V→C→M→V（经 C 更新） | **双向**：V↔Presenter↔M（Presenter 主动更新 V） | **双向**：V↔ViewModel↔M（**自动同步**，基于绑定） |
| **View 与其它** | 可调控制器，可能访问 Model | 通过**接口**与 Presenter 通信，**不直接访问** Model | 通过**数据绑定**与 ViewModel 交互，**无直接逻辑调用** |

- **MVC**：单线 C 协调，V 可能碰 M。
- **MVP**：P 控 V，V 不碰 M，通过接口。
- **MVVM**：VM 双向绑定，V 不调逻辑。

**数据流示意**：
```
  MVC:   View ——→ Controller ——→ Model ——→ (Controller 更新) View
  MVP:   View ←→ Presenter ←→ Model   （Presenter 主动更新 View）
  MVVM:  View ←→ ViewModel ←→ Model   （绑定自动同步）
```

## 三、中间层架构（★★★☆☆）

### 3.1 三层 + DomainModel—Service—Control

- **整体**：表示层、**业务层**、数据库层。业务层内部按 **DomainModel — Service — Control** 解耦。
- **作用**：职责分离、服务与服务控制隔离，提高**可重用性**和**灵活性**。

| 组件 | 职责 |
|------|------|
| **DomainModel** | 封装**业务实体及核心逻辑**（如用户、订单的属性和行为），保证业务规则一致 |
| **Service** | **协调跨领域**业务操作（事务、权限等），可调用多个 DomainModel 完成复杂功能 |
| **Control** | **服务控制器**：根据业务状态和参数决定 Service 的**执行顺序与依赖**，实现服务间松耦合 |

**示意**：
```
  表示层
     ↓
  业务层：  Control → Service → DomainModel（多实体）
     ↓
  数据库层
```

## 四、数据访问层（★★★★★）

### 4.1 数据访问模式对比

| 模式 | 核心思想 | 特点 |
|------|----------|------|
| **在线访问模式** | 应用与数据库**持续连接**，实时执行数据操作 | 实时性强、直接操作库；依赖网络与连接稳定性 |
| **DAO（数据访问对象）** | 将**数据访问逻辑与业务逻辑分离**，用接口封装数据库操作 | 解耦数据层与业务层，可换库，代码复用 |
| **DTO（数据传输对象）** | 跨进程/网络传输数据的**轻量容器**，**无业务逻辑** | 减少传输次数、可序列化、与数据源结构解耦 |
| **离线数据模式** | 数据从数据源取走后按**预定义结构存本地**，与数据源**断开** | 离线处理，支持 XML/JSON，不依赖持续连接 |
| **ORM（对象/关系映射）** | 在关系库与对象间建**映射**，像操作对象一样操作库 | 面向对象、自动 SQL、减少数据访问层开发 |

**DAO**：把“和数据库打交道”封装在 DAO 类里，业务层只调接口（如 `BookDAO`：addBook、deleteBook、getBookById），实现类里用 JDBC 等做具体 SQL。  
**DTO**：像“快递包裹”，只装数据、不装逻辑；例如后端查库后只把前端需要的字段放进 ProductDTO 传给前端。

### 4.2 在线访问 vs ORM（简答常考）

| 方式 | 优点 | 缺点 |
|------|------|------|
| **程序在线访问** | 性能较优、能处理复杂查询 | 要求程序员懂 SQL，修改维护相对困难 |
| **ORM** | 降低学习与开发成本、无需手写 SQL、减少代码量与 SQL 质量风险 | 复杂查询处理困难、性能一般不如直接 SQL |

**选题 ORM 的常见理由**：团队缺乏数据库经验、避免 SQL 质量与维护成本、应用相对简单、性能非瓶颈。

### 4.3 增加数据访问层的原因（约 100 字）

实现**数据与业务逻辑分离**，提高可维护性、扩展性和灵活性。数据访问层负责数据库操作、封装访问细节，使**业务逻辑层与具体存储技术解耦**，简化开发；便于对数据访问统一管理、优化，提升整体性能与可靠性。

### 4.4 ORM 落地：Hibernate（★★★☆☆）

- **作用**：ORM 框架，完成 **Java 对象 ↔ 数据库表** 的映射。
- Hibernate 结构中常见**持久化对象/实体**、**Session / SessionFactory**、**持久化层/数据访问层**等。

## 五、物联网层次架构（★★★☆☆）

| 层次 | 介绍 | 典型设备/技术 |
|------|------|----------------|
| **感知层** | 物联网基础，**识别物体、采集信息**，解决物理世界数据获取；传感器采集后经**短距离传输** | 传感器、RFID 标签与读写器、二维码与识读器、摄像头、GPS、M2M 终端、传感器网关等 |
| **网络层** | **连接感知层与应用层**，传递与处理信息，将感知数据安全可靠传到应用层，按需处理 | 通信网、互联网、网络管理中心；接入网（光纤/无线等）、传输网（电信网、广电网）；3G/4G、IPv6；路由器、交换机、网关等 |
| **应用层** | 最顶层，**行业智能化应用**，与行业技术深度融合，对感知数据**计算、处理、知识挖掘**，实现控制、管理与决策 | 智能操控、安防、远程医疗等应用；终端产品；物联网中间件、云计算 |

```
  感知层（传感器、RFID、二维码、GPS 等）
```

## 一、云原生架构原则（★★★★★）

### 1.1 七条原则一览

| 原则 | 描述 |
|------|------|
| **服务化原则** | 规模大时拆分服务，提升复用与流量治理能力 |
| **弹性原则** | 系统**自动伸缩**，**适配业务需求**（资源动态适配） |
| **可观测性原则** | 通过日志、链路跟踪等掌握系统状态 |
| **韧性原则** | **抵御软硬件异常**，保障系统稳定（故障场景下的保护） |
| **全流程自动化原则** | 自动化交付，提升效率、降低风险 |
| **零信任原则** | 基于身份访问控制，替代传统网络边界安全 |
| **持续演进原则** | 架构适应技术与业务变化，渐进式升级 |

**弹性 vs 韧性（必区分）**：
- **弹性**：侧重**资源动态适配业务需求**（扩缩容、负载适配）。
- **韧性**：侧重**故障场景下的系统保护**（容错、降级、恢复）。

## 二、云原生架构模式（★★★★★）

### 2.1 七种模式核心要点

| 模式 | 核心定义 | 典型场景/工具 |
|------|----------|----------------|
| **服务化架构**（SOA/微服务/小服务） | 按业务域或功能**拆分服务**，通过 API/协议交互；常结合 DDD、容器化与 CI/CD。优点：灵活扩展、独立部署、低耦合；缺点：分布式治理成本高 | 电商订单、支付、库存拆微服务 |
| **Mesh 化**（Service Mesh） | 将 **RPC、负载均衡、限流熔断、安全认证** 等从业务进程剥离，由 **Sidecar/Proxy（Mesh）** 统一处理；业务只写逻辑。优点：统一治理、解耦；缺点：额外性能开销 | Istio、Linkerd |
| **Serverless** | **无需管理服务器**，写业务逻辑即可；平台负责部署、运行、弹性伸缩与关闭；事件/流量触发，按调用量或时长计费。优点：运维成本低、按需伸缩；缺点：冷启动、厂商锁定 | 图片处理、日志分析、IoT 采集 |
| **存储计算分离** | **计算与存储解耦**：计算无状态，数据在分布式/云存储；有状态恢复靠 **WAL + Snapshot**。优点：弹性、成本优化；缺点：分离带来延迟 | Snowflake、TiDB、云数仓 |
| **分布式事务** | 跨库/跨服务下保持一致性。常见：**XA**（两阶段、强一致、性能差）、**消息最终一致**（MQ 异步、性能好）、**TCC**（Try/Confirm/Cancel、侵入强）、**SAGA**（本地事务+补偿）、**SEATA AT**（代理自动回滚） | — |
| **可观测架构** | **Logging**（多级日志）、**Tracing**（分布式链路）、**Metrics**（QPS、响应时间、并发等）；结合 SLO/SLA 做监控与告警 | Prometheus、Grafana、ELK、Jaeger |
| **事件驱动（EDA）** | 以**事件**为核心；事件带 Schema，可发布/订阅；有 QoS 与失败补偿。优点：解耦、异步、可扩展 | Kafka、Pulsar、EventBridge；订单完成触发发货、库变更触发缓存更新 |

**示意**：
```
  服务化：  [订单服务] [支付服务] [库存服务]  ←→ API/协议
  Mesh：    业务进程 + Sidecar(通信/治理/可观测)
  Serverless： 事件/流量 → 云函数 → 按量计费
  存储计算分离：  [计算节点-无状态]  ←→  [分布式存储]  （WAL+Snapshot 恢复）
  可观测：  Logging + Tracing + Metrics → 监控/告警
  EDA：     生产者 → 事件总线 → 消费者（异步、解耦）
```

## 三、云原生反模式 / 不好的实践（★★★☆☆）

| 反模式 | 问题描述 | 解决方案 |
|--------|----------|----------|
| **庞大的单体应用** | 依赖隔离不足、代码耦合、模块接口缺乏治理、发布难以协调、单模块故障影响整体 | **服务化适度拆分**：梳理聚合根，明确服务边界与接口，使组织与架构匹配 |
| **单体“硬拆”为微服务** | 过度拆分导致与组织能力不匹配、服务间数据依赖、性能下降 | **合理评估粒度**：考虑组织能力与业务需求，避免过度拆分；优化服务间数据交互与性能 |
| **缺乏自动化的微服务** | 规模大时人工开发/测试/运维导致交付慢、风险高、成本增 | **建立自动化**：自动化测试、发布、环境管理等，以应对复杂度 |

## 面向服务架构（SOA）

---

## 一、什么是 SOA（概念简答）

**从应用角度**：SOA 是一种**应用框架**，将日常业务应用划分为**单独的业务功能与流程**（即服务），使用户可构建、部署和整合这些服务，且**不依赖具体应用与运行平台**，从而提升业务灵活性、资产重用、管理简便与开发部署速度。

**从软件原理角度**：SOA 是一个**组件模型**，将应用的不同功能单元（**服务**）通过**定义良好的接口与契约**联系起来。接口以**中立方式**定义，**独立于**硬件平台、操作系统和编程语言，使不同系统中的服务能以**统一、通用**的方式交互。

**默写要点**：组件模型 + 服务 + 良好接口与契约 + 中立/独立于平台与语言 + 统一通用交互。

## 二、SOA / 微服务 / 单体对比（★★★★★）

### 2.1 微服务 vs SOA（五维度）

| 特性 | 微服务 | SOA |
|------|--------|-----|
| **服务粒度** | **极细**，单服务单业务功能（如商品展示、订单、支付各一服务） | **较大**，一服务可含多业务功能（如 ERP 中采购+库存一服务） |
| **通信方式** | 以 **HTTP/RESTful API** 为主，语言平台无关、易集成 | **多样化**，除 HTTP 外常用 **SOAP** 等，规范与安全性要求高 |
| **数据管理** | **每服务独立数据库**，数据源唯一 | 多服务**共享库**或经 **ESB** 交换整合 |
| **架构风格** | **去中心化**，无中心服务总线，轻量通信 | 可有**中心化 ESB**，负责注册、发现、路由、消息转换等 |
| **复杂性** | 服务独立、去中心，单服务简单；但**服务多**带来治理、监控、运维挑战 | 服务间耦合与 ESB 使设计、开发、维护相对复杂，对团队要求高 |

### 2.2 SOA vs 单体架构

| 特性 | SOA | 单体架构 |
|------|-----|----------|
| **服务粒度** | 业务功能**分解为多服务**，粒度较细，可复用 | **整体紧耦合**，功能难单独拆分与复用 |
| **通信方式** | **多协议**（SOAP、HTTP 等），标准接口，**分布式部署** | **方法调用**，同一应用内部，无复杂外部协议 |
| **数据管理** | 多服务**共享库**或经 **ESB** 交换 | **单一数据库**，所有模块共一数据源 |
| **架构风格** | **分布式**，服务可独立部署；可有 **ESB** 协调 | **集中式**，功能集中一应用、同一环境 |
| **故障影响** | 单服务故障**隔离较好**，仅影响依赖该服务的功能 | 任一部分故障可能导致**整体不可用** |

## 三、服务注册模式（★★★★★）

### 3.1 目的与三角色

**引入目的**：解决**服务提供者**与**服务请求者**之间的**寻址与发现**。通过注册中心统一发布与存储服务信息，调用方**不直接依赖具体地址**，而是**动态查询**获取调用方式，实现解耦，支持动态发现、负载均衡与版本演进。

| 角色 | 操作内容 |
|------|----------|
| **服务提供者** | 使用 **UDDI** 将 Web 服务**发布**到注册中心，用 **WSDL** 描述服务 |
| **服务注册中心** | **接收并存储**提供者通过 UDDI 发布的服务信息，供请求者**查找** |
| **服务请求者** | 借助 **UDDI** 在注册中心**查找**服务，依据 **WSDL** 了解调用方式，通过 **SOAP** 与提供者建立连接并使用服务 |

**流程示意**：
```
  提供者 --publish/UDDI--> 注册中心 <--bind/UDDI-- 请求者
       |                        |                        |
    WSDL 描述              存储/发现                 WSDL 了解
       |                        |                        |
       +-------- SOAP 调用 -----+------------------------+
```

### 3.2 UDDI、WSDL、SOAP（必背）

| 协议 | 概念 | 主要作用 |
|------|------|----------|
| **UDDI**（通用描述、发现和集成） | 基于 **XML** 的、用于发现与集成 Web 服务的协议/规范 | **发布与搜索** Web 服务信息；提供者注册，使用者查找与连接 |
| **WSDL**（Web 服务描述语言） | 描述 Web 服务**接口**的 **XML** 格式语言 | 定义功能、输入输出参数、绑定协议、访问端点等，供调用方了解**如何调用** |
| **SOAP**（简单对象访问协议） | 基于 **XML** 的**消息传输**协议，用于 Web 服务间通信 | 定义消息格式、编码规则、RPC 约定；可与 HTTP、SMTP 等结合 |

### 3.3 REST 规范要点（★★★★★）

- **全称**：Representational State Transfer（表述性状态转移）。
- **RESTful**：符合 REST 约束与原则的架构；未创造新技术，而是更好利用现有 Web 准则。

| 主题 | 要点 |
|------|------|
| **资源与 URI** | 资源是任何可被引用的事物（实体或抽象）；**URI** 为资源唯一标识（地址/名称） |
| **统一资源接口** | 使用标准 **HTTP 方法**（GET、PUT、POST、DELETE 等）访问资源 |
| **资源的表述** | 客户端获得的是资源的**表述**，形式多样（如 HTML、XML、JSON） |
| **资源的链接** | 用**超媒体**在表述中加入链接，引导客户端，增强连通性 |
| **状态的转移** | **服务端不保存客户端状态**；每次请求自带所需信息，实现状态转移 |

## 四、ESB 模式（★★★★★）

### 4.1 ESB 概念

**企业服务总线（ESB）**：由**消息中间件**发展而来，采用**“总线”模式**管理与简化应用集成拓扑，基于**开放标准**支持应用在**消息、事件和服务**级别的互联互通。

**ESB 提供**：
- **位置透明**的消息路由与寻址、**服务注册与命名**管理；
- **多种消息传递范型**（请求/响应、发布/订阅等）；
- **多种传输协议**、**多种数据格式及相互转换**；
- **日志与监控**功能。

**与服务注册**：ESB 与服务注册模式可**混用**（如架构图中 ESB 与 UDDI 注册中心并存）。

### 4.2 SOA 与 ESB 简答（约 200 字）

**SOA**：见第二节“什么是 SOA”的软件原理定义（组件模型、服务、良好接口与契约、中立、统一通用交互）。

**ESB 在 SOA 中的作用与特点**：提供位置透明性的消息路由和寻址服务、服务注册和命名的管理功能；支持多种消息传递范型（如请求/响应、发布/订阅）、多种传输协议、多种数据格式及其相互转换；还提供日志和监控功能。

## 五、SOA 设计原则（★★★★★）

| 原则 | 简要概括 |
|------|----------|
| **无状态** | 避免请求者依赖提供者状态 |
| **单一实例** | 防止功能冗余 |
| **明确定义的接口** | 由 **WSDL** 定义，WS-Policy 描述规约，XML Schema 定义消息格式 |
| **自包含和模块化** | 封装组件，功能实体可独立部署与管理 |
| **粗粒度** | 服务数量适度，靠消息交互 |
| **松耦合** | 使用者仅见接口，服务位置、实现不可见 |
| **重用能力** | 服务可重用 |
| **互操作性、兼容和策略声明** | 不同系统、应用、设备或组件能相互通信 |

## 六、典型架构

### 6.1 ESB 层次结构（四层）

| 层次 | 作用 |
|------|------|
| **应用工具层** | **流程编排**与 **ESB 管理**；流程编排负责业务流程设计与协调 |
| **流程控制层** | 通过**节点**控制流程走向，对**服务调用顺序、条件分支**等管理，保证流程按预期执行 |
| **数据转换与适配器层** | **适配器**连接不同系统/服务，实现通信；**数据转换**在不同格式间转换，保证数据可被正确识别与处理 |
| **服务总线层** | 基于 **JMS**（Java 消息服务），提供**可靠、异步、安全**的消息传递，为 ESB **基础通信层** |

**示意**：
```
  应用工具层（流程编排、ESB 管理）
       ↓
  流程控制层（节点、调用顺序、分支）
       ↓
  数据转换与适配器层（适配器、数据转换）
       ↓
  服务总线层（JMS、消息传递）
```

### 6.2 微服务六模式（★★★★★）

| 模式 | 定义 | 示意 |
|------|------|------|
| **聚合器微服务** | 将**多个微服务的 API 组合成统一接口**，整合响应后返回客户端，简化交互 | 客户端 → 聚合器 → 服务A、B、C → 聚合器 → 客户端 |
| **代理微服务** | 在**客户端与微服务间**增加中间层，处理**路由、负载均衡、安全验证**等 | 客户端 → 代理 → 微服务 |
| **链式微服务** | 微服务**按顺序串联**，每服务处理部分功能并传递结果，形成完整流程 | 客户端 → S1 → S2 → S3 → 响应 |
| **分支微服务** | 按**业务子系统**拆成独立微服务，通过 **API 或消息队列**通信，模块化开发 | 多个并列微服务，API/MQ 互联 |
| **数据共享微服务** | 提供**统一数据访问接口或复制机制**，支持多应用/服务共享数据 | 多服务 → 统一数据层/复制数据 |
| **异步消息微服务** | 通过**消息队列**异步通信，**解耦**服务依赖，提升吞吐与响应能力 | 服务 → MQ → 服务 |

**微服务优势**：解耦、独立部署、技术选型灵活、容错性好、易扩展。

## 一、软件架构风格（★★★★★）

### 1.1 风格总览（概念辨析表）

| 一级风格 | 二级风格 | 核心概念 | 典型场景 |
|----------|----------|----------|----------|
| **数据流** | 批处理 | 分步执行，前一步完成**整体传递**数据，**无并行** | 传统数据处理、批处理脚本 |
| **数据流** | 管道-过滤器 | 前一步**输出即后一步输入**，可**并行** | Unix Shell、编译器各阶段 |
| **调用/返回** | 主程序/子程序 | **单线程**控制，分层调用 | 早期结构化程序 |
| **调用/返回** | 面向对象 | 数据与操作**封装**，**消息**传递 | Java/C# 应用 |
| **调用/返回** | 层次型 | 每层为上层服务、作下层客户，**相邻层**可见 | 分层系统 |
| **调用/返回** | 客户端/服务器 | 表示层/功能层/数据层，**网络**交互 | Web、数据库应用 |
| **以数据为中心** | 仓库 | **中央数据存储**，独立构件操作 | IDE、企业数据管理 |
| **以数据为中心** | 黑板 | **分级解空间**，多**知识源**协同 | 语音识别、模式识别 |
| **虚拟机** | 解释器 | 自定义语言**解析执行**，可动态扩展 | 游戏脚本引擎 |
| **虚拟机** | 规则系统 | 规则集+规则解释器+选择器+**工作内存** | 专家系统 |
| **独立构件** | 进程通信 | 构件为**独立进程**，连接件为**消息传递** | 分布式、微服务 |
| **独立构件** | 事件驱动 | **隐式调用**，事件**广播**触发响应 | UI 系统 |
| **特殊** | C2 | 构件经连接件**分层**，**顶部到底部**通信 | 嵌入式、分布式 |
| **特殊** | 过程控制 | **反馈循环**调节物理状态 | 空调、巡航 |

**批处理 vs 管道-过滤器**：批处理需“落盘”（Tape/文件），整体传递；管道-过滤器**数据连续流动**、可并行、更高效。

**进程通信 vs 事件驱动**：进程通信是**显式**消息传递（点对点、RPC 等）；事件驱动是**隐式调用**——构件**注册**到事件，事件触发时系统**自动调用**已注册过程。

### 1.2 若干风格示意

**批处理**：`[步骤1] → Tape → [步骤2] → Tape → [步骤3]`（顺序、整体传递）

**管道-过滤器**：`[Filter1] → Pipe → [Filter2] → Pipe → [Filter3]`（流式、可分支/并行）

**仓库**：多个**独立构件** ↔ **中央数据库**（仅通过中心交换，构件间无直接连接）

**黑板**：**知识源 1～N** ↔ **黑板**（共享解空间）；实线=读写黑板，虚线=访问外部数据/计算

**解释器**：**存储器**（程序+当前状态）↔ **执行引擎**（取指、解释、更新状态）→ 输出

**层次型**：上层 → 调用 → 下层；仅**相邻层**可见，接口一致可替换实现。

**二层/三层 C/S、B/S**：二层=客户端(表示+业务)+数据库服务器（胖客户端）；三层=表示+**应用服务器**(业务)+数据（瘦客户端）；B/S=浏览器+Web 服务器+数据库，零安装、易推广，动态交互与高并发 OLTP 弱于 C/S。

---

### 1.3 解释器架构 vs 面向对象架构（选型比较）

**结论**：选**解释器**。  
- **可修改性**：解释器规则为独立语法，由解释器解析，**修改更容易**；面向对象需改具体类。  
- **灵活性**：解释器可按用户**灵活解释执行规则**，优于面向对象。  
- **性能**：面向对象一般略优（解释器多一层）。综合前两项优势更大，故选解释器。

## 二、面向架构评估的九大质量属性（★★★★★）

### 2.1 九大属性及对策

| 属性 | 描述 | 对策（简） |
|------|------|------------|
| **性能** | 响应能力、处理时间、单位时间处理量 | 优先级队列、增加资源、减少开销、并发、资源调度 |
| **可靠性** | 在错误/异常使用下维持功能；**平均失效时间** | 冗余、错误检测与恢复、异常处理、备份与恢复、负载均衡 |
| **可用性** | **正常运行时间比例**；故障间隔/恢复速度 | **心跳**（检测在线）、冗余/双机热备/集群、**快速恢复**、**故障转移**、内置监控与预警 |
| **安全性** | 合法服务+阻止非授权；机密性、完整性、不可否认、可控 | 认证、授权、加密、防火墙、入侵检测、审计与日志 |
| **可修改性** | 快速、高性价比变更；含可维护、可扩展、可移植等 | 模块化、接口标准化、抽象与封装、配置管理、版本控制 |
| **互操作性** | 与其他系统/环境交互；跨语言、跨系统 | 标准协议与接口（REST、SOAP）、数据格式转换、中间件、服务注册与发现 |
| **可移植性** | 适应不同运行环境；跨 OS/硬件/云 | 抽象层/虚拟机、减少平台依赖、兼容性测试 |
| **可测试性** | 易于发现缺陷、验证功能与性能 | 分层与模块化、测试接口、可执行语句覆盖率等度量 |
| **易用性** | 好学、好找、好操作、少犯错 | 统一交互与视觉、信息架构、即时反馈、快捷操作、多语言与无障碍、引导与帮助 |

**可用性 vs 可靠性**：若场景强调短时**切换备用 / 故障转移 / 重定向仍可访问**，多归为**可用性**；若强调**平均失效间隔**等度量，则更贴近**可靠性**语境。实务中两项常并列出现。

### 2.2 效用树与质量属性归类

- **效用树结构**：树根 → **质量属性** → 属性分类 → **质量属性场景**（叶子）。  
- **ATAM 默认四类**：**性能、可用性、安全性、可修改性**。  
- 判别线索：**响应时间/吞吐**→性能；**故障恢复/重定向/备用**→可用性；**审计/日志/攻击检测/访问控制**→安全性；**规则可配置/界面可调/扩展人天**→可修改性。  

## 三、质量属性场景描述（六要素）（★★★★★）

### 3.1 公式与语义

**公式**：质量属性场景 = **刺激源** + **刺激** + **环境** + **制品** + **响应** + **响应度量**

| 要素 | 理论 | 白话 | 例 |
|------|------|------|-----|
| **刺激源** | 人、计算机系统或其他刺激器 | 谁触发 | 大量用户、开发人员 |
| **刺激** | 刺激到达时需考虑的条件 | 触发条件 | 同时点击秒杀、请求到达 |
| **环境** | 刺激发生时系统可能处于的状态 | 何时/何状态 | 正常访问、降级模式 |
| **制品** | 受刺激的模块或系统 | 测谁 | 商品展示模块、订单模块 |
| **响应** | 激励到达后采取的行动 | 系统怎么做 | 处理请求、返回结果、切换备用 |
| **响应度量** | 可度量的指标，用于验证 | 如何衡量 | 响应时间 200ms、吞吐 500QPS、故障修复时间 |

**故事串**：**谁**（刺激源）在**什么条件**下（刺激）、**系统处于何状态**（环境）、对**谁**（制品）**做了什么**（响应），并用**什么指标**衡量（响应度量）。

### 3.2 各质量属性六要素简表

| 质量属性 | 刺激源 | 刺激 | 环境 | 制品 | 响应 | 响应度量 |
|----------|--------|------|------|------|------|----------|
| **可用性** | 系统内/外部 | 疏忽、错误、崩溃、时间 | 正常→降级 | 处理器、信道、存储器、进程 | 检测、记录、通知、隔离故障、恢复 | 故障修复时间 |
| **可修改性** | 最终用户、开发、管理员 | 改变功能、扩容等 | 设计时/编译时/构建时/运行时 | 界面、平台、交互系统 | 修改且不影响其他功能 | 成本、努力、对其它功能影响 |
| **性能** | 用户请求、他系统 | 事件到达 | 正常→超载 | 系统 | 处理刺激、改变服务级别 | 等待时间、期限、吞吐量、抖动、缺失率 |
| **可测试性** | 开发、测试、客户 | 分析/架构/设计/集成交付时触发 | 设计时/开发时/编译时/部署时 | 设计、代码段、应用 | 提供状态访问、计算值、测试环境 | 语句覆盖率、缺陷触发概率、测试时间等 |
| **易用性** | 最终用户 | 想学习系统、有效使用 | 运行时或配置时 | 系统 | 友好界面与适应性设计，错误影响最小 | 用户满意度、成功操作占比 |
| **安全性** | 内/外部个人或系统 | 试图访问/修改数据、访问服务 | 网络环境 | 系统服务、数据 | **认证、授权、审计、加密、监控** | 检测、溯源、恢复 |

**可用性六要素示例**（模板）：刺激源：系统内部、系统外部。刺激：疏忽、错误、崩溃、时间。环境：正常操作、降级模式。制品：系统处理器、通信信道、持久存储器、进程。系统响应：检测事件并记录、通知相关方、禁止故障源、一段时间不可用后恢复、在正常或降级模式下运行。响应度量：故障修复时间。

## 四、系统架构评估（★★★★★）

### 4.1 敏感点、权衡点、风险点、非风险点

| 概念 | 定义 | 句式/例 |
|------|------|----------|
| **敏感点** | **一个或多个构件（及关系）**的特性 | “**…将影响…**”；如：对查询处理时间的要求**将影响**传输协议与处理设计 |
| **权衡点** | **影响多个质量属性**的特性，是多个敏感点的综合 | “**…能提升 A 但会降低 B**”；如：加密级别高→安全性↑、性能↓ |
| **风险承担者** | 影响架构决策的利益相关者 | 财务部门、销售部门 |
| **场景** | 风险承担者角度的交互描述；常用刺激、环境、响应 | 客户在公共 Wi-Fi 下登录网银，10 秒内完成认证并显示账户 |
| **风险点** | **可能**导致架构出问题或不满足质量要求的特性/决策 | “**如果…，可能…，影响…**” |
| **非风险点** | 对质量属性无负面影响或可接受的特性/决策 | 选用成熟 MQ（如 RabbitMQ），经市场验证，当前架构下无明显风险 |

### 4.2 三种架构评估方法

| 方法 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **调查问卷/检查表** | 用问卷或检查表，凭经验与知识评估 | 利用经验与知识 | 依赖主观推断 |
| **基于场景的评估** | 分析架构对场景的支持程度；**SAAM、ATAM** 属此类 | 系统化评估对场景的支持 | 需详细定义场景，可能耗时 |
| **基于度量** | 建立质量属性与度量的映射，从架构文档推导 | 客观、可量化 | 需精确度量与映射，专业要求高 |

---

### 4.3 SAAM（★★★☆☆）

- **思想**：用**实际使用场景**对架构“挑刺”，检查是否扛得住。  
- **主要分析**：**可修改性**；协调风险承担者达成共识。  
- **输入**：问题描述、需求声明、架构描述。  
- **过程**：场景开发 → 架构描述 → 单场景评估 → 场景交互评估 → 总体评估。  
- SAAM 是 **ATAM** 的基础。

### 4.4 ATAM 与效用树（★★★★★）

- **ATAM**：在 SAAM 基础上，针对**性能、安全性、可修改性、可用性**，在开发前**评价与折中**；用**效用树**等工具做**优先级排序**再评估。  
- **四阶段**：场景与需求收集 → 架构视图与场景实现 → 属性模型构造与分析 → 权衡（迭代）。  
- **效用树**：树根 → **质量属性**（上述四类）→ 属性分类 → **场景**（叶子）。  
- **修剪与优先级**：保留重要场景（通常≤50），按**重要度 H/M/L** 与**实现难易度 H/M/L** 标优先级对，如 (H,L)=重要且易实现。

### 4.5 ATAM 实践四阶段

| 阶段 | 主要活动 |
|------|----------|
| **演示** | 介绍 ATAM 流程、业务目标、待评估架构 |
| **调查和分析** | 确定架构设计方法；**生成质量属性效用树并排序**；分析架构与质量属性匹配度 |
| **测试** | 头脑风暴候选场景；筛选高优先级场景并验证架构支持能力 |
| **报告** | 汇总**风险、非风险、敏感点、权衡点**；提交效用树、场景列表及改进建议 |

## 安全架构设计理论与实践

---

## 一、安全架构三性（★★★☆☆）

| 特性 | 定义 | 典型例子 |
|------|------|----------|
| **可用性** | 确保系统资源**在需要时可正常访问**，防止故障或攻击导致服务中断 | 数据备份与恢复、负载均衡、双机热备 |
| **完整性** | 保护数据**不被未经授权修改或破坏**，确保数据一致性和准确性 | 文件哈希校验（MD5/SHA）、**数字签名**、数据库事务回滚 |
| **机密性** | **防止敏感信息在未授权情况下被泄露或获取** | 数据加密存储、HTTPS、访问控制列表（ACL） |

**辨析**：**完整性** = 防篡改、保一致；**机密性** = 防泄露、保密。二者不同，答题时分别写清概念即可。

## 二、数据加密技术（★★★★★）

### 2.1 对称加密

- **原理**：**同一密钥**既加密又解密；发送方用密钥把明文变密文，接收方用**同一密钥**还原。
- **优点**：加解密**速度快、效率高**，适合**大量数据**。
- **典型算法**：AES、DES、3DES。
- **数学基础**：替代/置换等运算。

---

### 2.2 非对称加密

- **原理**：**一对密钥**——**公钥**（可公开，用于加密）、**私钥**（用户自持，用于解密）。发送方用**接收方公钥**加密，接收方用**自己私钥**解密。
- **优点**：**密钥管理方便**，无需在通信双方间传递私钥；安全性基于**数学难题**（如 RSA 大整数分解）。
- **典型算法**：RSA、ECC、DH。
- **用途**：数字签名、密钥交换、身份验证等。

---

### 2.3 非对称的两种应用场景

| 场景 | 谁加密 / 谁解密 | 传递内容 | 典型用途 |
|------|-----------------|----------|----------|
| **数字签名（信源验证）** | **私钥加密，公钥解密** | 明文 + 签名（对摘要签名） | 验证来源与完整性，防抵赖 |
| **数据加密** | **公钥加密，私钥解密** | 密文 | 只有拥有私钥的接收方能解密；也可用于**交换对称密钥** |

**记忆**：签名 = 私钥签、公钥验；加密 = 公钥加密、私钥解密。

**密钥交换**：非对称加密慢，适合小数据；实际大量数据用对称加密。双方可用**公钥加密“对称密钥”**传给对方，对方用私钥解密得到密钥，后续用对称密钥通信。

### 2.4 对称 vs 非对称对比

| 对比项 | 对称加密 | 非对称加密 |
|--------|----------|------------|
| **密钥** | **单一密钥** | **密钥对**（公钥 + 私钥） |
| **速度** | **快**，适合大数据 | **慢**，适合小数据 |
| **密钥管理** | 需**安全共享**密钥 | 公钥公开，私钥保密 |
| **主要用途** | 数据加密 | 密钥交换、数字签名、身份验证 |
| **数学基础** | 替代/置换 | 数论难题（分解、离散对数等） |
| **典型算法** | AES、DES、3DES | RSA、ECC、DH |

## 三、认证技术（★★★★★）

### 3.1 Hash（杂凑）算法

- **特性**：**易压缩**（任意长→固定长）、**单向**（不能由哈希反推原文）、**高灵敏**（输入微变输出大变）、**抗碰撞**（不同输入极难同哈希）。
- **用途**：数据**完整性校验**，是数字签名的重要组成部分。
- **常用算法**：SHA-1（已不推荐）、SHA-256、SHA-384、国产 **SM3**。

---

### 3.2 数字签名原理

**创建签名**：  
1. 对文件做 **Hash** 得到**消息摘要**。  
2. 用**签名者私钥**对摘要加密，得到**数字签名**。  
3. 可加时间戳。  
4. 发送：**原文 + 数字签名**（注意：只对摘要签名，不对整个文件，省资源、提效率）。

**验证签名**：  
1. 接收方对**原文**再做 Hash，得**本地摘要**。  
2. 用**签名者公钥**解密数字签名，得**发送时摘要**。  
3. **两摘要一致** → 文件完整且签名有效；不一致 → 被篡改或签名无效。

**示意**：
```
  发送方：  原文 → Hash → 摘要 → 私钥加密 → 签名  ； 发送 [原文 + 签名]
  接收方：  原文 → Hash → 摘要A ； 签名 → 公钥解密 → 摘要B ； 比较 摘要A 与 摘要B
```

---

### 3.3 数字证书

- **问题**：公钥如何保证属于真正的发送方？若公钥被篡改，验证会失效。
- **解决**：**数字证书**——由 **CA（证书颁发机构）** 签发，对**公钥及其持有者信息**做认证；CA 用**自己的私钥**对“公钥+信息”签名，形成证书。
- **验证**：接收方用 **CA 的公钥**解密证书，验证证书有效性，即可信任其中的公钥。
- **标准**：X.509；CA 维护**证书吊销列表（CRL）**；不同 CA 间通过**证书链**建立信任。
- **典型流程**：客户端请求 → 服务器返回**数字证书** → 客户端用 CA 公钥验证证书 → 用服务器公钥加密后续数据（或交换对称密钥）。

### 3.4 身份认证方式简表

| 方式 | 特点 |
|------|------|
| **口令认证** | 用户名/密码，简单但不安全，易泄露、被猜、被截获 |
| **动态口令** | 按时间或次数变化，一次一密，较安全；可能同步问题、输入不便 |
| **生物特征** | 身体/行为特征，分身体特征、行为特征及不同层级 |

## 二、批处理 vs 流处理 vs 混合（★★★★★）

| 架构类型 | 概念 | 优势 | 局限 | 应用场景 |
|----------|------|------|------|----------|
| **批处理** | 大规模数据**批量**任务，**离线**执行；含采集、存储、处理、输出；按**时间间隔或触发**处理 | 可处理大规模数据，**吞吐高、稳定**，充分利用计算资源 | **实时性差**，无法满足实时分析 | 对实时性要求不高的大规模数据处理 |
| **流处理** | 数据**连续、无界**流式处理，**新数据到达即处理**；由数据流、运算符组成，流处理引擎管理 | **实时**处理，快速响应，适应数据快速变化 | 无法处理历史数据、可能漏数据，需考虑并发与一致性 | 实时分析、监控、推荐等 |
| **混合架构** | 同时使用批处理与流处理，数据在两种模式间按需转换 | 兼具批与流优点，可处理历史+实时，快速响应 | 需权衡架构与数据处理，管理维护成本高 | 既有历史又有实时需求的场景 |

## 三、Lambda 架构（★★★★★）

### 3.1 三层结构

| 层次 | 概念 | 支撑技术 |
|------|------|----------|
| **批处理层** | 负责**所有批处理**，存储**全量数据集**并计算**批处理视图** | Hive、Spark-SQL、MapReduce 等 |
| **加速层/速度层** | 用**流式计算**实时处理**当前数据**，弥补批处理高延迟 | Storm、Spark Streaming、Flink 等 |
| **服务层** | 以批处理层与加速层结果为数据源，提供**低延时查询与即席查询** | 关系型库，或 Kylin、Presto、Impala、Druid 等 OLAP |

**示意**：
```
  数据源 → [批处理层] → BatchView ──┐
       ↘ [加速层] → RealtimeView ──┼→ [服务层] → 查询
```

### 3.2 批处理层与加速层分开设计的原因

| 原因 | 说明 |
|------|------|
| **容错性** | 加速层数据持续写入批处理层；批处理层**重算全量**后可覆盖加速层数据，**修正加速层错误** |
| **复杂性隔离** | 加速层用**增量算法**处理实时数据，复杂度高于批处理层，需隔离 |
| **横向扩容** | 数据量或负载增大时，分层便于**增加机器**维持性能 |

**二者对比**：加速层处理**最近增量数据流**，不断更新 RealtimeView；批处理层处理**全量数据集**，得到 BatchView。

### 3.3 Lambda 实现中各技术角色

| 层次 | 技术 | 功能 |
|------|------|------|
| **批处理层** | Hadoop | 大规模**批量**数据处理；从 Kafka 读数据进行批处理 |
| **加速层** | Spark / Storm | Spark：微批准实时、吞吐与复杂计算（ML、图）；Storm：**纯流**、亚秒级、低延迟与可靠性 |
| **服务层** | HBase / Cassandra | 实时访问与更新；HBase 基于 HDFS 列存；Cassandra 宽列、高吞吐 |
| **服务层** | Hive | 在 Hadoop 上建**可查询视图**，HiveQL→MapReduce，便于离线查询分析 |

**Spark vs Storm**：Spark 偏吞吐与复杂逻辑，Storm 偏**极致实时**。

### 3.4 Lambda 优缺点

| 类型 | 描述 |
|------|------|
| **优点** | **容错性好**：可修复算法或重算视图；**查询灵活**：批/实时存不同存储，多接口（SQL、REST 等）；**易伸缩**：分布式、弹性调度；**易扩展**：分层解耦、插件化 |
| **缺点** | 技术栈复杂，**两套系统**开发维护成本高、代码重复；全场景覆盖编码开销大；重部署与迁移成本高 |

## 四、Kappa 架构（★★★★★）

### 4.1 核心思想与组件

- **思想**：**去掉批处理层**，数据通道用**消息队列**替代；以**流处理为主**，数据存数据湖，需离线或重算时从消息队列**重放**数据即可。
- **数据流**：数据视为**持续事件流** → 进入 **Kafka/Pulsar**（高吞吐、持久化、可重放）→ **Flink/Kafka Streams/Spark Structured Streaming** 流处理（窗口、聚合、状态管理）→ 结果写入 **Redis/ClickHouse/ES** 或数据湖（Hudi、Iceberg）。
- **可回放**：算法或程序变更时，**重置消费位点、重放历史流**即可重新出结果，无需单独批处理程序。

| 核心组件 | 内容 |
|----------|------|
| **消息传输层** | Kafka 等，高吞吐、低延迟、**持久化与重放** |
| **流处理层** | Flink、Storm 等，实时计算，低延迟结果输出至服务层 |
| **数据存储** | HBase、Cassandra 等，支持快速查询 |

### 4.2 Kappa 优缺点

| 类型 | 描述 |
|------|------|
| **优点** | **实时与离线代码统一**，维护方便，**统一数据口径**，避免两套逻辑不一致 |
| **缺点** | 消息中间件**缓存与回溯**有性能瓶颈；多实时流**关联时顺序**问题易导致数据丢失；**抛弃离线模块**，失去离线计算更稳定的特点 |

---

### 4.3 Lambda vs Kappa（★★★★★ 必背）

| 对比内容 | Lambda | Kappa |
|----------|--------|-------|
| **复杂度与维护成本** | **两套系统**（批+流），复杂度高，开发维护成本高 | **一套系统**（流），复杂度低，成本低 |
| **计算开销** | 批处理和实时**同时常开**，计算开销大 | **必要时**全量计算（重放），计算开销相对小 |
| **实时性** | 满足实时性 | 满足实时性 |
| **历史数据处理能力** | **批式全量**，吞吐大，**历史数据处理能力强** | **流式全量**，吞吐相对低，历史数据处理能力**较弱** |

## 五、IOTA 架构（★★★☆☆）

- **特点**：**边缘计算** + **统一数据模型**；数据流连续、一致。
- **思路**：设定**标准数据模型**（如“主-谓-宾”“对象-事件”），在**数据产生、计算、查询**各环节分散计算，**统一模型贯穿**，支持即时查询。

## 一、DDD 三层边界总览

```
  大的边界（业务版图）   →  领域、子域（核心/支撑/通用）
  小的边界（团队协作）   →  统一语言、限界上下文、上下文映射（防腐层等）
  代码边界               →  实体、值对象、聚合/聚合根、领域服务、领域事件、仓储与工厂、应用层编排
```

**出处**：Eric Evans《领域驱动设计：软件核心复杂性应对之道》——在软件核心处应对复杂性。

## 二、大的边界：领域与子域（★★★☆☆）

### 2.1 领域与子域

| 概念 | 含义 |
|------|------|
| **领域** | 系统要解决的**业务问题空间**，如“电商交易” |
| **子域** | 领域中**可相对独立**的问题区域，如“商品管理”“交易下单”“支付结算”“库存管理”“营销优惠”“售后服务” |

**目的**：先划子域，把复杂度拆小，把精力集中在**最能体现业务价值**的部分。

---

### 2.2 子域分类（★★★★★）

| 类型 | 英文 | 含义 | 策略 | 电商示例 |
|------|------|------|------|----------|
| **核心子域** | Core | **企业差异化竞争力**所在 | 做深模型、持续演进 | 优惠计算引擎、库存与履约协同、售后规则 |
| **支撑子域** | Supporting | 业务需要但非核心竞争力 | 介于核心与通用之间 | 内容运营、活动页搭建 |
| **通用子域** | Generic | 通用能力、可替代 | **尽量复用**成熟方案，少占研发资源 | 统一登录、短信邮件、通用支付渠道接入 |

## 三、小的边界：统一语言与限界上下文（★★★★★）

### 3.1 统一语言（Ubiquitous Language）

- **定义**：业务与研发**共用**的一套业务术语与定义，并落实到**代码命名、接口字段、测试、文档、监控**中。
- **目的**：保证“**讨论的概念**”和“**实现的概念**”一致，减少误解与口径漂移；用语言约束模型一致性。
- **示例**：“取消订单”与“关闭订单”口语相近，规则不同——取消（用户主动）、超时关闭（系统自动）、风控关闭（风险判定）；代码中应体现为 `OrderStatus`、`CloseReason`、`CancelReason` 等，而非混为一谈。

### 3.2 限界上下文（Bounded Context）

- **定义**：在**明确边界内**，模型与语言一致；**跨边界**可同名不同义，通过**契约**协作。
- **作用**：避免不同业务语义混在一起导致模型“污染”（例如把营销规则写进订单模型，营销一变就要改订单逻辑）。

**电商示例（若干限界上下文）**：

| 上下文 | 典型概念 |
|--------|----------|
| **商品上下文** | 商品、SKU、类目、上下架、图文详情 |
| **交易上下文** | 订单、订单项、价格快照、收货地址、订单状态机 |
| **营销上下文** | 优惠券、活动、满减规则、凑单、会员价 |
| **库存上下文** | 可售库存、预占、扣减、回补、库存流水 |
| **支付上下文** | 支付单、渠道流水、退款单、对账 |
| **履约物流上下文** | 发货单、包裹、物流轨迹、签收 |

**污染示例**：把“优惠计算”直接写进交易上下文的订单类 → 营销规则（活动级、常变）会倒逼频繁改订单算价逻辑，即**订单被营销污染**。

### 3.3 上下文映射与防腐层（★★★☆☆）

**上下文映射**：描述多个限界上下文之间的**集成方式与依赖关系**，使跨团队协作与演进成本可控。

| 关系类型 | 含义 |
|----------|------|
| **客户/供应商** | 下游提需求，上游配合 |
| **顺从者** | 下游被迫适配上游 |
| **防腐层（ACL）** | **翻译与适配层**，隔离外部模型，防止外部模型污染核心领域 |
| **共享内核** | 少量共享模型由多上下文共同维护 |

**防腐层**：对接外部系统或遗留系统时，外部模型与本系统语言不一致。防腐层把外部模型**翻译**成本系统的领域概念，保证核心领域不被污染。例如第三方支付渠道返回渠道特有状态码、错误码 → 支付上下文的 ACL 翻译为统一的“支付成功/支付失败/待确认”等领域概念再向交易上下文发布事件；更换渠道时只需改 ACL，不动交易模型。

## 四、代码边界（★★★★★）

### 4.1 实体（Entity）

- **定义**：以**身份标识**区分；同一实体在生命周期内**属性可变**，**身份不变**；通常承载业务行为与状态演化。
- **示例**：订单（Order）有 `orderId`；地址、状态、备注可改，仍是同一订单。商品（Product）有 `productId`；**交易侧**宜用“价格快照/商品快照”，避免下单后商品改价影响历史订单。

### 4.2 值对象（Value Object）

- **定义**：由**属性**描述，**无独立身份**，通常**不可变**；相等性由属性值决定。
- **示例**：价格快照、收货地址（某次下单时的快照）、金额等。

### 4.3 聚合与聚合根（Aggregate & Aggregate Root）

- **聚合**：**一致性边界**。聚合内的业务不变式（强约束）应在**一次事务或一次一致性操作**内维护；**外部不能直接改内部对象**，必须通过**聚合根**统一入口修改。
- **聚合根**：聚合对外的唯一入口，负责保证聚合内规则不被绕开。
- **示例**：订单聚合（Order 为聚合根）含订单项、价格明细、收货信息。不变式示例：已支付后不许改应付金额；关闭后不许发货；同一订单同一时刻只能一个状态。应由聚合根方法如 `pay()`、`close(reason)`、`ship()` 保证，而非在控制器里直接改字段。

**示意**：
```
  [外部]  →  只能通过聚合根  →  [Order]  →  内部  OrderItem、PriceDetail、Address 等
```

### 4.4 领域服务（Domain Service）

- **定义**：当业务操作**涉及多个聚合**，或**难以归属**于单一实体/值对象时，用领域服务承载；仍在**领域层**，表达**业务规则**而非技术流程。
- **示例**：优惠分摊——订单多订单项，会员折扣后需把优惠分摊到各订单项（便于按项退款）；分摊规则不属于单个订单项，也不宜放在应用层脚本，适合做成领域服务如 `PricingService` / `PromotionApportionService`。

### 4.5 领域事件（Domain Event）

- **定义**：表示“**领域内发生的重要业务事实**”，用于在边界内外**传播变化、解耦协作**；强调**业务语义**，而非技术日志。
- **用法**：上游**发布**事件，下游**订阅**后执行自己的业务反应；常与限界上下文协作结合。

### 4.6 仓储与工厂（Repository & Factory）

| 概念 | 作用 |
|------|------|
| **仓储（Repository）** | 以“**集合**”方式**获取与保存聚合**，隐藏持久化细节；领域层不直接写 SQL |
| **工厂（Factory）** | **创建复杂聚合**，隐藏构造过程与校验逻辑 |

**目的**：领域模型不被数据库结构牵着走，保持领域层业务表达的纯粹性。

**示例**：`OrderRepository` 提供 `findById`、`save` 等；`OrderFactory` 在创建订单时完成：生成订单号、固化价格快照、校验库存可售、写入初始状态等，使“创建订单”不散落在控制器与多个服务类中。

### 4.7 应用层编排（Application Layer）

- **应用层**：负责**用例编排**——把一次业务请求串起来：校验权限、加载聚合、调用领域方法、持久化、发布事件。
- **领域层**：负责**业务规则本身**（“能不能做、做了以后状态怎么变”）。

**示例**：“支付订单”流程：读取订单聚合 → 调用 `order.pay()` → 生成支付单 → 调支付网关 → 收到回调后发布 `PaymentSucceeded` 事件。**编排**在应用层；**订单能否支付、支付后状态如何变化**在领域层。

## 五、小结

**DDD 三步**：  
1）大边界：领域、子域（核心/支撑/通用）；  
2）小边界：统一语言、限界上下文、上下文映射与防腐层；  
3）代码边界：实体、值对象、聚合/聚合根、领域服务、领域事件、仓储与工厂、应用层编排。

## 分布式系统

---

## 一、CAP 理论（★★★★★）

### 1.1 三性定义

**结论**：分布式系统**最多同时满足两个**，无法三者兼得；实践中**分区容错必然存在**，只能在**一致性 C** 与**可用性 A** 之间权衡。

| 概念 | 定义 |
|------|------|
| **一致性（Consistency）** | 写请求后**各相关节点数据均变化**，读请求能读到**变化后的数据**；多副本一致 |
| **可用性（Availability）** | 节点**收到请求就要处理并返回结果**，即使数据可能有问题（如返回旧数据）也不能拒绝 |
| **分区容错性（Partition Tolerance）** | 节点间**通信出现故障形成分区**时，系统仍能继续运行，**各分区可独立对外服务** |

**一致性**：一次写后，所有节点都更新，读能拿到新数据 → 称“发生了一致性变化”。  
**可用性**：能收请求就必须响应，哪怕返回的是过期数据（如三天前的数据）。  
**分区**：网络不可靠，节点间通信出问题即“分区”；**分区容错** = 出现分区后系统仍运行、分区可独立服务。  
**分区容错 vs 可用性**：分区容错强调“**通信故障导致分区**后各分区独立服务”；可用性强调“**非通信故障下**系统整体对外服务”。

---

### 1.2 CAP 组合与系统实现

| 组合 | 含义 | 典型例子 |
|------|------|----------|
| **CP**（一致性+分区容错） | 分区期间**宁可不可用**，也要保证数据一致 | **Zookeeper**：写经 Leader、多数派确认；Leader 失联或不足多数则**暂停服务**，直到重新选主 |
| **AP**（可用性+分区容错） | 分区期间**允许数据不一致**，保证继续可用 | **Eureka**：心跳丢失时**自我保护**，不剔除实例、不删注册信息，仍可查询，保持可用 |
| **CA**（一致性+可用性） | **分布式环境下无法实现**（无法避免分区） | — |

**示意**：
```
  分区发生时：
  CP → 暂停部分服务，保证一致（如 ZK 停写）
  AP → 继续服务，允许各分区数据不一致（如 Eureka 保护）
```

## 二、BASE 理论（★★★★★）

- **来源**：eBay 架构师 Dan Pritchett，2008 年 ACM；基于 CAP 的**一致性与可用性权衡**，牺牲强一致换可用性，允许**一段时间不一致**，**最终一致**。
- **思路**：通过业务设计，牺牲强一致性获得可用性，允许中间状态，最终达到一致。

| 概念 | 定义 |
|------|------|
| **基本可用（Basically Available）** | 遇不可预知故障时，**允许损失部分可用性**（如响应变慢、功能降级） |
| **软状态（Soft State）** | 允许系统存在**中间状态**，且该状态不影响整体可用性；**不同节点副本同步可有延迟** |
| **最终一致性（Eventually Consistent）** | 所有数据副本**经一段时间同步后**，**最终达到一致**，不要求实时强一致 |

## 三、分布式事务（★★★★★）

### 3.1 什么是分布式事务

- **定义**：事务的参与者、资源、事务管理器等分布在**不同节点**；一次业务涉及**多个服务/多库**，要么**全成功**，要么**全失败**。
- **典型场景**：下单 = 扣库存 + 扣优惠 + 生成订单，跨订单服务、库存服务、优惠服务等；需保证这些操作的整体一致性。
- **本质**：保证**不同数据库/服务**间的**数据一致性**。

### 3.2 刚性事务：2PC（二阶段提交）（★★★★★）

**两阶段**：**投票阶段** → **提交/回滚阶段**。

| 阶段 | 内容 |
|------|------|
| **阶段一（投票/预处理）** | ① 协调者向所有参与者发送事务内容，**询问**能否提交；② 各参与者**执行事务**，写 Undo/Redo 日志；③ 参与者向协调者反馈 **Yes/No** |
| **阶段二（提交/执行）** | 协调者根据反馈决定：**全 Yes** → 发 **Commit**，参与者提交并释放资源，回 ACK，协调者完成事务；**有 No 或超时** → 发 **Rollback**，参与者用 Undo 回滚并释放资源，回 ACK，协调者中断事务 |

**示意**：
```
  协调者          参与者1    参与者2
     | 询问         |          |
     |------------->| 执行+日志 |
     |------------->| 执行+日志 |
     |<--Yes--------|          |
     |<--Yes-------------------|
     | Commit/Rollback         |
     |------------->|          |
     |------------->|          |
     |<--ACK--------|          |
     |<--ACK-------------------|
```

**2PC 优缺点**：

| 优点 | 缺点 |
|------|------|
| 实现原理简单 | **性能**：两阶段内参与者**阻塞**，占资源 |
| | **单点故障**：协调者挂了，参与者无法释放资源或完成事务 |
| | **数据不一致**：发 Commit 后若网络异常或协调者崩溃，可能只有部分参与者提交，导致不一致 |
| | **保守**：依赖超时判断，容错差，难适应动态网络 |

### 3.3 柔性事务：TCC（Try-Confirm-Cancel）（★★★★★）

- **来源**：Pat Helland 2007 年论文《Life beyond Distributed Transactions: an Apostate's Opinion》。
- **与 2PC 对比**：① **无单点**，由主业务方发起，业务活动管理器可**集群**；② **同步阻塞**减轻，引入超时与补偿，资源以业务逻辑形式预留，**粒度更小**；③ **补偿机制**由业务活动管理器控制一致性。

| 阶段 | 含义 | 要点 |
|------|------|------|
| **Try** | **尝试执行** | 完成**业务检查**（一致性），**预留**必需业务资源（准隔离）；如库存检查+预留“可用库存” |
| **Confirm** | **确认执行** | **真正执行业务**，不再做检查，只使用 Try 预留的资源；**幂等**，失败需重试 |
| **Cancel** | **取消执行** | **释放** Try 阶段预留的资源；**幂等**；异常处理同 Confirm |

**示例**：订单+库存。Try：检查库存是否够、预留可用库存（如扣减“可用数量”）；Confirm：真正扣减库存、生成订单；Cancel：释放预留库存。

**缺点**：需把原一个接口拆成 **Try、Confirm、Cancel 三个接口**，**开发复杂度高**。

### 3.4 柔性事务：本地消息表（★★★★★）

- **来源**：eBay Dan Pritchett，2008 年 ACM；通过**消息**异步确保分布式任务执行，实现**最终一致性**。
- **核心**：**本地库中消息表**记录发送状态 + **MQ** 异步投递；业务数据与消息表**同一库、同一本地事务**写入。

**发送方**：  
① 本地事务内：写**业务数据** + 写**消息表**；② 投递消息到 **MQ**；③ 若消费方 **ACK**，可删除消息表记录；④ 未 ACK 则**重试**；消费方需**幂等**防重复处理。

**消费方**：  
① 消费 MQ 消息，执行业务逻辑；② 本地事务成功即处理成功；③ 失败则重试；④ 业务失败可向生产方发**补偿消息**（回滚等）。

**优缺点**：

| 优点 | 缺点 |
|------|------|
| 实现简单，开发成本低 | 与业务**高耦合**，占业务库资源，量大可能影响库性能 |
| | 消息**异步**，存在**延迟**；需处理重复消息（幂等） |

## 数据库系统

---

## 一、数据库性能优化（★★★★★）

### 1.1 索引建立注意事项

| 准则 | 说明 |
|------|------|
| 选**常查少改**的属性建索引 | 常作为查询条件、不常更新的属性适合建索引；**常更新的属性**建索引会严重影响性能 |
| **索引不宜过多** | 一个关系上索引过多会拉低 UPDATE、INSERT、DELETE 性能，因为更新时所有索引都要调整 |
| **按查询频度优化** | 分析重要查询的使用频度，优先优化使用最多的索引 |
| **小表可不建索引** | 数据量很小的关系不必建索引，全表扫描往往更快、耗资源更少 |

### 1.2 数据分片（Shard）

- **目的**：**横向扩展**（Scale Out），突破单节点 I/O 与扩展性限制；把“整库”打碎成多块，每块存到不同物理节点。
- **分片分类**（按拆分维度）：

| 类型 | 原理 | 重构方式 |
|------|------|----------|
| **水平分片** | 按**行**划分，元组分成多个子集，每个子集为一个片段；条件由**关系属性值**表示 | **并**操作重构全局关系 |
| **垂直分片** | 按**列（属性）**划分，满足不相交 | **连接**运算重构 |
| **导出分片** | 水平分片条件**来自其他关系**的属性（如按学生表的性别对选课表分片） | — |
| **混合分片** | 水平 + 垂直混合 | — |

### 1.3 分片算法

| 算法 | 原理 | 适用数据 |
|------|------|----------|
| **哈希分片** | 对键（如 id）哈希，再对节点数**取模**，余数对应存储节点 | 分布均匀、无强顺序/时间相关性、关联弱、可独立读写 |
| **范围分片** | 按**范围区间**（时间、ID 等）划分，同一范围放一起 | 有明显顺序（时间、编号）、可划分区间 |
| **地理位置分片** | 按**地理位置**（城市、地域）分配 | 数据与地理位置强相关，不同地域访问模式不同 |
| **融合算法** | 组合多种分片算法 | 同时有随机读写、范围查询或多维特征 |

**中间件常用**（如 MyCat）：**取模**（随机均匀）、**范围**（范围查询）、**列表**（枚举值指定分片）、**一致性哈希**（节点动态增删时只影响相邻数据）。

### 1.4 数据分区

- **含义**：单表内按规则（范围、哈希、列表等）划分为多个**逻辑子集**（分区）；查询时优化器只访问相关分区。
- **好处**：**查询**—减少 I/O、加快查询；**管理**—大表拆成小块，便于维护、备份、恢复；**隔离**—按类型或时间分隔，便于权限与生命周期管理。
- **分区算法**：与分片类似——**范围分区**（时间、数值区间）、**列表分区**（离散枚举值）、**哈希分区**（均匀、随机读写）、**复合分区**（先一种再另一种）。

### 1.5 分片 vs 分区（★★★★★）

| 特征 | 分片 | 分区 |
|------|------|------|
| **范围** | **跨多个数据库实例/服务器** | **单实例内** |
| **目的** | 提高**可扩展性、并发** | 优化**查询性能** |
| **实现** | 多个**物理表**（多节点） | **PARTITION BY** 子句（单库内） |
| **管理难度** | 相对复杂 | 相对简单 |

### 1.6 分库分表（★★★★★）

- **动机**：单机连接数、I/O、网络瓶颈；单表过大导致 B+ 树变高、I/O 增多；热点集中拖慢整体。
- **目的**：**分库**—分散请求到多台服务器，突破单机能力；**分表**—减小单表数据量，查询在更小范围完成。
- **分库分表是分片的物理实现**：“做了分片”的实现手段 = 分库 + 分表。

| 类型 | 定义 | 举例 |
|------|------|------|
| **水平分表** | 同一张表按**行**划分到多张表 | 订单表按订单 ID 取模拆成 orders_0、orders_1、orders_2 |
| **垂直分表** | 同一张表按**列**拆到多张表，减少单表字段 | 商品表：product_base（基本信息）、product_detail（描述、图片等大字段） |
| **水平分库** | 相同表结构分布到**多个库**，分摊压力 | 消息表按用户 ID 拆到 message_db1、message_db2 |
| **垂直分库** | **不同业务**的数据放到**不同库**，按模块拆 | 用户库 user_db、课程库 course_db、支付库 payment_db |

## 二、数据库高可用（★★★★★）

### 2.1 读写分离

- **目的**：把**读、写**分散到不同节点；**小幅提升写、大幅提升读**；符合写少读多。
- **常见**：**一主多从**—主库写，从库读；主从之间**数据同步**保证从库数据正确。
- **实现**：应用层如 **Shardingsphere**；数据库层中间件如 **MyCat**。

**示意**：
```
  写请求 → 主库  ←── 同步 ──→  从库  ← 读请求
```

### 2.2 主从复制

- **手段**：利用**日志**（如 MySQL 的 **binlog**）；主库写操作记入 binlog，推给从库，从库**重放**完成复制。
- **流程（MySQL）**：① 主库把数据变化写入 **binlog**；② 从库连接主库请求复制；③ 从库 **I/O 线程**向主库请求 binlog；④ 主库 **dump 线程**发送 binlog，从库 I/O 线程接收；⑤ 从库 I/O 线程将 binlog 写入**中继日志（relay log）**；⑥ 从库 **SQL 线程**从 relay log 读事件并应用到本地库。

| 优势 | 描述 |
|------|------|
| **数据冗余** | 多副本，主库故障时减少数据丢失风险 |
| **负载均衡** | 读分发到从库，减轻主库负载 |
| **高可用** | 主库故障时可切换到从库 |

### 2.3 主从一致性问题（主从延迟）（★★★★★）

- **现象**：主库写入后，同步到从库有**时间差**，读从库可能拿到**旧数据**。
- **解决思路**：减少延迟或延迟窗口内**强制读主**。三种方式对比：

| 方式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **半同步复制** | 主库提交事务后，等**至少一个从库**把事务 binlog 写入 relay log 并**回 ACK**，主库才提交并返回客户端 | 用数据库**原生**能力，实现简单 | 主库**写时延增加**，**吞吐下降** |
| **数据库中间件** | 读写都走中间件；写走主库，读走从库；**记录写库的 key**，在**主从同步时间窗**（如 500ms）内对该 key 的**读请求走主库**，过后再走从库（如 Canal、Otter） | **强一致** | 中间件**成本高** |
| **缓存记录** | 写前把 **key 写入 cache** 并设超时（如 500ms），再写主库；读时**先查 cache**，命中则说明刚写过，**读主库**，未命中则读从库 | 比中间件**成本低** | 多 **cache** 组件，读写多一步缓存操作 |

**key**：能唯一标识写操作的主键或业务键（如主键 ID、用户 ID、订单号），用于在延迟窗口内把“刚写过”的读请求路由到主库。

**示意**：
```
  半同步：  主库 --等从库 ACK--> 提交
  中间件：  写 key 记录 → 时间窗内该 key 读主库
  缓存：    写前 key 入 cache(500ms) → 读时命中则主库
```

| 方式 | 含义 | 特点 |
|------|------|------|
| **RDB** | Redis DataBase，**快照** | 按时间间隔将数据**快照**写入磁盘；**恢复快**；宕机可能丢一段数据；**默认**开启 |

## 三、Redis 集群（★★★★★）

### 3.1 三种模式概览

| 模式 | 作用 | 优点 | 缺点 | 适用 |
|------|------|------|------|------|
| **主从复制** | 主节点数据复制到从节点；主写从读 | 配置简单、数据冗余、读写分离 | **不能自动故障转移**，受单机内存限制 | 备份、读写分离、在线升级 |
| **哨兵（Sentinel）** | 在主从基础上加哨兵，**监控+自动故障转移** | 高可用，具备主从优点 | 配置管理较复杂 | 高可用要求高 |
| **Cluster** | **数据分片（slot）+ 故障转移**，多主多从 | 水平扩展、高可用、突破单机容量 | 架构与运维更复杂 | 大数据量、高并发、需分片 |

---

### 3.2 主从复制流程

**全量同步**（从库首次或需全量时）：  
① **握手**：从库发 `PSYNC -1`，主库回 `FULLRESYNC <runid> <offset>`（runid=主库身份，offset=复制进度）；  
② 主库 **BGSAVE** 生成 RDB，期间新写命令进 **replication buffer**；  
③ 主库把 **RDB** 发给从库，从库清空旧数据、加载 RDB；  
④ **补齐**：主库把 replication buffer 中的命令发给从库，从库执行后与主库一致；  
⑤ **常态复制**：后续写命令通过长连接实时同步。

**增量同步（部分重同步）**（断线重连后）：  
① 从库带 `runid + offset` 发 `PSYNC <runid> <offset>`；  
② 主库校验 runid 是否一致、缺口数据是否还在 **repl_backlog_buffer**；在则返回 CONTINUE，执行部分重同步；不在则全量；  
③ 主库从 backlog 中取“>offset”的日志发给从库执行；  
④ 进入常态复制。

---

### 3.3 哨兵（Sentinel）

- **作用**：**监控**主从健康；主节点下线时**自动故障转移**（选新主、通知从库与客户端）；**通知**管理员或其它服务。
- **建议**：至少 **3 个**哨兵节点，奇数便于投票。

**故障转移流程**：  
① **选 Sentinel Leader**：主节点客观下线的哨兵发起选举，得**过半数**票者成为 Leader；  
② **选新 Master**：从存活从节点中按 **slave-priority → 复制 offset → 实例 ID** 选数据最完整者为新主；  
③ **重建主从**：其它从节点复制新主；通过**发布/订阅**把新主地址通知客户端；  
④ **旧主恢复**：旧主上线后执行 `slaveof` 降级为新主的从节点。

---

### 3.4 Cluster 模式

- **目的**：解决单机内存与性能上限、主从+哨兵无法**水平分片**的问题；通过**数据分片（slot）+ 自动故障转移**实现扩展与高可用。
- **槽位**：键空间划分为 **16384** 个槽；每个主节点负责一部分槽；键定位：**CRC16(key) % 16384**；扩容缩容时迁移对应槽数据。
- **结构**：多个 Master（各负责一部分槽），每个 Master 可有多个 Slave；主库故障时从库可被提升为主库。

---

### 3.5 哨兵 vs Cluster

| 对比 | 哨兵 | Cluster |
|------|------|--------|
| **数据分片** | **不分片**，所有数据在一个主库 | **分片**，数据分布到多主 |
| **主要作用** | 监控 + **主从故障转移** | **分片 + 故障转移** |
| **选型** | 单 Redis 高可用、读写分离 | **横向扩展、大数据量、高并发** |

## 四、常见生产问题（★★★★★）

| 问题 | 原因 | 解决方案（简） |
|------|------|----------------|
| **缓存穿透** | 请求的 key 在**缓存和库都不存在**，请求直接打到库 | **校验**非法参数；**缓存空 key**（短过期）；**布隆过滤器**先判存在再查 |
| **缓存击穿** | **热点 key 过期瞬间**大量请求打到库 | 热点**永不过期**或长过期；**互斥锁**（只有一个请求查库）；**预热**热点数据 |
| **缓存雪崩** | **大量 key 同时失效**，请求压垮库 | **集群**避免单点；**限流**；**多级缓存**；**错峰过期**；**预热** |

**布隆过滤器**：多位数组 + k 个哈希函数；添加时把对应位标 1；判断时若有一位为 0 则**一定不存在**，全为 1 则**可能存在**（有假阳性，无假阴性）。

## 五、缓存与数据库一致性（★★★★★）

**旁路模式**：读先查缓存，没有则查库并回写缓存；**写先更新库，再删除缓存**，下次读时从库加载到缓存。

**主从延迟导致读旧缓存**：写库后删缓存前，其它请求可能读到旧缓存。常用做法：

| 方式 | 思路 | 一致性 | 备注 |
|------|------|--------|------|
| **延迟双删** | 删缓存 → 更新库 → **延迟后再删一次缓存** | 最终一致 | 延迟时间需根据业务权衡 |
| **先更新库再删缓存** | 更新库 → 删缓存 | 最终一致，概率上偶发不一致 | **最常用、最简单** |
| **分布式锁** | 写时加写锁（更新库+删缓存），读时加读锁，串行化 | **强一致** | 性能差 |
| **MQ 串行化** | 同一 key 的写请求进同一队列，顺序消费更新库与缓存 | 最终一致 | 无乱序覆盖 |
| **订阅 binlog** | 库更新后通过 **Canal** 等把 binlog 发 MQ，消费后更新缓存 | 最终一致 | 顺序消费保证最终一致 |

## 六、过期删除与内存淘汰（★★★★★）

**过期删除策略**：  
- **惰性删除**：**取 key 时**才检查是否过期；对 CPU 友好，可能堆积大量过期 key。  
- **定期删除**：**周期性**抽一批 key 删过期；通过限制执行时长和频率降低对 CPU 影响。

**内存淘汰策略**（达 **maxmemory** 时）：

| 策略 | 描述 | 适用 |
|------|------|------|
| **noeviction**（默认） | 内存满时**拒绝写入** | 要数据完整、对可用性要求不高 |
| **volatile-lru** | 在**设了过期**的 key 中淘汰**最近最少使用** | 缓存，优先淘汰不常用 |
| **allkeys-lru** | **全部** key 中淘汰最近最少使用 | **通用缓存** |
| **volatile-random** | 设了过期的 key 中**随机**淘汰 | 对顺序无要求 |
| **allkeys-random** | 全部 key 中**随机**淘汰 | 均匀淘汰 |
| **volatile-ttl** | 设了过期的 key 中淘汰**剩余 TTL 最小** | 优先淘汰即将过期 |

## 七、大 Key 与热 Key（★★★★★）

### 7.1 大 Key

- **定义**：Key 对应的 **Value 大**（如 String 5MB、Zset 成员上万、Hash 成员 value 总和大）。
- **危害**：命令变慢、易触发 maxmemory 淘汰或阻塞、集群下某分片内存倾斜、删除易阻塞引发主从中断。
- **处理**：**拆分**大 Key（如大 Hash 拆成多个小 Key）；**清理**不适于 Redis 的数据；**定期清理**过期数据防堆积。

### 7.2 热 Key

- **定义**：**QPS/带宽/CPU** 集中在一个或少量 key（如单 key QPS 远高于其它）。
- **危害**：占用大量 CPU、集群下某分片被打满、易引发击穿。
- **处理**：**多级缓存**（本地缓存分担）、**热 key 备份**（多节点备份分散读）、**热 key 拆分**（带后缀分散到多实例）、**业务隔离**（热点业务独立集群）。

## 八、分布式锁（★★★★★）

- **目标**：分布式环境下，保证同一时刻**只有一个进程**能访问共享资源。
- **三种实现**：

| 实现 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **数据库表** | 锁表插入一条记录，释放删记录 | 简单 | 依赖库可用性、无失效时间、非阻塞、非重入 |
| **Redisson** | 基于 Redis，支持单机/主从/Sentinel/Cluster；**看门狗**定时续期防提前过期；**可重入**（hset 记录加锁次数） | 性能好、可重入、续期 | 非阻塞需自旋 |
| **Zookeeper** | 在 /lock 下建**临时有序**子节点，序号最小者获锁；释放删节点；监听前序节点 | 无单点、可重入、阻塞等待、锁可释放 | 高并发下性能不如 Redis（ZAB 协议） |

## 九、Redis 事务（★★★★★）

- **用法**：**MULTI** 开启事务 → 命令入队 → **EXEC** 顺序执行；**DISCARD** 放弃事务；**WATCH** 监视 key，被改则事务失败。
- **特点**：**不保证原子性**—EXEC 后某条失败**不回滚**，后续仍执行；**持久性**依赖 RDB/AOF 配置。适合**批量命令串行执行**，而非严格 ACID 事务。

## Elasticsearch

## 一、ES 概述（★★★★★）

- **特点**：以**全文检索**为基础；使用**分析器、分词**处理文本，支持词干、停用词、同义词等；建立索引时**分词**，写入**倒排索引**；查询时对查询语句分词，在倒排索引中找文档，再按**相关性评分**（如 BM25、TF-IDF）排序。
- **流程**：文档 → **分词** → 词项 → **倒排索引**；查询 → 分词 → 查倒排索引 → **相关性打分** → 排序返回。

## 二、分词原理（★★★★★）

### 2.1 分析器（Analyzer）三部分

分析器是一条**流水线**，依次为：

| 环节 | 作用 |
|------|------|
| **字符过滤器（Character Filters）** | 对**原始文本**预处理：去/换特殊字符、HTML 转可读等；**可选** |
| **分词器（Tokenizer）** | **核心**：按规则把文本切成**词元（Token）**；如标准分词器按空格、标点切分 |
| **标记过滤器（Token Filters）** | 对词元再加工：**小写**、去**停用词**、**词干提取**等，便于索引与搜索 |

**示意**：
```
  原始文本 → [字符过滤器] → [分词器] → [标记过滤器] → 词元序列
```

---

### 2.2 常用 Tokenizer（分词器）

| 类型 | 输入示例 | 输出示例 | 特点 |
|------|----------|----------|------|
| **standard**（标准） | Hello, this is Elasticsearch! | ["hello","this","is","elasticsearch"] | 去标点、转小写，适合自然语言 |
| **whitespace**（空白） | Hello World, 你好 Elasticsearch | ["Hello","World,","你好","Elasticsearch"] | **仅按空格**切，标点保留 |
| **keyword**（关键词） | OrderID-12345 | ["OrderID-12345"] | **不切分**，整段作为一个词元，适合 **ID、手机号**等精确匹配 |
| **uax_url_email** | 请联系 support@example.com 或 https://cheko.cc | 邮箱和 URL 为**整体** | 识别 URL、邮箱，保持完整 |

**记忆**：standard 自然语言，whitespace 只看空格，keyword 不切（精确），uax_url_email 保 URL/邮箱。

## 三、倒排索引（★★★★★）

- **定义**：**反向索引**；先对文档**分词**得到词项，再建立「**词项 → 文档 ID 列表**」的索引；查词即可得包含该词的文档。
- **与正排**：正排是 **key（文档）→ value（内容）**；倒排是 **value（词项）→ key（文档列表）**。
- **优势**：无需扫描全部文档，通过**词典 + 倒排表**快速定位包含某词的文档，**全文检索效率高**。

**示例**：  
文档 1：苹果是一种水果。 文档 2：香蕉是一种热带水果。 文档 3：苹果和香蕉都是水果。  
倒排索引：**苹果 → [1, 3]**；**香蕉 → [2, 3]**。搜「苹果」即得文档 1、3。

**示意**：
```
  正排： 文档ID → 内容
  倒排： 词项 → [文档ID1, 文档ID2, ...]
```

## 四、相关性算法（★★★★★）

- **作用**：**相关性得分**决定排序，得分越高越符合查询意图。
- **算法**：由 **Lucene** 计算；常用 **TF-IDF**、**BM25**（ES 默认）。BM25 为 TF-IDF 改进版，长文档与参数可调性更好。

### 4.1 影响相关性的主要因素

| 因素 | 含义 | 影响 | 举例 |
|------|------|------|------|
| **查询类型** | 不同查询方式对应不同打分规则 | **match** 会算相关性；**term** 偏精确匹配，不参与复杂评分 | match「手机」考虑词频与相关性；term 只匹配完全一致的「手机」 |
| **词频（TF）** | 词在**当前文档**中出现次数 | 出现越多，得分越高 | 文档 A 出现 10 次「手机」比 B 出现 1 次得分高 |
| **逆文档频率（IDF）** | 词在**所有文档**中的稀有程度 | 越稀有，贡献得分越高 | 「iPhone17」比「手机」更稀有，区分度更强 |
| **文档长度归一化（Norms）** | 关键词在**短文档 vs 长文档**中的代表性 | 短文档中出现一次更有代表性 | 标题里出现一次「iPhone」比长文中一次更突出 |
| **字段权重（Boost）** | 手动提高某字段重要性 | 被 Boost 的字段匹配得分更高 | title boost=2，优先于 description |

## 五、ES 与传统关系型数据库对比（★★★★★）

| 对比维度 | Elasticsearch（ES） | 传统关系型数据库（如 MySQL） |
|----------|---------------------|------------------------------|
| **类型** | **分布式搜索引擎**，**文档型**存储 | **关系型**，**表结构**存储 |
| **核心功能** | **全文搜索**、复杂查询、日志分析、**实时检索** | 结构化存储、**事务**、**数据一致性** |
| **应用场景** | 全文检索（电商搜索）、日志分析、监控查询 | 订单、财务、用户信息等**强事务**场景 |
| **优点** | 搜索性能高、实时搜索、**分布式扩容方便**、全文检索强 | **ACID 事务**、一致性强、生态成熟、SQL 标准 |
| **缺点** | 部署运维复杂、查询学习成本高、**一致性弱** | 实时搜索弱、大数据扩展难、全文能力有限 |
| **数据结构** | **文档（JSON）**，半结构化 | **二维表**，结构化 |
| **分布式** | **原生**分布式集群，**横向扩展**能力强 | 需**分库分表**，架构复杂 |
| **事务** | 仅**文档级**简单操作，**事务弱** | **完整 ACID**，原子性、一致性有保障 |

## MongoDB

---

## 一、文档模型要点

- **文档**：一条记录 = 一个文档，结构为「字段-值」对，可嵌套文档与数组。
- **优势**：与编程语言中对象类型对应自然；**嵌入式文档和数组**减少多表连接，适合半结构化、嵌套数据。

## 二、数据类型（★★★★★）

### 2.1 BSON（Binary JSON）

- **定义**：**二进制**形式的类 JSON 存储格式；支持内嵌文档和数组，并扩展了 JSON 没有的类型（如 **Date、BinData** 等）。
- **与 JSON 对比**：

| 对比项 | BSON | JSON |
|--------|------|------|
| **数据表示** | **二进制** | **纯文本** |
| **类型支持** | 更多原生类型（日期、整型、浮点、**二进制**等） | 类型相对有限 |
| **存储与速度** | **更紧凑、读写更快**；体积可能略大 | 相对不紧凑、较慢 |
| **易读性** | 需专门工具查看 | **人类可读**，便于调试与日志 |
| **适用场景** | **数据库存储**，要求读写速度、复杂结构及扩展类型 | **网络传输**、开发调试 |
| **压缩** | 紧凑，压缩优化不如文本明显 | 文本压缩算法优化效果更好 |

### 2.2 GeoJSON（★★★★★）

- **定义**：描述**地理数据**的开放标准格式，**非 MongoDB 独有**；MongoDB **原生支持** GeoJSON，可存储与查询地理空间数据。
- **用途**：地理空间**索引与查询**，实现**位置服务、位置分析**等（如附近的人、范围查询）。

## 三、分片集群（★★★★★）

- **目的**：解决**海量数据**存储与读写扩展——单机磁盘、CPU、内存、网卡等瓶颈；通过**分片**将数据分布到多节点。
- **两种分片策略**：

| 分片类型 | 原理 | 适用场景 |
|----------|------|----------|
| **哈希分片** | 对**片键**字段做**哈希**，用哈希值在分片间划分数据 | **随机访问**为主，无明确范围；数据均匀分布 |
| **范围分片** | 按片键的**连续范围**划分，同一范围的数据在同一分片 | 数据**按某属性连续增长**（如**时间顺序**）；**范围查询**高效；片键选择不当会导致热点与性能下降 |

**记忆**：哈希→随机、均匀；范围→连续、范围查询（如时间）。

**示意**：
```
  哈希分片：  key → hash(key) → 分片索引 → 落在某分片
  范围分片：  key 在 [A,B) → 分片1；[B,C) → 分片2；...
```

## 四、高可用：副本集（Replica Set）（★★★★★）

- **组成**：一组 MongoDB 实例，包含 **1 个 Primary** 和 **多个 Secondary**。
- **读写**：**客户端（Driver）写只到 Primary**；Secondary 从 Primary **同步数据**，使副本集内数据一致，提供**高可用**。
- **同步机制**：通过 **oplog**（类似 MySQL 的 **binlog**）。Primary 完成写操作后，向 **local.oplog.rs** 集合写入一条 oplog；Secondary **持续从 Primary 拉取并应用** oplog，保持与 Primary 数据一致。

**示意**：
```
  客户端  →  写  →  Primary  →  oplog  →  Secondary 1
                        ↓                    Secondary 2
                  local.oplog.rs
```

**要点**：一主多从、写主读可从从、oplog 同步、local.oplog.rs。

## 大模型应用

---

## 一、检索增强生成（RAG）（★★★★★）

### 1.1 核心思想与流程

- **思想**：**“检索 + 生成”**——用大模型生成能力，同时用**外部知识**弥补模型记忆与时效局限。
- **三阶段**：**索引阶段** → **检索阶段** → **生成阶段**。
- **五环节**：**数据准备** → **向量化** → **索引存储** → **检索增强** → **生成答案**。

**示意**：
```
  数据准备 → 向量化 → 索引存储（向量库）
       ↑                    ↓
  用户问题 → 检索增强（向量检索/重排）→ 问题+文档 → 大模型 → 生成答案
```

### 1.2 数据准备

- **目标**：把原始文档变成**结构化、可处理**的文本，质量直接影响检索与回答。
- **常见步骤**：
  1. **清洗**：去掉 HTML、PDF 页眉页脚等多余格式。
  2. **分块**：长文档切成小块再向量化与检索；三种策略：
     - **语义切分**：每块**语义独立**（如问答对、完整段落）。
     - **结构切分**：按**标题层级**切，保留文档结构。
     - **递归切分**：文本过长时按**字符数**递归切，兼顾连贯与长度。

### 1.3 向量化与索引存储

- **向量化**：用 **Embedding 模型**（如 BERT、Sentence-BERT、BGE）把**文本块**转成**向量**（高维语义表示）；语义相近的文本向量接近（如“苹果手机”与“iPhone”）。
- **索引存储**：向量存入**向量数据库**（如 Milvus、Faiss、Elasticsearch、Pinecone），支持**相似度检索**；通常同时存**原始文本**，便于检索后回溯内容。

### 1.4 检索增强

- **步骤**：① **问题向量化**—用户问题转为向量；② **向量检索**—在库中找相似文本块；③ **重排序（Reranker）**—用 Reranker 模型（如 Cohere Reranker）对初筛结果精排，把最相关的放前面。
- **增强手段**：
  - **多阶段检索**：先用 **BM25** 等**关键词**粗筛，再用**向量检索**精排。
  - **混合检索**：关键词 + 向量检索结果用 **RRF（倒数排名融合）** 合并，兼顾“语义相似但缺词”和“有词但语义偏”的问题。

### 1.5 生成答案

- **输入**：**用户问题 + 检索到的相关文档** 一起交给大模型。
- **关键**：**Prompt 设计**——明确“仅依据文档作答、不知道就说不知道、标注来源、处理冲突”等；示例：先结论后依据、用[编号]标来源、冲突时分别说明、无关则说明无法回答并建议更精确提问。

## 二、模型上下文协议（MCP）（★★★★★）

- **提出**：**Anthropic**，2024 年 11 月。
- **定位**：为 **LLM 与外部世界**提供统一的**“通用接口”**（类似 USB-C），让模型**即插即用**地访问各类数据源与工具。
- **机制**：符合 MCP 标准的外部资源（数据库、API、文档库、本地文件等）可直接接入；模型**不需关心底层实现**，通过 MCP 即可理解和使用，从而**实时获取数据、调用工具、执行任务**，减少对静态知识的依赖。
- **价值**：① **标准化接入**—一次集成多场景复用；② **增强模型能力**—如从 GitHub 拉代码、读日历、操作本地文件；③ **架构模块化**—协议统一后扩展与维护更简单。

## 三、智能体（Agent）（★★★☆☆）

### 3.1 智能体简介

- **定义**：以**目标为导向**的智能系统；能**感知环境**，基于观测**推理与规划**，通过**行动**（调用工具/接口）影响环境，使任务向目标收敛；本质是**“决策—行动—反馈”**闭环。
- **与 Chatbot、Workflow 对比**：

| 类型 | 特点 | 局限 |
|------|------|------|
| **Chatbot** | 以**对话生成与交互**为主，可拟人、可聪明 | 不必调用外部系统，不必对结果负责 |
| **Workflow** | **步骤可复现、路径可控**，适合稳定流程与合规 | **适应性弱**，输入/约束/目标一变常需改流程 |
| **Agent** | **目标牵引**，在循环中选工具、执行、读反馈、修正计划，**自适应、可迭代** | 依赖工具质量与观测可靠性，需成本与回合约束 |

**示意**：Agent = 感知 → 推理/规划 → 行动（工具/接口）→ 反馈 → 再决策（闭环）。

### 3.2 智能体架构

- **含义**：为让 LLM 系统像“会做事的人”一样完成目标，在**模型外**搭建的**结构与组件组织**；解决的是“系统如何感知、决策、调用工具、记忆与修正、何时交付”，模型是其中的“大脑”。
- **两种常见模式**：**ReAct**、**规划-执行**。

---

### 3.3 ReAct 模式（Reason + Act）

- **思想**：**“推理”与“行动”交替**；每步从环境获取反馈，再根据反馈做下一步决策，形成**闭环**，而不是一次性生成答案。
- **过程**：模型通过**工具调用、检索、计算**等得到“地面事实”，再据此一步步推进直到完成。
- **劣势**：每步都可能回模型决策，**调用次数与时延**易偏高；可能**循环失控**（不停检索/尝试）；依赖**工具质量与观测可靠**，否则易在错误反馈上越走越偏；工程上需**预算约束**（最大回合、最大工具调用、超时降级）。
- **适用**：**信息不完整、需边做边看**的任务（多源检索综合、按日志排障、试探性改代码并观察、需计算与验证的问答）。**不适合**：流程固定、强一致性、低成本的场景（更适合预编排工作流）。

### 3.4 规划-执行模式（Plan-and-Execute）

- **思想**：把任务明确分为**规划阶段**和**执行阶段**。规划阶段根据目标生成**可执行计划**（步骤顺序、依赖、资源与工具、每步验收标准）；执行阶段**按计划逐步执行**，每步产出结果作为下一步输入，把复杂任务拆成可管理、可检查的子任务链。
- **劣势**：**计划与现实易偏差**——信息不全、环境变化时，规划易理想化，执行中易遇依赖不可用、约束改变，导致偏航与返工；若死守计划反而拖慢。需配套**动态重规划**：观测不满足假设时，可调整后续步骤或回到规划阶段重做，并保留已验证成果。
- **适用**：**可拆解、可验收、目标明确**的多步骤任务（如先列提纲与资料再写报告）。**不适合**：**强探索式**、目标模糊、路径不确定、需频繁试错的场景（易变成“计划—推翻—重来”），此类更适合 ReAct 或执行阶段嵌入 ReAct。
