# 数据资产智能管理平台 PRD

## 1. 文档基本信息

- 文档名称：数据资产智能管理平台产品需求文档（PRD）
- 项目名称：国有航空集团有限公司数据资产智能管理平台建设项目
- 文档版本：V1.0
- 文档属性：需求整理稿
- 适用对象：业务负责人、产品经理、架构师、研发团队、测试团队、实施团队、运维团队、管理层`

## 2. 项目背景

国有航空集团有限公司是国内大型航空运输集团，拥有总部数据中心、五大区域中心和 23 个分公司节点，业务覆盖航空客运、货运物流、航空食品、飞机维修、地面服务等多个领域。随着集团数字化转型持续推进，集团内部沉淀了大量覆盖航班运行、旅客服务、机务维修、货运物流、财务人力等核心业务的数据资产。

当前集团在数据管理方面仍面临多个关键问题：一是各业务系统独立建设，数据标准不统一，数据孤岛严重，集团级共享和协同能力不足；二是缺乏统一的数据质量管理机制，数据准确性、完整性、一致性难以保障；三是数据资产家底不清，无法实现统一盘点、分类分级、血缘追踪和价值评估；四是敏感数据识别和安全保护能力不足，存在较高合规与安全风险；五是总部、五大区域中心及 23 家分公司之间数据管理模式分散，难以形成统一的数据治理体系和协同机制。

在民航局《智慧民航建设数据管理政策标准体系》等行业政策指引下，以及数据治理、安全合规要求推动下，集团需要建设一套覆盖全集团的数据资产智能管理平台，实现数据资产“可见、可用、可管、可控”，并借助集团自建大模型提升数据资产管理全过程智能化能力，支撑业务创新、经营分析与管理决策。

## 3. 产品定位

数据资产智能管理平台是面向集团总部、区域中心与分公司的一体化数据资产管理平台，定位为集团级数据资产治理与运营中枢。平台聚焦“统一目录、统一规则、统一流程、统一安全、统一运营、智能赋能”，为集团提供数据资产盘点、目录服务、质量管理、安全管理、血缘分析、价值评估、数据共享和智能问答等核心能力。

## 4. 建设目标

### 4.1 总体目标

建设覆盖集团总部、五大区域中心和 23 个分公司的数据资产智能管理平台，构建统一的数据资产管理体系，实现数据资产全生命周期治理、跨区域协同共享和智能化管理能力落地。

### 4.2 业务目标

1. 建立覆盖全集团的数据资产统一目录，实现数据资产全面盘点和分类管理。
2. 实现数据资产从产生、采集、存储、加工、使用到销毁的全生命周期管理。
3. 建立统一的数据质量规则库，实现数据质量自动检测、预警、分析与整改闭环。
4. 建立数据安全分级分类与敏感数据保护体系，实现敏感数据识别、脱敏、访问控制和审计。
5. 建立数据资产价值评估模型和运营机制，支撑数据资产价值释放和数据产品化运营。
6. 实现总部、五大区域中心、23 个分公司之间的数据资产协同管理和安全共享。
7. 基于集团自建大模型，实现数据资产管理全流程智能辅助和智能决策支持。

### 4.3 量化目标

1. 数据资产目录盘点规模不少于 10 万项。
2. 数据质量规则不少于 5000 条。
3. 数据质量合格率提升至 98% 以上。
4. 敏感数据自动识别准确率不低于 95%，智能增强后不低于 98%。
5. 年数据服务调用量不低于 10 亿次。
6. 年数据产品化收入目标不低于 5000 万元。
7. 智能问答准确率不低于 90%。
8. 智能分析响应时间不高于 3 秒。
9. 智能推荐匹配度不低于 85%。

## 5. 产品范围

### 5.1 范围内

1. 数据资产目录管理。
2. 数据质量管理。
3. 数据安全管理。
4. 数据血缘与影响分析。
5. 数据价值评估与运营监控。
6. 智能能力模块建设。
7. 与集团自建大模型的私有化集成。
8. 总部、区域中心、分公司分层部署与协同接入。
9. 与 IAM、工单、消息、日志、CMDB、DevOps 等现有系统集成。

### 5.2 范围外

1. 外部公共大模型服务接入。
2. 与本项目无直接关系的业务系统功能重构。
3. 非集团范围外部单位的数据平台统一建设。

## 6. 目标用户与角色

### 6.1 用户类型

1. 集团管理层：关注数据资产总体规模、价值、质量和安全态势。
2. 数据治理委员会/数据管理部门：负责制度、标准、策略、考核和治理推进。
3. 数据 Owner：负责业务域数据资产认责、审批、质量整改和共享授权。
4. 数据管家/数据 Steward：负责资产维护、分类分级、规则配置、质量跟踪。
5. 数据分析师/数据开发人员：需要检索、申请、使用、订阅数据资产。
6. 安全管理员：负责敏感数据识别、脱敏策略、访问控制和审计分析。
7. 运维人员：负责平台运行、告警处理、性能与容量保障。
8. 区域中心管理员/分公司接入人员：负责区域部署、接入配置、同步与本地运营。

### 6.2 权限角色

1. 平台超级管理员。
2. 集团级治理管理员。
3. 区域中心管理员。
4. 分公司管理员。
5. 数据 Owner。
6. 数据 Steward。
7. 数据安全管理员。
8. 普通查询/申请用户。
9. 审计查看角色。

平台需同时支持基于角色的访问控制（RBAC）和基于属性的访问控制（ABAC）。

## 7. 业务痛点

1. 数据资产分散在数据库、数据仓库、大数据平台、文件系统等多个异构环境中，缺乏统一盘点入口。
2. 数据分类分级标准不统一，资产可见性不足，数据申请与授权流程不规范。
3. 数据质量规则缺乏统一管理，问题发现滞后，整改闭环效率低。
4. 敏感数据识别依赖人工和简单规则，难以应对复杂语义和上下文关联场景。
5. 血缘关系不完整，数据变更影响范围难以评估，发布和治理风险高。
6. 数据价值难量化，无法为共享运营、产品化和管理决策提供充分依据。
7. 总部、区域中心和分公司之间缺乏统一协同机制，治理策略执行不一致。
8. 当前数据管理过程自动化和智能化程度不足，人员投入大、效率低。

## 8. 产品价值

1. 帮助集团建立统一的数据资产底座，实现数据资产家底清晰可查。
2. 提升数据质量与安全治理能力，降低数据风险和合规风险。
3. 强化区域协同和共享能力，提升集团级数据资源复用效率。
4. 支撑数据价值评估与数据产品化运营，促进数据变现。
5. 通过智能能力降低治理成本，提高问题定位、分析和决策效率。

## 9. 产品原则

1. 集团统一管理，区域分级运营。
2. 标准先行，流程闭环，审计可追溯。
3. 安全合规优先，最小权限访问。
4. 平台能力模块化、可扩展、可复用。
5. 兼顾总部集中治理与区域就近处理。
6. 全量私有化部署，智能能力不得依赖外部公共模型服务。

## 10. 核心业务流程

### 10.1 数据资产接入流程

1. 管理员配置数据源接入信息。
2. 平台发起自动扫描和元数据采集。
3. 平台识别数据对象并生成初始目录。
4. 数据 Steward 对业务属性、管理属性进行补充完善。
5. 平台执行分类分级和敏感识别。
6. 资产进入统一目录并可被搜索、申请、订阅。

### 10.2 数据申请与授权流程

1. 用户通过统一搜索门户检索资产。
2. 用户查看资产详情并发起申请。
3. 系统按预设流程流转至数据 Owner、数据安全管理员等审批节点。
4. 审批通过后，平台根据权限策略完成授权。
5. 用户在授权范围内访问资产，平台保留完整审计日志。

### 10.3 数据质量治理流程

1. 治理人员配置质量规则或引用规则模板。
2. 平台按实时、定时或事件触发方式执行检测。
3. 平台自动评分、评级并生成异常结果。
4. 异常触发多渠道告警和工单。
5. 责任人整改并回填处理结果。
6. 平台跟踪趋势、复发率和整改 SLA。

### 10.4 数据安全治理流程

1. 平台自动执行敏感数据识别。
2. 安全管理员对识别结果复核并修正。
3. 配置脱敏策略和访问控制策略。
4. 用户访问资产时，平台按策略执行脱敏和鉴权。
5. 平台对访问行为进行全链路审计和留痕。

### 10.5 血缘与影响分析流程

1. 平台从 SQL、ETL 工具、作业日志等来源采集血缘。
2. 系统构建全链路血缘图谱。
3. 当数据对象或规则发生变更时，平台自动触发影响分析。
4. 输出受影响资产、系统、作业、API 消费者清单及风险提示。

### 10.6 智能问答与分析流程

1. 用户以自然语言提出问题。
2. 智能助手调用集团自建大模型与知识库进行语义检索。
3. 平台生成结构化结果、建议或图表。
4. 用户可进一步追问，完成多轮分析。

## 12. 功能需求详述

### 12.1 数据资产门户

#### 12.1.1 统一门户

1. 提供集团统一访问入口，支持 Web 和移动端访问。
2. 提供资产目录、数据地图、数据搜索、数据服务、智能助手等统一入口。
3. 支持消息提醒、待办审批、订阅通知和个性化首页。

#### 12.1.2 全局搜索

1. 支持资产名称、描述、标签、业务域、字段、规则等全文检索。
2. 支持按业务主题、数据类型、敏感级别、区域、所属系统等条件筛选。
3. 支持搜索推荐、热门资产推荐、最近访问记录。

#### 12.1.3 资产详情展示

1. 展示元数据、业务属性、技术属性、管理属性。
2. 展示质量评分、安全级别、血缘关系、访问统计、价值评分。
3. 支持资产使用说明、申请入口、变更记录和订阅信息查看。

### 12.2 数据资产目录管理

#### 12.2.1 资产自动发现

1. 支持自动扫描数据库、数据仓库、大数据平台、文件系统、对象存储、消息主题等数据源。
2. 支持自动识别表、字段、视图、存储过程、分区、作业定义等对象。
3. 支持实时、小时、天级采集频率配置。
4. 支持全量采集和增量采集。
5. 支持大规模库表分片并行采集。

#### 12.2.2 分类分级管理

1. 支持按业务主题分类，如航班、旅客、机务、货运、财务人力等。
2. 支持按结构化、半结构化、非结构化分类。
3. 支持按公开、内部、敏感、机密进行分级。
4. 支持集团自定义分类分级标准、标签体系和扩展规则。

#### 12.2.3 目录服务

1. 提供统一目录视图和数据地图。
2. 支持资产申请、审批、授权、订阅、变更通知完整闭环。
3. 支持资产收藏、关注、分享和常用目录管理。
4. 支持目录版本管理和资产变更追踪。

### 12.3 数据质量管理

#### 12.3.1 质量规则管理

1. 支持空值检查、重复检查、格式检查、范围检查、一致性检查等规则类型。
2. 支持 SQL 规则、正则表达式规则和规则模板库。
3. 支持规则复用、规则版本化、灰度发布和影响评估。
4. 支持按业务域和数据源分级维护规则库。

#### 12.3.2 质量检测执行

1. 支持实时检测和批量检测。
2. 支持定时调度、事件触发、上游作业成功回调等方式执行。
3. 支持检测结果自动评分和评级。
4. 支持趋势分析、基线管理和历史结果归档。

#### 12.3.3 质量问题处理

1. 支持短信、邮件、钉钉、企业微信等多渠道告警。
2. 支持自动生成质量工单并流转。
3. 支持异常趋势分析、根因分析和整改建议输出。
4. 支持整改进度、处理时效、复发率跟踪。

### 12.4 数据安全管理

#### 12.4.1 敏感数据识别

1. 支持正则匹配、关键字匹配、机器学习等多种识别方式。
2. 内置民航业敏感数据识别模型，覆盖旅客身份信息、航班动态信息、机组信息等典型场景。
3. 支持人工确认、修正和反馈学习。
4. 支持基于集团自建大模型的模糊语义和上下文关联敏感识别增强。
5. 支持识别规则智能生成与持续优化。

#### 12.4.2 数据脱敏

1. 支持静态脱敏和动态脱敏。
2. 支持替换、掩码、加密、泛化等多种脱敏算法。
3. 支持按用户、角色、场景配置细粒度脱敏策略。
4. 支持开发、测试、分析、对外共享等不同场景策略差异化生效。

#### 12.4.3 数据访问控制与审计

1. 支持 RBAC 和 ABAC。
2. 支持按环境、数据等级、终端、地理位置等属性进行控制。
3. 支持记录完整访问日志。
4. 支持审计检索、审计报表、异常访问分析与追溯。

### 12.5 数据血缘与影响分析

#### 12.5.1 血缘自动采集

1. 支持 SQL 解析自动采集血缘。
2. 支持 DataWorks、Kettle、Informatica 等 ETL 工具血缘采集。
3. 支持作业日志解析。
4. 支持手工维护、补录和冲突仲裁。

#### 12.5.2 血缘图谱展示

1. 支持全链路血缘图谱展示。
2. 支持向上追溯和向下追踪。
3. 支持表级、字段级、作业级和服务级血缘关系查看。
4. 支持变更影响范围分析。

#### 12.5.3 智能血缘增强

1. 支持基于大模型的跨系统、跨层级血缘智能补全。
2. 支持血缘关系校验和风险提示。
3. 支持数据变更影响智能评估与报告输出。

### 12.6 数据价值评估与运营

#### 12.6.1 价值评估模型

1. 支持成本法、收益法、市场法等多种评估方法。
2. 支持基于质量、应用、稀缺性、调用热度、替代成本、合规成本等维度综合评分。
3. 支持模型配置、权重调整和版本管理。

#### 12.6.2 价值看板

1. 提供集团级数据资产价值总览。
2. 支持按部门、业务域、数据类型钻取分析。
3. 支持价值趋势分析和高价值低质量预警清单。
4. 支持数据产品化、共享运营和收入目标跟踪。

### 12.7 智能能力中心

#### 12.7.1 智能问答与助手

1. 基于集团自建大模型构建数据资产管理专属智能助手。
2. 支持数据资产查询、质量问题咨询、安全规则解读、血缘关系分析等问答场景。
3. 支持自然语言交互、多轮对话、上下文理解。
4. 支持生成结构化查询结果和可视化图表。
5. 支持私有化部署，交互数据不得对外泄露。

#### 12.7.2 智能数据质量分析与预警

1. 支持自动识别质量问题根因并生成整改建议。
2. 支持基于趋势和业务语义进行风险预警。
3. 支持自动生成质量分析报告。
4. 支持自定义报告模板和推送频率。

#### 12.7.3 智能数据资产推荐与运营

1. 支持基于角色、行为、业务语义的资产推荐。
2. 支持为业务人员、数据分析师、管理人员提供差异化推荐结果。
3. 支持生成共享策略优化建议、数据产品化建议和价值提升路径建议。
4. 支持非结构化数据资产智能解析与结构化提取，并纳入目录管理。

#### 12.7.4 智能敏感识别增强

1. 支持对隐式旅客隐私、机组涉密信息等复杂场景进行精准识别。
2. 支持自动生成安全规则，并由人工审核后生效。

#### 12.7.5 智能影响分析增强

1. 支持自动分析数据变更对下游资产、业务系统、数据产品的影响。
2. 支持生成影响评估报告和风险等级提示。

### 12.8 系统管理与集成支撑

#### 12.8.1 组织与权限管理

1. 支持集团、区域、分公司多级组织架构管理。
2. 支持用户、角色、权限、审批链配置。

#### 12.8.2 消息与流程集成

1. 对接短信、邮件、钉钉、企业微信等通知渠道。
2. 对接工单系统，实现问题闭环流转。
3. 对接统一流程引擎，支撑申请审批流程。

#### 12.8.3 平台集成

## 14. 非功能需求

### 14.1 性能需求

1. 简单查询页面响应时间不高于 2 秒。
2. 复杂查询页面响应时间不高于 5 秒。
3. 数据导入速度不低于 10 万条/秒。
4. 元数据采集速度不低于 1000 个表/小时。
5. 并发在线用户数不少于 5000。
6. 系统可用性不低于 99.99%。
7. 数据存储容量支持不少于 5PB。

### 14.2 智能能力性能需求

1. 智能问答响应时间：简单问题不高于 3 秒，复杂问题不高于 5 秒。
2. 智能分析报告生成时间：单业务域不高于 10 秒，全集团域不高于 30 秒。
3. 大模型推理并发量不少于 200 请求/秒。
4. 知识库更新与数据资产同步时效不高于 1 小时。

### 14.3 安全需求

1. 满足国家信息安全等级保护三级要求。
2. 满足《民用航空信息安全管理规定》。
3. 支持国密算法 SM2、SM3、SM4。
4. 支持 TLS 1.3 加密传输。
5. 支持操作审计和日志留存，留存期不少于 6 个月。
6. 大模型推理过程需满足等保三级和私有化安全要求。
7. 禁止调用外部公共大模型服务。

### 14.4 兼容性需求

1. 支持鲲鹏、飞腾 CPU。
2. 支持麒麟、统信操作系统。
3. 支持达梦、人大金仓等国产数据库。
4. 支持阿里云、腾讯云、华为云和私有云部署。
5. 支持 Chrome、Firefox、Edge、360 安全浏览器。
6. 支持 iOS、Android、鸿蒙移动端。

### 14.5 可运维性需求

1. 支持监控、日志、告警和审计统一接入。
2. 支持健康检查、自动故障转移和弹性伸缩。
3. 支持版本发布、回滚和灰度策略。
4. 支持容量规划与年度健康检查。

## 15. 技术与架构约束

### 15.1 总体架构要求

平台需采用云原生微服务架构，支持容器化部署和弹性伸缩，整体分为门户层、应用层、能力层、大模型集成层、数据中台基础层和基础设施层。

### 15.2 技术路线要求

1. 开发语言支持 Java、Python、Go 等主流语言。
2. 微服务框架采用 Spring Cloud Alibaba 或同等能力框架。
3. 容器编排基于 Kubernetes。
4. 数据库支持 MySQL、PostgreSQL、TiDB、OceanBase 等。
5. 大数据组件支持 Hadoop、Spark、Flink、Kafka 等。
6. 中间件支持 Redis、RocketMQ、Elasticsearch 等。
7. 前端采用 Vue.js 或 React。
8. 大模型集成兼容集团自建大模型 RESTful/gRPC 接口。
9. 支持私有化推理与知识库向量检索能力。

### 15.3 部署要求

1. 总部作为全局逻辑控制中心。
2. 五大区域中心承担区域级采集、缓存、执行与就近处理能力。
3. 23 个分公司采用轻量接入模式。
4. 支持同城双活和异地灾备。
5. 区域之间默认禁止横向直连，共享需经过受控交换区。

## 16. 数据与知识库要求

1. 构建集团级数据资产目录库、规则库、策略库、审计库、知识库。
2. 知识库需覆盖制度规范、操作手册、规则说明、资产目录、质量规则、安全策略等内容。
3. 知识库需支持清洗、分词、向量化、增量更新和权限隔离。

## 21. 风险与依赖

### 21.1 主要依赖

1. 集团 IAM、工单、日志、消息、CMDB、DevOps 等基础平台可按期配合对接。
2. 各业务系统与区域中心配合提供数据源接入与元数据采集支持。
3. 集团自建大模型接口和推理资源池可按计划提供。
4. 网络、专线、VPN、安全策略可满足跨域协同要求。

### 21.2 主要风险

1. 历史系统文档不完整，导致接入与血缘梳理周期延长。
2. 区域和分公司网络条件不一致，影响同步和接入效果。
3. 数据标准和资产认责机制推进不足，影响目录质量。
4. 大模型能力效果与算力资源不足，影响智能模块体验。
5. 多系统协同发布和权限治理复杂，存在上线风险。

## 22. 成功标准

1. 平台形成集团统一的数据资产管理入口和治理机制。
2. 数据资产规模、质量、安全、价值、共享等核心指标达到招标目标。
3. 总部、五大区域中心和 23 个分公司形成统一协同管理模式。
4. 智能能力在问答、质量、安全、血缘、推荐等场景实现实际应用并可量化评估。
5. 平台通过验收并进入稳定运营阶段，为集团经营管理和数据运营提供持续支撑。
