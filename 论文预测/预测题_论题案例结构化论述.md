## 论题 1：论大模型及 AI Coding 技术的应用

**题目对齐：** 概要项目与职责 → LLM 选型与架构（私有化、RAG、提示工程、安全）+ AI Coding 与研发流程（IDE、CI、评审）→ 落地问题、应对与影响。

### 论点一：大模型技术选型与架构要点

**【母案例 A｜角度：敏感航务数据上的模型与知识工程】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 航班动态、机组排班、旅客服务依赖机型手册、航路规则、气象等强领域知识；数据敏感，不宜依赖公网大模型裸用。 |
| **技术方案** | 采用**私有化部署 LLM**；以 **RAG** 构建航务知识库（机型、航路、气象）；**提示工程**支撑延误预测辅助、旅客话术草稿等场景；对旅客身份、机组隐私做**脱敏与访问控制**。 |
| **落地实施** | 知识切片与向量检索策略与航司数据分级审批联动；提示模板按场景评审后入库；敏感字段在入向量库前脱敏。 |
| **成果** | 航务问答与辅助输出可在内网闭环；合规边界清晰，为后续 AI Coding 产出代码的审查提供同一套数据治理基线。 |

### 论点二：AI Coding 工具链与研发流程集成

**【母案例 A｜角度：航务关键代码的人机协同交付】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 航班调度、行李分拣、载重平衡等逻辑迭代快，手写效率低；燃油、载重等属**安全关键代码**，不能「生成即上线」。 |
| **技术方案** | IDE 集成 **AI Coding 插件**辅助算法与业务代码草稿；**CI/CD** 中增加 **AI 辅助代码审查**规则，突出变更中的数值与安全敏感点；核心航务算法实行**人工复核与双人策略**等人机协同治理。 |
| **落地实施** | 在流水线中固化「关键模块清单」与评审门禁；AI 审查结果作为合并请求必看项之一，不替代人工签核。 |
| **成果** | 研发侧形成可审计链路：谁生成、谁修改、谁复核可追溯；安全关键路径不因提效而失守。 |

### 论点三：落地过程、问题与应对及实际影响

**【母案例 A｜角度：幻觉、知识产权与效率的折中】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 试运行中出现**模型幻觉导致航路/限制类建议不可靠**；团队担心**代码知识产权与训练数据外泄**。 |
| **技术方案** | 引入**领域知识图谱与规则**约束 LLM 输出边界；代码侧优先基于**内部代码库与私有模型/企业版工具**降低外泄风险；对 AI 参与模块建立**航安合规审查**清单。 |
| **落地实施** | 对「可自动采纳」与「必须人工确认」分级；调度等模块上线前增加对抗样例与专家抽检。 |
| **成果** | 调度等相关模块开发效率**约提升 40%**（可作为论文量化）；同时明确**效率不能以牺牲航安复核为代价**。 |

## 论题 2：论自动化测试的应用

**题目对齐：** 概要项目与职责 → 接口与 UI 自动化选型与策略 → 体系构建与 **AI 辅助**（用例生成、自愈等）。

### 论点一：接口自动化测试

**【母案例 B｜角度：对外集成与延误高峰下的 API 质量】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 航班查询、订票、行李追踪等 **API** 与空管、**GDS** 等交互；大面积延误时退改签**并发陡增**，易出现契约不一致与性能问题。 |
| **技术方案** | 搭建覆盖核心链路的**接口自动化测试套件**；与外部系统采用**契约测试**保障字段与语义一致；对退改签等接口做**压力与峰值场景**用例。 |
| **落地实施** | 契约变更纳入发布门禁；核心接口在 CI 中每日构建与关键发布前必跑。 |
| **成果** | 外部接口变更可提前暴露；高峰场景下减少「线上才发现不一致」的风险。 |

### 论点二：UI 自动化测试

**【母案例 B｜角度：多机场、多终端上的同一套航务操作】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 机组排班、塔台态势、旅客 **kiosk** 等界面多、分辨率与浏览器内核不一，手工回归成本高。 |
| **技术方案** | 采用 **POM** 封装航班号输入、舱位、登机牌打印等**航空业务组件**；UI 自动化覆盖主流程与关键异常提示。 |
| **落地实施** | 页面对象与业务术语对齐，减少脚本因文案微调整体失效；多机场抽样环境上做兼容性矩阵。 |
| **成果** | 界面迭代时回归范围可控，脚本可读、可维护性提升。 |

### 论点三：AI 辅助测试与体系成效

**【母案例 B｜角度：历史延误数据驱动的智能测试】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 边界条件多（天气、限载、衔接航班），纯人工设计用例易遗漏；航显等界面变更快，脚本频繁失效。 |
| **技术方案** | 利用 **AI 生成/补充测试用例**（如基于历史延误样本构造边界）；**脚本自愈**降低选择器变更带来的维护成本；**测试左移**，从航务需求文档生成测试大纲与检查点。 |
| **落地实施** | AI 产出用例经测试负责人审核后入库；自愈规则与 CI 报告联动，避免静默失败。 |
| **成果** | **回归测试周期由约 3 天缩短至约 4 小时**（建议作为本题核心量化成果）。 |

## 论题 3：论服务网格（Service Mesh）的应用

**题目对齐：** 概要项目与职责 → 服务治理、**全链路可观测**、**零信任** → 落地问题与可维护性 / 稳定性。

### 论点一：服务治理机制

**【母案例 C｜角度：流量与发布从业务代码中外移】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 航班动态、机组、机务、旅客等**微服务**数量多，灰度发布、超时与重试策略若散落在各仓库，治理成本高。 |
| **技术方案** | 采用 **Istio** 等 **Sidecar** 模式；用 **VirtualService** 做**金丝雀发布**（如新航路算法服务）；用 **DestinationRule** 配置**熔断、超时**（如气象服务异常时降级）。 |
| **落地实施** | 发布按权重逐步切流，监控错误率与延迟门禁；熔断阈值结合业务容忍度调参。 |
| **成果** | 新版本可控上线，依赖抖动时避免级联故障。 |

### 论点二：全链路可观测性

**【母案例 C｜角度：一条旅客业务链上的可关联观测】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 跨服务排障依赖日志拼接，难以回答「从订票到行李」哪一环变慢。 |
| **技术方案** | **分布式追踪**覆盖「订票→值机→登机→行李托运」；**Prometheus + Grafana** 看航班准点率、接口 SLO；**日志聚合**定位某机场离港系统慢等局部问题。 |
| **落地实施** | TraceId 贯穿网关与 Sidecar；关键 KPI 与告警规则与运行手册对齐。 |
| **成果** | 故障从「各服务各自猜」变为「同一条链路上下钻」。 |

### 论点三：零信任安全与落地效果

**【母案例 C｜角度：机组与旅客相关链路的传输与身份】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 内部服务间若默认互信，一旦单点被突破易横向移动；机组排班等数据在链路上需防窃听与伪造。 |
| **技术方案** | 服务间 **mTLS**；**零信任**策略下每次调用校验身份与策略；Sidecar 带来额外延迟与资源占用时，用 **eBPF** 等手段优化数据面。 |
| **落地实施** | 证书轮换与身份源（如服务账号）纳入运维常规；性能优化前后对比航班实时计算路径延迟。 |
| **成果** | **服务间调用故障定位时间由小时级降至分钟级**（本题可背量化）；安全基线与可观测性共同提升可维护性。 |

## 论题 4：论云原生架构下的可靠性设计

**题目对齐：** 概要项目与职责 → 云原生**可靠性关键技术** → 落地与突发流量 / 故障下的**业务连续性**。

### 论点一：Kubernetes 自愈与弹性

**【母案例 D｜角度：春运与峰值下的容量与发布安全】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 春运、暑运等**流量高峰**；单机房或单可用区异常时，航班核心服务不能长时间不可用。 |
| **技术方案** | 核心服务部署于 **K8s**，**多可用区**部署；**HPA** 按 CPU/自定义指标扩缩容；**Pod Disruption Budget** 控制节点维护时的最小可用副本。 |
| **落地实施** | 容量基线按历史峰值与演练结果设定；发布窗口与 PDB 策略协调。 |
| **成果** | 峰值时自动扩容，维护滚动升级时不拖垮核心业务。 |

### 论点二：微服务容错设计模式

**【母案例 D｜角度：外部空管与内部峰值双不确定】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 空管等外部数据服务可能超时；刷票与批量改签带来**瞬时高峰**；行李、舱单等环节存在**异步堆积**。 |
| **技术方案** | **熔断降级**：空管不可用时降级使用缓存航路数据；**限流**保护舱位超售等接口；**重试 + 退避**应对网络抖动；**异步消息队列**对舱单生成等做削峰填谷。 |
| **落地实施** | 降级与限流策略与客服话术、业务规则联动，避免「技术上保住了、业务上说不通」。 |
| **成果** | 局部依赖故障时核心改签、查询能力仍可分级可用。 |

### 论点三：混沌工程与业务连续性保障

**【母案例 D｜角度：机场级故障与极端天气】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 需验证「数据中心级故障、极端天气」下是否仍能保障旅客改签与关键查询。 |
| **技术方案** | **混沌工程**演练：模拟某机场数据中心故障，验证调度与数据**转场/切换**；**故障注入**发现如机组排班库主从切换**约 30 秒**等隐患并优化至 **RTO < 10 秒**级目标。 |
| **落地实施** | 演练在业务低峰窗口执行，事前公告与回滚预案齐备。 |
| **成果** | 实际案例中雷暴导致大量航班取消时，依靠**弹性扩容 + 降级策略**，旅客改签等服务**不长时间中断**（与论题 6 的「运维处置」分工：本题强调**架构级**连续性与演练）。 |

## 论题 5：论大数据 Kappa 架构的设计与应用

**题目对齐：** 概要项目与职责 → Kappa **核心思想** + 与 **Lambda 对比** → 平台建设及**运维成本、实时性**效果。

### 论点一：Kappa 架构核心设计思想

**【母案例 E｜角度：把航班动态视为可重放的无限流】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 航班实时态势与历史航迹、统计既要**低延迟**又要**可重算**，若批流两套逻辑易不一致。 |
| **技术方案** | 采用 **Kappa**：以 **Kafka** 等为日志型总线，**ADS-B 位置**、旅客行为等统一进入可重放流；**Flink** 做**流批一体**计算，同一套作业支撑实时监控与历史分析。 |
| **落地实施** | 明确 Topic 保留周期与合规要求；状态后端与 Checkpoint 策略按 SLA 设计。 |
| **成果** | 架构上「一条管道、一套语义」，为后续按需重算奠基。 |

### 论点二：与 Lambda 架构的对比与优势

**【母案例 E｜角度：从 Hive+Storm 双栈到单栈维护】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 传统 **Lambda** 下，离线 Hive 算延误统计、实时 Storm 做预警，**两套代码、两套运维**，口径易漂移。 |
| **技术方案** | 统一到 **Flink 作业**后，通过调整**消费位点**对历史窗口**重算**（如近 30 天延误原因分布）；批流结果同源。 |
| **落地实施** | 迁移期并行核对关键指标，确认一致后下线旧链路。 |
| **成果** | **运维成本约降 50%**（可作本题对比量化）；**批流结果不一致问题显著减少**。 |

### 论点三：平台建设与实际效果

**【母案例 E｜角度：总线保留、状态与成本】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 原始流数据保留要支撑重放与审计，但存储成本高；延误预测等需要**增量状态**与低延迟输出。 |
| **技术方案** | Kafka 保留**约 7 天**原始数据支持重放；Flink **状态管理**支撑延误预测等增量计算；**热数据 SSD / 冷数据对象存储**分层，控制长期成本。 |
| **落地实施** | 冷热迁移策略与合规留存要求对齐；监控消费滞后与状态大小。 |
| **成果** | **航班动态信息延迟由分钟级降至秒级**；支持任意历史时刻态势**回溯**（事故调查、运营复盘）；成本可控。 |

## 论题 6：论智能运维（AIOps）的应用

**题目对齐：** 概要项目与职责 → AIOps **核心能力框架** → **平台建设**与**运维效率**效果。

### 论点一：AIOps 核心能力框架

**【母案例 F｜角度：多源数据上的异常、根因与容量】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 飞机 **ACARS**、机场 **A-CDM**、旅客服务等多源异构数据分散，传统阈值告警噪声大。 |
| **技术方案** | 建**运维数据中台**汇聚指标、日志、链路；**时序异常检测**识别准点率等异常波动；**根因分析**区分天气、空管、机组、机务等因素；**容量预测**支撑未来 72 小时资源准备。 |
| **落地实施** | 数据质量与字段字典先行；模型输出与专家经验规则结合。 |
| **成果** | 从「告警轰炸」转向「可解释的异常与方向」。 |

### 论点二：AIOps 平台建设过程

**【母案例 F｜角度：日志模式、知识图谱与自愈】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 一次离港系统故障可触发**数百条**相似告警；排障依赖老师傅经验，难以复制。 |
| **技术方案** | **日志聚类**提炼故障模式；**知识图谱**关联机型—故障—维修方案；**智能告警收敛**将 **200+ 告警收敛为少量根因事件**；对数据源异常做**故障自愈**（如切换备用航路数据源）。 |
| **落地实施** | 自愈动作分级：自动切换与仅建议人工；图谱与 CMDB、维修工单打通。 |
| **成果** | 告警可读性提升，重复劳动减少。 |

### 论点三：运维效率提升效果

**【母案例 F｜角度：定位时间与 MTTR】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 航班大面积延误时，定位慢直接影响恢复与旅客体验。 |
| **技术方案** | 以根因分析与自动化编排缩短路径；与值班体系、回滚剧本联动。 |
| **落地实施** | 落地前延误相关故障平均定位约 **45 分钟**；落地后 **AIOps 自动根因约 5 分钟级定位**，准确率约 **85%**；**MTTR 由约 2 小时降至约 20 分钟**。 |
| **成果** | 典型如凌晨**航显**故障：自动触发回滚并通知值班，**旅客值机中断约 3 分钟**即恢复（本题核心量化与故事收尾）。 |

## 论题 7：论分布式缓存技术的应用与实践

**题目对齐：** 概要项目与职责 → **实现原理与选型** + **数据一致性** → 应用中的问题与**效果**。

### 论点一：分布式缓存实现原理与选型

**【母案例 E｜角度：航班动态读路径与多级缓存】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 航班状态、行程、资质等**高频读**若直达数据库，春运峰值下 DB 成为瓶颈且跨机房延迟高。 |
| **技术方案** | **Redis Cluster** 承载航班动态、行程、机组资质等热点；**Memcached** 承载相对静态航路图；**本地缓存（如 Caffeine）+ Redis** 组成**多级缓存**降低跨机房与热点穿透 DB。 |
| **落地实施** | 按数据变更频率与一致性要求划分进哪一级；连接池与分片策略按峰值压测调优。 |
| **成果** | 读路径延迟与 DB 压力同时下降，与论题 5 的实时管道形成「算得动、读得快」的闭环。 |

### 论点二：数据一致性保障机制

**【母案例 E｜角度：状态变更与并发写】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 延误、登机、起飞等**状态变更频繁**；机组排班存在**并发修改**；取消航班时需全局一致感知。 |
| **技术方案** | **Cache-Aside + 延迟双删**降低旧值残留；机组等场景用 **Redisson 分布式锁**防并发冲突；航班取消等采用**先写库、MQ 异步失效缓存**的**最终一致性**，秒级不一致在业务可接受范围内明确公示。 |
| **落地实施** | 关键读路径对「可能读到旧值」有兜底提示或短轮询；锁粒度与死锁超时策略评审。 |
| **成果** | 高并发下仍保持可解释的一致性与可运维性。 |

### 论点三：缓存典型问题应对与效果

**【母案例 E｜角度：热点航班与故障重建】**

| 维度 | 内容 |
|:---|:---|
| **场景问题** | 无效航班号查询**穿透** DB；**热点航班**键过期引发击穿；批量相同 TTL 可能**雪崩**；Redis 故障后**重建风暴**。 |
| **技术方案** | **布隆过滤器**拦截不存在键；热点用**互斥锁 + 逻辑过期**；TTL **随机化**；节点故障时**限流重建 + 备份拓扑**等缓解风暴。 |
| **落地实施** | 春运高峰前专项压测与演练 Redis 切换。 |
| **成果** | **数据库查询量约降 80%**，航班信息查询 **P99 由约 200ms 降至约 20ms**（本题主量化）；与论题 4、5 同故事线下分别强调**韧性、管道、读性能**。 |
